# knowledge_base_basic 框架严格映射基线

## 1. 目的

本文件把以下两份框架标准落实到 `knowledge_base_basic` 的当前实现：

- [前端框架标准模块](../../framework/frontend/L2-M0-前端框架标准模块.md)
- [知识库工作台场景模块](../../framework/knowledge_base/L2-M0-知识库工作台场景模块.md)
- [认证入口保护检查与反馈场景模块](../../framework/auth/L1-M0-认证入口保护检查与反馈场景模块.md)

目标不是口头对应，而是把当前产品从框架到代码的收敛链写清楚：

1. `framework -> product spec`
2. `framework -> implementation config`
3. `framework -> runtime/code`
4. `framework -> validation`

当前实现的主链已经调整为：

`framework markdown -> framework IR -> product_spec.toml -> implementation_config.toml -> contracts/runtime bundle -> rule validation -> generated evidence`

关键落点：
- IR 解析：[src/framework_ir/](../../src/framework_ir/)
- 产品编译：[knowledge_base.py](../../src/project_runtime/knowledge_base.py)
- 前端通用契约与规则：[src/frontend_kernel/](../../src/frontend_kernel/)
- 知识库工作台契约与规则：[src/knowledge_base_framework/](../../src/knowledge_base_framework/)
- Product Spec：[product_spec.toml](./product_spec.toml)
- Implementation Config：[implementation_config.toml](./implementation_config.toml)
- 生成产物：`generated/framework_ir.json`、`generated/product_spec.json`、`generated/implementation_bundle.py`、`generated/generation_manifest.json`

状态约定：
- `已落地`：当前实现已有稳定代码和验证落点
- `部分落地`：当前已落代码，但仍有可继续抽象或扩展的空间
- `缺口`：当前没有稳定代码或验证落点

## 2. framework -> product spec

### 2.1 frontend 边界 -> `product_spec.toml`

约定：
- 框架边界保持大类建模
- `product_spec.toml` 以同名大类 `section` 承载产品值
- section 内字段为该边界下的小类变化项
- 跨边界联动不通过字段归属解决，而通过 `R*` 校验解决

### 2.2 auth 场景 -> `implementation_config.toml`

补充约定：
- `knowledge_base_basic` 的受保护工作台语义不在知识库层重复发明，而是承接 `framework/auth/L1-M0-认证入口保护检查与反馈场景模块.md`
- `product_spec.toml` 中的 `[auth]` 系列 section 用于把 auth 场景模块实例化为当前产品真相；`implementation_config.toml` 中的 auth section 用于把该真相细化为当前实现路径
- `[auth]` 只回答工作台访问前的认证入口、会话放行、登录后目标恢复与失效提示，不回答具体身份提供方、cookie/header、SDK 或协议实现；`[return]` 继续只承接知识引用后的来源返回与 anchor restore

| 归属 | 当前配置落点 | 状态 | 说明 |
|---|---|---|---|
| `frontend auth runtime` | `[frontend.auth].auth_runtime/guard_strategy/login_surface_runtime/session_storage/return_strategy` | 已落地 | 细化前端如何实现登录页、路由守卫、会话存放与工作台目标恢复 |
| `frontend auth style` | `[frontend.auth_style].style_profile/action_emphasis_profile/motion_profile/theme_binding` | 已落地 | 细化登录入口页如何落为具体样式 profile，不改产品层入口结构 |
| `backend auth runtime` | `[backend.auth].provider/session_transport/verification_mode` | 已落地 | 细化后端如何理解认证提供方、会话传输与校验方式 |
| `backend auth api` | `[backend.auth_api].login_endpoint/logout_endpoint/session_endpoint/login_method/logout_method/session_method/session_header` | 已落地 | 细化 auth 动作如何暴露为当前实现路径下的 HTTP 接口 |

## 5. 编译产物映射

| 产物 | 来源 | 用途 |
|---|---|---|
| `generated/framework_ir.json` | `framework markdown -> framework IR` | 冻结被选中的 primary modules 与闭包模块 |
| `generated/product_spec.json` | `framework + product_spec.toml -> compiled product spec` | 输出语言无关的产品真相 |
| `generated/implementation_bundle.py` | `product spec + implementation config + contracts/runtime bundle` | 生成后的实现侧导入入口 |
| `generated/generation_manifest.json` | `framework + product spec + implementation config + generator output hashes` | 证明当前产物只能由源输入重新生成 |

## 6. 结论

`knowledge_base_basic` 当前已经具备以下严格映射能力：

- 框架文档被解析为可追溯 IR
- `product_spec.toml` 只负责产品真相，`implementation_config.toml` 只负责技术细化
- `projects/<project_id>/generated/*` 只允许由生成器物化，不允许手改
- 前端通用层和知识库工作台层都产出显式 contract
- 受保护工作台语义显式承接 auth 场景模块，`[auth]` 负责产品真相，`[frontend.auth]/[backend.auth]` 负责实现细化
- 主界面保持 ChatGPT 风格：会话侧栏 + 聊天主区 + 按需来源抽屉
- `R1-R4` 在启动期变成硬校验
- 编译产物会把 IR、产品真相和实现 bundle 分开写出
