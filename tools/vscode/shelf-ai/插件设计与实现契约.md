# Shelf AI 插件设计与实现契约

## 0. 文档定位

本文档是 `tools/vscode/shelf-ai` 的正式实现契约。

它不负责讲发布历史，也不负责介绍用户怎么安装；它只回答一件事：

> 以后 Shelf AI 插件应该按什么边界、什么真相源、什么职责切分、什么验证门槛来继续设计和实现。

契约与现状分离规则：

- 本文档只陈述契约事实与目标，不陈述“当前已经做到什么”。
- 插件当前实现状态与可核验证据统一记录在 `tools/vscode/shelf-ai/插件现状与证据.md`。

作者语法说明：

- framework 作者源中的 `Boundary` 现统一改称 `Parameter`
- 当前 canonical / runtime / compatibility layer 仍保留 `boundary_*`、`Boundary*Class` 等历史机器字段与类名；凡涉及这些实现字段时，本文继续按实际符号名书写

后续凡是涉及以下内容的改动，都应先对照本文档：

- 新命令
- 新导航能力
- 新树视图
- 自动物化 / 校验 / 保护逻辑
- 插件配置项
- 发布流程

并且要把下面这条当成硬约束，而不是建议：

> 以后凡是插件相关代码发生变化，都必须同步审查并在需要时更新本文档；若实现已变而本文档未更新，则该实现视为未完成。

若插件 README、零散注释或临时讨论与本文档冲突，以本文档为准；若本文档与仓库更高层规范冲突，则以上位规范为准。

## 1. 插件身份

Shelf AI 不是独立产品，也不是新的仓库真相中心。

它的正式身份是：

> 面向当前仓库作者与 AI 协作者的 canonical-first VSCode 工作台插件。

它服务的对象是当前仓库的作者流，而不是抽象的任意 VSCode 工具市场用户。它应帮助作者在仓库既有主链中完成：

- framework 作者入口
- Config / Code / Evidence 的可追溯导航
- canonical 校验与物化
- workspace 保护
- graph/tree 视图浏览
- framework 草稿发布

它不应承担以下职责：

- 充当新的项目配置主入口
- 维护仓库外的业务数据模型
- 生成新的平行真相源
- 在插件内部重建一套独立于 canonical 的规则系统

## 2. 主链契约

插件必须明确服从仓库主链：

`Framework -> Config -> Code -> Evidence`

对应到插件实现时，必须长期成立：

- `framework/*.md` 仍是作者源
- `projects/*/project.toml` 仍是唯一项目配置入口
- `projects/*/generated/canonical.json` 仍是唯一机器真相源
- 机器持久化产物只允许 canonical graph；tree/report/view 只能是运行时投影或进程内缓存
- framework tree / evidence tree / 导航 / 校验 / 守卫 / 提示说明都必须围绕主链组织：framework tree 取作者源运行时投影，evidence tree 与跨层机器判定关系取 canonical

插件不允许把自己做成“任意跳转图”。

它在 UI 和交互上必须突出两件事：

- 正向收敛：`Framework -> Config -> Code -> Evidence`
- 反向追溯：`Evidence / Code -> Config -> Framework`

## 3. 真相源与允许读取的输入

### 3.1 唯一机器真相源

插件内凡是“机器判定关系”，必须只来自 canonical。

包括但不限于：

- framework 到 config 的投影关系
- config 到 code 的模块连接
- code 到 evidence 的模块连接
- base binding / boundary binding 的追溯信息
- evidence tree / 跨层 graph 的结构输入
- 生成物保护所识别的 canonical 派生路径

### 3.2 允许直接读取的作者侧输入

插件允许直接读取原始文件，但仅限于作者侧编辑体验，而不是重新定义机器真相：

- 当前打开的 `framework/*.md`
  用于本地 token hover / go to definition / completion / snippet 插入
- `projects/*/project.toml`
  用于把 canonical 已经确定的 section 跳转落到真实文件行
- `framework_drafts/*.md`
  用于草稿发布流程
- 当前打开的 `.sf` 文件
  仅用于实验性 keyword-first 语言的本地高亮 / lint / completion 预览；不进入 canonical，不替代 `framework/*.md` 作者源

### 3.3 明确禁止

下列行为禁止出现：

- 在插件侧维护新的 boundary -> config 硬映射表作为长期真相
- 在 canonical 缺失时自动猜测 config section 并冒充正式跳转结果
- 重新读取旧 graph / registry / manifest 作为导航中心
- 为了“更方便”绕过 canonical 读取别的 JSON

### 3.4 canonical freshness 契约

插件必须区分“canonical 存在”与“canonical 新鲜可用”。

对单个 `projects/<project_id>/generated/canonical.json` 而言，只有在以下条件同时满足时，才可视为 `fresh`：

- canonical 文件存在且 JSON 可解析
- 当前 `projects/<project_id>/project.toml` 未晚于 canonical
- 当前 project 选择的 `framework/*.md` 作者源未晚于 canonical
- canonical 中声明的 `config / code / evidence` 层 `source_ref.file_path` 未晚于 canonical

若上述任一条件不满足，则该 canonical 必须视为 `stale` 或 `invalid`，不得继续冒充正式机器真相。

Shelf AI 对 freshness 采用严格模式，而不是提示后继续默认使用旧结果：

- canonical-backed 正式跨层结果只允许建立在 `fresh canonical` 上
- framework 本地 hover / definition / references / completion / snippet 仍允许继续工作
- framework -> config 正式跳转在 canonical `missing / stale / invalid` 时必须退回作者侧本地体验，不得继续给出旧 config 结果
- evidence tree 属于 canonical-derived 正式视图；当 canonical `missing / stale / invalid` 时，插件不得继续打开或刷新正式 evidence tree
- 插件不得因为用户触发 hover、跳转或树视图而偷偷自动 materialize；刷新 canonical 只能通过仓库正式 materialize / validation 流程完成
- 当保存 framework 文件触发的自动流程遇到下游编译失败时，允许先写入“framework-only canonical 快照”作为临时作者反馈；但该快照必须显式标记为 degraded，并继续按 stale canonical 处理（例如 evidence tree 仍阻断）

freshness 的判定结果必须在 UI 中可见，至少要能明确告诉用户：

- 当前 canonical 是否 fresh
- 若不 fresh，是 `missing / stale / invalid` 中的哪一种
- 下一步应先 materialize / validate，而不是继续信任当前正式跨层结果

### 3.5 持久化边界契约

为了保持“canonical graph 是唯一机器持久化真相源”，插件必须满足：

- `projects/*/generated/` 下只允许持久化 `canonical.json`
- framework tree / evidence tree 只能在运行时从作者源或 canonical 投影，不得再落盘 `docs/hierarchy/*.json|*.html`
- 证据清单、校验汇总、导航索引等都只能在命令执行期临时构建；若使用缓存，缓存必须是可丢弃的运行时缓存，不得作为机器权威输入
- 插件禁止新增任何“读取派生持久化文件作为正式输入”的路径

## 4. 职责切分契约

插件代码必须继续保持真实模块边界，不得重新收敛成单一大文件里的隐式中心。

### 4.1 `extension.js`

职责：

- 命令注册
- watcher 编排
- 运行时 UI 触发
- 与 VSCode API 的集成

禁止：

- 内嵌仓库规则真相
- 内嵌 boundary / base / module 关系表
- 把导航、树生成、校验逻辑长期塞回 orchestrator

### 4.2 `framework_navigation.js`

职责：

- framework markdown 的本地符号导航
- module ref / rule ref 跳转
- canonical-backed config section 跳转

禁止：

- 维护独立于 canonical 的结构映射真相
- 在缺失 canonical 时推断 project config 结构

### 4.2.1 `config_navigation.js`

职责：

- `project.toml` 中边界 section 到 code anchor 的 canonical-backed 跳转
- 仅在 canonical fresh 时开放 Config -> Code 正式跳转

禁止：

- 绕过 canonical 直接猜测代码文件与行号
- 把配置跳转做成与边界无关的关键字搜索

### 4.3 `evidence_tree.js`

职责：

- 从 canonical 派生 evidence tree payload
- 为 change context 与 evidence 视图提供索引

禁止：

- 读取旧 evidence report 作为源
- 把 evidence tree 做成平行真相

### 4.4 `guarding.js`

职责：

- 识别受保护生成物
- 识别 watched path
- 识别需要物化 / 校验 / 类型检查的文件变化

禁止：

- 把仓库规则散落到 extension 其他位置重复维护

### 4.5 `validation_runtime.js`

职责：

- 规范化插件触发的验证命令
- 统一解析校验结果

禁止：

- 发明与仓库脚本不一致的伪验证入口

### 4.6 `framework_completion.js`

职责：

- 维持 `@framework` 模板与作者入口
- 提供基于章节与符号上下文的 framework 补全（含默认编号推断）

禁止：

- 移除直达模板入口而不给出同等级替代方案

### 4.7 `framework_lint.js`

职责：

- 对受控 framework Markdown 做实时语法诊断：包括 `framework/**`、`framework_drafts/**` 下的 Markdown，以及正文显式带 `@framework` 的任意 Markdown
- 输出可定位行列的 diagnostics（Problems / 波浪线）并与 canonical 项目校验解耦
- 为常见语法问题提供 Quick Fix（lightbulb）修复动作，且修复动作只作用于当前文档
- 对 `framework/**` / `framework_drafts/**` 中的非 module Markdown 执行“附件闭包”检查：只有被 module 直接引用的 Markdown 才是有效作者源，其余必须报错

禁止：

- 将 framework 文档语法诊断绑定到“当前 project 是否选择该模块”的前提
- 通过静默跳过格式错误来伪造“无问题”状态
- 将 framework 附件放宽为任意 README / 任意目录说明而不经过 module 直接引用

## 5. 功能面契约

Shelf AI 当前允许继续扩展的功能面只有以下几类：

### 5.1 作者入口

- `Shelf: Insert Framework Module Template`
- framework draft 发布入口

要求：

- 必须继续把 `@framework` 作为可直接触达的硬入口
- 不得把作者入口退化成隐藏命令或深层菜单

### 5.2 导航

- framework token hover / definition / references
- framework module ref 跳转
- canonical-backed project config section 跳转

要求：

- local symbol 可从当前 markdown 解析
- 跨层关系必须从 canonical 来
- canonical-backed 跳转必须先通过 freshness 检查；旧 canonical 不得继续充当正式跳转依据
- boundary 到 config 的跳转必须体现逐层感，而不是任意连线感

### 5.3 Tree / Graph 视图

- framework tree
- evidence tree

要求：

- framework tree 是作者视图
- evidence tree 是 canonical-derived workspace 证据视图
- tree 视图的交互实现应放在插件 Webview 运行时（可拖拽/缩放/节点检查），不得回退到仓库持久化 HTML/JSON 产物
- framework tree 必须以 `framework/**` 作者源作为正式关系来源，并维持“模块级作者图”语义：模块来自文件名；`B*` 与规则参与基只作为模块内部作者信息和模块间关系提取输入，不作为画布上的一等节点/边；当基定义引用上游模块（如 `Lx.My[...]`）时，画布必须折叠为模块到模块的关系边
- framework 保存触发的“快照级 canonical 刷新”用于保持机器主链产物更新，不得伪装成 full materialization；必须保留可判定的 degraded 标记，直到 full materialize 成功覆盖
- framework 保存触发的自动树刷新只能做后台重渲染，不得强制 reveal / 抢焦点弹出框架树面板；是否打开面板必须由用户主动动作决定
- 状态栏点击行为应可配置；当配置为打开框架树时，必须打开可停靠/可关闭/可调整大小的 WebviewPanel，而不是侧栏杂项视图；当配置为中间分叉模式时，必须先弹出“打开框架树 / 查看问题”二选一菜单
- 当插件从旧 canonical links 接入新的 correspondence 导航协议时，默认应优先做“无界面改动的接线集成”：
  - 不新增 panel / explorer / problem view
  - 不重画现有 Framework Tree 布局
  - 不调整现有 Shelf 首页、状态区、问题区和 inspector 布局
  - 仅替换当前界面的数据来源、默认跳转行为与详情内容
- 现有 inspector、状态区与问题区若已存在，应优先消费 `canonical.correspondence` 归一化协议（`/correspondence`、`/correspondence/tree`、`/correspondence/object/{object_id}`）；Framework Tree 只把 correspondence 作为可选导航增强，不得反向主导作者源图结构
- evidence tree 的打开与刷新必须建立在 fresh canonical 之上；stale canonical 只能先提示物化 / 校验
- 两者都不能倒退成新的真相源

### 5.4 自动化协作

- auto-materialize
- validate now
- codegen preflight
- generated artifact guard
- optional mypy

要求：

- 命令默认值必须与仓库现行脚本一致
- 变更触发条件必须可测试
- 任何新配置项都必须对应明确下游行为
- 若本地设置显式启用“framework 变化仅 lint”，则 framework 受控 Markdown 的自动化链路必须停在 lint，不得继续隐式跑 canonical 校验、物化或 mypy；手动 full validate / preflight 不受影响
- 提交/推送边界的强约束继续依赖仓库侧 `validate_canonical.py` 与 git hooks
- 受保护路径内的代码若本地引用到受保护路径外的实现文件，必须被判定为违规（越界引用）

## 6. 配置契约

插件设置项必须满足以下规则：

- 只暴露真实生效的配置
- 默认值必须指向仓库当前支持的脚本与产物路径
- 变更后必须同步 README、测试与实现
- 若配置项改变了机器行为，必须有对应回归测试
- 树图布局类设置若被暴露，只能影响运行时 layout / render 行为，不得改变 canonical 判定、导航真相或 freshness 语义
- 凡是“用户会按工作区反复调节、且不会改变仓库机器真相”的运行时行为，默认应优先做成 VSCode `shelf.*` setting，而不是长期写死在实现常量里
- 设置变更若只需要运行时重算，优先支持不 reload 生效；至少应能自动刷新当前已打开的相关 view / webview
- 右下角消息弹窗必须支持统一 setting 开关（例如 `shelf.showMessagePopups`）；关闭弹窗时，仍必须保留 output / status bar / problems 的同等状态信号
- 插件允许提供仓库内可见的本地配置文件投影（如 `.shelf/settings.jsonc`）作为“注释化工作区覆盖层”，但该文件不能替代 VSCode `shelf.*` setting，也不能成为新的机器真相源
- 本地覆盖层与 VSCode setting 同时存在时，优先级必须是：显式 VSCode `shelf.*` > `.shelf/settings.jsonc` > `package.json` 默认值

禁止：

- 暴露已经失效或仓库不再支持的旧命令
- 同时维护新旧两套路由 / 产物路径 / 校验入口
- 把用户偏好型参数长期留在 `extension.js` / Webview / layout 常量里，却不给正式 setting 入口

### 6.1 settings-first 适用范围

下列类型默认属于“应优先配置化”的范围：

- 命令字符串与可替换执行入口
- 自动化触发开关、通知开关、保护模式
- framework 变化时是否只执行 lint 的本地自动化偏好
- 只影响本地工作台体验的 tree / graph 布局与交互密度参数
- tree 自动刷新开关、状态栏点击动作等本地交互偏好
- 超时、节流、抑制窗口等运行时调优阈值
- 只影响本地显示与交互，不改变 canonical 语义的面板偏好

### 6.2 明确不得配置化的边界

下列内容必须继续由契约与实现直接定义，不得暴露成用户可改 setting：

- `Framework -> Config -> Code -> Evidence` 主链本身
- `projects/*/generated/canonical.json` 作为唯一机器真相源的地位
- canonical freshness 判定规则
- generated artifact guard 与 watched path 的主边界
- framework token 语法、作者入口与 `@framework` 模板硬入口
- 任何会让插件脱离仓库正式脚本、正式真相源或正式导航语义的“方便开关”

### 6.3 现状与证据分离规则

- 本节只定义 settings-first 的契约边界，不记录“当前已落地能力”清单。
- 配置化覆盖范围、待配置化候选项与对应证据，统一记录在 `tools/vscode/shelf-ai/插件现状与证据.md`。
- 新增或调整 setting 时，必须同时更新：  
  - 本文档（约束与目标）  
  - 现状与证据文档（实际状态与证据）

## 7. UI 与交互契约

插件 UI 必须服务于“逐层理解仓库”，而不是制造第二个产品前端。

必须长期保持：

- 侧栏首页以工作台与导航为主
- tree 视图强调主链分层
- 提示文案围绕 canonical、validation、materialize、framework draft、git hooks

禁止：

- 引入与仓库架构无关的大型功能面
- 让 UI 命名脱离 `Framework / Config / Code / Evidence`
- 把插件做成独立业务面板或低信息密度营销页

## 8. 测试与验收契约

涉及插件实现的改动，至少应覆盖相应层面的验证。

### 8.1 最低回归集合

- `node tools/vscode/shelf-ai/test_guarding.js`
- `node tools/vscode/shelf-ai/test_framework_navigation.js`
- `node tools/vscode/shelf-ai/test_framework_lint.js`
- `node tools/vscode/shelf-ai/test_evidence_tree.js`
- `node tools/vscode/shelf-ai/test_validation_runtime.js`
- `node tools/vscode/shelf-ai/test_snippets.js`

### 8.2 仓库联动验证

- `uv run mypy`
- `uv run python scripts/materialize_project.py`
- `uv run python scripts/validate_canonical.py`
- `uv run python scripts/validate_canonical.py --check-changes`

### 8.3 必测场景

至少应保证：

- canonical 缺失时，插件不会伪造正式 config 跳转结果
- canonical stale / invalid 时，插件不会继续给出正式 config 跳转或正式 evidence tree
- canonical 存在时，framework -> config 跳转稳定
- 当 `communication / exact` 出现超出 framework projection 的路径时，`validate_canonical.py` 必须以 `FRAMEWORK_VIOLATION` 失败，并提示先由人修改 framework
- 当 canonical 中存在非一对一 boundary projection 时，`validate_canonical.py` 也必须以 `FRAMEWORK_VIOLATION` 失败，并提示先由人修改 framework
- 受保护路径内文件若引用受保护路径外本地实现文件，`validate_canonical.py` 必须以 `FRAMEWORK_VIOLATION` 失败，并给出 `from -> to` 越界链路
- evidence tree 只从 canonical 派生
- generated artifact guard 不会放过 `projects/*/generated/*`
- 插件默认命令与 README 描述一致

## 9. 发布契约

Shelf AI 的发布必须继续遵守仓库发布标准。

每次正式版本至少要有：

- `tools/vscode/shelf-ai/package.json` 中对应版本号
- `tools/vscode/shelf-ai/CHANGELOG.md` 更新
- `tools/vscode/shelf-ai/release-notes/<version>.md`
- 可安装的 `.vsix`

发布路径必须是：

- 本地打包安装验证
- 推送版本 tag
- 由 GitHub 发布流程创建对应 release 与资产

禁止：

- 只改版本号不改 release notes
- 只推 tag 不校验安装产物
- 把样例项目产物当成公开发布对象

## 10. 变更流程契约

以后凡是对插件做实质设计变更，默认流程如下：

1. 先判断是否触碰本文档边界
2. 若触碰，先更新本文档
3. 再改实现与测试
4. 再跑插件回归与仓库联动验证
5. 若涉及发布，再补版本说明与 `.vsix`

这意味着：

> 本文档不是事后总结，而是插件未来设计与实现的前置契约。

## 11. 文档同步强制规则

下列文件或区域发生变化时，默认必须同步检查本文档是否需要更新：

- `tools/vscode/shelf-ai/extension.js`

- `tools/vscode/shelf-ai/framework_navigation.js`

同步要求不是“有空再补”，而是：

也就是说，本文档与插件代码必须保持同版本语义，而不是长期落后于实现。

## 12. 一句话总原则

Shelf AI 必须始终是：

> 一个 canonical-first、四层主链优先、作者工作流优先、模块边界真实存在的仓库协作插件。

而不是：

> 一个靠插件内硬编码和临时推断维持表面可用性的“方便工具集合”。

## 13. 仓库技术方案（当前讨论基线）

本节是“以后看仓库实现方式”的唯一讨论入口。凡涉及 `framework / project.toml / runtime code / shelf-ai` 的改造，先对齐本节。

### 13.1 目标与不变项

- 不变主链：`Framework -> Config -> Code -> Evidence`。
- 不变真相源：`projects/*/generated/canonical.json` 仍是唯一机器真相源。
- 不变持久化边界：tree/view/report 仅是运行时投影，不新增平行持久化真相；其中 framework tree 来自作者源投影，evidence tree 来自 canonical 投影。
- 允许零项目 bootstrap：当仓库尚不存在任何 `projects/*/project.toml` 时，`validate` / `materialize` / preflight 可 no-op success 并明确提示当前处于 bootstrap / no-project 状态；但 runtime serve、证据树与正式跨层导航仍要求至少一个真实项目。
- 变更目标：把“边界一对一映射”从约定升级为机器可验证的实现标准，并可在插件中逐层跳转。

### 13.2 参数命名一对一契约（硬约束）

每个 framework 边界项必须满足同名映射：

1. Framework：`boundary_id` 允许 `snake_case`，并兼容既有全大写 / 混合大小写作者 token；大小写本身不表达额外语义
2. Communication：`communication.<framework>.<normalize_lower(boundary_id)>`
3. Exact：`exact.<framework>.<normalize_lower(boundary_id)>`
4. Code Anchor：`exact_export.modules.<module_key>.static_params.<field_name>`

禁止：

- 把多个边界长期归并到一个“大而全” section（例如“全部挂在 library/chat”）来掩盖边界语义。
- 依赖仅靠大小写差异来区分两个边界（例如同模块内同时出现 `CHAT` 与 `chat`）。
- `boundary_id`、配置 section 名、代码 anchor 名字不一致。
- 使用 `exact_export.boundaries.*` 作为运行时主锚点或兼容回退。

允许：

- 不同模块复用同一个 `boundary_id` 时，共享同名 section（例如多个模块都使用 `a11y`），但必须语义一致。

### 13.3 参数双 Class 契约（静态 + 运行时）

每个边界在三层都要显式落成双 class：

- Framework 层：`FrameworkBoundarySpecClass` + `FrameworkBoundaryRuntimeClass`
- Config 层：`ConfigBoundaryStaticClass` + `ConfigBoundaryRuntimeClass`
- Code 层：`CodeBoundaryStaticClass` + `CodeBoundaryRuntimeClass`

解释：

- Static class 承担“静态配置与锚点定义”。
- Runtime class 承担“运行时投影与消费锚点”。
- 两者都必须写入 canonical，供导航、审计与校验消费。

### 13.4 导航跳转链契约

插件导航必须支持并强调“逐层感”：

1. Framework token 跳到 Config section（canonical fresh 前提）
2. Config section 再跳到 Code anchor（canonical anchor）

当插件已接入 correspondence 协议时，现有点击/打开/跳转逻辑应按以下优先级解释，而不是继续直接吃旧字段：

1. `object_id`
2. `primary_nav_target_kind`
3. `primary_edit_target_kind`
4. `navigation_targets`
5. `correspondence_anchor`
6. `implementation_anchor`

要求：

- 现有 Framework Tree 双击 / Enter / inspector 默认打开行为：若命中 correspondence 对象则按 `primary_nav_target_kind`；否则回退作者源文件锚点打开。
- 若当前对象是 `runtime_dynamic_type`，必须仍能通过非动态 fallback target 稳定跳转。
- 现有 issue/status 汇总逻辑应优先接入 `validation_summary`，但不新增新的问题面板。
- 若旧 canonical links 仍保留，只能作为兼容 fallback，不得继续主导默认跳转。
- Framework Tree 必须始终基于作者源实时解析（模块/B*/R* 与模块引用），不因 canonical freshness 切换到并行数据源；canonical `stale` 只影响跨层正式跳转与 evidence tree，不影响 framework tree 的作者渲染。
- 右下角通知弹窗属于本地交互偏好：默认可用、可统一关闭；关闭时不得吞掉校验失败/阻断类信号，必须继续写入 output 并维持状态栏/问题列表可见性。

禁止从 Framework 直接“凭猜测”跳到代码实现细节来绕过配置层。

### 13.5 临时停检与恢复契约

允许在本地临时停检以完成大规模迁移，但必须满足：

- 仅允许通过 `.shelf/settings.jsonc` 的本地设置停检，不改仓库规则本身。
- 迁移完成前必须恢复严格模式。
- 恢复后必须重新通过：
  - `npm --prefix tools/vscode/shelf-ai test`
  - `uv run mypy`
  - `uv run python scripts/materialize_project.py --project-file ...`
  - `uv run python scripts/validate_canonical.py --check-changes`

### 13.6 迁移执行清单（framework/config/code 全链目标）

1. Framework 文档
- 在 `framework/frontend|knowledge_base|backend/**` 统一声明“一对一边界映射 + 边界双 class 承接”。

2. Config（`project.toml`）
- 补齐所有边界的同名 section：`communication.<framework>.<normalize_lower(boundary_id)>` 与 `exact.<framework>.<normalize_lower(boundary_id)>`。
- 根模块运行需要的边界字段必须可执行，不能只留空壳。

3. Code Runtime
- `framework_layer.py` 改为同名直映射，不再维护派生归并表。
- `config_layer.py` 输出边界 static/runtime class。
- `code_layer.py` 输出边界 static/runtime class 与 code anchor。

4. Shelf AI
- framework 作者体验默认提供“实时补全 + 诊断 Quick Fix”，且不引入保存阻断。

### 13.7 验收标准（当前标准）

- 边界映射：每个边界均能在 canonical 中看到同名映射链（Framework -> Config -> Code anchor）。
- Class 完整性：每个边界在 Framework/Config/Code 三层都存在 static/runtime 双 class 记录。
- 门禁一致性：非一对一映射会在仓库校验与插件意图门禁中被识别。
- 作者体验：framework 文档具备实时语法诊断、上下文补全与可执行 Quick Fix，不依赖保存门禁；默认本地自动化对 framework 变化停在 lint，不隐式拉起 full-chain 校验。
- 语法单一真相：framework 作者语法必须由同一 grammar 定义统一驱动 lint、snippet/completion 与 highlight，三者不得各自维护分叉规则。
- 实验语言边界：可额外提供独立 language id 的实验作者语言（如 `.sf` / `shelf-framework`），但该语言必须明确标记为编辑器预览能力；不得冒充正式 `framework/*.md` 作者源，不得直接进入 canonical / materialize / publish 主链。
- 实验语言激活：独立 language id 的实验作者语言若主要依赖扩展侧 provider 提供 highlight/lint/completion，则插件必须在打开该语言文档时直接激活；不得只依赖 `onStartupFinished`、命令或视图事件，导致首屏作者体验缺失。
- 实验语言高亮兜底：独立 language id 的实验作者语言若没有宿主内建语法高亮，必须提供由共享 grammar 派生的 TextMate grammar 或等价兜底，并默认开启该语言的 semantic highlighting；不得让作者在 provider 未就绪或主题关闭语义高亮时看到完全无高亮的首屏。
- 实验语言块命名：`.sf` 顶层块命名必须保持单数一致性，当前正式口径为 `Goal / Base / Principle / Space / Boundary`；`Principles` 与 `Spaces` 不再是正式拼写。
- 实验语言注记语义：`.sf` 中 `op[shape]` 的 shape 与 `in/out/param<subtype>` 的 subtype 必须保持不同写法与不同高亮语义；声明名也必须作为独立 token 可被主题区分，不得把它们全部压成与 block 相同的颜色层级。
- 实验语言映射/操作语义：`.sf` 中 `map` 与 `op[...]` 不得混为同一种概念；`map` 表示显式映射、索引或传播关系，可出现在 `Base` 或 `Principle`，而 `op[...]` 表示带 shape 的操作子，只属于 `Base`。
- 实验语言边界语义：`.sf` 的 `Boundary` 章节只允许输入、输出、参数边界，不得把操作签名再写成 `op[...]`；`op[...]` 只属于 `Base` 章节中的操作子定义。
- framework 标题协议：`FWL012`（标准二级标题内容/顺序）必须提供可执行 Quick Fix；框架主章节固定为 `## Goal / ## Base / ## Combination Principles / ## Combination Space / ## Boundary`，且 `## Boundary` 下必须包含 `### Input / ### Output / ### Parameter`。
- framework 附件协议：`framework/**` 与 `framework_drafts/**` 下的非 module Markdown 只有在被同域 module 直接引用时才是有效作者源；孤立 Markdown、跨域 Markdown 引用与目录外 Markdown 旁路都必须被诊断。
- 可运行性：零项目 bootstrap 状态下，`validate` / `materialize` / preflight 允许 no-op success；存在真实项目后，materialize/validate/mypy/插件测试均通过。

### 13.8 全模块静态对象落地（契约要求）

- `framework/**` 下可解析的全部 framework module，均必须以真实静态 class 直接定义 `M/B/R/D`，不再允许“仅某个模块试点”。
- Code 层静态定义入口改为“按 framework module 一一落地”：
  - 每个 framework module 对应独立 Python 文件：`src/project_runtime/static_modules/modules/<framework>/l<level>_m<index>.py`
  - 全量注册入口：`src/project_runtime/static_modules/registry.py`
  - 通用静态契约基类与 bundle：`src/project_runtime/static_modules/common.py`
- 禁止回退到“单文件承载所有模块”模式（例如 `all_module_contracts.py`）。
- `code_layer` 的模块合约类型装配必须只通过静态注册表（`STATIC_MODULE_CONTRACTS`），不得回退到运行时 `type(...)` 动态生成。
- `codegen_consistency_guard` 的 effective extraction 适用范围必须覆盖全部模块，不再保留 pilot 模块豁免。
- canonical/correspondence 中全部模块的 module/base/rule/static_param/runtime_param，应优先落到上述静态类符号与文件锚点。
- 测试层必须保留“framework module 集合 == `STATIC_MODULE_CONTRACTS` 键集合”的回归断言，防止回退到局部覆盖。
- 当前是否满足以上要求，不在本文断言；统一以 `tools/vscode/shelf-ai/插件现状与证据.md` 与对应验证输出为准。
