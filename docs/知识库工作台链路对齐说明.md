# 知识库工作台链路对齐说明

## 1. 结论

当前 `projects/knowledge_base_basic/` 这条链已经切到统一的新主线：

> `Framework -> Product -> Implementation -> Code -> Evidence`

它不是“框架文档写一套、代码再手写一套”的双轨制。当前默认知识库项目已经做到：

- Framework 层由 Markdown-first 文档驱动
- Product 层只实例化产品真相
- Implementation 层只细化实现路径
- Code 层只消费 Implementation 导出
- Evidence 层只消费 Code 导出

如果你要看更细的逐层跳转和反查方式，直接看：

- [框架到代码映射与反查覆盖说明.md](./框架到代码映射与反查覆盖说明.md)

## 2. 机器真相源

当前知识库主链只有一个机器真相源：

- `projects/knowledge_base_basic/generated/canonical_graph.json`

它是分层的，不是扁平 bundle：

- `layers.framework`
- `layers.product`
- `layers.implementation`
- `layers.code`
- `layers.evidence`

其它生成文件仍然保留，但已经降级为派生视图，例如：

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`

## 3. 五层如何串起来

### 3.1 Framework

当前知识库项目依赖三份框架模块：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

它们会先被解析为 `FrameworkModule`，每个模块内部显式包含：

- `FrameworkCapability`
- `FrameworkBoundary`
- `FrameworkBase`
- `FrameworkRule`
- `FrameworkVerification`

### 3.2 Product

产品真相来自：

- `projects/knowledge_base_basic/product_spec.toml`
- `projects/knowledge_base_basic/product_spec/*.toml`

编译后会进入 `KnowledgeBaseProductModule`，形成稳定产品导出。

### 3.3 Implementation

实现细化来自：

- `projects/knowledge_base_basic/implementation_config.toml`

编译后会进入 `KnowledgeBaseImplementationModule`，只保留实现选择，不回改产品真相。

### 3.4 Code

编译器入口：

- `src/project_runtime/knowledge_base.py`

它会把上游三层收成 `KnowledgeBaseCodeModule`，当前稳定导出的核心对象包括：

- `frontend_contract`
- `workbench_contract`
- `ui_spec`
- `backend_spec`
- `product_view`
- `implementation_effects`

运行时代码位于：

- `src/knowledge_base_runtime/app.py`
- `src/knowledge_base_runtime/frontend.py`
- `src/knowledge_base_runtime/backend.py`

这些文件只消费 Code 层导出。

### 3.5 Evidence

Evidence 层会把 Code 导出继续固化为：

- `canonical_graph.json`
- 治理清单 / 治理树
- strict zone 报告
- object coverage 报告

## 4. 为什么说这条链是对得上的

判断依据不是口头解释，而是下面几件事同时成立：

- 框架输入只有一条 Markdown-first 主线
- 产品真相和实现细化已拆成独立 TOML 文件
- 编译入口只有一条：`project_runtime.knowledge_base.materialize_knowledge_base_project`
- 运行时代码不再直接读取 Product / Implementation 内部细节，而是消费 Code 层导出
- 高风险 symbol 已绑定回 framework / product / implementation 来源
- 生成文件中明确声明 `canonical_graph.json` 是唯一机器真相源

## 5. 一句话总结

当前知识库工作台不是“框架说明 + demo 代码”的组合，而是一条已经落到工程里的分层编译链：

> `Framework Markdown -> Layered Modules -> Code Binding -> Canonical Evidence`
