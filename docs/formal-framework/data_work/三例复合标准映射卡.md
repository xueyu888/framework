# data_work 三例复合标准映射卡

本文用于对 `framework/data_work/` 的 `L1` 组合能力做复合场景压测。

- 用途：在单例标准映射卡之外，再用 3 个更接近真实交互的复合场景检查 `L1` 之间的组合方向是否稳定。
- 判定原则：先判断场景的主导语义，再判断哪些 `L1` 是辅助组合单元；不要因为场景里出现多个模块，就丢失“谁是主语义”。
- 通过标准：复合场景应能在不扭曲各模块职责的前提下自然拆分；若必须靠“把主单元改写成兄弟单元”才能成立，则说明组合边界仍未收敛。

## 1. 可搜索下拉

- 主导单元：`L1-M2 选择单元模块`
- 辅助单元：`L1-M1 输入单元模块` + 展示子单元
- 场景主语义：用户最终要完成的是“在候选对象集合中进行选择”；搜索框只是帮助缩小候选范围的辅助输入。
- 结构映射：
  - 根层：`候选对象承载基 + 选择状态基 + 选择关系基`
  - 辅助层：搜索框作为输入子单元负责形成筛选条件
  - 展示层：候选项标题、说明、图标、空结果文案等作为展示子单元
- 通过条件：
  - 搜索输入只改变候选集合的可见范围，不直接替代选择关系。
  - 单选、多选、切换、取消等仍由选择单元负责解释。
  - 搜索框移除后，场景仍然成立为一个完整的选择单元。
- 回归警戒：
  - 不要因为存在输入框，就把整个场景误判为输入单元。
  - 如果允许用户输入一个完全不在候选集合中的新值，那就已经不是纯“可搜索下拉”，而是新的混合场景，必须单独建模。

## 2. 提交表单并报错

- 主导单元：`L1-M1 输入单元模块`
- 辅助单元：`L1-M3 触发单元模块` + `L1-M4 反馈单元模块` + 展示子单元
- 场景主语义：用户在一组明确输入位置中录入内容，形成当前值集合，并通过触发动作尝试提交；若失败，再进入反馈阶段。
- 结构映射：
  - 根层：一个或多个输入单元共同构成输入主场景
  - 动作层：提交按钮作为触发子单元，负责发起提交、维持提交过程、结束提交过程
  - 反馈层：服务器报错、提交流程失败、保存成功提示等作为反馈子单元
  - 展示层：字段标签、说明文案、分组标题、帮助提示等作为展示子单元
- 通过条件：
  - 字段内容形成和更新当前值，仍由输入单元解释。
  - 点击提交、loading、完成、取消等仍由触发单元解释。
  - 提交失败、成功提示、流程结果等仍由反馈单元解释。
  - 即使按钮和错误消息同时存在，也不会把整个表单场景误写成触发或反馈单元。
- 回归警戒：
  - 不要因为有提交按钮，就把整张表单降格为触发单元。
  - 不要把“提交流程失败”混写成输入状态；字段级输入限制和流程级反馈结果要分开。

## 3. 确认弹窗

- 主导单元：`L1-M4 反馈单元模块`
- 辅助单元：`L1-M3 触发单元模块` + 展示子单元
- 场景主语义：某个对象进入“待确认”的反馈阶段，直到用户确认、取消或关闭，场景才结束。
- 结构映射：
  - 根层：反馈对象承载基 + 反馈阶段基
  - 动作层：确认、取消、关闭按钮作为触发子单元
  - 展示层：标题、正文、风险说明、强调图标等作为展示子单元
- 通过条件：
  - “待确认” 被解释为反馈阶段，而不是纯展示状态。
  - “确认 / 取消 / 关闭” 被解释为触发动作，而不是反馈本身。
  - 弹窗文本、图标、布局不会反过来定义根语义，它们只是反馈场景中的展示内容。
- 回归警戒：
  - 不要因为界面上最显眼的是标题和按钮，就把确认弹窗误写成“展示 + 触发”而丢失反馈阶段语义。
  - 如果弹窗内部允许用户编辑内容或重新选择对象，应视为反馈根单元内部再组合输入/选择子单元，而不是让反馈单元吞并这些职责。

## 4. 使用方式

- 每次修改 `framework/data_work/` 后，先过单例映射卡，再过本文件中的 3 个复合场景。
- 对每个复合场景，依次问 4 个问题：
  - 谁是主导单元？
  - 哪些是辅助组合单元？
  - 如果拿掉某个辅助单元，主语义是否仍然成立？
  - 是否出现“为了让场景成立，不得不把主导单元改写成兄弟单元”的情况？
- 若答案显示主语义不稳定，优先回到 framework 收边界，而不是先在 config 或 code 层用实现细节掩盖问题。
