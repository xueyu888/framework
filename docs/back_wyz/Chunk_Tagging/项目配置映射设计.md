# Chunk_Tagging 项目配置映射设计

## 1. 文档目的

本文件记录 `Chunk_Tagging` 从 framework 作者源落到 `projects/<project_id>/project.toml` 时的配置映射约定，作为本次“框架 -> Config -> Code -> Evidence”转换的中间说明。

## 2. 需要解决的问题

`framework/Chunk_Tagging/**` 的各模块边界统一使用 `P1 / P2 / ...` 命名，但这些 `P*` 在不同模块中的语义并不一致。

若仍沿用仓库中“`exact.<framework>.<boundary_id_lower>`”的默认路径，以下条目会发生语义冲突：

- `Chunk_Tagging.L0.M0.P1`：标签 ID 类型
- `Chunk_Tagging.L1.M0.P1`：文本块字段边界
- `Chunk_Tagging.L2.M1.P1`：文档 ID 边界

因此，本次落地采用“重复 boundary_id 自动切到模块级 section”的规则。

## 3. 映射规则

### 3.1 communication 路径

- 无重复边界：`communication.<framework>.<boundary_id_lower>`
- 重复边界：`communication.<framework>.l<level>_m<module>.<boundary_id_lower>`

### 3.2 exact 路径

- 无重复边界：`exact.<framework>.<boundary_id_lower>`
- 重复边界：`exact.<framework>.l<level>_m<module>.<boundary_id_lower>`

### 3.3 Code anchor 路径

- 仍保持：`exact_export.modules.<module_key>.static_params.<field_name>`

也就是说，本次变化只影响 Config 层 section 路径，不改变 Code 层静态锚点形式。

## 4. 本项目 root 选择

本次项目配置选择以下 root：

- `Chunk_Tagging.L3.M0`
  承载完整审核闭环
- `Chunk_Tagging.L1.M2`
  承载文本-标签评分记录结构；当前 framework upstream 链未直接把它纳入 `L3 -> L2 -> L1` 闭包，因此显式补选

其余模块通过 framework upstream 自动收敛。

## 5. 关键映射示例

- `module_id = Chunk_Tagging.L0.M0`
  - `boundary_id = P1`
  - `exact_path = exact.Chunk_Tagging.l0_m0.p1`
- `module_id = Chunk_Tagging.L1.M0`
  - `boundary_id = P1`
  - `exact_path = exact.Chunk_Tagging.l1_m0.p1`
- `module_id = Chunk_Tagging.L2.M1`
  - `boundary_id = P2`
  - `exact_path = exact.Chunk_Tagging.l2_m1.p2`
- `module_id = Chunk_Tagging.L3.M0`
  - `boundary_id = P3`
  - `exact_path = exact.Chunk_Tagging.l3_m0.p3`

## 6. exact 字段设计原则

本次 `exact` 不使用空表占位，而是按以下原则填充：

- 标签定义、评分记录、数据库读写等结构边界：写成字段名、字段集合、允许值、主外键或数量约束
- 阈值、状态、流程阶段等运行边界：写成可直接消费的数值、枚举和顺序列表
- 评分算法相关边界：引用 `docx/Chunk_Tagging/置信度计算*.md` 中已经明确的阈值和结构字段

## 7. 当前限制

- `Chunk_Tagging` 中仍有部分外部引用写在说明句尾，当前 parser 已补充“句内同 framework 引用可识别”，但跨 framework 的嵌入式引用仍要求更明确的作者表达
- `L1.M2` 目前通过项目 root 显式补选进入 canonical；如果未来 framework 作者将该结构显式接入 upstream，则可回收这项项目级补选
