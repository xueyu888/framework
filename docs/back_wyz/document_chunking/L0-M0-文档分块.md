# 文档分块:DocumentChunking

@framework

## 1. 能力声明（Capability Statement）

- `C1` 分段边界检测能力：检测分段点的能力。
- `C2` 分段结果记录能力：记录所有分段点的能力。

## 2. 边界定义（Boundary / 参数）

- `NORMDOC` 归一文本边界：输入必须为单文档纯文本，且已完成换行、空白与编码归一。来源：`C1`。
- `CUTRELY` 分段依据边界：分段依据必须明确，来源：`C1`。
- `BLOCKNUM` 分段数量边界：段落块具有一定数量范围限制，来源：`C1`。
- `ROLEJUDGE` 角色判型依据边界：判型依据必须明确，来源：`C3`。
- `ROLETYPE` 段落块角色边界：段落块只能被判定为标题或正文，来源：`C2 + C3`。
- `BLOCKCOMBINE` 段落块组合边界：段落块的组合依据必须明确，来源：`C2 + C3 + C4`。
- `OUTPUT` 输出状态边界：模块输出的结果状态、文件名和路径必须明确，来源：`C5 + C6 + C7`。


## 3. 最小可行基（Minimum Viable Bases）

- `B1` 段落分界模板（分段依据，标明分段位置）：L0.M0[R1]。来源：`C1 + CUTRELY + BLOCKNUM`。
- `B2` 段落块（段落文本容器，存放通过 `B1` 拆分出来的一段文本，并保留顺序，定位信息，与原文本ID）：L0.M1[R3]。来源：`C2 + C7 + BLOCKNUM`。
- `B3` 文本角色识别模块（依据 `ROLEJUDGE` 对已装载文本的 `B2` 进行角色判型并赋予 title/body 标签）：L0.M2[R1]。来源：`C3 + ROLEJUDGE + ROLETYPE`。
- `B4` 段落块打包器（将段落块按照1标题+n正文的方式打包）：L0.M3[R1]。来源：`C4 + BLOCKCOMBINE`。
- `B5` 结果文档（文档分块模块最终结果）：L0.M4[R1]。来源：`C5 + C6 + C7 + OUTPUT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 段落切块组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：`B1` 只能作用于满足 `NORMDOC` 的单文档纯文本；每次合法分界命中必须生成一个按原文顺序排列的 `B2`，且各 `B2` 不得重叠、跳序或跨文档混合。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 输出结构：`ParagraphBlock_set`（元素：`PB`，每个 `PB` 必须包含 `block_id`、`order_index`、`text`、`start_offset`、`end_offset`）。
  - `R1.5` 边界绑定：`NORMDOC/CUTRELY/BLOCKNUM`。
- `R2` 段落块角色识别组合
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：每个 `B2` 在进入后续组合前都必须经 `B3` 完成角色判型，并获得唯一的 title/body 角色标签；未完成角色识别的段落块不得进入后续匹配。
  - `R2.3` 输出能力：`C3`。
  - `R2.4` 输出结构：`LabeledParagraphBlock_set`（元素：`LPB`，每个 `LPB` 必须包含 `block_id`、`order_index`、`text`、`start_offset`、`end_offset`、`block_role`）。
  - `R2.5` 边界绑定：`ROLEJUDGE/ROLETYPE`。
- `R3` 段落块打包组合
  - `R3.1` 参与基：`B2 + B3 + B4`。
  - `R3.2` 组合方式：`B4` 只能消费已完成角色判型且保持稳定顺序的 `B2` 序列；每个文本块必须由且仅由一个标题块作为锚点，并吸收其后直到下一个标题块出现前的连续正文块；任一 `B2` 不得同时归属多个文本块，也不得跳序归组。
  - `R3.3` 输出能力：`C4`。
  - `R3.4` 输出结构：`ChunkMatch_set`（元素：`CM`，每个 `CM` 必须包含 `text_chunk_id`、`title_block_id`、`body_block_id_set`、`start_order`、`end_order`）。
  - `R3.5` 边界绑定：`ROLEJUDGE/ROLETYPE/BLOCKCOMBINE`。
- `R4` 输出封装组合
  - `R4.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R4.2` 组合方式：全部合法文本块必须按 `start_order` 输出为稳定序列；`B5` 必须先给出 `text_id`，再按稳定顺序输出由 `chunk_id + chunk_text + title_block_id + body_block_id_set` 构成的文本块单元，并同时保留原始 `B2` 序列与必要的回溯字段，使段落块到文本块的归属关系能够直接从结果对象回连，而不再单独依赖归属关系表。
  - `R4.3` 输出能力：`C5 + C6 + C7`。
  - `R4.4` 输出结构：`DocumentChunkingOutput`（必须包含 `text_id`、`ordered_chunk_item_set`、`paragraph_block_set`、`trace_meta`；其中每个 `chunk_item` 都必须包含 `chunk_id`、`chunk_text`、`title_block_id`、`body_block_id_set`）。
  - `R4.5` 边界绑定：`OUTPUT/BLOCKNUM/BLOCKCOMBINE/ROLETYPE`。

## 5. 验证（Verification）

- `V1` 文本切分与段落块承载能力有效性：`R1` 必须能够在 `NORMDOC/CUTRELY/BLOCKNUM` 边界内，对同一输入文档稳定地产出同一组 `ParagraphBlock_set`，从而同时兑现 `C1 + C2`。
- `V2` 角色判型能力有效性：`R2` 必须能够在 `ROLEJUDGE/ROLETYPE` 边界内，对同一段落块稳定给出同一 title/body 角色结果，从而兑现 `C3`。
- `V3` 段落块打包能力有效性：`R3` 必须能够在 `ROLEJUDGE/ROLETYPE/BLOCKCOMBINE` 边界内，稳定地产出满足“一个标题锚点 + 零个或多个连续正文块”的 `ChunkMatch_set`，从而兑现 `C4`。
- `V4` 输出封装能力有效性：`R4` 必须能够在 `OUTPUT/BLOCKNUM/BLOCKCOMBINE/ROLETYPE` 边界内，稳定输出 `text_id + ordered_chunk_item_set`，并保证文本块顺序、字段结构与组合结果一致，从而兑现 `C5 + C6`。
- `V5` 归属回溯能力有效性：`R4` 产出的 `DocumentChunkingOutput` 必须显式保留 `paragraph_block_set` 与必要的 `trace_meta`；每个 `chunk_item` 都必须显式携带 `title_block_id` 与 `body_block_id_set`，使段落块到文本块的归属关系能够直接回溯到中间 `B2/CM` 状态，从而兑现 `C7`。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失败。
- `V7` 结论表达：验证结果必须逐条输出 `R* -> 通过/失败 / 边界 / 原因` 的结论，而不是只给出汇总布尔值。
