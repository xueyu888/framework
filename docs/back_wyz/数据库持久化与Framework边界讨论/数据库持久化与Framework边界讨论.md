# 数据库持久化与 Framework 边界讨论

## 1. 背景

本文件整理一轮围绕以下问题展开的讨论：

- 对关系型数据库中的表，是否应该由“创建该表的程序”负责维护
- 在 shelf-ai 的四层结构里，如果某个模块需要维护数据库，应该怎样从 framework 角度描述
- 当数据库中还没有对应表，需要运行中建表、后续继续迁移、维护约束与事务时，这些内容应该落在哪一层
- shelf 的 framework 是否属于“静态框架”，是否能够承载运行过程中发生变化时的应对措施

## 2. 关于“谁建表谁维护”

更准确的原则不是“谁建表谁维护”，而是：

谁拥有这张表背后的业务语义，谁维护这张表的结构与演进。

也就是说，如果 A 程序创建了一个表，并且这个表本质上属于 A 的业务领域模型，那么通常由 A 负责：

- 表结构定义
- 约束设计
- 索引策略
- 数据迁移与演进
- 写入语义与事务边界

其他程序更适合：

- 按约定读取数据
- 按明确契约写入数据
- 不擅自修改 schema、约束与核心语义

这个原则在 shelf 语境下可以进一步收敛为：

- 表的“业务语义所有者”负责 framework 边界
- 当前项目实例负责 exact 参数落点
- 具体实现模块负责 code 层数据库工程

## 3. 在 shelf 中，数据库问题应该怎样理解

shelf 的主链是：

`Framework -> Config -> Code -> Evidence`

其中：

- `framework` 负责稳定结构、能力边界、合法组合空间
- `communication` 负责人与 AI 的结构化沟通要求
- `exact` 负责 Code 层精确消费字段
- `code` 负责真正的运行时实现

因此，对于“某模块需要维护数据库”这件事，不应该直接理解为“要不要搭一个数据库底层框架”，而应该先判断：

数据库在这个模块里，是不是一个需要被 framework 显式声明的业务边界。

如果数据库读写本身就是该模块的对外能力，比如：

- 需要写入某类业务记录
- 需要按业务身份查询记录
- 需要更新记录状态
- 需要保证某些更新原子完成

那么应该在 framework 中描述“持久化接口”或“记录仓储接口”。

如果数据库只是模块内部实现细节，比如：

- 最终只是把结果存起来
- 对外并不暴露数据库交互语义
- 上层只关心模块输入输出，不关心数据库边界

那么 framework 通常不需要单独起数据库模块。

## 4. framework 该承载什么，不该承载什么

### 4.1 应该写进 framework 的内容

应该写那些一旦变化，就会改变业务语义的内容，例如：

- 这类记录是什么业务对象
- 哪些字段构成业务身份
- 哪些字段是写入必填
- 哪些状态是合法值
- 状态允许怎样流转
- 哪些字段更新必须视为同一次业务提交
- 什么叫幂等写入
- 结构演进时哪些核心语义不能被破坏

换句话说，framework 负责把“业务约束”钉住。

### 4.2 不应该写进 framework 的内容

不应该把实现型细节直接写进 framework，例如：

- 具体表名、列名
- PostgreSQL / MySQL / SQLite 的选型
- ORM 类、SQL 文件组织方式
- 索引类型与索引实现细节
- migration 工具名和脚本组织方式
- 事务隔离级别的具体枚举值
- 锁策略、连接池、重试参数
- 启动时自动建表还是发版前执行 migration

这些属于 `exact` 或 `code` 的职责范围。

## 5. 为什么说 framework 本质上是“偏静态”的

这里的“静态”，不是指 framework 只能描述一张不变的图纸，而是指它主要定义：

- 稳定结构
- 长期成立的边界
- 合法变化空间
- 动态行为必须遵守的规则

它不直接承载运行时的具体实现策略。

因此，framework 更像“动态系统的静态宪法”，而不是运行时调度器本身。

## 6. 什么叫“一个需求的大头是运行时演化和存储实现”

如果一个需求的主要难点不在“业务结构是什么”，而在“系统运行起来以后如何安全地把结构落到真实数据库里并持续演进”，那么它的大头就属于运行时演化和存储实现。

典型问题包括：

- 表不存在时怎么创建
- 线上已有旧表时怎么迁移
- 新旧 schema 如何兼容一段时间
- 哪些更新必须放在同一事务里
- 失败后如何重试、回滚或补偿
- 索引什么时候加、怎么避免锁表
- 并发写入时怎样保证唯一性与一致性

这些问题高度依赖：

- 具体数据库产品
- 具体部署环境
- 具体实现策略

所以它们天然不会主要靠 framework 搭起来，而主要落在 `exact` 和 `code`。

framework 在这里的职责是：

- 定义业务边界
- 定义合法状态空间
- 定义哪些更新必须原子
- 定义哪些演进不能破坏原有语义

它的作用不是完成数据库工程，而是防止实现层失控。

## 7. framework / exact / code 的职责分工

### 7.1 framework

回答的问题：

- 这张表在业务上承载什么对象
- 一条记录怎样才算合法
- 哪些字段构成稳定业务身份
- 哪些状态和值域合法
- 哪些流转是允许的
- 哪些写入或更新必须原子完成
- 演进时哪些语义必须保持兼容

### 7.2 exact

回答的问题：

- 当前项目决定采用什么数据库方案
- 表名、字段映射、唯一键映射是什么
- migration 策略是什么
- 事务策略是什么
- 当前代码实现需要精确消费哪些数据库参数

### 7.3 code

回答的问题：

- 怎样建表
- 怎样改表
- 怎样迁移
- 怎样写 SQL / ORM
- 怎样加索引
- 怎样包事务
- 怎样回填、补偿、回滚、重试

## 8. 一个简单判断法

如果某条内容回答的是下面这些问题，它更像 `framework`：

- 这类数据是什么业务对象
- 哪些字段决定业务身份
- 哪些字段缺失后记录不成立
- 哪些状态流转合法
- 哪些操作必须具备原子语义
- 哪些失败可以重试，哪些失败必须人工介入

如果某条内容回答的是下面这些问题，它更像 `exact` 或 `code`：

- 表到底叫什么名字
- 用哪一种数据库
- 用 ORM 还是手写 SQL
- migration 怎样组织
- 索引怎样设计
- 事务隔离级别怎样配置
- 启动时自动建表还是单独执行初始化脚本

## 9. 当模块需要在运行中建表、迁移、维护事务时，framework 里该写什么

建议把这类内容写成“记录对象 + 持久化接口”的 framework 模块，而不是写成“数据库底层实现说明”。

可以使用下面这个通用模板。

```md
# 记录持久化接口:RecordPersistenceInterface

@framework

0. 目标（Target）
声明一个持久化接口模块，用于将某类业务记录写入持久化存储，并支持后续按业务身份查询、状态更新与版本演进约束。

## 1. 能力声明（Capability Statement）

- `C1` 记录写入能力：支持写入一条或一批合法业务记录。
- `C2` 记录查询能力：支持按业务身份或业务作用域查询已有记录。
- `C3` 状态更新能力：支持对已有记录执行合法状态流转。
- `C4` 原子提交能力：支持将一次业务提交中要求同时成立的字段更新作为同一原子操作处理。
- `C5` 演进兼容能力：支持在结构演进过程中保持既有业务身份与核心状态语义不失真。

- `N1` 不提供具体建表实现能力：不负责 DDL、ORM model 或 migration 脚本实现。
- `N2` 不提供具体数据库选型能力：不负责声明 PostgreSQL/MySQL/SQLite 等具体产品。
- `N3` 不提供性能优化实现能力：不负责索引类型、分区策略、连接池和锁实现细节。

## 2. 边界定义（Boundary / Parameter 参数）

- `P1` 记录身份边界：每条记录必须能被一组稳定业务字段唯一定位。
- `P2` 写入完整性边界：新写入记录必须至少包含最小必填字段集合。
- `P3` 状态合法值边界：状态字段只能落入已声明的合法值域。
- `P4` 状态流转边界：状态只能按声明的流转路径变化，不允许越级或逆向非法变更。
- `P5` 原子更新边界：同一次业务提交要求同时成立的字段，必须作为同一原子语义处理。
- `P6` 幂等边界：重复提交同一业务请求时，不得生成语义重复且彼此冲突的记录。
- `P7` 演进兼容边界：结构调整后，不得破坏既有业务身份字段、核心状态字段及其解释语义。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 记录身份：唯一定位单条业务记录的身份结构。
- `B2` 最小记录载荷：创建记录所需的最小字段集合。
- `B3` 查询接口：按业务身份或作用域读取记录的接口。
- `B4` 写入接口：创建记录的接口。
- `B5` 状态更新接口：执行合法状态流转的接口。
- `B6` 原子提交单元：要求在一次提交中共同成立的字段更新集合。

## 4. 基组合原则（Base Combination Principles）

- `R1` 创建记录组合
  - `R1.1` 参与基：`B1 + B2 + B4`
  - `R1.2` 组合方式：接收最小合法记录载荷后，以稳定业务身份写入一条新记录。
  - `R1.3` 输出能力：`C1`
  - `R1.4` 参数绑定：`P1 + P2 + P6`

- `R2` 查询记录组合
  - `R2.1` 参与基：`B1 + B3`
  - `R2.2` 组合方式：按记录身份或业务作用域查询已有记录。
  - `R2.3` 输出能力：`C2`
  - `R2.4` 参数绑定：`P1`

- `R3` 状态流转组合
  - `R3.1` 参与基：`B1 + B5`
  - `R3.2` 组合方式：对已有记录执行一次合法状态更新，并保持状态解释一致。
  - `R3.3` 输出能力：`C3`
  - `R3.4` 参数绑定：`P3 + P4`

- `R4` 原子提交组合
  - `R4.1` 参与基：`B1 + B5 + B6`
  - `R4.2` 组合方式：对同一次业务提交要求共同成立的字段更新，以单次原子语义完成。
  - `R4.3` 输出能力：`C4`
  - `R4.4` 参数绑定：`P5`

- `R5` 演进兼容组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`
  - `R5.2` 组合方式：当记录结构演进时，仍保持业务身份、核心状态和值域解释稳定。
  - `R5.3` 输出能力：`C5`
  - `R5.4` 参数绑定：`P1 + P3 + P7`

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须可由对应 `C* + P*` 推导得到。
- `V2` 规则完整性：每个 `R*` 必须明确参与基、组合方式、输出能力与参数绑定。
- `V3` 状态合法性：任意有效状态更新都必须满足 `P3 + P4`。
- `V4` 原子语义符合性：任意声明为同次提交的字段更新都必须满足 `P5`。
- `V5` 幂等符合性：任意重复提交都不得破坏 `P6`。
- `V6` 演进兼容性：结构演进后仍必须满足 `P1 + P3 + P7`。
```

## 10. 当模块需要在运行中建表、迁移、维护事务时，framework 里不该写什么

framework 不应该直接描述下面这些内容：

- 建表 SQL
- migration 文件与执行脚本
- ORM Model 定义
- 索引实现细节
- 外键实现细节
- 事务隔离级别具体配置
- 锁机制与连接池参数
- 回滚脚本与补偿脚本实现
- 灰度迁移、双写、回填的具体执行流程

这些内容即使非常重要，也仍然属于实现层问题。

## 11. 一句话总结

对于“运行中建表、迁移、维护事务”这类需求：

- `framework` 负责写清楚业务边界
- `exact` 负责写清楚项目采用的精确数据库方案
- `code` 负责把这套方案真正实现出来

所以，framework 不是用来承载整套数据库工程的；它的职责是给数据库工程划清不可越界的业务红线。

## 12. 哪些结论是标准明文规定，哪些是工程推论

这轮讨论中，有些内容来自仓库标准的直接约束，有些内容则是基于这些约束做出的工程推论，两者需要分开看。

### 12.1 标准明文规定的部分

标准直接支持以下判断：

- 仓库唯一主链为 `Framework -> Config -> Code -> Evidence`
- `Framework` 负责结构、边界、规则与合法展开空间
- `exact` 负责 Code 层精确消费的字段、路径、参数、策略与槽位
- `Config` 只能把 framework 允许的空间具体化为 `communication` 与 `exact`
- `exact` 不得改写 framework 已限定的结构参数
- 若 `communication / exact` 超出 framework projection，必须以 `FRAMEWORK_VIOLATION` 失败
- 若需求在 framework 中找不到对应约束映射，AI 不得直接通过修改 config 或 code 绕过 framework 前置

### 12.2 属于工程推论的部分

以下说法是基于现有分层精神得出的“合理落法”，但不是标准逐字钉死的固定做法：

- schema readiness 通常落在 `exact + code`
- 表存在性检测一般不必单独做一个 framework 模块
- 可以用某种 schema 管理策略字段去表达“启动时建表 / 发布前迁移 / 仅校验版本”等做法

这些都属于工程设计建议，而不是仓库已经提供的统一保底协议。

## 13. exact 是怎么来的，schema readiness 机制又是怎么来的

### 13.1 exact 的来源

`exact` 不是 framework 的全文翻译，也不是自动推导出的唯一答案。

更准确地说，`exact` 是：

在 framework 已经定义好合法空间之后，当前项目对这些边界的具体落点与实例化选择。

也就是说：

- `framework` 说“必须满足什么”
- `exact` 说“当前项目具体怎么落”
- `code` 说“实际怎么实现”

因此，`exact` 一般来自：

- 人先定义 framework
- 人或 AI 在 `projects/<project_id>/project.toml` 中补入当前项目的精确配置
- 编译链将这些 exact 字段导出给 Code 层消费

### 13.2 schema readiness 机制的来源

所谓 schema readiness，是在回答一个工程问题：

系统启动或运行时，如何确认数据库 schema 已经处于当前代码可安全消费的状态。

它通常来自项目级实现决策，而不是 framework 本体直接生成。常见来源包括：

- 发布前统一执行 migration
- 启动时执行 schema 检测与初始化
- 提供单独的 `init-db / upgrade-db` 管理命令
- 只做版本校验，不做自动升级

换句话说，schema readiness 机制通常是项目对“怎么落库运行”的策略选择。

## 14. 能否保证 AI 在 project.toml 里补的 exact 一定满足建表需求

不能保证。

这一点需要说得非常明确：

如果 framework 只描述了“记录有哪些字段”，但没有对“存储结构是否需要初始化、检测、迁移、版本兼容”给出明确的结构化约束，那么 AI 生成 `project.toml` 时，未必会自动把这部分考虑进去。

原因有三点：

- `exact` 只能在 framework 允许的空间内具体化
- `exact` 不能反向发明 framework 本来没有声明的正式业务语义
- 当 `exact` 超出 framework projection 时，标准要求校验失败，而不是默认放行

所以，不能把“AI 也许会想到”当成正式保证。

### 14.1 这意味着什么

如果你希望“表不存在时检测并建表”“schema 版本不兼容时拒绝启动或升级”成为一个：

- 能被稳定推导
- 能被 AI 一致处理
- 能被仓库校验
- 能被不同项目复用

的正式能力，那么更稳的做法是满足下面两者之一：

- framework 中已经存在对应的持久化就绪 / schema 初始化 / schema 演进能力边界
- 仓库级标准统一规定这类持久化模块都允许声明哪些 exact 槽位

如果两者都没有，那么这件事仍然只能算“实现层临时约定”，而不是 framework 驱动的正式能力。

## 15. 如果已经上线运行，后来又想给记录加一个字段

假设 framework 已经存在，代码也已运行，此时想修改某个持久化记录的字段，对数据库而言常见动作就是“给已有表增加一列”。

这时要分三层看：

### 15.1 是否一定要停机

不一定。

如果新增的是兼容式字段，例如：

- 先允许为空
- 老代码不依赖该字段
- 新代码可以兼容该字段暂时为空
- 不立刻引入重型索引或强约束

那么通常可以设计为不停机或近似不停机演进。

真正危险的通常不是“增加一列”本身，而是同时引入：

- 非兼容读写
- 大量历史数据回填
- 新唯一约束或外键约束
- 高频查询路径变化
- 更重的事务语义

### 15.2 应该怎样修改 framework

关键在于：这个字段是不是业务语义字段。

如果新字段会改变下面这些内容，就应该先修改 framework：

- 记录的业务身份
- 记录最小必填字段集合
- 状态机或合法值域
- 上下游理解该记录的稳定语义

如果它只是实现型字段，例如技术追踪字段、时间戳、分片键、内部重试计数，那么不一定需要先改 framework。

### 15.3 程序差异会不会变大，稳定性和性能会不会受影响

有可能，但关键不在 framework 本身，而在迁移策略。

如果只是兼容式新增、可选字段、非热点路径，程序差异通常较小。

如果字段牵涉：

- 唯一性定义
- 查询条件
- 状态流转
- 事务边界
- 大规模回填
- 新索引

那差异就可能明显变大，风险也会相应增加。

更稳妥的方式通常是采用兼容式演进：

- 先扩表
- 再发兼容代码
- 再回填
- 再收紧约束
- 最后清理旧逻辑

## 16. 如果表还不存在，要不要为“检测表存在 + 建表”再写一个 framework

通常不需要单独为“表存在性检测 + 建表动作”再写一个 framework 模块。

更准确地说：

如果 framework 已经说明了“该记录应该存在怎样的业务结构”，那么：

- 表存在性检测
- 不存在时建表
- 已存在时版本检查
- 升级或拒绝启动

这些一般属于 schema readiness 的实现策略，更像 `exact + code` 的职责。

只有在“存储结构就绪能力”本身已经成为一个稳定、可复用、需要被上层显式依赖的正式能力时，才值得把它提升为 framework 模块。

例如，你明确要求某类模块都必须具备：

- 自举初始化持久化结构
- 启动时校验 schema 兼容性
- 幂等升级结构版本

那么这时可以考虑引入一个正式的“存储结构就绪”类 framework 模块。

否则，通常没有必要为了“检测是否有表 + 没有则创建”这个动作单独新增 framework。
