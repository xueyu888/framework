# 当前仓库完整实现方案与提示词对照审计

## 1. 文档定位

这份文档回答两件事：

1. 当前仓库现在到底是怎么实现的。
2. 这套实现和 `docs/full_rewrite_exec_plan.md` 附录里的重构提示词相比，还有哪些差异。

结论先写在前面：

- 当前仓库已经完成了主骨架切换。
- 但还不能诚实地说“已经百分之百达到提示词里的最终形态”。

## 2. 当前仓库的真实主链

当前真实主链不是旧的 `product_spec.toml + implementation_config.toml` 双轨，也不是旧的 template registry。  
当前主链已经切成下面这条：

```text
framework/*.md
  -> framework_ir parser / loader
    -> FrameworkRegistry
      -> FrameworkPackageRegistry
        -> projects/*/project.toml
          -> config slicing
            -> package compile results
              -> internal compilation state
                -> KnowledgeBaseRuntimeBundle
                -> runtime projection
                  -> generated/canonical_graph.json
                    -> derived governance / tree / report / runtime bundle
```

这里最关键的事实有六个：

1. `framework/*.md` 仍然是作者源。
2. `project.toml` 已经是唯一项目配置入口。
3. 每个 framework 文件都已有对应 package 注册。
4. `generated/canonical_graph.json` 已经是唯一机器真相源。
5. 其他 manifest / tree / report 已经是 canonical 派生视图。
6. 编译期内部仍保留场景化 compilation state，但运行时公开对象已经收敛成 `KnowledgeBaseRuntimeBundle`。

- framework modules: `29`

对应代码入口：

## 4. Package layer

### 4.1 package 目录结构

当前 package 都收在：

- [src/framework_packages/modules](../src/framework_packages/modules)

例如：

- `framework/frontend/L2-M0-前端框架标准模块.md`
  - 对应 [src/framework_packages/modules/frontend_l2_m0/__init__.py](../src/framework_packages/modules/frontend_l2_m0/__init__.py)
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
  - 对应 [src/framework_packages/modules/knowledge_base_l2_m0/__init__.py](../src/framework_packages/modules/knowledge_base_l2_m0/__init__.py)
- `framework/backend/L2-M0-知识库接口框架标准模块.md`
  - 对应 [src/framework_packages/modules/backend_l2_m0/__init__.py](../src/framework_packages/modules/backend_l2_m0/__init__.py)

### 4.2 统一 contract

统一 contract 入口：

- [src/framework_packages/contract.py](../src/framework_packages/contract.py)

当前 `FrameworkPackageContract` 已经显式承载：

- `framework_file()`
- `module_id()`
- `config_contract()`
- `child_slots(...)`
- `compile(...)`

`PackageCompileResult` 里也已经稳定收口：

- `framework_file`
- `module_id`
- `entry_class`
- `package_module`
- `config_contract`
- `child_slots`
- `config_slice`
- `export`
- `evidence`

### 4.3 当前 package 的实际实现方式

当前 package 入口类大多是薄声明类，真正共用逻辑集中在：

- [src/framework_packages/static.py](../src/framework_packages/static.py)

这意味着：

- 好处：framework file -> module id -> package entry class 的绑定很统一。
- 代价：package 级 compile/export/evidence 逻辑目前还比较模板化，不是每个 package 都有自己独立鲜明的实现边界。

### 4.4 与提示词的对齐度

已经满足：

- 一份 framework 文件对应一个 package
- registry 中一份 framework 文件只对应一个 entry class
- compile/export/evidence 已经有统一结果结构

还差一点：

- contract 目前只有 dotted-path 级配置声明，没有字段级默认值、禁用字段、严格 schema 约束
- “每个 package 目录只能有一个入口类”目前主要靠结构约定和人工复核，不是独立验证器

## 5. Config layer

### 5.1 统一项目配置

当前默认项目配置文件是：

- [projects/knowledge_base_basic/project.toml](../projects/knowledge_base_basic/project.toml)

它已经替代了旧的：

- `product_spec.toml`
- `implementation_config.toml`

### 5.2 逻辑分区

`project.toml` 当前已经明确分成四块：

- `selection`
- `truth`
- `refinement`
- `narrative`

这和提示词要求基本对齐。

### 5.3 当前配置分发方式

配置加载入口：

- [src/project_runtime/project_config_source.py](../src/project_runtime/project_config_source.py)

package contract 当前通过 dotted path 做切片分发，例如：

- `selection.preset`
- `truth.surface`
- `truth.library`
- `truth.preview`
- `truth.chat`
- `refinement.frontend`
- `refinement.backend`

但要注意一件事：

- package compile 阶段的确在按 contract 切片
- 运行时投影阶段仍有若干 builder 直接读取编译期内部 state 上的字段

所以它不是“全链完全只靠 package config slice 驱动”，而是“compile 阶段已经切片，runtime assembly 阶段仍有聚合读取”

## 6. Compile / assembly layer

### 6.1 编译主入口

主入口仍在：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)

当前 `_compile_runtime_bundle(...)` 的真实流程是：

1. 读取 `project.toml`
2. 加载 framework registry
3. 加载 builtin package registry
4. 解析 root modules
5. 展开 framework closure
6. 按 package contract 做 config slicing
7. 编译 package results
8. 构造内部 `KnowledgeBaseCompilationState`
9. 手工补建 `backend_spec`
10. 手工补建 `frontend_contract`
11. 手工补建 `workbench_contract`
12. 手工补建 `ui_spec`
13. 构造公开的 `KnowledgeBaseRuntimeBundle`
14. 收集 validation reports
15. 输出 canonical graph 与派生视图

### 6.2 当前 compile 层的真实评价

这部分既有完成的地方，也有没切干净的地方。

已经完成：

- 编译已经是 registry-driven
- 编译已经显式展开 framework module tree
- config slicing 已经进入 package compile
- canonical graph 已经直接从新的主链收口

还没完全完成：

- 编译期内部仍保留 `KnowledgeBaseCompilationState`
- `RootModuleSelection` 仍固定为 `frontend / knowledge_base / backend`
- runtime projection 不是由 package export 自动装配，而是后置手工汇总

换句话说，当前 compile 层已经完成“新骨架”，但还没有完全摆脱“知识库场景大编排器”

## 7. Runtime layer

但它也没有只消费 package exports，而是消费已经收口后的 `KnowledgeBaseRuntimeBundle`：

- [src/frontend_kernel/contracts.py](../src/frontend_kernel/contracts.py)
- [src/knowledge_base_framework/workbench.py](../src/knowledge_base_framework/workbench.py)

这正是当前和提示词差异最大的地方之一：

- 提示词要求 runtime assembly 更接近“package export -> assembled runtime”
- 当前实现仍是“package compile -> internal compilation state -> `KnowledgeBaseRuntimeBundle` -> runtime app”

## 10. 和提示词的逐项对照

### 10.1 已经满足或基本满足

1. Framework Markdown 仍是作者源
2. 每个 framework 文件都有对应 package
3. 每个 package 当前都有唯一入口类
4. entry class 已注册到统一 registry
5. registry 已成为正式绑定真相
6. 项目配置已统一为 `project.toml`
7. 配置逻辑分区已具备 `selection / truth / refinement / narrative`
8. canonical graph 已成为唯一机器真相源
9. governance / tree / report 已降级为 canonical 派生视图
10. evidence 已纳入 canonical 主层

### 10.2 仍有明确差异

1. 大编排器没有完全删除
   - 同一个 [knowledge_base.py](../src/project_runtime/knowledge_base.py) 仍承担 scene-specific compiler、runtime projection builder、materializer 等多重职责。
2. runtime assembly 还不是纯 package export 装配
   - `frontend_contract / workbench_contract / ui_spec / backend_spec` 仍靠手工 builder 收口。
3. 配置切片没有贯穿到 runtime assembly 末端
   - compile 阶段是切片的，但 runtime builder 继续直接消费编译期 internal state。
4. config contract 还偏粗
   - [src/framework_packages/contract.py](../src/framework_packages/contract.py) 还没有字段级必填/默认值/禁用字段建模。
5. 根模块树仍是三槽位硬编码
   - `RootModuleSelection` 仍写死 `frontend / knowledge_base / backend`。

## 11. 最终审计结论

如果把提示词理解为“主骨架必须落地”，那当前仓库已经达标。  
如果把提示词理解为“旧知识库场景大聚合器也必须完全退出主路径”，那当前仓库还没有完全达标。

因此，最诚实的结论只能是：

- 当前仓库已经完成了大部分 destructive rewrite。
- 但还没有完全完成提示词要求的最终收口态。
- 差异主要集中在 `knowledge_base.py` 大编排器、runtime projection 手工汇总、固定三槽位根模块选择，以及粗粒度 config contract。
