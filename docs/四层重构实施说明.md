# 四层重构实施说明

## 0. 文档定位

本文记录本次仓库级重构的实施方案、删除范围、重建结果，以及当前仓库在新架构下的具体形态。

它回答四个问题：

1. 这次重构到底改了什么
2. 旧架构删掉了哪些核心部件
3. 新的 `Framework -> Config -> Code -> Evidence` 主链如何落地
4. 当前仓库里每个关键文件分别承担什么职责

本文不是抽象原则说明；它描述的是这次已经落到代码库中的真实实现。

## 1. 重构目标

本次重构的目标不是“在旧架构外面补一层新名字”，而是把仓库彻底收敛到一条新的单一主链：

```text
Framework -> Config -> Code -> Evidence
```

落地要求如下：

- `framework/*.md` 继续作为唯一 Framework 作者源
- `projects/<project_id>/project.toml` 成为唯一 Config 入口
- Code 层第一次承担真实实现绑定
- Evidence 层正式承担验证、追溯、快照和 GUI 材料职责
- `projects/*/generated/canonical.json` 成为唯一机器真相源
- 所有树图、校验、导航、扩展命令和派生视图都从 canonical 收敛

## 2. 本次实施方案

本次实施按照四步执行，而不是边兼容边迁移。

### 2.1 先删除旧核心

先删除所有继续承担核心职责的旧部件，而不是给它们补兼容字段：

- 删除旧 `src/framework_packages/**`
- 删除旧 `project_runtime/pipeline.py`
- 删除旧 `project_runtime/governance.py`
- 删除旧 `project_runtime/project_governance.py`
- 删除旧 `project_runtime/module_tree.py`
- 删除旧 `project_runtime/package_config.py`
- 删除旧 `project_runtime/config_loader.py`
- 删除旧 `project_runtime/project_config_source.py`
- 删除旧 `workspace_governance.py`
- 删除旧 `scripts/validate_strict_mapping.py`
- 删除旧 `scripts/generate_governance_tree_hierarchy.py`
- 删除旧 `scripts/generate_module_hierarchy_html.py`
- 删除旧测试：
  - `tests/test_framework_package_registry.py`
  - `tests/test_package_config_contract.py`
  - `tests/test_project_runtime.py`
  - `tests/test_project_governance.py`
- 删除旧生成物：
  - `generated/canonical_graph.json`
  - `generated/derived_governance_*`
  - `generation_manifest.json`
  - `strict_zone_report.json`
  - `object_coverage_report.json`

### 2.2 重建新的四层编译链

在旧核心删掉之后，重新建立四层链路：

- Framework 层：从 markdown 解析成正式模块类
- Config 层：从 `project.toml` 生成 `communication` 和 `exact`
- Code 层：从 `exact` 生成真实代码绑定和运行时 export
- Evidence 层：从 Code export 生成验证、追溯和运行蓝图

### 2.3 统一 canonical JSON

新的 `canonical.json` 不再是旧 graph 的改名版，而是天然四层结构：

- `project`
- `framework`
- `config`
- `code`
- `evidence`
- `links`

### 2.4 让工具链全部改吃 canonical

重写或替换下列入口：

- `scripts/materialize_project.py`
- `scripts/validate_canonical.py`
- `scripts/generate_framework_tree_hierarchy.py`
- `scripts/generate_evidence_tree_hierarchy.py`
- VS Code 扩展的默认命令、守卫逻辑、树图入口和 README

## 4. 新架构落地后的具体样子

### 4.1 Framework 层

Framework 层由两部分组成：

1. 作者源
2. 机器化模块类

作者源仍然是：

- `framework/<domain>/Lx-Mn-*.md`

解析入口仍然在：

- [parser.py](/home/xue/code/shelf/src/framework_ir/parser.py)

但 parser 输出的聚合对象已经不再服务旧注册表链，而是供新的 `FrameworkModuleClass` 动态生成器消费。

正式的 Framework 层机器对象定义在：

- [framework_layer.py](/home/xue/code/shelf/src/project_runtime/framework_layer.py)

当前落地形式是：

- 每个 framework markdown 文件都会生成一个 `FrameworkModuleClass`
- 每个 `B*` 都会生成一个 `FrameworkBaseClass` 子类
- 每个 `R*` 都会生成一个 `FrameworkRuleClass` 子类
- `FrameworkModuleClass` 显式持有：
  - `base_classes`
  - `rule_classes`
  - `boundaries`
  - `capabilities`
  - `verifications`
  - `upstream_module_ids`

这意味着：

- Base 不再是注释项
- Rule 不再是临时结构
- framework 文件和类之间已经形成正式一一对应关系

### 4.2 Config 层

Config 层唯一入口是：

- [project.toml](/home/xue/code/shelf/projects/knowledge_base_basic/project.toml)

它现在只保留四个总入口：

- `[project]`
- `[framework]`
- `[communication.*]`
- `[exact.*]`

加载与切片逻辑在：

- [config_layer.py](/home/xue/code/shelf/src/project_runtime/config_layer.py)

Config 层当前做的事有三类：

1. 读取项目元信息
2. 读取选中的 framework 根模块
3. 按 boundary 生成 `communication_export` 和 `exact_export`

Config 层正式机器对象包括：

- `ProjectConfig`
- `ConfigModuleClass`
- `ConfigModuleBinding`

其中 `ConfigModuleClass` 对每个模块导出：

- `communication_export`
- `exact_export`
- `compiled_config_export`

当前 boundary 到 config 的正式路径映射由 `BOUNDARY_CONFIG_PATHS` 固化，例如：

- `SURFACE -> communication.frontend.surface / exact.frontend.surface`
- `CHAT -> communication.knowledge_base.chat / exact.knowledge_base.chat`
- `RESULT -> communication.backend.result / exact.backend.result`

这一步的意义是：

- Framework 只定义边界
- Config 把边界具体化
- Code 不再直接读 framework 或 communication

### 4.3 Code 层

Code 层定义在：

- [code_layer.py](/home/xue/code/shelf/src/project_runtime/code_layer.py)

它的核心规则是：

- `CodeModuleClass` 只消费对应模块的 `exact_export`

当前正式机器对象包括：

- `CodeModuleClass`
- `CodeModuleBinding`

Code 层做的事情不是“把配置原样透传”，而是第一次真正产生实现绑定和运行时 export。

当前 root 模块已经有明确落地产物：

- frontend root 导出 `frontend_app_spec`
- knowledge_base root 导出 `knowledge_base_domain_spec`
- knowledge_base root 同时导出 `runtime_documents`
- backend root 导出 `backend_service_spec`

同时，每个 Code module 都会产生：

- `code_bindings`
- `base_bindings`

当前 canonical 里的 base 绑定形式是：

- `base_id -> code_module_class`

也就是说，Base 已经进入正式代码绑定链，而不是停留在 Framework 注释层。

### 4.4 Evidence 层

Evidence 层定义在：

- [evidence_layer.py](/home/xue/code/shelf/src/project_runtime/evidence_layer.py)

它只消费 Code export，不再回头拼接多份旧 report。

当前正式机器对象是：

- `EvidenceModuleClass`

当前 Evidence export 主要包括：

- `runtime_blueprint`
- `document_digests`
- `validation_reports`

其中：

- `runtime_blueprint` 给运行时和 GUI 一个正式的页面与 API 蓝图
- `document_digests` 给知识库文档一个稳定摘要证据
- `validation_reports` 统一承载当前前端与知识库规则校验

这意味着：

- Evidence 不再是附属报告
- Evidence 已经成为第四主层

## 5. canonical.json 现在长什么样

canonical 生成逻辑位于：

- [compiler.py](/home/xue/code/shelf/src/project_runtime/compiler.py)

当前 schema 为：

```json
{
  "schema_version": "four-layer-canonical/v1"
}
```

当前 canonical 顶层结构为：

```json
{
  "schema_version": "four-layer-canonical/v1",
  "project": { ... },
  "framework": { ... },
  "config": { ... },
  "code": { ... },
  "evidence": { ... },
  "links": { ... }
}
```

### 5.1 `framework`

记录：

- `author_source`
- `selected_modules`
- `root_module_ids`
- `modules`

其中每个 module 里已经展开：

- boundaries
- bases
- rules
- verifications
- export_surface

### 5.2 `config`

记录：

- `project_file`
- `communication`
- `exact`
- `modules`

每个 config module 里明确包含：

- `communication_export`
- `exact_export`
- `compiled_config_export`

### 5.3 `code`

记录：

- `modules`
- `runtime_exports`

每个 code module 里明确包含：

- `code_exports`
- `code_bindings`

### 5.4 `evidence`

记录：

- `modules`
- `exports`
- `validation_reports`

### 5.5 `links`

当前 `links` 用来表达层间正式连接关系，而不是保留旧 graph 命名：

- `framework_to_config`
- `config_to_code`
- `code_to_evidence`
- `boundary_bindings`
- `base_bindings`

其中：

- `boundary_bindings` 记录 boundary 到 communication/exact/code/evidence 的落点
- `base_bindings` 记录 base 到 code module class 的正式绑定

这也是 GUI 和导航未来继续深化的基础。

## 11. 这次重构后仓库的判断标准

如果后续要判断仓库有没有偏离这次重构结果，可以用下面的标准快速审查：

1. 是否仍然只保留 `Framework -> Config -> Code -> Evidence` 一条主链
2. 是否仍然只保留 `generated/canonical.json` 一个机器真相源
3. 是否每个 framework markdown 都仍然能落到正式 `FrameworkModule / Base / Rule` class
4. 是否 Code 层仍然只读 `exact`
5. 是否 Evidence 层仍然只读 Code export
6. 是否树图、扩展、校验和派生视图仍然全部从 canonical 派生
7. 是否没有重新引入旧 `registry / governance / strict-mapping / canonical_graph` 体系

## 12. 总结

这次重构不是对旧架构加一层包装，而是完成了三件实事：

1. 把旧核心链路删掉
2. 把四层主链真正重建出来
3. 把工具链、运行时、派生视图和扩展入口全部切到新的 canonical 主链

当前仓库已经不是“旧系统外面套新文档”，而是实际运行结构已经收敛到新架构。
