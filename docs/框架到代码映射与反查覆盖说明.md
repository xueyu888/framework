# 框架到代码映射与反查覆盖说明

## 1. 这份文档回答什么

这份文档只回答一件事：

> 这个仓库是怎么把 `Framework -> Product Spec -> Implementation Config -> Code -> Evidence` 串成一条可追踪、可反查的路径的。

如果你更想先看图，再回来看文字说明，可以直接打开：

- [framework-to-code-mapping-cn.pptx](../docs/presentations/framework-to-code-mapping-cn.pptx)

如果你想顺着文件真正看懂“一一对应”，最稳的阅读方式不是先看运行时代码，而是按下面这条主线走：

1. 看框架输入是什么
2. 看产品真相和实现细化是什么
3. 看编译器如何把上游收敛成项目对象
4. 看项目对象如何继续收敛成 contract / spec / route
5. 看运行时代码如何只消费这些编译结果
6. 看治理清单如何把代码再反查回上游

---

## 2. 整条链的总图

当前知识库项目的真实主链路可以压成下面这张图：

```text
framework/*.md
  -> FrameworkModuleIR
product_spec.toml + product_spec/*.toml
  -> KnowledgeBaseProductSpec
implementation_config.toml
  -> KnowledgeBaseImplementationConfig
      -> KnowledgeBaseProject
          -> frontend_contract
          -> workbench_contract
          -> ui_spec
          -> backend_spec
              -> runtime code
              -> generated/*
              -> governance manifest / governance tree
                  -> strict mapping / change propagation
```

也就是说：

- `Framework` 先决定结构空间
- `Product Spec` 决定这个项目到底是哪一个产品
- `Implementation Config` 决定这个产品走哪一条实现路径
- `Code` 只消费编译结果
- `Evidence` 再把这条链固化成可比对产物

## 4. 第一层：Framework 是怎么进入编译链的

知识库项目当前直接声明了 3 个框架入口：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

这些入口在产品规格里先被声明出来，见：

- [projects/knowledge_base_basic/product_spec.toml](../projects/knowledge_base_basic/product_spec.toml)

真正进入代码时，走的是 [knowledge_base.py](../src/project_runtime/knowledge_base.py) 里的这一组函数：

- `_resolve_framework_module()`
- `_collect_framework_closure()`
- `_build_framework_ir_text()`

对应的真实编译入口在：

- `_compile_project()`：把三份 framework module 解析并收进项目对象

解析后的结构被写入：

- [framework_ir.json](../projects/knowledge_base_basic/generated/framework_ir.json)

你在这份文件里能直接看到：

- `primary_modules`
- `resolved_modules`
- 每个 module 的 `capabilities / boundaries / bases / rules / verifications`

所以第一层一一对应关系是：

```text
framework/*.md
  -> parse_framework_module(...)
  -> FrameworkModuleIR
  -> generated/framework_ir.json
```

这一步的意义是：

- 运行时代码并不直接读 Markdown
- Markdown 先被编译成结构化 IR
- 后面的 contract / spec / governance 都基于同一份 IR 展开

## 5. 第二层：Product Spec 和 Implementation Config 是怎么进入项目对象的

### 5.1 Product Spec

产品真相来自：

- [projects/knowledge_base_basic/product_spec.toml](../projects/knowledge_base_basic/product_spec.toml)
- [projects/knowledge_base_basic/product_spec/](../projects/knowledge_base_basic/product_spec/)

加载入口在 [knowledge_base.py](../src/project_runtime/knowledge_base.py)：

- `_load_product_spec()`

这一步会把产品层信息装进 `KnowledgeBaseProductSpec`，包括：

- `surface`
- `visual`
- `route`
- `showcase_page`
- `a11y`
- `library`
- `preview`
- `chat`
- `context`
- `return`
- `documents`

### 5.2 Implementation Config

实现细化来自：

- [projects/knowledge_base_basic/implementation_config.toml](../projects/knowledge_base_basic/implementation_config.toml)

加载入口同样在 [knowledge_base.py](../src/project_runtime/knowledge_base.py)：

- `_load_implementation_config()`

它会被装进 `KnowledgeBaseImplementationConfig`，包括：

- `frontend`
- `backend`
- `evidence`
- `artifacts`

### 5.3 两者如何合流

真正合流发生在：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 的 `_compile_project()`

这一步会同时做 4 件事：

1. 解析 framework module
2. 校验产品规格是否仍落在 framework 边界里
3. 校验实现配置是否只是实现细化，没有回改产品真相
4. 组装成统一的 `KnowledgeBaseProject`

所以第二层一一对应关系是：

```text
product_spec.toml + product_spec/*.toml
  -> _load_product_spec()
  -> KnowledgeBaseProductSpec

implementation_config.toml
  -> _load_implementation_config()
  -> KnowledgeBaseImplementationConfig

FrameworkModuleIR + ProductSpec + ImplementationConfig
  -> _compile_project()
  -> KnowledgeBaseProject
```

## 6. 第三层：KnowledgeBaseProject 是怎么继续收敛成代码可消费结构的

`KnowledgeBaseProject` 是当前知识库模板的核心中间结构。

它不是最终代码，但它已经把上游真相压成了代码可以消费的稳定对象。定义见：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py)

在 `_compile_project()` 里，`KnowledgeBaseProject` 会继续被收敛成 4 类关键结构：

### 6.1 前端 contract

入口：

- [frontend_kernel/contracts.py](../src/frontend_kernel/contracts.py)
- `build_frontend_contract(project)`

它对应的高风险治理对象是：

- `kb.frontend.surface_contract`

它回答的是：

- 页面骨架有哪些区域
- 基础交互动作有哪些
- 路由约束是什么
- 扩展槽位挂哪几个 framework module

### 6.2 领域 workbench contract

入口：

- [knowledge_base_framework/workbench.py](../src/knowledge_base_framework/workbench.py)
- `build_workbench_contract(project)`

它对应的高风险治理对象是：

- `kb.workbench.surface_contract`

它回答的是：

- 知识库工作台有哪些 stage
- citation return contract 是什么
- library / preview / chat / return 如何闭环

### 6.3 UI spec

入口：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 中的 `_build_ui_spec()`

它对应的高风险治理对象是：

- `kb.ui.surface_spec`

它回答的是：

- 具体页面有哪些
- 具体组件有哪些
- 样式 token 和实现 profile 怎样落地

### 6.4 Backend spec

入口：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 中的 `_build_backend_spec()`

它对应的高风险治理对象是：

- `kb.backend.surface_spec`

它回答的是：

- API 路由契约
- retrieval / answer policy
- return policy
- implementation backend profile 怎样落地

所以第三层可以压成：

```text
KnowledgeBaseProject
  -> build_frontend_contract()
  -> build_workbench_contract()
  -> _build_ui_spec()
  -> _build_backend_spec()
```

## 7. 第四层：Implementation Config 是怎么保证“配置即功能”的

这条链里最容易漂移的是：

- 实现配置写了
- 代码其实没用

当前仓库防这个问题靠的是：

- `build_implementation_effect_manifest(project)`

定义见：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py)

它会把实现配置字段一条条映射到下游生效位。

最直接的证据就是：

- [generation_manifest.json](../projects/knowledge_base_basic/generated/generation_manifest.json)
  里的 `configuration_effects`

比如当前已经明确记录：

- `frontend.renderer -> ui_spec.implementation.frontend_renderer`
- `frontend.style_profile -> ui_spec.implementation.style_profile`
- `frontend.script_profile -> ui_spec.implementation.script_profile`
- `backend.renderer -> backend_spec.implementation.backend_renderer`
- `backend.transport -> backend_spec.transport.mode`
- `backend.retrieval_strategy -> backend_spec.retrieval.strategy`
- `evidence.product_spec_endpoint -> backend_spec.transport.product_spec_endpoint`

也就是说，Implementation Config 不是“写在那里给人看”的，而是会被自动验证：

```text
implementation_config field
  -> build_implementation_effect_manifest()
  -> generation_manifest.configuration_effects
  -> validate_strict_mapping.py 逐条校验
```

## 8. 第五层：Code 是怎么只消费这些编译结果的

### 8.1 应用入口

应用入口在：

- [app_factory.py](../src/project_runtime/app_factory.py)
- [knowledge_base_runtime/app.py](../src/knowledge_base_runtime/app.py)

其中：

- `build_project_app(...)` 会先加载项目
- `build_knowledge_base_runtime_app(...)` 会把 `KnowledgeBaseProject` 装成 FastAPI 应用

### 8.2 页面路由

页面路由全部走：

- [app.py](../src/knowledge_base_runtime/app.py)

这些函数上带有：

- `@governed_symbol id=kb.runtime.page_routes ...`

它们消费的是：

- `project.route`
- `backend_spec.transport.product_spec_endpoint`

而不是直接自己再发明一套路径。

### 8.3 页面 HTML

页面 HTML 主要来自：

- [frontend.py](../src/knowledge_base_runtime/frontend.py)

它消费的是：

- `ui_spec`
- `frontend_contract`
- `workbench_contract`
- `validation_reports`

### 8.4 后端 API 和回答行为

API 和回答行为主要来自：

- [backend.py](../src/knowledge_base_runtime/backend.py)

这里带有这些高风险治理对象：

- `kb.api.library_contracts`
- `kb.api.chat_contract`
- `kb.answer.behavior`

它们消费的是：

- `backend_spec`
- `documents`
- `return policy`
- `chat/context/return` 相关产品真相

所以第五层可以压成：

```text
frontend_contract / workbench_contract / ui_spec / backend_spec
  -> runtime app
  -> runtime pages
  -> runtime api
```

## 9. 第六层：Evidence 是怎么生成的

Evidence 不是附属品，而是这条链能反查的关键。

Evidence 生成入口在：

- `materialize_knowledge_base_project(...)`

它会最终写出：

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`

这里最关键的是 3 份：

### 9.1 framework_ir.json

证明 framework Markdown 已经被编译成结构 IR。

### 9.2 generation_manifest.json

证明：

- 当前唯一生成入口是什么
- framework 输入是什么
- implementation config 的 effect 是什么
- 当前 canonical generated files 是什么

### 9.3 governance_manifest.json / governance_tree.json

证明：

- 高风险结构对象有哪些
- 每个对象的 framework / product / implementation 上游来源是什么
- 这些对象在代码里绑定到了哪些 symbol
- 哪些文件属于 strict zone

## 10. 反查链：Code 是怎么被拉回上游的

这部分是当前仓库最关键的“不是只靠自觉”的地方。

### 10.1 代码里先显式打点

高风险代码对象先通过注释显式打点：

- `# @governed_symbol id=...`

你在这些文件里都能看到：

- [frontend_kernel/contracts.py](../src/frontend_kernel/contracts.py)
- [knowledge_base_framework/workbench.py](../src/knowledge_base_framework/workbench.py)
- [knowledge_base_runtime/app.py](../src/knowledge_base_runtime/app.py)
- [knowledge_base_runtime/backend.py](../src/knowledge_base_runtime/backend.py)
- [knowledge_base.py](../src/project_runtime/knowledge_base.py)

### 10.2 治理闭包会收集这些打点

治理主入口在：

- [governance.py](../src/project_runtime/governance.py)

核心函数是：

- `build_governance_closure(project)`
- `build_governance_manifest(project)`
- `build_governance_tree(project)`

其中又会走到：

- `scan_governed_python_bindings(...)`
- `collect_governed_bindings(...)`

也就是说，治理链不是靠“猜这个函数应该对应哪个 framework”，而是先从代码里把治理打点收上来。

### 10.3 结构对象本身也不是自由散落的

`build_governance_closure(project)` 会为项目构建：

- `structural_objects`
- `required_roles`
- `role_bindings`
- `strict_zone`
- `upstream_closure`
- `evidence_artifacts`

也就是说，代码绑定的不是原始 Markdown，而是“上游结构闭包”。

### 10.4 validate_strict_mapping.py 会做最终比较

最终校验入口在：

- [validate_strict_mapping.py](../scripts/validate_strict_mapping.py)

它做的不是单一检查，而是多层比较：

1. 重新物化项目 generated 产物
2. 比较 generated 产物是否漂移
3. 比较 implementation config 是否真的进入下游 effect
4. 比较 governance manifest/tree 是否与当前代码一致
5. 比较 framework tree / mapping registry / change propagation 是否闭合

所以反查链可以压成：

```text
code symbol
  -> governed_symbol annotation
  -> governance closure
  -> governance manifest/tree
  -> validate_strict_mapping.py compare
```

## 11. 两个具体例子

下面给两个最容易顺着追的例子。

### 11.1 例子 A：`/api/knowledge/product-spec`

这条链怎么追：

1. 产品层先给出 `route.api_prefix`
   - [projects/knowledge_base_basic/product_spec/route.toml](../projects/knowledge_base_basic/product_spec/route.toml)

2. 实现层再给出 `evidence.product_spec_endpoint`
   - [projects/knowledge_base_basic/implementation_config.toml](../projects/knowledge_base_basic/implementation_config.toml)

3. 编译器校验它必须 stay under `route.api_prefix`
   - [knowledge_base.py](../src/project_runtime/knowledge_base.py)
   - `_validate_implementation_config(...)`

4. 编译器把它写进 `backend_spec.transport.product_spec_endpoint`
   - [knowledge_base.py](../src/project_runtime/knowledge_base.py)
   - `_build_backend_spec()`

5. 运行时 app 再从 `backend_spec.transport` 读出来真正挂路由
   - [app.py](../src/knowledge_base_runtime/app.py)
   - `_require_backend_transport(...)`
   - `@app.get(transport.product_spec_endpoint)`

6. generation manifest 还会把这条 effect 写成正式证据
   - [generation_manifest.json](../projects/knowledge_base_basic/generated/generation_manifest.json)
   - `configuration_effects["evidence.product_spec_endpoint"]`

也就是：

```text
product_spec.route.api_prefix
  + implementation_config.evidence.product_spec_endpoint
    -> _validate_implementation_config
    -> backend_spec.transport.product_spec_endpoint
    -> app.py route
    -> generation_manifest.configuration_effects
```

### 11.2 例子 B：`kb.answer.behavior`

这条链怎么追：

1. framework 侧来源主要来自知识库工作台场景模块的 `R1~R4`
   - [framework/knowledge_base/L2-M0-知识库工作台场景模块.md](../framework/knowledge_base/L2-M0-%E7%9F%A5%E8%AF%86%E5%BA%93%E5%B7%A5%E4%BD%9C%E5%8F%B0%E5%9C%BA%E6%99%AF%E6%A8%A1%E5%9D%97.md)

2. product 层来源主要来自：
   - `chat`
   - `context`
   - `return`

3. implementation 层来源主要来自：
   - `backend`

4. 代码落点在：
   - [backend.py](../src/knowledge_base_runtime/backend.py)
   - 带有 `@governed_symbol id=kb.answer.behavior ...`

5. governance manifest 里会把它固化为一个 `StructuralObject`
   - [governance_manifest.json](../projects/knowledge_base_basic/generated/governance_manifest.json)
   - `object_id = "kb.answer.behavior"`

6. 这个对象里会明确列出：
   - framework sources
   - product sources
   - implementation sources
   - semantic expectation
   - actual evidence

也就是：

```text
framework rules
  + product sections
  + implementation backend section
    -> backend.py governed symbol
    -> governance structural object
    -> compare expected vs actual
```

这就是当前仓库里最接近“一一对应”的真实机制。

## 12. 你如果要顺着搞清楚，一步一步该怎么看

### 路径一：从上往下看

适合“我想知道某个配置最终变成了什么代码”。

顺序：

1. `product_spec.toml` / `product_spec/*.toml`
2. `implementation_config.toml`
3. `knowledge_base.py`
4. `generation_manifest.json`
5. `frontend.py` / `backend.py` / `app.py`

### 路径二：从下往上看

适合“我想知道某个函数/路由/行为到底来自哪里”。

顺序：

1. 在代码里找 `@governed_symbol`
2. 去 `governance_manifest.json` 找同一个 `object_id`
3. 看这个对象的 `sources.framework / sources.product / sources.implementation`
4. 再回到对应 framework file / section file

### 路径三：查配置是否真的生效

适合“我想知道 implementation_config 这一项是不是死配置”。

顺序：

1. 找 `implementation_config.toml` 的字段
2. 去 `generation_manifest.json` 看 `configuration_effects`
3. 找 `targets`
4. 回到 `ui_spec` / `backend_spec` / `generated_artifacts`
5. 再去 runtime 消费点确认

## 13. 当前已经覆盖到什么程度

现在这条链已经能强覆盖：

- framework 主链输入
- Product Spec 和 Implementation Config
- 编译器生成的 contract / spec / route
- 运行时的高风险 symbol
- generated canonical artifacts
- implementation config 的 effect 生效链

但它还没有做到：

- 仓库每一个 helper 函数都直接挂 framework
- 每一行代码都能反查到 framework

更准确地说，当前仓库做的是：

> 对稳定、可命名、会影响结构语义或行为边界的对象做治理闭包。

这也是当前最合理的粒度。

---

## 14. 一句话总结

这个仓库不是“先写框架文档，再手写一套代码碰碰运气”。

它当前真正做到的是：

> 先把 framework 文档编成 IR，再把产品真相和实现细化编成项目对象，再把项目对象收敛成 contract / spec / route / runtime / generated evidence，最后再用 governance manifest/tree 和 strict mapping 把代码反查回 framework、product 和 implementation。

如果你要顺着看一一对应，最有用的不是看单个文件，而是沿着：

> `product_spec / implementation_config -> knowledge_base.py -> generated manifest -> governed symbol -> governance manifest`

这条链读。它就是当前仓库最真实的映射路径。
