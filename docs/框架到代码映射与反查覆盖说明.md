# 框架到代码映射与反查覆盖说明

## 1. 最短答案

当前仓库已经不再走旧双轨配置和旧大聚合模块主线。  
现在真正生效的是：

```text
framework/*.md
  -> Framework IR
    -> Framework Package Registry
      -> package entry class
        -> projects/<project_id>/project.toml
          -> config slicing
            -> package compile
              -> runtime assembly
                -> generated/canonical_graph.json
                  -> derived governance / tree / report
```

完整展开版见：

- [全链实现框架与跳转逻辑详解.md](./全链实现框架与跳转逻辑详解.md)

## 2. 当前仓库的真实主链

必须记住三件事：

1. `framework/*.md` 仍是作者源。
2. `projects/<project_id>/project.toml` 是唯一项目配置入口。
3. `generated/canonical_graph.json` 是唯一机器真相源。

## 4. 当前从 framework 到代码怎么走

### 第一步：Framework Markdown 先变成 IR

入口：

- [src/framework_ir/parser.py](../src/framework_ir/parser.py)
- [src/framework_ir/models.py](../src/framework_ir/models.py)

产物：

- `FrameworkModule`
- `FrameworkBase`
- `FrameworkRule`

### 第二步：Framework file 绑定到 package

入口：

- [src/framework_packages/modules/](../src/framework_packages/modules)
- [src/framework_packages/builtin_registry.py](../src/framework_packages/builtin_registry.py)

要点：

- 每个 framework 文件对应一个 package
- 每个 package 只有一个入口 class
- 入口 class 必须注册到统一 registry

### 第三步：统一项目配置按 contract 分发

入口：

- [projects/knowledge_base_basic/project.toml](../projects/knowledge_base_basic/project.toml)
- [src/project_runtime/project_config_source.py](../src/project_runtime/project_config_source.py)

要点：

- 统一配置分成 `selection / truth / refinement / narrative`
- package 只能消费自己声明的 config slice
- 子包不能偷读全局配置

### 第四步：Compiler 用 registry 装配 package

真正的编排器是：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py) 的 `_compile_runtime_bundle(...)`

它做的事情是：

- 读取 module tree
- 读取 unified project config
- 用 registry 找 package
- 做 config slicing
- 调用 package compile
- 汇总 runtime projection

### 第五步：Evidence 汇入 canonical

核心产物：

- `canonical_graph.json`
- `runtime_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`

这些文件都由 canonical 派生，不再各自承担独立真相。

## 5. 当前从代码反查 framework 怎么走

现在有两条稳定路径。

### 路径 A：package 绑定反查

```text
runtime / compiled package
  -> module id
    -> registry binding
      -> framework file
```

这条链来自：

- `layers.framework.registry_binding`

### 路径 B：canonical 分层反查

```text
runtime projection
  -> code.package_results
    -> config.projection
      -> framework.module_tree
```

这条链来自：

- `generated/canonical_graph.json`

## 7. 一句话结论

当前仓库已经把“框架到代码”改造成真正的 package-assembly 编译链：

> framework markdown 先形成 IR，再通过 registry 绑定到 package；统一项目配置按 contract 切片分发给 package，编译结果收敛成 runtime projection，并统一沉到 `canonical_graph.json`，从而实现 framework 到代码的正向落地和代码到 framework 的反向追溯。
