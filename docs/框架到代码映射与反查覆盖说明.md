# 框架到代码映射与反查覆盖说明

## 1. 这份文档回答什么

这份文档只回答一件事：

> 当前仓库如何把 `Framework -> Product -> Implementation -> Code -> Evidence` 串成一条可逐层跳转、可逐层反查的路径。

如果你更想先看图，再回来看文字，可以直接打开：

- [framework-to-code-mapping-cn.pptx](../docs/presentations/framework-to-code-mapping-cn.pptx)

如果你要看更完整的全链路说明，包括 canonical source、逐层跳转和治理直跳逻辑，继续看：

- [全链实现框架与跳转逻辑详解.md](./全链实现框架与跳转逻辑详解.md)

## 2. 先记住一条总图

当前知识库主链可以压成下面这张图：

```text
framework/*.md
  -> FrameworkModule / FrameworkBase / FrameworkRule
product_spec.toml + product_spec/*.toml
  -> KnowledgeBaseProductModule
implementation_config.toml
  -> KnowledgeBaseImplementationModule
      -> KnowledgeBaseCodeModule
          -> runtime code symbols
          -> canonical_graph.json
              -> governance / report / view files
```

这条链里只有一个机器真相源：

- `projects/knowledge_base_basic/generated/canonical_graph.json`

其它生成文件都是派生视图。

## 4. 第一层：Framework 如何进入编译链

知识库项目当前直接声明三份 framework markdown：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

它们先进入：

- [parser.py](../src/framework_ir/parser.py)
- [models.py](../src/framework_ir/models.py)

解析结果不是一堆匿名 dict，而是正式结构对象：

- `FrameworkModule`
- `FrameworkCapability`
- `FrameworkBoundary`
- `FrameworkBase`
- `FrameworkRule`
- `FrameworkVerification`

所以第一层映射关系是：

```text
framework/*.md
  -> parse_framework_module(...)
  -> FrameworkModule
```

`FrameworkModule` 之后会被继续装入 `KnowledgeBaseFrameworkLayer`。

## 5. 第二层：Product 如何消费 Framework 导出

产品真相来源：

- [product_spec.toml](../projects/knowledge_base_basic/product_spec.toml)
- [product_spec/](../projects/knowledge_base_basic/product_spec)

加载入口在：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 的 `_load_product_spec()`

这里形成的是 `KnowledgeBaseProductSpec`，随后进一步收敛成：

- `KnowledgeBaseProductModule`

这个模块的职责是：

- 固定产品元信息
- 固定页面、路由、视觉、交互和文档种子
- 记录 section 到 source file 的来源
- 明确 derived_from 的 framework module

所以第二层映射关系是：

```text
FrameworkModule exports
  + product_spec.toml
  -> KnowledgeBaseProductModule
```

## 6. 第三层：Implementation 如何消费 Product 导出

实现细化来源：

- [implementation_config.toml](../projects/knowledge_base_basic/implementation_config.toml)

加载入口同样在：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 的 `_load_implementation_config()`

这里先形成 `KnowledgeBaseImplementationConfig`，然后收敛成：

- `KnowledgeBaseImplementationModule`

它只保存实现层决策，例如：

- frontend renderer / style / script profile
- backend renderer / transport / retrieval strategy
- evidence endpoint
- artifact naming

它不会回改 Product 真相。

## 7. 第四层：Code 如何消费 Implementation 导出

真正把上游三层收成代码侧稳定对象的入口是：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py) 的 `_compile_project()`

这一步会构建出：

- `KnowledgeBaseFrameworkLayer`
- `KnowledgeBaseProductModule`
- `KnowledgeBaseImplementationModule`
- `KnowledgeBaseCodeModule`

`KnowledgeBaseCodeModule` 是当前运行时代码唯一应该看的上游对象。

它当前稳定导出的关键字段包括：

- `frontend_contract`
- `workbench_contract`
- `ui_spec`
- `backend_spec`
- `product_view`
- `public_summary`
- `implementation_effects`

运行时代码对应关系：

- [contracts.py](../src/frontend_kernel/contracts.py)
  - 生成 `kb.frontend.surface_contract`
- [workbench.py](../src/knowledge_base_framework/workbench.py)
  - 生成 `kb.workbench.surface_contract`
- [app.py](../src/knowledge_base_runtime/app.py)
  - 消费 `product_view`、`route`、`backend_spec`
- [frontend.py](../src/knowledge_base_runtime/frontend.py)
  - 消费 `ui_spec`、`workbench_contract`
- [backend.py](../src/knowledge_base_runtime/backend.py)
  - 消费 `backend_spec`、`documents`

所以第四层的核心关系是：

```text
KnowledgeBaseImplementationModule
  -> KnowledgeBaseCodeModule
      -> runtime contracts / specs / route bindings / handlers
```

## 8. 第五层：Evidence 如何从 Code 导出生成

Evidence 层入口仍然在：

- [knowledge_base.py](../src/project_runtime/knowledge_base.py)

它会把 Code 层导出物化成：

- `canonical_graph.json`
- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`

其中只有：

- `canonical_graph.json`

是机器真相源。

其它文件的角色是：

- `framework_ir.json`
  - Framework 层视图
- `product_spec.json`
  - Product 层视图
- `implementation_bundle.py`
  - 代码导入友好视图
- `generation_manifest.json`
  - 配置 effect 与生成纪律视图
- `governance_manifest.json`
  - 高风险结构对象治理视图
- `governance_tree.json`
  - 树状追溯与 GUI 视图
- `strict_zone_report.json`
  - strict zone 视图
- `object_coverage_report.json`
  - 结构覆盖视图

## 9. 一个具体例子：`/api/knowledge/product-spec`

这条路由最适合用来理解逐层映射。

### 9.1 Framework

上游约束来自：

- 前端路由与承载边界
- 知识库场景里的返回/引用/证据闭环
- 后端接口模块里的 transport 与证据出口语义

### 9.2 Product

产品真相固定了：

- `route.api_prefix`

来源：

- [product_spec.toml](../projects/knowledge_base_basic/product_spec.toml)

### 9.3 Implementation

实现细化固定了：

- `evidence.product_spec_endpoint`

来源：

- [implementation_config.toml](../projects/knowledge_base_basic/implementation_config.toml)

### 9.4 Code

编译器把它们收进：

- `backend_spec.transport.api_prefix`
- `backend_spec.transport.product_spec_endpoint`
- `product_view`

运行时代码真正挂路由的位置在：

- [app.py](../src/knowledge_base_runtime/app.py)

### 9.5 Evidence

最后这条链会出现在：

- `canonical_graph.json`
- `generation_manifest.json`
- `governance_manifest.json`

所以这条路由既能从上往下追，也能从下往上追。

## 10. 反查是怎么做的

代码反查上游，不是靠 grep 注释，而是靠治理闭包。

入口在：

- [governance.py](../src/project_runtime/governance.py)

当前高风险结构对象，例如：

- `kb.runtime.page_routes`
- `kb.frontend.surface_contract`
- `kb.workbench.surface_contract`
- `kb.ui.surface_spec`
- `kb.backend.surface_spec`
- `kb.api.library_contracts`
- `kb.api.chat_contract`
- `kb.answer.behavior`

都会被记录：

- 代码绑定点
- upstream refs
- expected / actual evidence
- required bindings

然后落进：

- `governance_manifest.json`
- `governance_tree.json`

## 11. 代码改了以后，怎么检查 framework 有没有对应

这件事现在通过两条链完成：

### 11.1 重新物化 generated

命令：

```bash
uv run python scripts/materialize_project.py
```

这会重新生成 canonical 图及其派生视图。如果有人手改 generated 文件，或者代码与编译链不一致，就会暴露出来。

### 11.2 严格映射校验

命令：

```bash
uv run python scripts/validate_strict_mapping.py
uv run python scripts/validate_strict_mapping.py --check-changes
```

它会检查：

- framework 文法和引用是否合法
- project/product/implementation 分层是否越界
- generated 文件是否与重新物化结果一致
- implementation_config 字段是否真的进入下游行为
- 治理树和结构对象是否还能反查回上游

## 12. 能不能覆盖全部代码

当前答案是：

- **关键结构对象可以强覆盖**
- **全部低风险代码还不是逐对象治理**

也就是说，当前仓库已经做到：

- 关键 route / contract / spec / behavior / evidence builder 可反查
- 关键配置 effect 可验证

但还没有做到：

- 每个辅助函数、每个内部细节都逐对象上框架

这也是当前仓库刻意选择的边界：逐结构覆盖，而不是逐行覆盖。

## 13. 一句话总结

当前仓库已经不再是“Markdown 说明书 + 代码实现”两套语言，而是一条真正可追溯的编译链：

> Framework 先定义结构空间，Product 固定产品真相，Implementation 固定实现路径，Code 承接真实运行，Evidence 固化 canonical 图与反查视图。
