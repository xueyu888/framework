# 全链实现框架与跳转逻辑详解

## 1. 现在的完整链路

当前仓库的完整实现链路已经切成下面这条：

```text
Framework Markdown
  -> parser / Framework IR
    -> framework package registry
      -> package entry classes
        -> unified project config
          -> config slicing
            -> package compile
              -> runtime projection
                -> canonical_graph.json
                  -> governance / tree / report / workspace views
```

这里最重要的变化有四个：

1. 每个 framework 文件都对应一个 package。
2. package 之间只用组合，不用继承表达装配关系。
3. 项目配置统一到 `project.toml`，但仍按 `selection / truth / refinement / narrative` 分区。
4. `canonical_graph.json` 是唯一机器真相源。

## 2. Framework layer

Framework 作者源仍然是：

- `framework/*/*.md`

解析入口：

- [src/framework_ir/parser.py](../src/framework_ir/parser.py)
- [src/framework_ir/models.py](../src/framework_ir/models.py)

解析结果：

- `FrameworkModule`
- `FrameworkBase`
- `FrameworkRule`
- `FrameworkVerification`

编译器不直接读散乱 Markdown 文本，而是先把 Markdown 收敛成可装配 IR。

## 3. Package layer

### 3.1 一份 framework 文件对应一个 package

代码位置：

- [src/framework_packages/modules/](../src/framework_packages/modules)

例如：

- `framework/frontend/L2-M0-前端框架标准模块.md`
  - 对应 `src/framework_packages/modules/frontend_l2_m0/`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
  - 对应 `src/framework_packages/modules/knowledge_base_l2_m0/`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`
  - 对应 `src/framework_packages/modules/backend_l2_m0/`

### 3.2 每个 package 只有一个入口 class

入口 class 全部继承统一静态包基类，并实现统一 contract：

- [src/framework_packages/contract.py](../src/framework_packages/contract.py)
- [src/framework_packages/static.py](../src/framework_packages/static.py)

contract 至少承载：

- framework file 对齐
- module id
- config contract
- child slot
- compile
- export
- evidence contribution

### 3.3 registry 是正式绑定真相

入口：

- [src/framework_packages/registry.py](../src/framework_packages/registry.py)
- [src/framework_packages/builtin_registry.py](../src/framework_packages/builtin_registry.py)

registry 负责：

- framework file -> entry class
- module id -> entry class
- 冲突检测
- framework 未实现检测
- orphan package 检测

编译器不再靠扫目录猜入口类。

## 4. Config layer

### 4.1 统一项目配置入口

当前默认项目：

- [projects/knowledge_base_basic/project.toml](../projects/knowledge_base_basic/project.toml)

编译器只读这一份配置。

### 4.2 逻辑分区

这份配置物理上是一份 TOML，逻辑上分成：

- `selection`
  - 选择 root modules 和 preset
- `truth`
  - 固定产品真相
- `refinement`
  - 固定实现偏好和生成物命名
- `narrative`
  - 解释性说明

### 4.3 切片分发

每个 package 的 config contract 会声明它需要的 dotted paths。  
编译器只把这些切片分发给对应 package：

- `selection.preset`
- `truth.surface`
- `truth.library`
- `refinement.frontend`
- `refinement.backend`

子包不能绕过 contract 偷读全局配置。

## 5. Compile layer

核心入口：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)

`_compile_runtime_bundle(...)` 的真实工作顺序是：

1. 读取 `project.toml`
2. 加载 framework registry
3. 加载 package registry
4. 解析 root modules
5. 展开 module closure
6. 按 config contract 做 slicing
7. 递归按依赖顺序 compile package
8. 汇总 runtime projection
9. 汇总 canonical graph

这里已经接入了新的 `module tree + registry + config slicing + package compile` 主骨架，  
并且运行时公开对象已经收敛成 `KnowledgeBaseRuntimeBundle`。

当前实际情况是：

- package compile 已经是新主链
- 但 [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py) 里仍保留 `KnowledgeBaseCompilationState` 作为编译期内部聚合对象
- runtime projection 仍然由专门 builder 手工汇总，还没有完全演进成纯 package export 自动装配

## 6. Runtime layer

运行时入口：

- [src/project_runtime/app_factory.py](../src/project_runtime/app_factory.py)
- [src/knowledge_base_runtime/app.py](../src/knowledge_base_runtime/app.py)

运行时当前直接消费 `KnowledgeBaseRuntimeBundle`：

- `frontend_contract`
- `workbench_contract`
- `ui_spec`
- `backend_spec`
- `project_config_view`

它不再回头读取旧双轨配置，但也还没有完全演进成“只由 package export 自动装配 runtime”。

## 7. Canonical layer

唯一机器真相源：

- `projects/<project_id>/generated/canonical_graph.json`

它稳定分成四层：

- `layers.framework`
- `layers.config`
- `layers.code`
- `layers.evidence`

### 7.1 framework layer

记录：

- module tree
- root module ids
- registry binding

### 7.2 config layer

记录：

- project file
- section sources
- selection
- truth
- refinement
- narrative
- 每个 module 的 config projection

### 7.3 code layer

记录：

- package compile order
- root packages
- package compile results
- runtime projection

### 7.4 evidence layer

记录：

- validation reports
- document digests
- generated artifact paths
- derived view metadata

## 8. Derived views

派生入口：

- [src/project_runtime/governance.py](../src/project_runtime/governance.py)

当前派生视图包括：

- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`
- `runtime_bundle.py`

这些文件都必须在内容里显式带上：

- `derived_from = canonical_graph.json`

## 9. Workspace layer

workspace 级派生视图入口：

- [src/workspace_governance.py](../src/workspace_governance.py)

它不再读取旧 mapping registry 或旧 structural governance 对象模型，  
而是直接用：

- project discovery
- canonical artifact contract
- workspace hierarchy

生成：

- `docs/hierarchy/shelf_governance_tree.json`
- `docs/project_discovery_audit.md`

## 10. 验证链

物化：

- [scripts/materialize_project.py](../scripts/materialize_project.py)

严格校验：

- [scripts/validate_strict_mapping.py](../scripts/validate_strict_mapping.py)

现在校验的核心是：

1. framework registry 与 package registry 是否一一对应
2. project materialization 是否与仓库内生成物一致
3. canonical graph 是否只有 `framework / config / code / evidence`
4. derived views 是否都反指 canonical
5. workspace governance 视图是否同步
