# 全链实现框架与跳转逻辑详解

## 1. 当前主链

当前仓库的实现链路已经固定为：

```text
Framework Markdown
  -> Framework IR
    -> Package Registry
      -> selection.roots
        -> Module Tree
          -> Config Slicing
              -> Package Compile
                -> Child Runtime Fragments
                -> Root Runtime Exports
                -> Runtime Projection
                  -> Canonical Graph
                    -> Derived Views / Validation
```

## 3. 根模块选择

`selection` 已改为 `[[selection.roots]]`：

- `slot_id` 表达项目内稳定引用名
- `role` 表达当前根模块的语义位
- `framework_file` 才是进入 registry 与 module tree 的正式入口

当前知识库项目通过三个 root role 绑定：

- `frontend`
- `knowledge_base`
- `backend`

但编译器不再把它们写死成固定结构槽位。

## 4. runtime 收敛

runtime 主链不再依赖知识库专属 builder，也不再由场景名决定装配。

现在的主路径是：

1. child package 在 `compile(...)` 阶段先产出 runtime fragments
2. root package 只从 `payload.child_runtime_exports` 组合根级 runtime exports
3. `project_runtime.pipeline.assemble_runtime_projection(...)` 汇总通用 export graph
4. [src/project_runtime/runtime_app.py](../src/project_runtime/runtime_app.py) 只按根级 exports 构造运行时
