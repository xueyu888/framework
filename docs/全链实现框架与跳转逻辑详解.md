# 全链实现框架与跳转逻辑详解

## 1. 这份文档解决什么问题

这份文档只做一件事：

> 把当前 `shelf` 仓库里 `Framework -> Product -> Implementation -> Code -> Evidence` 这条主链如何真实落地、如何逐层跳转、如何从 code 反查回 framework，完整说清楚。

它不是理念文档，也不是单个模块说明，而是当前知识库主链的工程全景。

如果你想先看一张图，再回来看文字，可以先打开：

- [framework-to-code-mapping-cn.pptx](./presentations/framework-to-code-mapping-cn.pptx)

如果你想只看最短版本，可以先看：

- [architecture.md](./architecture.md)
- [框架到代码映射与反查覆盖说明.md](./框架到代码映射与反查覆盖说明.md)

## 2. 当前主线的最短总图

当前默认主线已经固定为五层：

```text
Framework Markdown
  -> Framework Module objects
Product Spec
  -> Product Module
Implementation Config
  -> Implementation Module
Code Binding / Runtime Surface
  -> Code Module
Evidence / Governance / Reports
  -> Evidence Module
```

在知识库项目里，这条链最终被固化成唯一机器真相源：

- `projects/knowledge_base_basic/generated/canonical_graph.json`

这是当前主链的唯一机器真相源。其它 manifest、tree、report 都只是它的派生视图。

## 3. 现在真正有哪些层

### 3.1 Framework 层

作者源：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

编译入口：

- [src/framework_ir/parser.py](../src/framework_ir/parser.py)
- [src/framework_ir/models.py](../src/framework_ir/models.py)

正式对象：

- `FrameworkModule`
- `FrameworkCapability`
- `FrameworkBoundary`
- `FrameworkBase`
- `FrameworkRule`
- `FrameworkVerification`

这里的含义很直接：

- 每个 framework markdown 文件先被解析成一个独立 `FrameworkModule`
- 每个 `B*` 被编译成 `FrameworkBase`
- 每个 `R*` 被编译成 `FrameworkRule`

也就是说，Framework 层现在不是匿名 dict，而是正式对象。

### 3.2 Product 层

作者源：

- `projects/knowledge_base_basic/product_spec.toml`
- `projects/knowledge_base_basic/product_spec/*.toml`

编译入口：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)

正式对象：

- `KnowledgeBaseProductModule`

Product 层只做产品真相具体化，例如：

- 产品元信息
- 页面与路由真相
- 聊天、预览、返回、上下文真相
- 文档种子与展示面真相

它不反向修改 framework 语义。

### 3.3 Implementation 层

作者源：

- `projects/knowledge_base_basic/implementation_config.toml`

编译入口：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)

正式对象：

- `KnowledgeBaseImplementationModule`

Implementation 层只做实现细化，例如：

- `frontend.renderer`
- `frontend.style_profile`
- `frontend.script_profile`
- `backend.renderer`
- `backend.transport`
- `backend.retrieval_strategy`
- `evidence.product_spec_endpoint`
- artifact 命名

### 3.4 Code 层

编译入口：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)
- [src/project_runtime/layered_models.py](../src/project_runtime/layered_models.py)

正式对象：

- `KnowledgeBaseCodeModule`

Code 层不再直接“看 Product + Implementation + Framework 的混合大对象”，而是只消费 `KnowledgeBaseImplementationModule` 的收敛导出，然后输出稳定 code surface。

当前 Code 层的主要落点在：

- [src/frontend_kernel/contracts.py](../src/frontend_kernel/contracts.py)
- [src/knowledge_base_framework/workbench.py](../src/knowledge_base_framework/workbench.py)
- [src/knowledge_base_runtime/app.py](../src/knowledge_base_runtime/app.py)
- [src/knowledge_base_runtime/backend.py](../src/knowledge_base_runtime/backend.py)
- [src/knowledge_base_runtime/frontend.py](../src/knowledge_base_runtime/frontend.py)
- [src/knowledge_base_runtime/frontend_style.py](../src/knowledge_base_runtime/frontend_style.py)
- [src/knowledge_base_runtime/frontend_script.py](../src/knowledge_base_runtime/frontend_script.py)
- [src/knowledge_base_runtime/workspace.py](../src/knowledge_base_runtime/workspace.py)

### 3.5 Evidence 层

Evidence 层也是由编译链直接导出，不是人工真相源。

主要入口：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)
- [src/project_runtime/governance.py](../src/project_runtime/governance.py)
- [scripts/validate_strict_mapping.py](../scripts/validate_strict_mapping.py)

Evidence 当前主要包含：

- `canonical_graph.json`
- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`

- `object_coverage_report.json`

其中只有 `canonical_graph.json` 是 canonical source；其余都只是 view。

## 4. 当前唯一机器真相源是什么

当前唯一机器真相源是：

- `projects/knowledge_base_basic/generated/canonical_graph.json`

它的结构不是平铺大杂烩，而是明确分成五层：

```json
{
  "layers": {
    "framework": {...},
    "product": {...},
    "implementation": {...},
    "code": {...},
    "evidence": {...}
  }
}
```

当前真实的逐层依赖关系是：

- `layers.product.derived_from` 指向 Framework 层模块文件
- `layers.implementation.derived_from` 指向 Product 层模块
- `layers.code.derived_from` 指向 Implementation 层模块
- `layers.evidence.derived_from` 指向 Code 层模块

也就是说，主链上真正允许的跳转是：

```text
Framework -> Product -> Implementation -> Code -> Evidence
```

而不是任意跨层乱跳。

## 5. 现在的跳转逻辑到底是怎样的

### 5.1 主跳转逻辑：逐层跳

这是当前系统默认、最干净、最推荐的跳转方式。

#### Framework -> Product

从 `canonical_graph.json` 的 `layers.product.derived_from` 可以看到，当前知识库产品模块来自三份 framework markdown：

- `framework/frontend/L2-M0-前端框架标准模块.md`
- `framework/knowledge_base/L2-M0-知识库工作台场景模块.md`
- `framework/backend/L2-M0-知识库接口框架标准模块.md`

也就是说，产品层是对这三份 framework 模块的产品化收敛。

#### Product -> Implementation

`layers.implementation.derived_from` 指向：

- `projects/knowledge_base_basic/product_spec.toml`

说明实现层不是直接消费 framework，而是先消费产品真相。

#### Implementation -> Code

`layers.code.derived_from` 指向：

- `projects/knowledge_base_basic/implementation_config.toml`

说明代码层当前是对实现层导出的代码绑定，不直接回看 product/framework 内部细节。

#### Code -> Evidence

`layers.evidence.derived_from` 指向：

- `knowledge_base_basic.code`

说明 Evidence 层只消费 Code 层输出，再把它派生成治理树、报告和快照。

### 5.2 直跳逻辑：Framework 直接跳到 Code 有没有

有，但它不是主链，而是**治理视图里的诊断跳转**。

当前仓库里，直接 `Framework -> Code` 的跳转不是写在 `canonical_graph.json` 主层级上的，而是写在：

- `projects/knowledge_base_basic/generated/governance_manifest.json`
- `projects/knowledge_base_basic/generated/governance_tree.json`

这两个文件明确声明自己只是 canonical 派生视图：

- `canonical_source = projects/knowledge_base_basic/generated/canonical_graph.json`
- `derived_view_of = projects/knowledge_base_basic/generated/canonical_graph.json`

这条直跳链的用途不是替代五层主链，而是为了：

- GUI 结构追溯
- governed symbol 精确定位
- strict mapping 对高风险结构对象做 expected/actual 比对

所以结论是：

- **主链跳转**：必须逐层跳
- **诊断直跳**：可以从 Framework 通过 governance object 直达 Code symbol

## 6. 从 Framework 一层层跳到 Code，当前怎么走

最稳的实际顺序是：

1. 选定 framework markdown 文件
2. 找到它在 `canonical_graph.json` 中进入哪一个 product module
3. 再看 product module 进入哪一个 implementation module
4. 再看 implementation module 进入哪一个 code module
5. 最后从 code export 或 governance object 找到真实代码符号

## 7. 具体例子一：`/api/knowledge/product-spec` 是怎么一路落下来的

这条链很适合看“逐层跳转”。

### Framework

它首先受前端和后端 framework 的规则约束，尤其是：

- 前端 framework 的路由/界面承载规则
- 后端 framework 的接口/证据暴露规则

### Product

产品真相在：

- `projects/knowledge_base_basic/product_spec/route.toml`

这里定义了产品层的 route truth，例如 API prefix 等。

### Implementation

实现细化在：

- `projects/knowledge_base_basic/implementation_config.toml`

这里的：

- `evidence.product_spec_endpoint`

决定产品规格证据以什么 endpoint 暴露。

### Code

编译入口在：

- [src/project_runtime/knowledge_base.py](../src/project_runtime/knowledge_base.py)

它把 route truth 和 implementation endpoint 合并进 `backend_spec.transport`，然后由：

- [src/knowledge_base_runtime/app.py](../src/knowledge_base_runtime/app.py)

里的路由注册函数真正挂出 `/api/knowledge/product-spec`。

### Evidence / Governance

在治理视图里，它归属于高风险结构对象：

- `kb.runtime.page_routes`

它的治理来源当前是：

- framework：`framework/frontend/L2-M0-前端框架标准模块.md` 的 `R1/R2/R3`
- product：`projects/knowledge_base_basic/product_spec/route.toml` 的 `route`
- implementation：`projects/knowledge_base_basic/implementation_config.toml` 的 `evidence`

它要求绑定的 code roles 包括：

- `function:root`
- `function:knowledge_base_page`
- `function:knowledge_base_list_page`
- `function:knowledge_base_detail_page`
- `function:document_detail_page`
- `function:product_spec`
- `function:build_knowledge_base_runtime_app`
- 以及治理用的 expected/actual extractor

这就是“逐层收敛之后，再通过治理对象把 framework/product/implementation 绑定回具体 code symbol”的真实实现。

## 8. 具体例子二：`kb.answer.behavior` 是怎么反查回 framework 的

这条链很适合看“从 code 反查 framework”。

### Code 落点

实际行为落在：

- [src/knowledge_base_runtime/backend.py](../src/knowledge_base_runtime/backend.py)

核心 governed role 是：

- `function:answer_question`

### Governance 绑定

在 `governance_manifest.json` 里，它对应结构对象：

- `kb.answer.behavior`

这个对象的来源当前是：

- framework：
  - `framework/knowledge_base/L2-M0-知识库工作台场景模块.md` 的 `R1/R2/R3/R4`
- product：
  - `projects/knowledge_base_basic/product_spec/chat.toml`
  - `projects/knowledge_base_basic/product_spec/context.toml`
  - `projects/knowledge_base_basic/product_spec/return.toml`
- implementation：
  - `projects/knowledge_base_basic/implementation_config.toml` 的 `backend`

它要求绑定的 role 有：

- `kb.answer.behavior:answer_question`
- `kb.answer.behavior:expected_builder`
- `kb.answer.behavior:actual_extractor`

所以从 `backend.py` 的 `answer_question` 可以反查到：

```text
code symbol
  -> governance object: kb.answer.behavior
      -> implementation section: backend
          -> product sections: chat/context/return
              -> framework rules: knowledge_base.L2.M0 R1/R2/R3/R4
```

这就是当前真正的 code -> framework 反查逻辑。

## 9. 现在的 GUI / 校验 / 严格映射到底依赖什么

### 9.1 Canonical 基础

当前 GUI、validator、change propagation、strict checking 的基础，首先都是：

- `canonical_graph.json`

它提供：

- 五层结构
- 逐层 derived_from
- 每层 export
- code/evidence 的稳定对外面

### 9.2 治理派生视图

在需要“对象级别追踪”和“框架直跳 code”时，系统会继续消费派生视图：

- `governance_manifest.json`
- `governance_tree.json`

但这两者当前都明确声明自己来自 canonical。

所以当前基础关系应理解为：

```text
canonical_graph.json
  -> governance_manifest.json
  -> governance_tree.json
  -> strict_zone_report.json
  -> object_coverage_report.json
```

而不是这些文件彼此平行。

## 10. 当前还保留哪些生成物，它们为什么没有被删

当前仍保留这些派生生成物：

- `framework_ir.json`
- `product_spec.json`
- `implementation_bundle.py`
- `generation_manifest.json`
- `governance_manifest.json`
- `governance_tree.json`
- `strict_zone_report.json`
- `object_coverage_report.json`

它们没有被删的原因不是“仍然是平行真相源”，而是各自承担派生职责：

- `framework_ir.json`
  - 方便只看 Framework 编译结果
- `product_spec.json`
  - 方便只看 Product 层展开结果
- `implementation_bundle.py`
  - 方便代码侧导入和调试查看 Implementation 派生内容
- `generation_manifest.json`
  - 方便看配置 effect 与生成纪律
- `governance_manifest.json / governance_tree.json`
  - 方便 GUI、治理、直跳与严格反查
- `strict_zone_report.json / object_coverage_report.json`
  - 方便做严格区域与对象覆盖审计

所以当前删除策略不是“只剩 canonical 一个文件”，而是：

- **只剩 canonical 一个机器真相源**
- **其它文件只允许作为 canonical 的派生视图存在**

## 12. 现在到底能不能直接从 framework 跳到 code

结论直接说：

- **可以**
- **但不是主链**

主链必须仍然是：

```text
Framework -> Product -> Implementation -> Code -> Evidence
```

直接 `Framework -> Code` 只应理解为：

- 由 governance 派生视图提供的结构追踪捷径
- 用于 GUI、审查和 strict mapping
- 不用于破坏层纯度

所以当前系统实际上同时支持两种阅读方式：

### 方式 A：标准逐层阅读

适合真正理解系统：

```text
Framework
  -> Product
  -> Implementation
  -> Code
  -> Evidence
```

### 方式 B：治理直跳阅读

适合快速定位：

```text
Framework rule / section
  -> governance object
  -> code role / code symbol
```

当前推荐做法是：

- 讲架构时只讲 A
- 做排障、验证、GUI 跳转时可以用 B

## 13. 如果你想最快顺着搞清楚一一对应，怎么读

最省力的真实阅读路径如下：

1. 打开 [canonical_graph.json](../projects/knowledge_base_basic/generated/canonical_graph.json)
2. 看 `layers.product.derived_from`
   - 先确定用了哪几份 framework markdown
3. 看 `layers.implementation.derived_from`
   - 确认 product 是怎样进入 implementation 的
4. 看 `layers.code.derived_from`
   - 确认 implementation 是怎样进入 code 的
5. 看 `layers.code.export`
   - 确认 code 当前到底导出了哪些稳定面
6. 打开对应 runtime 文件
   - `app.py`
   - `backend.py`
   - `frontend.py`
7. 如果要追某个高风险对象，再打开：
   - `governance_manifest.json`
   - `governance_tree.json`
8. 再看 [scripts/validate_strict_mapping.py](../scripts/validate_strict_mapping.py)
   - 确认机器是怎么验证这条链的

这就是当前仓库里最真实、最省力、最不绕的理解路径。
