# 用户管理框架重写设计稿

## 1. 目标

把 `framework/user-new-framework` 收敛成一套更小、更直白、能从 `L0` 往上重新搭起来的用户管理框架。范围只保留：

- 登录
- 注册
- 访问令牌生成
- 访问令牌解析

不覆盖权限、锁定、风控、刷新令牌或第三方身份接入。

## 2. 分层方案

### L0 原子单元

- `L0-M0 用户存取单元`
  - `B1` 用户记录
  - `B2` 登录名索引
- `L0-M1 密码摘要单元`
  - `B1` 密码规则表
  - `B2` 摘要条目
- `L0-M2 访问令牌单元`
  - `B1` 令牌文本
  - `B2` 令牌载荷
  - `B3` 令牌签名

### L1 组合单元

- `L1-M0 登录注册单元`
  - 由 `用户存取单元 + 密码摘要单元` 组合而成
- `L1-M1 令牌单元`
  - 由 `用户存取单元 + 访问令牌单元` 组合而成

### L2 模块

- `L2-M0 用户管理模块`
  - 由 `登录注册单元 + 令牌单元` 组合而成

## 3. 命名与写法约束

- `L0` 优先写最小结构件，不写大而全功能块。
- `B*` 必须让人一眼看出是什么件，例如“记录、索引、规则表、条目、载荷、签名”。
- `B*` 说明必须回答三件事：手里有什么、根据什么工作、稳定产出什么。
- `L1` 和 `L2` 不再发明新的底层结构，只负责把下层组合成注册、登录、发令牌、解令牌。
- 边界只保留真正参与组合和验证的长度、唯一性、算法、时长、签名方式等约束。

## 4. 预期结果

- 读者可以先看 `L0`，再顺着 `L1`、`L2` 在脑海里重新搭出完整结构。
- 每个 `B*` 都应当能单独解释清楚，且移除后会直接破坏上层至少一个能力。
