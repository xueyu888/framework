# 账号凭证存取模块：IdentifierCredentialAccessModule

@framework

组件目标：对账号与凭证记录执行存取操作；在需要时可组合既定格式校验模块对账号输入与凭证输入执行格式校验；在写入过程中保证账号在有效记录范围内满足唯一性约束，并输出存取通过或失败结论。

## 1. 能力声明（Capability Statement）

* `C1` 账号凭证输入承接能力：承接用于存取的账号输入与凭证输入。
* `C2` 账号格式校验编排能力：在启用时，组织账号输入进入已选字符串格式校验模块，形成可判定的账号格式校验结论。
* `C3` 凭证格式校验编排能力：在启用时，组织凭证输入进入已选字符串格式校验模块，形成可判定的凭证格式校验结论。
* `C4` 账号凭证写入能力：依据既定存取规则写入账号凭证记录。
* `C5` 账号凭证读取能力：依据账号输入读取对应的账号凭证记录。
* `C6` 账号唯一性约束承接能力：在写入过程中保证同一账号在有效记录范围内至多对应一条记录。
* `C7` 存取结论输出能力：输出存取通过或失败结论。

## 2. 非职责声明（Non-Capability Statement）

* `N1` 不负责账号格式规则本身的底层定义；当前模块只引用既定字符串格式校验模块。
* `N2` 不负责凭证格式规则本身的底层定义；当前模块只引用既定字符串格式校验模块。
* `N3` 不负责字符串表示变换。
* `N4` 不负责凭证匹配与身份结论判定。
* `N5` 不负责唯一性约束的底层实现细节。
* `N6` 不负责账户资料维护、账户状态管理和会话控制。

## 3. 边界定义（Boundary / Parameter 参数）

* `D1` 账号输入类型：账号输入必须是字符串。来源：`C1`
* `D2` 凭证输入类型：凭证输入必须是字符串。来源：`C1`
* `D3` 记录结构范围：存取对象必须是由账号字符串与凭证字符串构成的账号凭证记录。来源：`C1 + C4 + C5`
* `D4` 账号唯一性约束：同一账号在有效记录范围内至多对应一条账号凭证记录。来源：`C6`
* `D5` 账号格式校验实例范围：若启用账号格式校验，则对应字符串格式校验模块实例必须预先确定，并能够输出可判定的格式校验结论。来源：`C2`
* `D6` 凭证格式校验实例范围：若启用凭证格式校验，则对应字符串格式校验模块实例必须预先确定，并能够输出可判定的格式校验结论。来源：`C3`
* `D7` 编排顺序约束：当启用格式校验时，写入操作只允许在已启用的格式校验结论均为通过后进入。来源：`C2 + C3 + C4`
* `D8` 写入可判定性：写入操作必须能够形成可判定的写入结果。来源：`C4`
* `D9` 读取可判定性：读取操作必须能够形成可判定的读取结果。来源：`C5`
* `D10` 结论范围：最终输出只允许为通过、失败。来源：`C7`

## 4. 最小可行基（Minimum Viable Bases）

* `B1` 账号凭证输入结构：由账号字符串与凭证字符串构成。来源：`C1 + D1 + D2 + D3`
* `B2` 账号凭证记录结构：由账号到凭证的键值对应关系构成。来源：`C4 + C5 + D3`
* `B3` 账号唯一性约束结构：由账号唯一性判定范围与唯一性成立条件构成。来源：`C6 + D4`
* `B4` 账号格式校验模块：L0-M0-字符串格式校验模块。来源：`C2 + D5`
* `B5` 凭证格式校验模块：L0-M0-字符串格式校验模块。来源：`C3 + D6`

## 5. 基组合原则（Base Combination Principles）

* `R1` 账号格式校验编排组合

  * `R1.1` 参与基：`B1 + B4`
  * `R1.2` 组合方式：当启用账号格式校验时，账号输入必须进入已选账号格式校验模块；该模块必须返回可判定的账号格式校验结论。
  * `R1.3` 输出能力：`C1 + C2`
  * `R1.4` 边界绑定：`D1/D5`

* `R2` 凭证格式校验编排组合

  * `R2.1` 参与基：`B1 + B5`
  * `R2.2` 组合方式：当启用凭证格式校验时，凭证输入必须进入已选凭证格式校验模块；该模块必须返回可判定的凭证格式校验结论。
  * `R2.3` 输出能力：`C1 + C3`
  * `R2.4` 边界绑定：`D2/D6`

* `R3` 账号凭证写入组合（带格式校验）

  * `R3.1` 参与基：`B1 + B2 + B3 + B4 + B5`
  * `R3.2` 组合方式：当启用格式校验时，只有在已启用的账号格式校验结论与凭证格式校验结论均为通过后，账号字符串与凭证字符串才允许以账号凭证记录结构形成待写入记录；仅当该账号满足唯一性约束时，写入才允许成立；若账号已占用、格式校验失败或写入无法形成有效结果，则写入失败。
  * `R3.3` 输出能力：`C1 + C2 + C3 + C4 + C6 + C7`
  * `R3.4` 边界绑定：`D1/D2/D3/D4/D5/D6/D7/D8/D10`

* `R4` 账号凭证写入组合（不带格式校验）

  * `R4.1` 参与基：`B1 + B2 + B3`
  * `R4.2` 组合方式：当未启用格式校验时，账号字符串与凭证字符串可直接以账号凭证记录结构形成待写入记录；仅当该账号满足唯一性约束时，写入才允许成立；若账号已占用或写入无法形成有效结果，则写入失败。
  * `R4.3` 输出能力：`C1 + C4 + C6 + C7`
  * `R4.4` 边界绑定：`D1/D2/D3/D4/D8/D10`


## 6. 验证（Verification）

* `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个边界项推导得到。
* `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力和边界绑定。
* `V3` 闭环性：`R1~R6` 必须形成“输入承接 ->（可选）格式校验 -> 唯一性判定 -> 写入 / 读取 -> 存取结论”的完整存取链。
* `V4` 边界符合性：所有有效组合都必须满足 `D1~D10`。
* `V5` 最小必要性：移除任一 `B*` 后，至少一项能力无法成立。
