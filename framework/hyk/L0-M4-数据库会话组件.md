# 数据库会话管理组件:DatabaseSessionManager

@framework

组件目标：围绕同一数据库连接对象，统一管理数据库会话的取得、定位、复用和关闭。

## 1. 能力声明（Capability Statement）

- `C1` 会话取得能力：基于连接信息取得可用数据库会话。
- `C2` 会话定位能力：通过稳定句柄定位已取得的数据库会话。
- `C3` 会话复用能力：在同一会话来源上继续借出会话，或按独立方式生成新的借出链。
- `C4` 会话关闭能力：按关闭责任关闭单个已借出会话，或清理全部活跃借出关系。
- `C5` 非能力项：不负责事务提交与回滚，不负责 ORM 映射，不负责业务对象装配。

## 2. 边界定义（Boundary / 参数）

- `URL` 连接地址（database_url）：必须是可消费且能唯一定位数据库的连接地址。来源：`C1`。
- `PF` 配置标识（profile_name）：必须是非空配置名。来源：`C1`。
- `DK` 数据库类型（database_kind）：必须是非空数据库类型标识。来源：`C1`。
- `MD` 会话模式（mode）：只允许 `sync | async`。来源：`C1 + C3`。
- `PO` 复用方式（policy）：只允许 `shared | one-shot`。来源：`C3 + C4`。
- `CO` 关闭责任（owner）：只允许 `component | caller`。来源：`C3 + C4`。
- `HP` 句柄标识（handle）：取得、定位和关闭都必须以句柄为定位依据。来源：`C1 + C2 + C4`。
- `LS` 生命周期槽位（lifecycle_state）：只允许 `active | closed`。来源：`C2 + C4`。
- `AI` 管理入口约束（interface）：连接进入、会话取得、句柄定位和关闭都必须经由本组件稳定公开入口进入。来源：`C1 + C2 + C3 + C4`。
- `IN` 输入限制（constraint）：组件只接收连接信息和会话使用约束，不接收事务策略、ORM 映射或业务对象。来源：`C1 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 连接信息单元（自有基）：数据库连接信息的承载结构，由连接地址、配置标识和数据库类型构成。来源：`C1 + URL + PF + DK + IN`。
  - 注：这是“连接对象描述壳”，用来稳定描述一个数据库对象是谁。
  - 注：它内部只有三类信息：连接地址、配置标识、数据库类型；这三者合在一起，才构成一份完整的连接对象描述。
  - 注：在整条结构链里，它处在最前面，后面的连接承接、会话来源和持有关系都默认围绕这份连接对象描述展开。

- `B2` 连接适配单元（自有基）：数据库连接承接结构，由连接接入口、驱动适配层和连接载体位构成。来源：`C1 + URL + DK + MD`。
  - 注：这是“连接承接壳”，用来把 `B1` 的连接对象接到具体数据库服务上，并形成可继续被上层承接的连接载体。
  - 注：它内部有三层：连接接入口负责吃进连接对象，驱动适配层负责对接具体数据库服务，连接载体位负责持有承接后的连接结果。
  - 注：在结构链里，它接在 `B1` 后面；没有这一层，连接对象只是一份描述，不会自然变成后面可继续被会话结构承接的连接载体。

- `B3` 会话来源单元（自有基）：数据库会话来源的承载结构，由上下文承接位、来源实例位、上下文可用位、复用锚点和收敛挂点构成。来源：`C1 + C3 + C4 + MD + PO + CO`。
  - 注：这是“会话组织壳”，用来承接 `B2` 的连接载体，并把它组织成带生命周期标识的逻辑会话上下文来源。
  - 注：它内部有五个稳定位置：上下文承接位表示当前承接的是哪一类连接载体，来源实例位表示当前供给源是谁，上下文可用位表示当前哪些逻辑会话上下文处于可引用状态，复用锚点表示多个持有关系能否落回同一来源，收敛挂点表示持有关系结束后向哪里收敛。
  - 注：这里的会话不是把连接换个名字；连接载体仍然是驱动或服务层对象，会话上下文是在该载体之上被组织出来、带有生命周期标识的逻辑使用面。
  - 注：复用和独立展开不是规则临时造出来的，而是这个来源结构本身就允许的两种展开方式：多个持有关系落回同一来源实例时就是复用；每次形成新的来源实例时就是独立展开。

- `B4` 持有映射单元（自有基）：数据库会话持有关系的映射结构，由会话句柄、已借出会话、来源引用和生命周期槽位构成。来源：`C1 + C2 + C4 + HP + LS`。
  - 注：这是“当前持有关系表”，用来记录现在有哪些会话已经被借出、分别由哪个句柄持有、又来自哪个来源。
  - 注：它内部最核心的稳定关系是：`handle -> borrowed_session -> source_ref -> lifecycle_slot`。
  - 注：定位和关闭都直接建立在这张关系表上；句柄之所以有意义，不是因为它是一个 id，而是因为它在这张表里稳定地指向某个借出会话。

说明：`B1~B4` 先天带有区分连接对象、承接数据库服务、提供会话来源和记录持有关系的结构能力；下面的 `R1~R7` 只限定这些结构在本组件中哪些用法被承认为合法，不额外发明新能力。

## 4. 基组合原则（Base Combination Principles）

- `R1` 连接承接约束
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：当组件用于取得会话时，连接信息必须先由 `B1` 承载，再进入 `B2` 形成连接载体位；没有合法连接信息、连接信息与适配层不匹配或绕过适配层的用法，都不承认是合法连接承接。
  - `R1.3` 输出结构：`CONNECTION_CARRIER`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`URL/PF/DK/MD/AI/IN`。

- `R2` 会话取得约束
  - `R2.1` 参与基：`B2 + B3 + B4`。
  - `R2.2` 组合方式：当组件用于取得会话时，只允许从 `B2` 已形成的连接载体位进入 `B3` 的上下文承接位，由 `B3` 把该连接载体组织成带生命周期标识的逻辑会话上下文，再由 `B4` 为本次借出写入唯一句柄和开启态；跳过会话来源结构或跳过持有映射的取得都不承认是合法取得。
  - `R2.3` 输出结构：`MANAGED_SESSION_CONTEXT + MANAGED_HANDLE`。
  - `R2.4` 输出能力：`C1`。
  - `R2.5` 边界绑定：`MD/PO/CO/HP/LS/AI`。

- `R3` 会话定位约束
  - `R3.1` 参与基：`B4`。
  - `R3.2` 组合方式：当组件用于定位会话时，只允许使用句柄进入 `B4` 查询持有映射；没有句柄、句柄不存在或句柄已落入关闭态，都不承认是合法定位。
  - `R3.3` 输出结构：`MANAGED_SESSION_REF`。
  - `R3.4` 输出能力：`C2`。
  - `R3.5` 边界绑定：`HP/LS/AI`。

- `R4` 会话复用约束
  - `R4.1` 参与基：`B3 + B4`。
  - `R4.2` 组合方式：当组件用于继续使用已有会话或生成独立持有链时，必须先服从 `B3` 的复用锚点和收敛挂点，再通过 `B4` 建立或延续持有映射；复用型用法允许多个持有映射指向同一来源实例，独立型用法必须为新持有链生成新的来源实例。
  - `R4.3` 输出结构：`REUSED_SESSION_REF + ISOLATED_SESSION_REF`。
  - `R4.4` 输出能力：`C3`。
  - `R4.5` 边界绑定：`PO/CO/HP/LS/AI`。

- `R5` 会话关闭约束
  - `R5.1` 参与基：`B3 + B4`。
  - `R5.2` 组合方式：当组件用于关闭会话时，必须先通过句柄从 `B4` 找到持有映射，再按 `B3` 的收敛挂点收敛对应持有关系；复用型用法只能按共享边界收敛持有关系，独立型用法必须连同对应来源实例一起收敛。
  - `R5.3` 输出结构：`CLOSE_RESULT + CLEANUP_RESULT`。
  - `R5.4` 输出能力：`C4`。
  - `R5.5` 边界绑定：`PO/CO/HP/LS/AI`。

- `R6` 完整数据库会话管理约束
  - `R6.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R6.2` 组合方式：当组件以完整形态工作时，只承认这样一条路径为合法路径：先确定连接信息，再完成连接承接，再把连接载体组织成逻辑会话上下文来源，最后写入持有映射，并在该映射上完成取得、定位、复用和关闭；任何绕开这条路径的用法都不属于本组件承认的数据库会话管理。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R6.4` 边界绑定：`URL/PF/DK/MD/PO/CO/HP/LS/AI/IN`。

- `R7` 禁止组合
  - `R7.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R7.2` 组合方式：没有连接信息直接取得会话、绕过连接适配层直接伪造会话来源、没有来源结构直接建立持有映射、没有句柄直接定位或关闭会话、把事务策略或 ORM 语义混入本组件、让复用方式与关闭责任相互冲突的组合均无效。
  - `R7.3` 输出能力：`C5`。
  - `R7.4` 边界绑定：`URL/PF/DK/MD/PO/CO/HP/LS/AI/IN`。

## 5. 验证（Verification）

- `V1` 推导一致性：`B1/B2/B3/B4` 都必须能由至少一个 `C*` 和一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R6` 输出能力并集必须覆盖 `C1~C4`。
- `V4` 边界符合性：所有合法用法都必须满足连接信息、会话模式、复用方式、关闭责任、句柄标识、生命周期槽位和管理入口约束。
- `V5` 最小必要性：移除 `B1` 后无法确定连接对象；移除 `B2` 后无法把连接对象承接到数据库服务；移除 `B3` 后无法形成可借出会话来源；移除 `B4` 后无法定位当前持有关系。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
- `V7` 结构落点一致性：连接信息、连接适配、会话来源和持有映射必须分别在配置模型、运行时状态和公开接口中有稳定落点。
- `V8` 能力落点一致性：`L3` 文档能力声明、`package.py capabilities`、`api.py/component.py` 中的公开出口必须一一对应。

## 6. 能力暴露（Capability Exposure）

- `E1` 语义落点：本文件 `1. 能力声明`。
- `E2` 结构落点：本文件 `3. 最小可行基` 与 `4. 基组合原则`。
- `E3` 机器落点：组件包中的 `package.py::capabilities`。
- `E4` 接口落点：组件包中的 `api.py`。
- `E5` 运行时落点：组件包中的 `component.py`。
- `E6` 一致性要求：`E1~E5` 必须一一对应；新增能力或新增结构约束时必须同步修改。
