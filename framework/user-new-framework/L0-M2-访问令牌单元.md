# 访问令牌单元:AccessTokenUnit

@framework

## 0. 目标 (Goal)
- 访问令牌单元定义“jwt 处理器、令牌载荷和签名算法这一组最小结构”，用于承载访问令牌的生成结果和解析结果。
- 该单元只回答三件事：由谁处理 jwt、载荷里放什么、按什么算法完成签名和验签；不承担登录注册职责。

## 1. 能力声明（Capability Statement）

- `C1` 令牌生成能力：支持按用户编号和令牌载荷生成可携带的访问令牌文本。
- `C2` 令牌解析能力：支持按令牌文本解析用户编号和访问到期时间等载荷。
- `C3` 令牌验证能力：支持按声明签名算法判断令牌文本是否被篡改。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责登录注册或密码校验。
- `N2` 非职责声明：不负责权限判断或任何风控状态。

## 2. 边界定义（Boundary / Parameter 参数）

- `access_token_payload_fields` 令牌载荷字段集合：对应 `B2`。必须至少包含用户编号和访问到期时间。
- `access_token_ttl` 访问令牌时长：对应 `B2`。约束 token 的有效时长。
- `access_token_sign_kind` 访问令牌签名方式：对应 `B3`。必须给出有限枚举集合，如 `HS256`、`RS256` 等。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` jwt 处理器：负责把令牌载荷编码成token，并把token解析回可读载荷。
- `B2` token 载荷(payload)：token 中所存储的信息，例如用户 id、过期时间。
- `B3` 签名算法：声明 jwt 使用的签名方式，保证 token 不被篡改。

## 4. 基排列组合（Base Arrangement / Combination）

- `R1` `令牌生成组合`：由 `{B1, B2, B3}` 形成令牌生成路径，导出 `C1`。`B2` 先整理出本次 token 要携带的字段，`B3` 再给出签名方式，最后由 `B1` 生成完整 token 文本。
- `R2` `载荷解析组合`：由 `{B1, B2}` 形成载荷解析路径，导出 `C2`。
- `R3` `令牌验证组合`：由 `{B1, B2, B3}` 形成令牌验证路径，导出 `C3`。
- `R4` `非法组合`：载荷字段缺失、访问令牌时长超界或签名方式不匹配时，不导出 `C1 ~ C3`。

## 5. 验证（Verification）

- `V1` 生成路径验证：`R1` 必须同时落到 `B1 + B2 + B3`，并只导出 `C1`。
- `V2` 解析路径验证：`R2` 必须导出 `C2`，且解析结果能回到声明的载荷字段。
- `V3` 验证路径验证：`R3` 必须导出 `C3`，且验证过程使用声明的签名方式。
- `V4` 参数符合性：有效组合受 `access_token_payload_fields + access_token_ttl + access_token_sign_kind` 约束。
