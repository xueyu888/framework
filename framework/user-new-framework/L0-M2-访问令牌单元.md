# 访问令牌单元:AccessTokenUnit

@framework

## 0. 目标 (Goal)
- 访问令牌单元定义“令牌文本、令牌载荷和令牌签名这一组最小结构”，用于承载访问令牌的生成结果和解析结果。
- 该单元只回答三件事：令牌文本长什么样、载荷里放什么、如何确认令牌未被篡改；不承担登录注册职责。

## 1. 能力声明（Capability Statement）

- `C1` 令牌生成能力：支持按用户编号生成可携带的访问令牌文本。
- `C2` 令牌解析能力：支持按令牌文本解析用户编号和访问到期时间。
- `C3` 令牌验签能力：支持按声明签名方式判断令牌文本和载荷是否匹配。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责登录注册或密码校验。
- `N2` 非职责声明：不负责权限判断或任何风控状态。

## 2. 边界定义（Boundary / Parameter 参数）

- `access_token_text_uniqueness` 令牌文本唯一性：对应 `B1`。任一令牌文本只能定位一个访问令牌单元。
- `access_token_ttl` 访问令牌时长：对应 `B2`。必须给出正时长；任一访问到期时间都必须落在签发时刻加该时长内。
- `access_token_sign_kind` 访问令牌签名方式：对应 `B3`。必须给出有限枚举集合；不在集合内的签名方式视为非法。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` jwt 处理器：承载可被外部携带的访问令牌字符串，能作为一次令牌解析的输入。
- `B2` 令牌载荷(payload)：写入用户编号和访问到期时间，能在解析后输出这两个事实。
- `B3` 签名算法：记录令牌文本和令牌载荷对应的签名结果，能判断当前令牌是否被篡改。

## 4. 基组合原则（Base Combination Principles）

- `R1` 合法访问令牌组合：
  - `R1.1` 参与基：`B1 + B2 + B3`
  - `R1.2` 组合方式：`B1` 必须满足 `access_token_text_uniqueness`，`B2` 必须满足 `access_token_ttl`，`B3` 必须满足 `access_token_sign_kind`，从而形成可生成、可解析、可验签的访问令牌单元。
  - `R1.3` 输出能力：`C1 + C2 + C3`
  - `R1.4` 参数绑定：`access_token_text_uniqueness + access_token_ttl + access_token_sign_kind`

- `R2` 非法组合：
  - `R2.1` 若令牌文本无法唯一定位一个访问令牌单元，则组合非法。
  - `R2.2` 若载荷中的访问到期时间不满足 `access_token_ttl`，或签名不满足 `access_token_sign_kind`，则组合非法。
  - `R2.3` 非法组合不导出任何 `C*`。

## 5. 验证（Verification）

- `V1` 规则一致性：每个 `R*` 必须明确给出参与基、组合方式、输出能力或失效结论，以及绑定的参数边界。
- `V2` 结构闭环性：合法组合规则 `R1` 的输出能力必须覆盖 `C1 ~ C3`，且不得包含未声明能力。
- `V3` 参数符合性：任意有效访问令牌单元必须同时满足 `access_token_text_uniqueness + access_token_ttl + access_token_sign_kind`。
- `V4` 可度量性：每个边界项都必须可落到正时长、有限枚举或唯一性断言；禁止不可判定表述。
- `V5` 可测试性：每个边界项都必须对应至少一个可执行通过/失败断言，例如令牌文本唯一性断言、到期时间断言、签名方式断言。
- `V6` 最小必要性：移除任一 `B*` 后，`C1 ~ C3` 中至少一项无法成立。
- `V7` 非职责隔离性：访问令牌单元不得直接承担登录注册或权限判断职责。
- `V8` 结论表达：逐条输出 `R1 -> C* + 参数绑定` 或 `R2 -> 非法组合` 的通过或失败结论。
