# 登录注册单元:LoginRegistrationUnit

@framework

## 0. 目标 (Goal)
- 登录注册单元负责把用户存取单元、密码摘要单元和访问令牌单元接成一条登录入口，完成注册写入、登录校验和登录后访问令牌发放。
- 该单元只回答三件事：用户怎么注册、密码怎么核对、登录成功后怎么拿到访问令牌；不展开底层存储、摘要或 jwt 细节。

## 1. 能力声明（Capability Statement）

- `C1` 注册写入能力：支持把一条新用户信息与对应密码摘要写入系统。
- `C2` 登录校验能力：支持根据用户标识读取用户信息并完成密码核对。
- `C3` 登录发令牌能力：支持在登录校验通过后导出访问令牌。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责单独的访问令牌解析或令牌校验入口。
- `N2` 非职责声明：不负责失败锁定、角色权限或访问控制。

## 2. 边界定义（Boundary / Parameter 参数）

- `user_key_uniqueness` 用户标识唯一性：对应 `B1`。约束注册后的用户标识分配。
- `password_rule_pattern` 密码规则表达式：对应 `B2`。约束注册和登录时的密码规则校验。
- `password_hash_kind` 摘要算法：对应 `B2`。约束注册写入与登录核对使用的摘要算法。
- `access_token_ttl` 访问令牌时长：对应 `B3`。约束登录成功后导出的访问令牌时长。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 用户存取单元：`L0.M0[R1/R2]`。提供用户信息写入、定位与读取。
- `B2` 密码摘要单元：`L0.M1[R1/R2]`。提供密码规则校验与摘要生成。
- `B3` 访问令牌单元：`L0.M2[R1]`。提供访问令牌生成。

## 4. 基组合原则（Base Combination Principles）

- `R1` `注册组合`：由 `{B1, B2}` 形成注册路径，导出 `C1`。输入密码先在 `B2` 中完成规则校验和摘要生成，再由 `B1` 为这条用户信息分配用户标识并完成写入。
- `R2` `登录校验组合`：由 `{B1, B2}` 形成登录校验路径，导出 `C2`。
- `R3` `登录发令牌组合`：由 `{B1, B2, B3}` 形成登录发令牌路径，导出 `C3`。先沿用 `R2` 的校验结果确认当前用户，再把用户标识交给 `B3` 生成访问令牌。
- `R4` `非法组合`：注册写入失败、密码规则不通过、用户不存在、密码核对失败或令牌时长不满足边界时，不导出 `C1 ~ C3`。

## 5. 验证（Verification）

- `V1` 注册路径验证：`R1` 必须同时落到 `B1 + B2`，并只导出 `C1`。
- `V2` 登录校验路径验证：`R2` 必须导出 `C2`，且校验结果对应同一用户标识。
- `V3` 发令牌路径验证：`R3` 必须建立在登录校验通过的前提下，并只导出 `C3`。
- `V4` 参数符合性：有效组合受 `user_key_uniqueness + password_rule_pattern + password_hash_kind + access_token_ttl` 约束。
