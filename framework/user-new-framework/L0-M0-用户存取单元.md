# 用户存取单元:UserStorageUnit

@framework

## 0. 目标 (Goal)
- 用户存取单元定义“用户记录和登录名索引这一对最小结构”，用于承载注册和登录时的用户写入、定位与读取。
- 该单元只回答三件事：用户信息写到哪里、如何用登录名找到用户、如何读出用户信息；不承担密码摘要或令牌处理职责。

## 1. 能力声明（Capability Statement）

- `C1` 用户写入能力：支持写入新用户记录，并建立登录名到用户编号的对应。
- `C2` 用户定位能力：支持通过登录名唯一定位用户记录。
- `C3` 用户读取能力：支持读取用户记录中的用户编号、登录名和当前状态。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责密码规则校验、摘要写入或密码比对。
- `N2` 非职责声明：不负责访问令牌生成、令牌解析或权限判断。

## 2. 边界定义（Boundary / Parameter 参数）

- `user_id_uniqueness` 用户编号唯一性：对应 `B1`。任一用户编号只能绑定一条用户记录。
- `login_name_length` 登录名长度：对应 `B1 + B2`。必须给出正整数上限与计量单位（字符）；有效登录名长度必须落在闭区间 `[1, 上限]`。
- `login_name_uniqueness` 登录名唯一性：对应 `B2`。任一登录名只能命中一条用户记录。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 用户记录：存放用户编号、登录名和当前状态，能根据用户编号写入和读取一条完整用户信息。
- `B2` 登录名索引：记录登录名到用户编号的唯一对应，能根据登录名定位唯一用户记录。

## 4. 基组合原则（Base Combination Principles）

- `R1` 合法用户存取组合：
  - `R1.1` 参与基：`B1 + B2`
  - `R1.2` 组合方式：`B1` 必须满足 `user_id_uniqueness`，`B2` 必须满足 `login_name_length + login_name_uniqueness`，并且每个索引结果都必须能回到一条真实用户记录，从而形成可写入、可定位、可读取的用户存取单元。
  - `R1.3` 输出能力：`C1 + C2 + C3`
  - `R1.4` 参数绑定：`user_id_uniqueness + login_name_length + login_name_uniqueness`

- `R2` 非法组合：
  - `R2.1` 若用户编号重复，或登录名长度不满足边界约束，则组合非法。
  - `R2.2` 若任一登录名无法唯一命中一条用户记录，则组合非法。
  - `R2.3` 非法组合不导出任何 `C*`。

## 5. 验证（Verification）

- `V1` 规则一致性：每个 `R*` 必须明确给出参与基、组合方式、输出能力或失效结论，以及绑定的参数边界。
- `V2` 结构闭环性：合法组合规则 `R1` 的输出能力必须覆盖 `C1 ~ C3`，且不得包含未声明能力。
- `V3` 参数符合性：任意有效用户存取单元必须同时满足 `user_id_uniqueness + login_name_length + login_name_uniqueness`。
- `V4` 可度量性：每个边界项都必须可落到长度或唯一性断言；禁止不可判定表述。
- `V5` 可测试性：每个边界项都必须对应至少一个可执行通过/失败断言，例如用户编号唯一性断言、登录名长度断言、登录名唯一性断言。
- `V6` 最小必要性：移除任一 `B*` 后，`C1 ~ C3` 中至少一项无法成立。
- `V7` 非职责隔离性：用户存取单元不得直接承担密码摘要、令牌生成或令牌解析职责。
- `V8` 结论表达：逐条输出 `R1 -> C* + 参数绑定` 或 `R2 -> 非法组合` 的通过或失败结论。
