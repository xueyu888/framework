# 用户存取单元:UserStorageUnit

@framework

## 0. 目标 (Goal)
- 用户存取单元定义“k-v 键值对、全局唯一键和持久化接口这一组最小结构”，用于承载用户信息的存储、定位与读取。
- 该单元只回答三件事：用户信息存到哪里、用什么唯一键标识、通过哪个持久化入口完成写入和读取；不承担密码摘要或令牌处理职责。

## 1. 能力声明（Capability Statement）

- `C1` 用户建档能力：支持为一条用户信息分配全局唯一键，并通过持久化接口写入数据库。
- `C2` 用户定位能力：支持根据全局唯一键定位对应的用户信息。
- `C3` 用户读取能力：支持通过持久化接口读取全局唯一键对应的用户信息容器。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责密码规则校验、摘要写入或密码比对。
- `N2` 非职责声明：不负责访问令牌生成、令牌解析或权限判断。

## 2. 边界定义（Boundary / Parameter 参数）

- `user_key_uniqueness` 全局唯一键唯一性：对应 `B2`。任一全局唯一键只能绑定一条用户身份信息。
- `storage_write_mode` 持久化写入方式：对应 `B3`。必须给出固定写入方式，例如新增、覆盖或更新。
- `user_info_field_set` 用户信息字段集合：对应 `B1`。必须给出用户信息容器允许写入的有限字段集合。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` k-v 键值对：用户信息存储容器。
- `B2` 全局唯一键：用户身份信息唯一标识。
- `B3` 持久化接口：负责将 k-v 键值对写入和读取数据库中的固定位置。

## 4. 基排列组合（Base Arrangement / Combination）

- `R1` `用户建档组合`：由 `{B1, B2, B3}` 形成用户建档路径，导出 `C1`。先由 `B2` 给出这条用户信息的全局唯一键，再由 `B3` 按当前写入方式把键和值写入 `B1`；写入完成后，才算形成一条可持久化的用户记录。
- `R2` `用户读取组合`：由 `{B1, B2, B3}` 形成定位并读取同一条用户信息的路径，导出 `C2` 与 `C3`。
- `R3` `非法组合`：全局唯一键重复、写入方式不匹配或字段超出集合时，不导出 `C1 ~ C3`。

## 5. 验证（Verification）

- `V1` 建档路径验证：`R1` 必须同时落到 `B1 + B2 + B3`，且只导出 `C1`。
- `V2` 读取路径验证：`R2` 必须导出 `C2` 与 `C3`，且读取结果对应同一全局唯一键。
- `V3` 参数符合性：有效组合受 `user_key_uniqueness + storage_write_mode + user_info_field_set` 约束。
