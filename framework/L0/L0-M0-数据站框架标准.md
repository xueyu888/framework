# 数据站框架标准:DataStationFramework

@framework

## 1. 能力声明（Capability Statement）

- `C1` 数据接入登记能力：提供器件手册上传接入、文件接收、文档记录创建与初始状态登记能力。
- `C2` 结构化处理能力：提供清洗、提取、精炼、审核与结果回写能力。
- `C3` 状态治理与追溯能力：提供状态流转、版本管理、责任归属与全流程留痕能力。
- `C4` 展示检索复用能力：提供结果可见可达、列表检索、详情展示与复用导出能力。
- `C5` 统一鉴权治理能力：提供统一角色权限判定与范围控制能力。
- `C6` 非能力项：不负责外部业务结论生成，不替代客户内部人工判断。

## 2. 边界定义（Boundary / 参数）

- `D` 领域范围（domain_scope）：仅覆盖器件手册对象及其处理结果，不纳入外部无关业务对象。来源：`C1 + C2 + C4`。
- `L` 生命周期范围（lifecycle_scope）：覆盖接入、处理、审核、展示、检索五阶段闭环。来源：`C1 + C2 + C3 + C4`。
- `P` 路径范围（path_scope）：仅定义内的 UI 路径与 API 路由可触发平台能力。来源：`C1 + C4 + C5`。
- `I` 输入契约（input_contract）：仅文件流与会触发状态变化或副作用的必填元信息属于有效输入。来源：`C1 + C2 + C5`。
- `O` 输出契约（output_contract）：输出必须表现为界面变化、状态变化或副作用变化中的至少一种。来源：`C2 + C3 + C4`。
- `S` 状态契约（state_contract）：仅允许定义内状态集合与合法流转路径生效。来源：`C2 + C3 + C5`。
- `A` 权限契约（authority_contract）：客户公司内部全员可见，上传、处理、审核能力仅对内部工程师开放。来源：`C4 + C5`。
- `Q` 质量契约（quality_contract）：质量规则与阈值达标后方可进入下一状态。来源：`C2 + C3`。
- `T` 追溯契约（traceability_contract）：关键操作必须记录主体、动作、对象、结果、原因与版本信息。来源：`C1 + C3 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 接入入口基：承载上传入口、文件接收、文档记录创建与初始状态登记的结构骨架。来源：`C1 + I + P + A + T`。
- `B2` 处理审核基：承载结构化处理、人工审核与结果回写的结构骨架。来源：`C2 + I + O + Q + T`。
- `B3` 状态追溯基：承载状态机、版本管理、责任归属与全流程留痕的结构骨架。来源：`C3 + S + O + T`。
- `B4` 展示检索基：承载结果可见、列表检索、详情展示、版本信息与复用导出的结构骨架。来源：`C4 + D + P + O + A`。
- `B5` 权限治理基：承载统一鉴权、角色策略与范围控制的结构骨架。来源：`C5 + A + P + T`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 接入可见组合
  - `R1.1` 参与基：`B1 + B3 + B4 + B5`。
  - `R1.2` 组合方式：先由权限治理基完成入口鉴权，再由接入入口基接收文件并创建文档记录，随后状态追溯基提交已接入状态，最后展示检索基暴露可见可达结果。
  - `R1.3` 输出能力：`C1 + C3 + C4 + C5`。
  - `R1.4` 边界绑定：`P/I/O/S/A/T`。
- `R2` 处理回写组合
  - `R2.1` 参与基：`B2 + B3 + B4 + B5`。
  - `R2.2` 组合方式：处理审核基仅消费已接入对象，完成清洗、审核与结果回写后，由状态追溯基推进状态并保留版本关系，再由展示检索基同步呈现处理结果。
  - `R2.3` 输出能力：`C2 + C3 + C4 + C5`。
  - `R2.4` 边界绑定：`D/I/O/S/A/Q/T`。
- `R3` 状态权限组合
  - `R3.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R3.2` 组合方式：平台内所有上传、处理、审核、展示与检索动作都必须先经权限治理基判定，再由状态追溯基校验合法状态流转与责任归属，非法流转或越权操作直接拦截。
  - `R3.3` 输出能力：`C3 + C5`。
  - `R3.4` 边界绑定：`P/S/A/Q/T`。
- `R4` 闭环复用组合
  - `R4.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R4.2` 组合方式：平台必须形成上传、处理、审核、展示、检索的完整闭环，并保证结果在可见、可检索的同时具备复核与复用条件。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R4.4` 边界绑定：`D/L/O/S/A/Q/T`。

## 5. 验证（Verification）

- `V1` 接入闭环验证：样本文档必须能够完成上传接入、文档记录创建、状态登记与初始可见。
- `V2` 处理一致性验证：处理与审核结果必须成功回写，并与展示检索结果保持一致。
- `V3` 状态合法性验证：所有状态流转必须满足定义内的合法路径，非法流转全部被拦截。
- `V4` 权限一致性验证：上传、处理、审核、展示与检索操作必须统一经过同一鉴权判定来源。
- `V5` 追溯完整性验证：关键操作必须具备主体、动作、对象、结果、原因与版本字段的完整记录。
- `V6` 闭环复用验证：平台必须走通上传、处理、审核、展示、检索闭环，并保证结果可复核、可检索、可复用。
