# 前端视觉语义原子模块:FrontendVisualSemanticAtoms

@framework

浏览器对页面元素渲染会渲染成一个矩形框，因此会引出一个盒模型概念，在 L0-M0-容纳与绑定原子模块.md 会定义一个外壳结构基，这个结构基就是自定义的盒模型结构。
盒模型：
+------------------------------------------------------+
|                      margin                          |
|  +------------------------------------------------+  |
|  |                    border                      |  |
|  |  +------------------------------------------+  |  |
|  |  |                padding                   |  |  |
|  |  |  +------------------------------------+  |  |  |
|  |  |  |            content                 |  |  |  |
|  |  |  |   (文本 / 图片 / 子元素等)        |  |  |  |
|  |  |  +------------------------------------+  |  |  |
|  |  |                                          |  |  |
|  |  +------------------------------------------+  |  |
|  |                                                |  |
|  +------------------------------------------------+  |
|                                                      |
+------------------------------------------------------+

## 1. 能力声明（Capability Statement）

- `C1` 样式选择能力：规定盒模型相关的样式如何被选择，形成清楚的外观基础结构。
- `C2` 布局组织能力：规定多个视觉块如何排列、对齐和分配空间，形成清楚的布局结构。
- `C3` 排版组织能力：规定字体、字号、字重、行高与文本层级如何组织，形成清楚的排版结构。
- `C4` 主题映射能力：规定同一套视觉语义如何映射到不同主题与状态，形成清楚的主题映射结构。

### 非职责声明（Non-Responsibility Statement）

- `N1` 不负责宿主挂载、页面流程与实例品牌文案。
- `N2` 不负责页面级视觉编排、品牌资产治理与运行对象管理。

## 2. 主边界定义（Primary Boundaries）

- `ELEMENT` 元素类型边界：样式选择器所服务的元素类型如何划分，以及不同元素类型如何稳定对应视觉规则。
- `BOX` 盒模型边界：宽高、内外边距、边框、圆角、阴影等盒模型相关样式如何选择。
- `LAYOUT` 布局边界：对齐方式、排列方向、间距规则与区域占比如何组织。
- `TYPO` 排版边界：字体、字号、字重、行高、字距与文本层级如何组织。
- `THEME` 主题映射边界：同一套样式选择器在不同主题、状态和局部覆盖下如何映射。


## 3. 最小可行基（Minimum Viable Bases）

- `B1` 样式选择器结构基：由元素类型、宽高、内外边距、边框、圆角、背景、阴影等盒模型样式选择组成的最小稳定结构（元素-【盒模型要素】-md,元素-【盒模型要素】-lg, 元素-【盒模型要素】-default...集合）。来源：`C1 + ELEMENT + BOX`。
- `B2` 布局结构基：由对齐方式、排列方向、间距规则与区域占比组成的最小稳定布局结构。来源：`C2 + LAYOUT`。
- `B3` 排版结构基：由字体、字号、字重、行高、字距与文本层级组成的最小稳定排版结构。来源：`C3 + TYPO`。
- `B4` 主题映射来源基：由主题集、状态映射和可被主题配置消费的稳定映射位组成的最小稳定结构。后续配置层收集主题映射时，应以这一层为唯一来源。来源：`C4 + THEME`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 样式选择器先成立
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先按元素类型划分稳定的视觉宿主，再为每类元素组织盒模型相关样式选择，使样式选择器先有明确服务对象和选择依据。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`ELEMENT + BOX`。

- `R2` 布局附着在样式选择器上
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在已经成立的样式选择结构上，进一步说明多个块如何排列、对齐和分配空间，使外观基础与空间组织形成清楚且稳定的关系。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`BOX + LAYOUT`。

- `R3` 排版附着在样式选择器与布局上
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：在已经成立的样式选择和布局结构上，进一步说明文本如何组织、文本层级如何区分、阅读节奏如何形成，使外观基础、空间组织和文本表达形成清楚且稳定的关系。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`BOX + LAYOUT + TYPO`。

- `R4` 主题映射来源附着在样式选择器、布局与排版上
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：在已经成立的样式选择、布局和排版结构上，进一步整理可被主题配置消费的稳定映射位，使样式选择、布局组织和排版组织都能成为后续主题映射的统一来源。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`BOX + LAYOUT + TYPO + THEME`。

- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：如果没有先建立盒模型样式选择，就直接声明布局、排版或主题映射；或者布局关系混乱；或者排版层级不可解释；或者主题映射无法解释；或者同一视觉语义在不同主题下失去稳定对应，这样的方案都无效。
  - `R5.3` 失效结论：`N1 + N2`。
  - `R5.4` 边界绑定：`BOX + LAYOUT + TYPO + THEME`。

## 5. 验证（Verification）

- `V1` 样式选择完整性：这个结构必须能说清样式选择器服务于哪些元素类型，以及每类元素的盒模型相关样式如何选择，外观基础不能混乱。
- `V2` 布局完整性：这个结构必须能说清多个视觉块如何排列、对齐、和占据空间，布局关系不能混乱。
- `V3` 排版完整性：这个结构必须能说清字体、字号、字重、行高、字距与文本层级如何组织，排版关系不能混乱。
- `V4` 主题映射来源有效性：这个结构必须能说清哪些稳定映射位可被主题配置消费，以及这些映射位在不同主题、状态和局部覆盖下如何建立对应关系，映射来源不能混乱。
- `V5` 组合闭合性：当样式选择器、布局结构、排版结构和主题映射器一起成立时，不能出现“有外观但没有空间组织”“有布局但没有文本组织”“有排版但没有主题落点”“有主题映射但没有稳定语义依据”这类断链情况。
- `V6` 最小必要性：移除任一 `B*` 后，`V1 ~ V5` 至少一项失败，说明这四个基础部分都是必要的。
- `V7` 结论表达：验证结果应逐条说明哪条 `R*` 成立或失败、对应输出了什么能力或失效结论，并附上对应的边界证据。
