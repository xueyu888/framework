# 归属关系表:OwnershipRelationshipTable

@framework

## 1. 能力声明（Capability Statement）

- `C1` 记录能力：记录对象的归属关系。

## 2. 边界定义（Boundary / 参数）

- `HEADER` 表头边界：表头的内容必须明确。来源：`C1`。
- `COUNT` 数量边界：记录行的数量需要限制。来源：`C1`。
- `ROW` 行边界：一行数据必须规定哪些列数据是必要的，哪些列数据是可以为空的。来源：`C1`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 表头：关系表的列结构。来源：`C1 + HEADER`。
- `B2` 记录行：关系表的单条记录结构。来源：`C1 + COUNT + ROW`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 表头-记录装表组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先由 `B1` 给出稳定表头，再由全部 `B2` 按表头列结构写入记录行。每条记录行都必须满足 `B1` 声明的列约束，并在必填列中同时给出段落块与文本块的归属记录；记录行数量必须落在已声明范围内。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 输出结构：`OwnershipRelationshipTable`（必须包含 `header`、`row_set`；其中每个 `row` 都必须满足 `B1` 约束的列结构）。
  - `R1.5` 边界绑定：`HEADER/COUNT/ROW`。

- `R2` 禁止组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：表头缺失必要列、记录行脱离表头独立存在、记录行未填充必填列、记录行列数与表头不一致、或记录行数量超出已声明范围的方案均无效。
  - `R2.3` 输出能力：`C1`。
  - `R2.4` 输出结构：`InvalidOwnershipRelationshipTable_set`（元素：`IOT`，每个 `IOT` 必须包含 `invalid_reason`、`offending_header`、`offending_row_set`）。
  - `R2.5` 边界绑定：`HEADER/COUNT/ROW`。

## 5. 验证（Verification）

- `V1` 基完备性：`B1` 必须能够稳定定义表头列结构，`B2` 必须能够稳定定义单条记录行结构；缺少任一项都不能构成合法关系表。
- `V2` 表头合法性：`R1` 使用的 `header` 必须明确列名、列顺序与必填列，不得存在未声明列或语义不明的列。
- `V3` 记录行合法性：`R1` 产出的每条 `row` 都必须满足 `B1` 规定的列结构，并对必填列完成填充；记录行不得脱离表头单独成立。
- `V4` 数量合法性：`R1` 产出的 `row_set` 数量必须满足 `COUNT` 约束，不得出现超量、缺量或未声明数量范围的情况。
- `V5` 非法组合可检出性：`R2` 列举的无效方案都必须能够被识别为无效，并输出明确失效原因与对应表头或记录行。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失败。
- `V7` 结论表达：逐条输出 `R1/R2 -> 有效/无效 / 边界 / 失效原因` 的结论。
