# 输出格式:OutputFormat

@framework

## 1. 能力声明（Capability Statement）

- `C1` 输出格式约束能力：提供稳定的输出字段与字段顺序。

## 2. 边界定义（Boundary / 参数）

- `FORMAT` 格式边界：输出格式的字段顺序、组织形式必须明确。来源：`C1`。
- `FIELD` 字段边界：字段的种类必须明确。来源：`C1`。
- `COUNT` 数量边界：字段的数量必须存在限制。来源：`C1`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文本ID：用于标识当前输出结果属于哪个文本。来源：`C1 + FORMAT + FIELD`。
- `B2` 文本块编号：用于标识文本块在当前文本中的顺序编号。来源：`C1 + FORMAT + FIELD + COUNT`。
- `B3` 文本块内容：用于承载与文本块编号对应的文本内容。来源：`C1 + FORMAT + FIELD + COUNT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 文本分块结果格式组合
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：输出格式必须先给出一个 `B1`，再按稳定顺序给出零个或多个由 `B2 + B3` 构成的文本块单元；每个文本块单元中，文本块编号与文本块内容必须一一对应，且所有文本块单元必须按编号顺序出现；文本块单元数量必须落在已声明范围内。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 输出结构：`TextChunkOutputFormat`（必须包含 `text_id`、`ordered_chunk_item_set`；其中每个 `chunk_item` 都必须包含 `chunk_id`、`chunk_text`）。
  - `R1.5` 边界绑定：`FORMAT/FIELD/COUNT`。

- `R2` 禁止组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：缺失 `B1`、文本块编号脱离文本块内容单独出现、文本块内容脱离文本块编号单独出现、文本块单元顺序不稳定、或文本块单元数量超出已声明范围的方案均无效。
  - `R2.3` 输出能力：`C1`。
  - `R2.4` 输出结构：`InvalidTextChunkOutputFormat_set`（元素：`ITF`，每个 `ITF` 必须包含 `invalid_reason`、`offending_text_id`、`offending_chunk_item_set`）。
  - `R2.5` 边界绑定：`FORMAT/FIELD/COUNT`。

## 5. 验证（Verification）

- `V1` 基完备性：`B1` 必须能标识文本，`B2` 必须能标识文本块编号，`B3` 必须能承载对应文本块内容；缺少任一项都不能构成合法输出格式。
- `V2` 格式合法性：`R1` 产出的每个 `TextChunkOutputFormat` 都必须满足“先 `text_id`，后零个或多个按顺序排列的 `chunk_id + chunk_text` 单元”的结构。
- `V3` 单元合法性：每个 `chunk_item` 都必须同时包含 `chunk_id` 与 `chunk_text`，二者不得拆开独立出现。
- `V4` 数量合法性：`ordered_chunk_item_set` 的数量必须满足 `COUNT` 约束。
- `V5` 非法组合可检出性：`R2` 列举的无效方案都必须能够被识别为无效，并输出明确失效原因与触发文本块单元。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失败。
- `V7` 结论表达：逐条输出 `R1/R2 -> 有效/无效 / 边界 / 失效原因` 的结论。
