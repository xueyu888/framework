# 数据库接口:DatabaseInterface

@framework

## 1. 能力声明（Capability Statement）

- `C1` 文本块查询能力：支持根据文档ID查询对应的文本块集合。
- `C2` 文档领域查询能力：支持根据文档ID查询对应的文档领域。
- `C3` 标签定义查询能力：支持根据文档领域查询对应的标签定义集合。

- `C4` 标注记录查询能力：支持根据单个文档ID查询该文档下已有标注记录。

- `C5` 标注记录写入能力：支持将文本块-标签评分结果写入数据库。

- `C6` 人工审核数据关联查询能力：支持将标注记录与对应文本块内容关联后导出为人工审核数据。

- `C7` 标注状态写回能力：支持将系统判定结果或人工确认结果写回数据库。

- `N1` 不提供结构生成能力：不负责生成文档、文本块或标签定义，只负责读取已有结构。
- `N2` 不提供审核能力：不负责决定标注结果是否通过审核，只负责写入给定结果。
- `N3` 不提供筛选能力：不负责决定应查询哪些领域或哪些标签，只执行给定查询路径。

## 2. 边界定义（Boundary / Parameter 参数）

- `P1` 文档ID查询边界：输入查询条件必须为单个合法文档ID。来源：`C1 + C2 + C4`。
- `P2` 文本块查询结果边界：查询得到的文本块集合中的每个文本块都必须满足 `backend.L0.M0[R3]` 的组合规则。来源：`C1`。
- `P3` 领域查询结果边界：查询得到的文档领域必须为单个合法领域标识。来源：`C2`。
- `P4` 标签定义查询结果边界：根据领域查询得到的标签定义集合中的每个标签定义都必须满足 `L0.M0[R1]` 的组合规则。来源：`C3`。

- `P5` 标注记录查询结果边界：查询得到的标注记录必须属于单个文档ID下的已有标注记录，且至少包含 `doc_id`、`chunk_id`、`tag_name`、`confidence`、`reason`、`review_status` 与 `enable_status` 字段。来源：`C4`。
- `P6` 标注记录查询数量边界：单次查询的标注记录必须全部属于同一个文档ID。来源：`C4`。

- `P7` 标注记录写入字段边界：写入数据库的标注记录必须至少包含 `doc_id`、`chunk_id`、`tag_name`、`confidence`、`reason` 和 `review_status` 字段。来源：`C5`。
- `P8` 标注记录写入数量边界：单次写入的标注记录数量等于 `1`。来源：`C5`。

- `P9` 人工审核数据字段边界：导出的人工审核数据必须至少包含 `doc_id`、`chunk_id`、`tag_name`、`confidence`、`reason`、`enable_status`、`review_status` 与对应文本块内容。来源：`C6`。
- `P10` 文本块内容关联边界：每条标注记录都必须能通过 `doc_id + chunk_id` 关联到唯一文本块内容，且该文本块内容必须满足 `backend.L0.M0[R3]` 的组合规则。来源：`C6`。

- `P11` 状态写回字段边界：写回输入必须至少包含记录定位字段、`enable_status`、`review_status` 与 `review_source` 字段。来源：`C7`。
- `P12` 状态写回范围边界：写回结果中的 `enable_status` 只能为 `启用`、`不启用` 或 `待人工审核`；`review_status` 只能为 `已审核` 或 `未审核`。来源：`C7`。
- `P13` 状态写回字段更新边界：状态写回必须作用于 `enable_status`、`review_status` 和 `review_source` 字段。来源：`C7`。
- `P14` 状态写回作用域边界：单次状态写回必须作用于单个文档下的一条或多条已有标注记录。来源：`C7`。

- `P15` 数据库类型边界：数据库类型固定为postgres。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 文本块查询接口：根据文档ID查询文本块集合的接口。来源：`C1 + P1 + P2`。
- `B2` 领域查询接口：根据文档ID查询文档领域的接口。来源：`C2 + P1 + P3`。
- `B3` 标签定义查询接口：根据文档领域查询标签定义集合的接口。来源：`C3 + P4`。
- `B4` 标注记录查询接口：根据单个文档ID查询已有标注记录的接口。来源：`C4 + P1 + P5 + P6`。
- `B5` 标注记录写入接口：将文本块-标签评分结果写入数据库的接口。来源：`C5 + P7 + P8`。
- `B6` 标注状态写回接口：将系统判定结果或人工确认结果写回数据库的接口。来源：`C7 + P11 + P12 + P13 + P14`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 查询接口组合
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：通过文本块查询接口、领域查询接口和标签定义查询接口，形成面向单个文档标注任务的完整查询能力，支持先按文档ID获取文本块集合与文档领域，再按领域获取标签定义集合。
  - `R1.3` 输出能力：`C1 + C2 + C3`。
  - `R1.4` 参数绑定：`P1 + P2 + P3 + P4`。

- `R2` 文本块标签评分接口组合
  - `R2.1` 参与基：`B1 + B2 + B3 + B5`。
  - `R2.2` 组合方式：在完整查询能力基础上补充标注记录写入接口，形成完整的标注数据库接口，支持文档标注模块完成“查询输入数据 + 写入标注记录”的完整数据库交互。
  - `R2.3` 输出能力：`C1 + C2 + C3 + C5`。
  - `R2.4` 参数绑定：`P1 + P2 + P3 + P4 + P7 + P8`。

- `R3` 标注记录查询接口组合
  - `R3.1` 参与基：`B4`。
  - `R3.2` 组合方式：根据单个文档ID查询该文档下已有标注记录，作为系统判定或后续审核流转的输入。
  - `R3.3` 输出能力：`C4`。
  - `R3.4` 参数绑定：`P1 + P5 + P6`。

- `R4` 人工审核数据关联查询接口组合
  - `R4.1` 参与基：`B1 + B4`。
  - `R4.2` 组合方式：先查询该文档下已有标注记录，再通过 `doc_id + chunk_id` 关联查询对应文本块内容，最后整理为统一人工审核数据。
  - `R4.3` 输出能力：`C4 + C6`。
  - `R4.4` 参数绑定：`P1 + P2 + P5 + P6 + P9 + P10`。

- `R5` 标注状态写回接口组合
  - `R5.1` 参与基：`B6`。
  - `R5.2` 组合方式：接收系统判定结果或人工确认结果后，校验记录定位字段、状态字段与审核来源是否合法，再将对应结果写回数据库。
  - `R5.3` 输出能力：`C7`。
  - `R5.4` 参数绑定：`P11 + P12 + P13 + P14`。

## 5. 验证（Verification）

- `V1` 推导一致性：`B1` 必须可由 `C1 + P1 + P2` 推导得到，`B2` 必须可由 `C2 + P1 + P3` 推导得到，`B3` 必须可由 `C3 + P4` 推导得到，`B4` 必须可由 `C4 + P1 + P5 + P6` 推导得到，`B5` 必须可由 `C5 + P7 + P8` 推导得到，`B6` 必须可由 `C7 + P11 + P12 + P13 + P14` 推导得到。
- `V2` 规则完整性：`R1`、`R2`、`R3`、`R4` 与 `R5` 必须明确给出参与基、组合方式、输出能力与参数绑定。
- `V3` 结构闭环性：`R1 + R2 + R3 + R4 + R5` 的输出能力并集必须覆盖 `C1 + C2 + C3 + C4 + C5 + C6 + C7`。
- `V4` 查询结果边界符合性：任意有效查询接口组合都必须满足 `P2 + P3 + P4 + P5 + P6`。
- `V5` 标注记录写入边界符合性：任意有效标注记录写入都必须满足 `P7 + P8`。
- `V6` 人工审核数据关联边界符合性：任意有效人工审核数据查询都必须满足 `P9 + P10`。
- `V7` 状态写回边界符合性：任意有效状态写回都必须满足 `P11 + P12 + P13 + P14`。
- `V8` 最小必要性：移除 `B4`、`B5` 或 `B6` 任一结构基后，标注记录查询、标注记录写入或状态写回能力至少一项无法成立；移除 `B1` 后，人工审核数据关联查询能力无法成立。
