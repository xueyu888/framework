# 消息队列结构模块:MessageQueueAbstractStructureModule

@framework

## 1. 能力声明（Capability Statement）

- `C1` 完整消息暂存能力：导出对单条完整消息的保存、取出与回收能力。
- `C2` 确定性取用能力：导出消息在暂存结构内的确定性排列结构。
- `C3` 并发端点协调能力：导出读写端点相互独立、并可在并发访问下保持状态一致的协调结构。
- `C4` 非能力项：不负责消息内部外部引用目标的存在性、一致性与时效性。

## 2. 边界定义（Boundary）

<!-- 边界定义了本模块能力的合法变化空间与不变约束：即在实例化时“什么可以变”，但为了维持核心能力“必须满足什么底线”。 -->
- `SCALE` 容量扩展边界：总容量、扩展粒度需明确。来源：`C1`。
- `LIFETIME` 生命周期边界：消息添加移除条件需明确。来源：`C1 + C2`。
- `ORDER` 顺序规则边界：消息入队、排队与出队的确定性次序规则需明确。来源：`C2`。
- `FLOW` 入出队边界：消息从哪一端进入、从哪一端取出、头尾指针如何推进、空满状态下端点行为需明确。来源：`C3`。
- `PARALLEL` 并行访问边界：多读写端并发访问时的互斥、同步机制与并发一致性需明确。来源：`C3`。

## 3. 最小可行基（Minimum Viable Bases）
<!-- 存储槽只是存储的空间，只是存下来，不关心怎么排，也不关心谁来读怎么读 -->
- `B1` 消息承载节点：由可申请和释放的内存单元与容量计数、被引用计数字段构成的存储结构。在运行环境允许的范围内可申请，被引用技术归零销毁占用空间。来源：`C1 + SCALE + LIFETIME`。
- `B2` 链式索引：由头尾位置指针、内存关联指针与固定入队/出队端规则构成的链式索引结构。来源：`C1 + C2 + ORDER + LIFETIME + FLOW`。
- `B3` 并发协调：由端点角色与互斥锁构成的并发协调结构。读写锁分开，提高并发访问效率。来源：`C3 + FLOW + PARALLEL`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 节点暂存组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：实例化可申请、可释放的存储节点与容量计量结构，形成仅具备暂存与回收能力的基础存储池。
  - `R1.3` 输出能力：`C1`（暂存）。
  - `R1.4` 边界绑定：`SCALE/LIFETIME`。

- `R2` 链式顺序组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在`B1`之上叠加`B2`，形成具备确定性推进关系的链式队列结构，但尚未开放并发控制。
  - `R2.3` 输出能力：`C1 + C2`（暂存 + 有序）。
  - `R2.4` 边界绑定：`SCALE/LIFETIME/ORDER/FLOW`。

- `R3` 并发队列组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：在`B1`之上叠加`B2`，`B3`约束共享状态更新，形成并发访问安全的标准消息队列。
  - `R3.3` 输出能力：`C1 + C2 + C3`（暂存 + 有序 + 并发协调）。
  - `R3.4` 边界绑定：`SCALE/LIFETIME/ORDER/FLOW/PARALLEL`。

- `NR1` 禁止组合
  - `NR1.1` 参与基：`B1 + B2 + B3`。
  - `NR1.2` 组合方式：缺失链式顺序结构直接暴露端点、或绕过固定入队/出队端规则修改共享状态，或并发访问下无法判定节点追加/摘除结果的方案均无效。


## 5. 验证（Verification）
<!-- 检验边界是都闭合、检验组合能力导出、检验输出格式和证据落点 -->
- `V1` 并发协调性：在同时读写的情况下，必须满足`PARALLEL`边界定义的互斥访问规则。
- `V2` 顺序一致性：在读写访问成功时，必须满足`ORDER`边界定义的确定性顺序规则、`FLOW`边界定义的推进规则。
- `V3` 生命周期管理：在读写访问成功后，必须满足`LIFETIME`边界定义的消息移除条件。
- `V4` 容量扩展：在扩展容量时，必须满足`SCALE`边界定义的扩展粒度，且能在运行时获取当前总容量。
