# 图标与反馈原子模块:IconFeedbackAtomicModule

@framework

## 1. 能力声明（Capability Statement）

- `C1` 图标语义抽象能力：定义图标语义节点和映射结构。
- `C2` 反馈状态抽象能力：定义加载、成功、失败反馈结构。
- `C3` 原子治理能力：支持图标与反馈结构的演进与兼容约束。
- `C4` 非能力项：不负责登录表单骨架和流程编排。

## 2. 边界定义（Boundary / 参数）

- `ICONSEM` 图标语义约束：图标只表达状态语义，不绑定实例主题值。来源：`C1 + C3`。
- `FEEDBACK` 反馈状态约束：加载、成功、失败反馈节点需统一。来源：`C2 + C3`。
- `ACCESS` 可访问约束：反馈提示和图标替代文本需可读。来源：`C1 + C2`。
- `TOKEN` 令牌映射约束：图标与反馈样式映射需可替换。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 图标语义原子基：由语义图标集、映射表和替换入口构成。来源：`C1 + ICONSEM + ACCESS`。
- `B2` 反馈状态原子基：由反馈状态面、切换节点和提示插槽构成。来源：`C2 + FEEDBACK + TOKEN`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 原子定义组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先建立图标语义映射，再挂接统一反馈状态结构。
  - `R1.3` 输出能力：`C1 + C2 + C3`。
  - `R1.4` 边界绑定：`ICONSEM/FEEDBACK/ACCESS/TOKEN`。
- `R2` 禁止组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将具体业务文案或具体主题实例绑定到图标反馈原子的方案无效。
  - `R2.3` 输出能力：`C4`。
  - `R2.4` 边界绑定：`ICONSEM/FEEDBACK/ACCESS/TOKEN`。

## 5. 验证（Verification）

- `V1` 语义完整性：图标语义和反馈状态结构必须可独立复用。
- `V2` 映射一致性：同一状态在不同入口的反馈语义必须一致。
- `V3` 约束符合性：实例文案和主题值不得固化在 `B*` 中。
