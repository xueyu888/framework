# 中文模块名:数据清洗工作台前端模块

@framework

这个模块想解决的，其实不是“做一个页面”这么简单。

把数据清洗这类业务里经常重复出现的前端页面，整理成一套更稳定、更好复用的“工作台结构”。

以后不管是数据清洗、字段修正、批量处理，还是类似的数据处理页面，都不用每次从头拼，而是能站在同一套前端骨架上继续长。

## 1. 能力声明（Capability Statement）

- `C1` 工作台承接能力：把当前处理对象、页面区域、主操作入口和步骤路径组织清楚，让用户进入工作台后不迷路。
- `C2` 数据表达能力：把数据样本、字段结构、异常标记、统计摘要等信息转成统一可理解的前端表达，让用户看得懂当前数据状态。
- `C3` 配置组织能力: 把规则和配置统一起来，形成一个可视化的配置体系，包括：清洗规则、参数、条件、步骤顺序等。
- `C4` 结果预览能力：统一表达处理后产出的结果内容，让用户理解“结果是什么、改了什么、哪里异常”。
- `C5` 交互反馈能力：统一表达用户动作触发后的系统响应，让用户理解“刚才操作是否生效、系统当前处于什么状态、接下来能做什么”。
- `C6` 扩展接入能力: 以后换一种清洗规则、换一种字段类型、换一种预览方式，甚至换一种业务场景，还是能接进来，而不用把整个工作台推翻重写。

### 非职责声明（Non-Responsibility Statement）

- `N1` 不负责数据清洗算法本身。
- `N2` 不负责后端数据处理逻辑。

## 2. 边界定义（Boundary / 参数）

- `P1` 工作台承接能力约束：对象可见，区域可见，路径可见，不越权。来源：`C1`。
  - `P1.1` 对象可见：用户进入工作台后，必须能知道当前处理的数据对象是什么。
  - `P1.2` 区域可见：用户必须能清晰看到操作区、结果区、配置区等不同功能区域的划分。
  - `P1.3` 路径可见：用户必须能知道当前在整个清洗流程的哪个步骤，下一步应该做什么。
  - `P1.4` 不越权：工作台承接只负责页面分区与动作入口的组织，不负责规则、预览、执行等后续处理逻辑。
- `P2` 数据约束：所有进入页面的数据都不能直接拿原始响应就开始渲染，对象必须先被收敛为统一前端数据模型，才能继续展示、配置与预览。来源：`C2`。
- `P3` 配置边界：规则、参数、步骤等配置内容，必须先收敛为统一配置模型，再通过统一配置接口进入工作台；业务页面不得以私有表单状态、私有字段命名或局部配置逻辑直接驱动工作台核心结构。来源：`C3`。
- `P4` 结果预览边界：所有与数据处理结果相关的展示内容，必须通过统一结果预览模型进入工作台，包括样本结果、差异结果、异常结果与解释信息；不允许由各业务页面分别以私有结构定义结果展示区域。来源：`C4`。
- `P5` 反馈边界：工作台中的配置、预览、执行、保存、导出等动作，必须进入统一反馈通路，由统一反馈模型表达状态变化、错误信息与后续动作；不允许由各区域分别维护不可收敛的局部反馈逻辑。来源：`C5`。
- `P6` 扩展边界：工作台核心结构必须保持稳定，所有规则、字段、预览方式、业务场景的差异，都必须通过可注册、可替换、可局部挂载的模块方式接入，而不允许通过修改核心结构来适配差异。来源：`C6`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 结构基一：待定义或 L0.M0[R1] 或 frontend.L1.M0[R1,R2]。来源：`C1 + P1`。
- `B2` 结构基二：待定义或 L0.M1[R1]。来源：`C2 + P2`。
- `B3` 结构基三：待定义或 L0.M0[R2] + L0.M1[R2]。来源：`C2 + C3 + P2 + P3`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 结构装配组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先固定结构接口，再收敛状态通路与装配关系。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`P1 + P2 + P3`。
- `R2` 上层展开组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：在装配结果上叠加扩展约束，形成可继续生长的模块结构。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`P1 + P2 + P3`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：脱离下层结构直接按功能清单拼接，或违反边界约束的组合均无效。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`P1 + P2 + P3`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基/组合方式/输出能力或失效结论/边界绑定。
- `V3` 结构闭环性：`R1~R2` 输出能力并集必须覆盖 `C1~C3`。
- `V4` 边界符合性：所有有效组合必须满足绑定的参数边界。
- `V5` 最小必要性：移除任一 `B*` 后，`V3` 或 `V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 边界绑定` 或 `R* -> N* + 边界绑定` 的通过或失败结论。
