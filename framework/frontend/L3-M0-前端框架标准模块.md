# 前端框架标准模块:FrontendFrameworkStandard

@framework

本模块建立在“前端基础原子 -> 前端界面原子 -> 前端复合结构 -> 前端总标准 -> 运行时标准 / 页面与运行编排”这条压缩链上。承载绑定、状态交互与视觉语义位于 `frontend/L0-M0 ~ L0-M2`，触发、文本输入、展示容器、集合浏览、稳定标记与附着回流反馈、局部任务接管容器、互斥视图切换、临时展开选择面、锚点附着提示、展开披露、页序列导航、路径链导航、文件摄取、日期时间输入、媒体预览与离散值选择位于 `frontend/L1-M0 ~ L1-M15`，规则化二维集合浏览、结构化字段提交、条件组合检索工作面与高频动作编排带等由多个 `L1` 原子稳定组合出的复合结构位于 `frontend/L2-*`，本模块只定义前端通用框架如何在更高一层承接这些结构。项目实例应把本模块收敛成显式前端 contract 和规则校验，而不是重写这些共同结构。在当前前端通用框架中，`SURFACE`、`VISUAL`、`ROUTE`、`A11Y` 是稳定的大类边界；`INTERACT + STATE + EXTEND` 当前仍由框架 contract 固定，不额外开放新的大类边界。更具体的运行时成立条件由 `frontend/L3-M1-前端运行时与技术栈标准模块.md` 约束，页面如何装入既有运行承接面由 `frontend/L4-M0-页面与运行编排模块.md` 约束。本模块中的 `B*` 是当前模块的最小结构承接单元，可以由 `L1` 原子族或 `L2` 复合结构模块组合而成，并不要求在全框架引用图里都是全局叶子。本模块的 Markdown 表达、编号与作者入口约束遵循 `specs/框架文档Lint标准.md`，其中 plain `@framework` 直达模板入口属于硬约束。本模块只负责共同前端结构语言，不直接固化某个产品的最终真相、实例值或具体技术栈。

## 1. 能力声明（Capability Statement）

- `C1` 通用界面契约能力：把输入、展示、导航、反馈、局部任务接管、互斥视图切换与二维集合浏览等结构收敛为稳定前端界面骨架。
- `C2` 跨场景交互承接能力：以统一结构承接不同场景交互。
- `C3` 领域框架承接能力：为更高层的领域通用框架输出稳定 UI 承接面，而不泄漏底层实现细节。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责具体领域对象定义、后端存储模型与项目实例值。
- `N2` 非职责声明：不负责重定义运行时 profile、启动追踪主结构、页面集编排关系与实例页面装配细节。

## 2. 边界定义（Boundary / 参数）

- `SURFACE` 界面边界：主承载面、辅助承载面、导航承载面、局部反馈承载面与跨页面辅助承载面的职责、层级关系与落位范围需明确。来源：`C1 + C2`。
- `VISUAL` 视觉边界：颜色、字号级、圆角级、阴影级、密度档位与 token 到 recipe 的映射关系需通过受控结构表达。来源：`C1 + C3`。
- `INTERACT` 交互边界：输入、选择、切换、展开、创建、删除、提交等交互出口与交互结果类型需稳定。来源：`C1 + C2`。
- `STATE` 状态边界：加载、空态、错误与成功等结构状态集合、状态迁移关系与反馈窗口需统一表达。来源：`C1 + C2`。
- `EXTEND` 扩展边界：领域框架只能在既有插接口上扩展，且扩展到通用骨架的映射关系不得重写通用原子职责。来源：`C2 + C3`。
- `ROUTE` 路由边界：通用页面入口、返回路径、深链落点与路由到界面承载面的映射关系需可承接。来源：`C2 + C3`。
- `A11Y` 可访问边界：阅读顺序、键盘导航、提示关系与辅助技术可感知关系需可达。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 承载交互契约基：L1.M4[R2] + L1.M0[R2] + L1.M9[R1,R2]。来源：`C1 + SURFACE + VISUAL + INTERACT + STATE`。
- `B2` 导航反馈扩展契约基：L1.M12[R1,R2] + L1.M15[R1,R2]。来源：`C2 + C3 + INTERACT + ROUTE + EXTEND + A11Y`。
- `B3` 复合结构承接基：L2.M2[R2,R3,R4]。来源：`C1 + C2 + SURFACE + INTERACT + STATE + EXTEND + A11Y`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 通用骨架收敛
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定触发、文本输入与展示容器之间的主关系，再把界面承载层级关系与视觉边界内的实例差异收敛成稳定 layout recipe 和 token 映射。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SURFACE + VISUAL + INTERACT + STATE`。
- `R2` 场景扩展闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在统一骨架上接入集合浏览与稳定标记、附着回流反馈，约束交互出口、返回路径、扩展映射关系与可访问顺序，使不同场景共享同一前端结构骨架，并为领域框架提供稳定插接口。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`INTERACT + ROUTE + A11Y + STATE + EXTEND + SURFACE + VISUAL`。
- `R3` 复合结构接入闭合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：在通用骨架与导航反馈扩展之上，允许规则化二维集合浏览等 `L2` 复合结构作为显式插接口接入，但复合结构必须继续服从通用界面的交互、状态与可访问边界，不得重写总标准本身。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`SURFACE + INTERACT + STATE + EXTEND + ROUTE + A11Y + VISUAL`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：让领域框架直接改写通用原子职责、绕过交互契约、把运行时标准或页面编排细节回写到总标准层，或让 `L2` 复合结构脱离总标准边界独立生长的方案均无效。
  - `R4.3` 失效结论：`N1 + N2`。
  - `R4.4` 边界绑定：`SURFACE + VISUAL + INTERACT + STATE + EXTEND + ROUTE + A11Y`。

## 5. 验证（Verification）

- `V1` 骨架完整性：通用界面必须同时具备触发、输入、展示、导航和反馈承载面，并满足 `SURFACE + VISUAL + INTERACT + STATE` 约束。
- `V2` 交互一致性：不同场景必须共享统一交互出口与返回结构，且交互结果类型与状态迁移关系可判定。
- `V3` 扩展稳定性：领域扩展必须通过 `EXTEND` 插接口完成，不得破坏通用职责；扩展到通用骨架的映射关系必须可追溯。
- `V4` 复合结构承接稳定性：接入的 `L2` 复合结构必须继续服从总标准边界，不得绕过 `EXTEND` 独立定义主界面真相。
- `V5` 可访问符合性：有效组合必须满足 `A11Y` 约束，且对辅助技术可感知。
- `V6` 最小必要性：移除任一 `B*` 后，`V1 ~ V5` 至少一项失败。
- `V7` 边界归属稳定性：界面承载、视觉、路由、复合结构接入与可访问相关实例差异必须只落到对应大类边界，不得跨大类游离。
- `V8` 结论表达：逐条输出 `R* -> C* + 边界绑定` 或 `R* -> N* + 边界绑定` 的通过或失败结论，并附对应边界证据。
