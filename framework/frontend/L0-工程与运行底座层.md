# 前端抽象结构底座模块:FrontendAbstractBase

@framework

## 1. 能力声明（Capability Statement）

- `C1` 结构抽象能力：定义前端系统最小稳定结构单元与装配接口。
- `C2` 运行承载能力：提供构建、运行、资源加载与部署承载能力。
- `C3` 约束外化能力：把安全、兼容和观测约束外化为可验证规则。
- `C4` 非能力项：不负责具体业务页面编排与视觉主题实例取值。

## 2. 边界定义（Boundary / 参数）

- `ARCH` 架构分层约束：UI、状态、服务职责边界必须清晰。来源：`C1 + C2`。
- `RUNTIME` 运行时约束：浏览器、容器和执行模式需可识别。来源：`C1 + C2`。
- `BUILD` 构建策略：构建产物类型、分包策略和发布模式需确定。来源：`C2`。
- `ASSET` 资源加载策略：静态资源命名、缓存和回源机制需统一。来源：`C2 + C3`。
- `SEC` 安全约束：输入防护、依赖治理和内容安全策略需可执行。来源：`C2 + C3`。
- `OBS` 可观测约束：日志、指标和链路追踪需覆盖关键路径。来源：`C2 + C3`。
- `COMPAT` 兼容目标：最低运行环境与降级策略需明确。来源：`C1 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 分层基：由模块边界、依赖方向和抽象接口构成的骨架结构。来源：`C1 + ARCH + COMPAT`。
- `B2` 构建运行基：由构建管线、运行拓扑和资源装载通道构成的执行结构。来源：`C1 + C2 + RUNTIME + BUILD + ASSET`。
- `B3` 治理观测基：由安全控制面、观测指标面和故障定位面构成的治理结构。来源：`C2 + C3 + SEC + OBS + COMPAT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 结构先行组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固化模块分层和依赖方向，再接入运行与治理能力。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`ARCH/COMPAT`。
- `R2` 运行承载组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在结构基上叠加构建和运行策略，形成稳定执行链路。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`RUNTIME/BUILD/ASSET/ARCH`。
- `R3` 约束闭环组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将安全与观测约束绑定到构建运行链路并形成可验证闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`SEC/OBS/COMPAT/RUNTIME/BUILD`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：跳过结构分层直接拼装运行策略，或缺失安全观测约束的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`ARCH/RUNTIME/BUILD/ASSET/SEC/OBS/COMPAT`。

## 5. 验证（Verification）

- `V1` 结构一致性：依赖方向必须满足 `ARCH` 约束且无逆向引用。
- `V2` 承载完整性：构建产物、运行模式和资源策略必须全链路可执行。
- `V3` 治理有效性：安全与观测规则必须覆盖关键执行路径。
- `V4` 边界符合性：所有有效组合必须满足 `COMPAT` 的兼容目标。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
