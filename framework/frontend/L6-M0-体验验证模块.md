# 前端体验验证模块:FrontendExperienceVerification

@framework

## 1. 能力声明（Capability Statement）

- `C1` 体验指标建模能力：把性能、可读性、一致性与稳定性收敛为可验证指标结构。
- `C2` 多维验证能力：联合验证流程时延、阅读顺序、交互一致性与错误恢复质量。
- `C3` 回归治理能力：基于指标异常识别回归并形成持续治理闭环。
- `C4` 非能力项：不负责具体领域 KPI 设定与后端算法优化。

## 2. 边界定义（Boundary / 参数）

- `PERF` 性能边界：首屏、切换、提交与回退时延预算需可量化。来源：`C1 + C2`。
- `READ` 阅读边界：主信息顺序、辅助信息层级与阅读焦点需稳定。来源：`C1 + C2`。
- `INTERACT` 交互边界：同类动作的输入、反馈与恢复表达需一致。来源：`C1 + C2`。
- `A11Y` 可访问边界：键盘导航、语义提示与焦点回路需达标。来源：`C1 + C2`。
- `STABLE` 稳定性边界：错误率、回退成功率与重试成功率需可观测。来源：`C2 + C3`。
- `ALERT` 告警边界：指标越界触发条件与处置路径需明确。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 指标模型结构基：L5.M0[R1,R3]。来源：`C1 + PERF + READ + INTERACT`。
- `B2` 验证样本结构基：L5.M0[R1,R2]。来源：`C1 + C2 + A11Y + STABLE + PERF`。
- `B3` 告警回归结构基：L5.M0[R2,R3]。来源：`C2 + C3 + ALERT + STABLE + INTERACT`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 指标口径统一
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先统一体验指标口径，再绑定验证样本与执行流程。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`PERF/READ/INTERACT/A11Y`。
- `R2` 稳定性治理闭环
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：将异常样本映射到告警与回归跟踪，形成持续治理回路。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`STABLE/ALERT/INTERACT/A11Y`。
- `R3` 体验验证收敛
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：联合验证性能、可访问性、一致性和稳定性，并输出回归结论。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`PERF/READ/INTERACT/A11Y/STABLE/ALERT`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失统一指标口径、验证样本不可回放或异常无告警路径的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`PERF/READ/INTERACT/A11Y/STABLE/ALERT`。

## 5. 验证（Verification）

- `V1` 指标一致性：同一体验指标在不同页面和流程中的口径必须一致。
- `V2` 样本覆盖性：验证样本必须覆盖浏览、编辑、上传与对话四类关键流程。
- `V3` 告警有效性：指标异常必须具备稳定告警和回归处置链路。
- `V4` 边界符合性：验证结果必须满足 `PERF/A11Y/STABLE` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
