# 前端文件摄取原子模块:FrontendFileIngestAtoms

@framework

模块短标签：文件接纳
典型落地对象：upload input / dropzone / intake queue
不包括：纯触发入口、文件库管理、后端上传处理流程

本模块定义前端文件摄取原子。它承接上传入口、拖拽接纳区、待处理文件队列和摄取结果反馈这类“围绕文件接纳与摄取状态形成短闭环”的结构，不把领域文件库、后端存储协议或正文预览工作面混进同一模块。

本模块位于 `frontend` 的界面原子层。它在触发、展示容器与稳定标记、附着回流反馈等更底层前端原子之上，把文件接纳入口、文件队列、接纳约束与摄取状态沉成共享结构，为附件上传、资源导入、资产接纳和工作台侧文件摄取区提供统一底座。

## 1. 能力声明（Capability Statement）

- `C1` 文件接纳入口承接能力：把点击选择、拖拽接纳与再次添加入口收敛为稳定文件接纳入口结构。
- `C2` 摄取队列闭合能力：把待处理文件、当前处理项与队列状态收敛为稳定摄取队列闭合结构。
- `C3` 摄取反馈回流能力：把数量限制、类型约束、失败回流与重试提示收敛为稳定摄取反馈回流结构。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责领域文件目录组织、后端存储实现与文件正文预览工作面。

## 2. 边界定义（Boundary / 参数）

- `INGEST` 接纳边界：点击选择入口、拖拽接纳区、再次添加出口与接纳入口到队列的映射关系需稳定。来源：`C1 + C3`。
- `QUEUE` 队列边界：待处理文件列表、当前处理项、队列顺序与队列项到文件状态的映射关系需稳定。来源：`C2 + C3`。
- `FILE` 文件边界：文件名、文件类型、体量摘要、文件状态与文件归属关系需稳定。来源：`C1 + C2`。
- `LIMIT` 约束边界：数量限制、类型限制、体量限制、拒绝原因与约束判定条件需稳定。来源：`C1 + C3`。
- `STATE` 状态边界：待处理、处理中、成功、失败与重试中等状态集合、状态迁移关系与反馈窗口需稳定。来源：`C2 + C3`。
- `A11Y` 可访问边界：接纳区说明、队列提示、状态播报与辅助技术可感知关系需可达。来源：`C1 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文件接纳入口基：L0.M0[R1,R3] + L0.M1[R4] + L0.M2[R1,R3]。来源：`C1 + INGEST + LIMIT + A11Y`。
- `B2` 摄取队列闭合基：L0.M0[R1,R2] + L0.M1[R3] + L0.M2[R1,R2]。来源：`C2 + FILE + QUEUE + STATE`。
- `B3` 摄取反馈回流基：L0.M0[R3] + L0.M1[R3] + L0.M2[R1,R3]。来源：`C3 + LIMIT + STATE + A11Y`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 文件接纳入口先行
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定点击选择、拖拽接纳、再次添加入口与接纳入口到队列的映射关系，再允许待处理队列附着其上。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`INGEST + LIMIT + A11Y`。
- `R2` 摄取队列闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：待处理文件列表、当前处理项、队列顺序与队列项到文件状态的映射关系必须附着稳定接纳入口，形成统一文件摄取队列结构。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`FILE + QUEUE + STATE + INGEST`。
- `R3` 摄取反馈回流
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：数量限制、类型约束、失败回流、重试提示与约束判定条件必须附着在接纳面或队列结构之上，并随队列状态回流到接纳面，形成稳定文件摄取反馈回流闭环，但不得替代队列结构本身。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`LIMIT + STATE + A11Y + FILE + QUEUE + INGEST`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：只有上传入口没有队列、存在待处理文件却无状态说明、约束判定条件无法解释、失败回流缺位，或把领域文件库管理与后端上传处理流程误写为文件摄取原子的方案均无效。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`INGEST + QUEUE + FILE + LIMIT + STATE + A11Y`。

## 5. 验证（Verification）

- `V1` 文件接纳入口完整性：文件接纳入口必须满足 `INGEST + LIMIT` 约束；点击选择、拖拽接纳与再次添加入口完整，且接纳入口到队列的映射关系与约束判定条件可判定。
- `V2` 摄取队列闭合有效性：摄取队列结构必须满足 `QUEUE + FILE` 约束；待处理文件、当前处理项与队列顺序可解释，且队列项到文件状态的映射关系完整。
- `V3` 摄取反馈回流一致性：摄取反馈结构必须满足 `LIMIT + STATE + A11Y` 约束；数量限制、类型限制、失败回流与状态播报可解释，且必须附着于接纳面或队列结构而不替代队列结构本身，并对辅助技术可感知。
- `V4` 共享原子有效性：本模块必须能被附件上传、资源导入和资产接纳等场景复用，而不是被误写成某一个领域文件库的唯一真相源。
- `V5` 最小必要性：移除任一 `B*` 后，`V1 ~ V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 边界绑定` 或 `R* -> N* + 边界绑定` 的通过或失败结论，并附对应边界证据。
