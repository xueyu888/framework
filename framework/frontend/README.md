# Frontend Framework README

本文档不是 `framework/*.md` 真相源，而是 `framework/frontend/` 的阅读入口与协作索引。
`frontend` 的真相仍以各个 `framework/frontend/L*-*.md` 模块为准；本文件只帮助作者和 AI 更快进入这套结构语言，避免第一次阅读时在层级、模块边界与推荐顺序上迷路。

## 1. 这个目录在回答什么

`framework/frontend/` 不是某个具体前端项目的组件库文档，也不是某个技术栈的最佳实践集合。
它回答的是另一类问题：

- 前端共同结构最底层由哪些原子能力构成
- 这些原子如何组合成稳定可复用的界面复合结构
- 更高层前端总标准、运行时标准与页面编排分别管到哪里
- 项目实例应该把哪些内容写进 `framework`，哪些内容留给 `project.toml` 与真实 code

如果把这套目录压缩成一句话，可以理解为：

- `frontend` 在定义“前端共同结构语言”

而不是：

- 某个产品页面长什么样
- React / Vue / 浏览器 API 应该怎么直接使用
- 某个实例字段、页面文案、业务菜单或项目布局细节

## 2. 总体分层

推荐先把 `frontend` 理解成一条单向收敛链：

`L0 原子前提 -> L1 界面原子 -> L2 复合结构 -> L3 总标准 / 运行时标准 -> L4 页面与运行编排`

各层职责可以先这样抓：

- `L0`
  - 回答前端最底层的共同结构前提。
  - 例如承载与绑定、状态与交互、视觉语义。
- `L1`
  - 回答单一主关系成立的界面原子。
  - 例如文本输入、离散值选择、触发、展示容器、集合浏览、附着反馈。
- `L2`
  - 回答多个 `L1` 如何稳定组合成常见前端工作面。
  - 例如结构化提交（form表单）、条件检索（条件检索）、二维浏览（table）、高频动作编排带（接工具栏、命令栏）。
- `L3-M0`
  - 回答前端通用框架标准如何承接 `L1/L2`。
- `L3-M1`
  - 回答前端运行时如何成立，profile 如何落位，启动与回放如何闭合。
- `L4-M0`
  - 回答页面如何装入既有运行承接面，以及页面进入/返回如何编排。

一个实用的判断方法是：

- 如果你描述的是“一个独立前端对象的单一主关系”，优先看 `L1`
- 如果你描述的是“一个工作面由多个原子稳定组合”，优先看 `L2`
- 如果你描述的是“整个 frontend 的共同 contract / runtime / page orchestration”，优先看 `L3/L4`

## 3. 各层速览

### 3.1 L0：最底层共同前提

`L0` 不是可直接落地成具体业务组件的层，而是前端最底层的共同结构前提。

- `L0-M0`
  - 承载与绑定原子模块
  - 重点是“承载区 -> 槽位 -> 宿主绑定 -> 可达闭合”
- `L0-M1`
  - 状态与交互原子模块
  - 重点是交互、状态迁移与反馈闭环的最低前提
- `L0-M2`
  - 视觉语义原子模块
  - 重点是视觉 token、语义映射与受控视觉表达

如果你第一次进入这套文档，建议从 `L0-M0` 开始，因为它最能帮助建立“这套前端框架到底在描述什么”的感觉。

### 3.2 L1：单一主关系的界面原子

`L1` 是目录中最常被直接消费的一层。它的目标不是覆盖所有具体组件名称，而是把前端里反复出现的单一主关系抽成稳定原子。

可以先按下面几组理解：

- 输入族
  - `L1-M0` 文本输入
  - `L1-M1` 离散值选择
  - `L1-M2` 日期时间输入
  - `L1-M3` 文件摄取
- 动作与短时结构
  - `L1-M4` 触发
  - `L1-M5` 临时展开选择面
  - `L1-M6` 互斥视图切换
  - `L1-M7` 展开披露
  - `L1-M8` 锚点附着提示
- 承载族
  - `L1-M9` 展示与容器
  - `L1-M10` 局部任务接管容器
  - `L1-M11` 媒体预览
- 浏览与导航族
  - `L1-M12` 集合浏览
  - `L1-M13` 路径链导航
  - `L1-M14` 页序列导航
- 反馈族
  - `L1-M15` 标记与反馈

理解 `L1` 时最重要的不是记例子，而是记住一句话：

- 一个 `L1` 模块只能表达一个单一主关系

如果一个对象必须借多个 `L1` 才能成立，它通常已经更接近 `L2`，而不是新的 `L1`。

### 3.3 L2：前端常见工作面的复合结构

`L2` 负责把多个 `L1` 组合成稳定工作面。当前目录中的四个核心复合结构可以先这样理解：

- `L2-M0` 结构化字段提交
  - 重点不是“某个表单长什么样”
  - 而是“字段组 + 校验链 + 提交出口 + 错误回流 + 状态反馈”如何闭合
- `L2-M1` 条件组合检索工作面
  - 重点不是“某个搜索栏长什么样”
  - 而是“条件输入 + 条件标签 + 提交检索 + 结果回流 + 状态反馈”如何闭合
- `L2-M2` 规则化二维集合浏览
  - 重点不是“某个数据表长什么样”
  - 而是“行集合 + 列定义 + 单元格关系 + 浏览操作 + 状态反馈”如何闭合
- `L2-M3` 高频动作编排带
  - 重点不是“某排按钮长什么样”
  - 而是“主动作 + 次动作 + 溢出区 + 上下文动作 + 状态提示”如何闭合

一个常见误区是：

- 把 `L2` 写成某个产品页面的唯一真相源

这会破坏 `frontend` 作为共同结构语言的目标。`L2` 应当能被多个场景复用，而不是直接等于某个实例页面。

### 3.4 L3：总标准与运行时标准

`L3` 分成两层不同问题，不要混读：

- `L3-M0`
  - 前端框架总标准
  - 关心的是通用界面 contract、扩展方式、总边界
- `L3-M1`
  - 前端运行时与技术栈标准
  - 关心的是运行承接面、renderer/style/script profile、启动与回放

可以把两者粗略区分为：

- `L3-M0` 回答“前端共同结构如何成立为总标准”
- `L3-M1` 回答“这些结构最终如何装进运行时”

### 3.5 L4：页面与运行编排

`L4-M0` 站在 `L3` 之上，回答页面级组织问题：

- 页面集如何成立
- 页面如何挂到既有运行承接面
- 页面进入、恢复、返回与回放证据如何形成闭环

如果你的问题开始涉及：

- 页面入口
- 路由入口
- 返回路径
- 页面到运行承接面的装配

那就不应只停在 `L3-M1`，而要进入 `L4-M0`。

## 4. 推荐阅读顺序

### 4.1 第一次进入 `frontend`

推荐顺序：

1. `L0-M0`
2. `L1` 速查与目录分组
3. `L1` 里最容易混淆的相邻带
4. `L2-M0 ~ L2-M3`
5. `L3-M0`
6. `L3-M1`
7. `L4-M0`

更具体一点：

1. 先读 `L0-M0-承载与绑定原子模块.md`
2. 再扫一遍 `L1` 文件树，建立模块地图
3. 然后对照 `docs/verification/frontend_l1_模块速查表.md`
4. 再读 `docs/verification/frontend_l1_正交性检查表.md`
5. 最后进入 `L2/L3/L4`

### 4.2 如果你正在新增或修改某一类对象

- 遇到单一组件/原子对象
  - 先看对应 `L1`
- 遇到工作面、工作流或列表/表单/动作带
  - 先看对应 `L2`
- 遇到前端总 contract、跨场景通用规则
  - 先看 `L3-M0`
- 遇到运行时成立、profile 落位、启动追踪
  - 先看 `L3-M1`
- 遇到页面进入、返回、路由与运行装配
  - 先看 `L4-M0`

## 5. 常见边界误判

阅读或新增模块时，最常见的误判不是“写错一句话”，而是把不同层和不同主关系混写到同一模块。

高频误判包括：

- 把 `L1` 原子写成多轴复合结构
- 把 `L2` 工作面写成某个产品页面的实例真相
- 把运行时问题提前回写到 `L3-M0`
- 把页面编排问题压回 `L3-M1`
- 把通用附着反馈 `L1-M15` 误用成所有反馈的总出口
- 把路径链、分页、集合浏览继续混成“大导航”
- 把触发、临时选择面、附着提示、离散值选择混成同一类对象

如果你对某个对象归属拿不准，优先问自己三个问题：

1. 它是在表达单一主关系，还是多个原子的复合闭环
2. 它是在表达共同结构，还是某个实例页面/业务对象
3. 它是在表达界面结构，还是运行时/页面编排

## 6. 本目录与其他文档的关系

下面几类文件各有分工：

- `framework/frontend/L*-*.md`
  - 真相源
  - 负责定义模块能力、边界、最小可行基、组合原则与验证
- `docs/verification/frontend_l1_模块速查表.md`
  - 快速阅读索引
  - 负责帮助在文件树中快速建立 `L1` 地图
- `docs/verification/frontend_l1_正交性检查表.md`
  - 边界治理与扩展检查表
  - 负责在新增或拆分 `L1` 时检查是否仍保持正交
- 本 README
  - 目录级入口
  - 负责帮助第一次进入 `frontend` 的作者和 AI 建立整体地图与阅读顺序

## 7. 协作建议

对作者与 AI 来说，进入 `frontend` 时建议遵循以下习惯：

1. 先判断当前问题落在哪一层，不要一上来就改具体模块。
2. 先确认对象是在表达共同结构，还是在表达项目实例。
3. 若修改 `L1`，优先检查是否破坏相邻带正交性。
4. 若修改 `L2`，优先检查是否把实例真相写回 framework。
5. 若修改 `L3/L4`，优先检查 runtime、page orchestration 与总标准边界是否被混写。

如果一个改动让你同时想修改很多层，通常说明你还需要先回答：

- 这次变化究竟是在改共同结构语言
- 还是只是在改项目实例消费方式

## 8. 一句话导航

如果只保留一段最短导航，可以记住下面这组：

- `L0` 是共同前提
- `L1` 是单一主关系原子
- `L2` 是常见工作面复合结构
- `L3-M0` 是前端总标准
- `L3-M1` 是运行时标准
- `L4-M0` 是页面与运行编排

先分层，再读模块；先看主关系，再看例子；先守边界，再谈实例。
