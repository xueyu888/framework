# 账号密码登录校验组件：AccountPasswordLoginVerificationComponent

@framework

组件目标：组合已选字符串格式校验模块和已选身份校验模块，完成账号密码登录校验，并输出最终通过或失败结论。

## 1. 能力声明（Capability Statement）

* `C1` 账号格式校验编排能力：组织账号输入进入已选字符串格式校验模块。
* `C2` 账号密码身份校验编排能力：在账号格式校验通过后，组织账号输入与密码输入进入已选身份校验模块。
* `C3` 登录结论收敛能力：根据账号格式校验结论和身份校验结论收敛为最终通过或失败结论。

## 2. 非职责声明（Non-Capability Statement）

* `N1` 不负责账户创建、账户资料维护和登录后信息修改。
* `N2` 不负责密码存储设计与持久化实现。
* `N3` 不负责 token、会话与权限控制。
* `N4` 不在本组件中重复定义字符串格式规则、字符串表示变换规则或匹配规则。
* `N5` 不负责字符串格式校验模块和身份校验模块本身的底层实现设计；当前组件只引用既定模块实例。

## 3. 边界定义（Boundary / Parameter 参数）

* `D1` 登录输入范围：当前校验必须提供账号输入与密码输入，且二者均作为字符串参与编排。来源：`C1 + C2`
* `D2` 账号格式校验实例范围：账号输入对应的字符串格式校验模块实例必须预先确定，并能够对账号输入输出可判定的格式校验结论。来源：`C1`
* `D3` 身份校验实例范围：身份校验模块实例必须预先确定，并能够对账号输入与密码输入输出可判定的身份校验结论。来源：`C2`
* `D4` 编排顺序约束：身份校验只允许在账号格式校验结论为通过后进入。来源：`C1 + C2`
* `D5` 最终结论范围：最终输出只允许为通过、失败。来源：`C3`

## 4. 最小可行基（Minimum Viable Bases）

* `B1` 登录输入结构：由账号输入和密码输入构成。来源：`C1 + C2 + D1`
* `B2` 已选字符串格式校验模块：由预先确定的字符串格式校验模块实例构成；当前组件只引用，不重复展开其内部。来源：`C1 + D2`
* `B3` 已选身份校验模块：由预先确定的身份校验模块实例构成；当前组件只引用，不重复展开其内部。来源：`C2 + D3`

## 5. 基组合原则（Base Combination Principles）

* `R1` 账号格式校验编排组合

  * `R1.1` 参与基：`B1 + B2`
  * `R1.2` 组合方式：账号输入必须先进入已选字符串格式校验模块；该模块必须返回可判定的账号格式校验结论。
  * `R1.3` 输出能力：`C1`
  * `R1.4` 边界绑定：`D1/D2`

* `R2` 账号密码身份校验编排组合

  * `R2.1` 参与基：`B1 + B2 + B3`
  * `R2.2` 组合方式：仅当账号格式校验结论为通过时，账号输入与密码输入才允许进入已选身份校验模块；该模块必须返回可判定的身份校验结论。
  * `R2.3` 输出能力：`C2`
  * `R2.4` 边界绑定：`D1/D3/D4`

* `R3` 登录结论收敛组合

  * `R3.1` 参与基：`B2 + B3`
  * `R3.2` 组合方式：账号格式校验结论为失败时，最终结论直接为失败；账号格式校验结论为通过且身份校验结论为通过时，最终结论为通过；账号格式校验结论为通过但身份校验结论为失败时，最终结论为失败。
  * `R3.3` 输出能力：`C3`
  * `R3.4` 边界绑定：`D5`

## 6. 验证（Verification）

* `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个边界项推导得到。
* `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力和边界绑定。
* `V3` 闭环性：`R1~R3` 必须形成“账号格式校验 -> 身份校验 -> 登录结论收敛”的完整登录校验链。
* `V4` 边界符合性：所有有效组合都必须满足 `D1~D5`。
* `V5` 最小必要性：移除任一 `B*` 后，至少一项能力无法成立。
