# 数据库连接适配组件:DatabaseConnectionAdapter

@framework

组件目标：把连接规格承接到具体数据库服务，并形成可继续被上层承接的连接载体。

## 1. 能力声明（Capability Statement）

- `C1` 连接规格承接能力：接收合法连接规格并进入数据库服务适配层。
- `C2` 连接载体形成能力：把承接后的连接结果稳定组织为可用连接载体。
- `C3` 连接载体释放能力：释放由本层持有的连接载体。
- `C4` 非能力项：不负责逻辑会话上下文组织，不负责句柄追踪。

## 2. 边界定义（Boundary / 参数）

- `CS` 连接规格（connection_spec）：输入必须是完整连接规格。来源：`C1 + C2`。
- `DK` 数据库类型（database_kind）：适配层必须与数据库类型匹配。来源：`C1 + C2`。
- `MD` 会话模式（mode）：只允许 `sync | async`。来源：`C1 + C2`。
- `LS` 载体状态（carrier_state）：只允许 `ready | released`。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 规格接入口单元（自有基）：连接规格进入适配层的承接结构，由输入口和规格承接位构成。来源：`C1 + CS + DK`。
- `B2` 驱动适配桥单元（自有基）：数据库服务适配结构，由驱动桥和类型匹配位构成。来源：`C1 + C2 + DK + MD`。
- `B3` 连接载体单元（自有基）：承接后连接结果的持有结构，由载体实例位和状态位构成。来源：`C2 + C3 + MD + LS`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 规格进入组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：所有连接适配都必须先吃入完整连接规格；没有合法连接规格不承认进入适配层。
  - `R1.3` 输出结构：`SPEC_PORT`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`CS/DK/MD`。
- `R2` 连接载体形成组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：连接规格必须先经过驱动适配桥，再写入连接载体单元；绕过适配桥直接伪造载体的用法不承认有效。
  - `R2.3` 输出结构：`CONNECTION_CARRIER`。
  - `R2.4` 输出能力：`C1 + C2`。
  - `R2.5` 边界绑定：`CS/DK/MD/LS`。
- `R3` 连接载体释放组合
  - `R3.1` 参与基：`B3`。
  - `R3.2` 组合方式：释放动作只能作用于本层持有的连接载体，并把状态从就绪态收敛到已释放态。
  - `R3.3` 输出结构：`CARRIER_RELEASE_RESULT`。
  - `R3.4` 输出能力：`C3`。
  - `R3.5` 边界绑定：`LS`。
- `R4` 完整连接适配组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：先承接连接规格，再形成连接载体，最后在需要时释放连接载体；不允许跳过中间承接层。
  - `R4.3` 输出能力：`C1 + C2 + C3`。
  - `R4.4` 边界绑定：`CS/DK/MD/LS`。
- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：把逻辑会话、句柄映射或事务语义直接混入连接适配层的组合均无效。
  - `R5.3` 输出能力：`C4`。
  - `R5.4` 边界绑定：`CS/DK/MD/LS`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R4` 输出能力并集必须覆盖 `C1~C3`。
- `V4` 边界符合性：所有有效连接承接都必须满足连接规格、数据库类型、模式和载体状态边界。
- `V5` 最小必要性：移除任一 `B*` 后，连接规格承接、连接载体形成或连接载体释放至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
