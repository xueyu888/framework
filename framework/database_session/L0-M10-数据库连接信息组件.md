# 数据库连接信息组件:DatabaseConnectionSpec

@framework

组件目标：稳定承载一个数据库连接对象的描述信息。

## 1. 能力声明（Capability Statement）

- `C1` 连接对象描述能力：承载可唯一定位数据库对象的连接描述。
- `C2` 连接规格投影能力：把地址、配置标识和数据库类型稳定投影为统一连接规格。
- `C3` 非能力项：不负责连接建立，不负责驱动适配，不负责会话生成。

## 2. 边界定义（Boundary / 参数）

- `URL` 连接地址（database_url）：必须是可消费且能唯一定位数据库的连接地址。来源：`C1 + C2`。
- `PF` 配置标识（profile_name）：必须是非空配置名。来源：`C1 + C2`。
- `DK` 数据库类型（database_kind）：必须是非空数据库类型标识。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 地址记录单元（自有基）：数据库连接地址的承载结构，由地址值和地址位形构成。来源：`C1 + URL`。
- `B2` 配置标识单元（自有基）：数据库配置身份的承载结构，由配置名和值域约束构成。来源：`C1 + PF`。
- `B3` 类型标记单元（自有基）：数据库类型的承载结构，由类型标记和值域约束构成。来源：`C2 + DK`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 地址进入组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：所有连接对象描述都必须先形成合法地址记录；没有连接地址的描述不承认有效。
  - `R1.3` 输出结构：`ADDRESS_RECORD`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`URL`。
- `R2` 连接规格组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：连接规格只能由地址记录、配置标识和类型标记共同形成；缺任一结构都不承认是完整连接规格。
  - `R2.3` 输出结构：`CONNECTION_SPEC`。
  - `R2.4` 输出能力：`C1 + C2`。
  - `R2.5` 边界绑定：`URL/PF/DK`。
- `R3` 禁止组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把驱动适配、连接建立或会话生成直接混入连接规格的组合均无效。
  - `R3.3` 输出能力：`C3`。
  - `R3.4` 边界绑定：`URL/PF/DK`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R2` 输出能力并集必须覆盖 `C1~C2`。
- `V4` 边界符合性：所有有效规格都必须满足地址、配置标识和数据库类型边界。
- `V5` 最小必要性：移除任一 `B*` 后，连接对象描述或连接规格投影至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
