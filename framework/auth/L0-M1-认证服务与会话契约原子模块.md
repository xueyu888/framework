# 认证服务与会话契约原子模块:AuthServiceSessionContracts

@framework

本模块定义 auth 子域里偏后端承接面的最小结构语言：认证请求如何形成稳定契约、会话结果如何被稳定判定、已认证主体如何被统一表达、认证失败如何被结构化返回。它不定义登录页长什么样，也不定义具体产品要求使用用户名密码、验证码、SSO 还是某个身份提供商；这些分别属于更上层的场景语义与产品真相。本模块只负责把“后端认证服务应稳定提供什么叶子结构”沉成可复用的原子，供 auth 领域更高层编排复用。需要特别说明的是：本模块收敛于“认证是否成立、会话是否有效、当前主体是谁”这一闭环，不负责“已登录但是否有权访问某资源或执行某动作”的授权判定；授权应在后续独立模块中承接。本模块中的主体信息只作为认证结果输出，不在此定义资源、动作、策略与访问控制规则。


## 1. 能力声明（Capability Statement）

- `C1` 认证请求契约能力：把登录建立、会话探测和登出结束组织成稳定的请求契约。
- `C2` 会话状态表达能力：把已登录、未登录、已失效和已结束这些状态组织成稳定的会话语义。
- `C3` 已认证主体承载能力：把主体标识、主体类别和主体认证上下文组织成稳定的已认证主体表达。
- `C4` 失败结果结构化能力：把认证失败、会话失效、重新认证要求和服务拒绝组织成稳定错误结构。
- `C5` 非能力项：不负责具体协议、Header/Cookie/Token 名称、SDK、加密算法、存储介质、第三方身份服务选型，或把服务结果直接编排成前端页面与交互，也不负责资源、动作、策略与访问控制判定。

## 2. 边界定义（Boundary / 参数）

- `AUTHREQUEST` 认证请求边界：登录建立、会话探测、登出结束和刷新重验这些请求类型要稳定。来源：`C1 + C2`。
- `SESSIONSTATE` 会话状态边界：已登录、未登录、已失效、已结束和待重验这些状态值域要稳定。来源：`C1 + C2`。
<!--   说明：这一段回答“如果认证已经成功，系统现在到底认定是谁在使用当前会话”。它不是用户详情页模型，也不是权限模型，而是认证成功后最小、稳定、可复用的主体身份快照。上层拿到它之后，至少应该能回答‘是谁登录了’、‘通过什么方式完成认证’、‘当前认证强度到什么级别’。例如一个最小可理解示例可以是：主体标识为 `user_123`，主体类别为 `user`，认证来源为 `sso`，认证强度为 `mfa`。如果想进一步落成最小字段形态，可以理解成：`subject_id`、`subject_type`、`auth_source`、`auth_strength`，以及一个只承接最小展示或识别信息的 `subject_snapshot`。 -->
- `IDENTITYSUBJECT` 已认证主体边界：主体标识、主体类别、认证来源、认证强度与主体快照的表达方式要稳定。这里的主体快照只服务于“当前已认证主体是谁”的表达，不在本模块内展开为资源访问规则。来源：`C2 + C3`。
- `AUTHERROR` 认证错误边界：失败类别、失败原因、是否可重试和是否需要重新认证的表达方式要稳定。这里不承载独立授权模块中的访问拒绝语义，只负责认证链路内的失败与拒绝结果。来源：`C1 + C4`。
- `AUTHSERVICE` 服务承接边界：认证服务必须能被上层以统一契约调用，而不是依赖某个具体协议细节。它回答的是“服务如何被稳定调用”，不回答“前端如何展示这些结果”。来源：`C1 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

<!--   说明：`B1` 回答“系统到底收到了什么认证请求”。它不关心登录页怎么画，而关心后端收到的请求是否能被稳定识别，例如这是登录、探测会话、刷新重验还是登出；请求里带了什么输入、处在什么调用上下文，也都要能说清楚。没有这一基，上层就只能靠零散接口约定或临时字段猜当前在做什么认证动作。 -->
- `B1` 认证请求结构基：由请求类型、输入载荷和调用上下文构成的请求契约结构。来源：`C1 + AUTHREQUEST + AUTHSERVICE`。
<!--   说明：`B2` 回答“这次认证或会话检查之后，系统当前到底处于什么状态”。它要求后端不要只返回模糊的成功或失败，而要明确说明现在是已登录、未登录、已失效、已结束还是待重验。这样上层才能稳定决定要继续放行、提示重登，还是结束流程。 -->
- `B2` 会话结果结构基：由会话标识、当前状态、有效性判定和结束语义构成的会话结果结构。来源：`C1 + C2 + SESSIONSTATE + AUTHSERVICE`。
<!--   说明：`B3` 回答“如果已经认证成功，现在能稳定确认是谁”。它要求系统在会话成立后，能够导出统一的已认证主体信息，例如主体标识、主体类别、认证来源和认证强度。这里的重点不是访问控制，而是让上层不用再从 token、cookie 或第三方返回里各自拆解‘当前用户是谁’。你可以把它理解成‘认证成功后，系统给出的当前主体身份快照’。若用最小字段示例表达，可以是：`subject_id=user_123`、`subject_type=user`、`auth_source=sso`、`auth_strength=mfa`，再加一个 `subject_snapshot` 承接展示名或最小识别信息。 -->
- `B3` 已认证主体结构基：由主体标识、主体类别、认证来源、认证强度和主体快照构成的已认证主体结构。来源：`C2 + C3 + IDENTITYSUBJECT`。
<!--   说明：`B4` 回答“认证没走通时，失败要怎么被稳定表达”。它要求失败不能只剩一句模糊报错，而要说明失败类别、原因、能不能重试、是否需要重新认证。这样上层才能把‘账号密码错误’、‘会话过期’、‘暂时服务拒绝’这些情况分开处理，而不是到处写特判。 -->
- `B4` 认证错误结果结构基：由失败类别、失败原因、可重试标记和重新认证标记构成的错误结果结构。来源：`C1 + C4 + AUTHERROR + AUTHSERVICE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 认证请求必须回到稳定会话结果
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：将认证请求结构与会话结果结构串联后，每个认证请求都必须落到可判定的会话结果；因此登录建立、会话探测、刷新重验和登出结束都不得只返回模糊成功，而必须同时给出稳定的会话状态。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/AUTHSERVICE`。
- `R2` 已登录会话必须能承接已认证主体
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：将会话结果结构与已认证主体结构串联后，凡是可被判定为已登录或可继续放行的会话，都必须能导出稳定的当前主体身份快照；因此主体信息不得脱离会话结果孤立存在。这里导出的只是认证后的主体表达，不在本模块内继续推导访问控制规则。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`SESSIONSTATE/IDENTITYSUBJECT/AUTHSERVICE`。
- `R3` 认证失败必须可被结构化处理
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：将认证请求结构与认证错误结果结构串联后，请求失败、会话失效、服务拒绝和重新认证要求都必须导出可区分的失败结果；因此上层不得从自由文本里猜测失败类型。至于这些失败结果在前端落成登录提示、重新认证提示还是其他拒绝承接，不由本模块负责。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`AUTHREQUEST/AUTHERROR/AUTHSERVICE`。
- `R4` 完整认证服务闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：将认证请求、会话结果、已认证主体和失败结构统一到同一服务承接边界后，认证服务才能稳定支持“发起请求 -> 判定会话 -> 导出主体 -> 失败回传”闭环；缺失任一结构都会使上层编排退化为实现细节拼装。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/IDENTITYSUBJECT/AUTHERROR/AUTHSERVICE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：只有认证请求而没有会话结果结构，无法判断服务结果是否可继续放行；只有会话结果而没有已认证主体结构，已登录结果无法承接稳定主体；只有失败文本而没有失败类别结构，上层无法稳定区分重新认证、重试或服务拒绝；只有主体结构而没有请求契约结构，主体快照缺少可信来源。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/IDENTITYSUBJECT/AUTHERROR/AUTHSERVICE`。

## 5. 验证（Verification）

- `V1` 认证请求契约有效性：通过条件：登录建立、会话探测、刷新重验和登出结束都能被 `AUTHREQUEST` 稳定表达。失败原因：请求类型缺失、请求载荷不可判定或调用上下文混乱。证据：请求契约映射、服务调用样例与状态结果说明。
- `V2` 会话状态有效性：通过条件：`SESSIONSTATE` 能稳定覆盖已登录、未登录、已失效、已结束和待重验。失败原因：关键状态缺失、状态切换不可解释或有效性判定缺失。证据：状态结构映射、状态图与运行样例。
- `V3` 已认证主体有效性：通过条件：已登录会话能导出 `IDENTITYSUBJECT` 对应的主体标识、主体类别、认证来源和认证强度。失败原因：主体快照缺失、主体来源不明或认证上下文表达不稳定。证据：主体结构映射、服务返回样例与主体导出样例。
- `V4` 失败结构有效性：通过条件：`AUTHERROR` 能稳定区分认证失败、会话失效、需要重新认证和服务拒绝。失败原因：失败类别混淆、失败结果只剩自由文本或无法判断是否需要重试/重新认证。证据：错误结构映射、失败样例与处理结论。这里验证的是错误结果能否被稳定表达，不验证前端场景如何呈现，也不验证独立授权场景如何呈现。
- `V5` 服务闭环必要性：通过条件：保留全部 `B*` 时，`V1~V4` 可同时成立。失败原因：移除任一 `B*` 后，认证请求、会话结果、已认证主体或失败结构至少一项失效。证据：`B1~B4` 到 `R1~R5` 的组合映射与校验结论。
- `V6` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
