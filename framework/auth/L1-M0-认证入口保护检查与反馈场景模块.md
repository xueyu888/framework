# 认证入口保护检查与反馈场景模块:AuthEntryGuardCheckFeedbackScenarioModule

@framework

本模块是跨项目复用的认证与会话场景标准模块。它只定义一类通用结构语言：需要登录的页面怎么先拦住，用户怎么进入登录入口，登录成功后怎么恢复到原页面或转到默认页面，失败或失效时前端怎么反馈，整条会话流程怎么闭环。项目实例应把本模块编译成明确的 auth contract、session contract 和验证结论，而不是在具体项目代码或生成器里临时拼装这些规则。本模块在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中只负责认证与会话的共同结构、边界、场景基和组合规则；具体采用用户名密码、邮箱密码还是验证码登录，属于 `Product Spec`；具体采用 header、cookie、token 还是第三方身份服务，属于 `Implementation Config`。认证入口与返回恢复壳编排沉在 `auth.L0.M0`，偏后端承接面的认证服务与会话契约沉在 `auth.L0.M1`，本模块则负责把这两类 L0 原子组合成可讨论、可约束的场景语义。本模块聚焦“是否已完成认证”，不负责“已登录但无权限、角色不足、租户未开通”这类访问拒绝场景；后者应由更高一层权限标准单独承接。


## 1. 能力声明（Capability Statement）

- `C1` 认证入口承接能力：支持在受保护页面进入前，稳定展示登录入口或认证请求触发点。
- `C2` 会话完整流程能力：支持会话建立、会话检查、会话结束，以及基于会话结果决定是否继续访问。
- `C3` 返回恢复能力：支持登录成功后恢复到原目标页，或者转到默认页面。
- `C4` 反馈能力：支持登录失败、会话失效、取消登录或重新认证时给出稳定反馈。
- `C5` 非能力项：不负责具体字段文案、UI 样式、接口路径、SDK 选型、存储介质、加密实现和权限不足判定。

## 2. 边界定义（Boundary / 参数）

- `AUTHENTRYFLOW` 认证入口流边界：定义登录入口是否承载账号输入、授权触发、提交区和辅助动作区。来源：`C1`。合法入口变体包括：`username_password`、`email_password`、`phone_code`、`passwordless_code`、`sso_redirect`、`oauth_button`、`magic_link`。

  - `username_password`：用户输入用户名和密码后直接提交登录。这是最常见的账号口令入口。
  - `email_password`：用户输入邮箱和密码后直接提交登录。它和 `username_password` 的区别主要在于身份标识改成邮箱。
  - `phone_code`：用户输入手机号，并通过短信验证码完成登录。主验证因子不再是固定密码，而是一次性手机验证码。

  - `sso_redirect`：用户从当前产品跳转到统一身份平台或企业身份系统，完成认证后再返回当前产品。当前入口本身主要承担跳转触发与返回恢复承接。
  - `oauth_button`：用户点击“使用某第三方账号登录”这类按钮后，跳转到外部身份提供方完成认证，再回到当前产品。它通常强调第三方身份按钮，而不是本地输入表单。
  - `magic_link`：用户输入邮箱等可接收链接的身份标识，系统发送一次性登录链接；用户点击链接后直接完成登录，无需输入密码。

<!--   说明：这一段回答“系统怎么判断用户现在算不算已经登录，以及这个登录状态什么时候建立、失效和结束”。 -->
- `SESSIONFLOW` 会话流边界：定义认证请求如何接入会话结果、会话结果如何决定能不能继续访问，以及失效后如何重新进入。来源：`C2`。
<!--   说明：这一段回答“登录成功后把用户送回哪里，是回原页面，还是去默认页面”。 -->
- `RETURNFLOW` 返回恢复边界：定义认证成功后恢复原页面、转到默认页面和恢复原上下文的语义。来源：`C3`。
<!--   说明：这一段回答“登录过程中出了什么情况，要怎么把信息明确告诉用户”。这里最好把反馈分成两类：一类是入口内反馈，例如账号密码错误、验证码错误、提交失败；另一类是场景级反馈，例如会话已过期、登录被取消、需要重新认证。前者跟着登录入口走，后者跟着整个认证流程走。 -->
- `FEEDBACKFLOW` 反馈边界：定义登录失败、会话失效、取消登录和重新认证时用户能看到什么反馈。反馈至少应区分 `entry_feedback` 与 `scenario_feedback` 两类。来源：`C4`。
<!--   说明：这一段回答“哪些页面必须先查登录，不满足条件时要先拦住并带去登录入口”。除了页面或路由进入前的保护，也要覆盖动作触发前保护，以及运行中会话失效后的切回入口。否则场景会只覆盖首屏进入，覆盖不到真实运行过程。 -->
- `GUARDFLOW` 保护检查边界：定义哪些页面是受保护目标、什么时候必须先登录以及检查失败后如何切回入口。它至少应能覆盖 `page_route`、`action_trigger` 和 `runtime_reauth` 三类保护检查场景。来源：`C2 + C3`。

- `B2` 保护检查结构基：L0.M0[R1,R3] + L0.M1[R1,R2]。来源：`C2 + C3 + SESSIONFLOW + RETURNFLOW + GUARDFLOW`。

- `B3` 反馈承接结构基：L0.M0[R3] + L0.M1[R2,R4]。来源：`C2 + C4 + SESSIONFLOW + FEEDBACKFLOW`。
## 4. 基组合原则（Base Combination Principles）

- `R1` 受保护页面进入前先做认证检查
  - `R1.1` 参与基：`B2`。
  - `R1.2` 组合方式：将受保护目标场景、会话结果场景和返回恢复语义场景绑定到同一承接结构后，受保护页面、路由和动作在进入前必须先被保护壳包裹，再由会话结果决定是否继续进入；当判定不通过时，当前流程不得继续，而必须切换到登录入口与返回恢复语义所定义的承接回路。
  - `R1.3` 输出能力：`C2 + C3`。
  - `R1.4` 输出结构：`GUARDFLOW + RETURNFLOW + SESSIONFLOW`。
  - `R1.5` 边界绑定：`GUARDFLOW/SESSIONFLOW/RETURNFLOW`。
- `R2` 登录入口必须接上完整流程
  - `R2.1` 参与基：`B1`。
  - `R2.2` 组合方式：将认证入口场景绑定到已接入完整回路的入口承接结构后，认证入口的输入、触发和提交出口必须同时接上认证请求发起、入口内失败反馈和认证成功后的返回恢复链路；因此登录入口不能只是静态表单或孤立按钮。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 输出结构：`AUTHENTRYFLOW + SESSIONFLOW + RETURNFLOW`。
  - `R2.5` 边界绑定：`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW`。
- `R3` 认证成功后优先回原目标
  - `R3.1` 参与基：`B2`。
  - `R3.2` 组合方式：将会话结果场景、受保护目标场景和返回恢复语义场景绑定到同一承接结构后，认证成功后的出口必须优先恢复原目标；当原目标不可恢复时，流程必须切换到默认页面语义所定义的稳定页面。
  - `R3.3` 输出能力：`C3`。
  - `R3.4` 输出结构：`RETURNFLOW + GUARDFLOW + SESSIONFLOW`。
  - `R3.5` 边界绑定：`RETURNFLOW/GUARDFLOW/SESSIONFLOW`。
- `R4` 登录失败和会话失效都要提示
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：将认证入口接入结构、保护检查结构和反馈承接结构串联后，登录失败、会话失效和取消登录的状态切换必须沿既有入口、检查和返回恢复回路导出场景级用户可观察反馈；其中入口内失败反馈由认证入口接入结构承接，流程中的会话失效、取消登录和重新认证反馈由反馈承接结构稳定导出。因此反馈内容必须和当前会话结果、主体状态与重新认证要求保持一致，而不能脱离既有结构单独定义。更直白地说：表单报错走“入口内反馈”，流程报错走“场景级反馈”，两者不能混写成一团。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 输出结构：`FEEDBACKFLOW + AUTHENTRYFLOW + SESSIONFLOW`。
  - `R4.5` 边界绑定：`FEEDBACKFLOW/AUTHENTRYFLOW/SESSIONFLOW`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：以下方案无效：缺少 `B1` 时，认证入口场景无法形成稳定的输入、触发和提交承接；缺少 `B2` 时，受保护目标判定、会话检查和返回恢复无法形成稳定回路；缺少 `B3` 时，认证失败、会话失效和重新认证要求无法稳定落到主体与反馈语义；只有入口与检查结构但缺少与之绑定的反馈承接结构时，认证失败、会话失效和取消登录会退化为不可观察状态切换。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 输出结构：`AUTHENTRYFLOW + SESSIONFLOW + RETURNFLOW + FEEDBACKFLOW + GUARDFLOW`。
  - `R5.5` 边界绑定：`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW/FEEDBACKFLOW/GUARDFLOW`。

## 5. 验证（Verification）

- `V1` 保护检查语义验证：通过条件：所有需要登录后才能访问的页面或动作，都能落入 `GUARDFLOW` 的受保护目标解释，并在进入前被稳定包裹到保护壳与检查判定语义中。失败原因：受保护目标缺失、进入前包裹不清、检查条件不明或保护范围与返回恢复语义冲突。证据：`GUARDFLOW` 对应的产品 section、规则映射与页面进入样例。
- `V2` 会话流语义验证：通过条件：`SESSIONFLOW` 能解释认证请求如何接入会话结果，以及已登录、未登录、已失效和结束后的状态切换。失败原因：关键状态缺失、状态切换不可判定或结束语义无法落点。证据：`SESSIONFLOW` 对应的产品 section、状态图和运行样例。
- `V3` 返回恢复语义验证：通过条件：`RETURNFLOW` 能解释“恢复原目标”和“转到默认页面”两类结果，且切换条件清晰。失败原因：返回目标记录缺失、默认页面未定义或两类返回条件互相冲突。证据：`RETURNFLOW` 对应的产品 section、路由回路说明与运行样例。
- `V4` 反馈语义验证：通过条件：`FEEDBACKFLOW` 能同时覆盖入口内失败反馈，以及会话失效、取消登录和重新认证这类场景级用户可见反馈，且反馈语义与当前状态一致。失败原因：反馈类别缺失、反馈不可观察、入口内反馈与场景级反馈职责混乱，或反馈内容与状态不一致。证据：`FEEDBACKFLOW` 对应的产品 section、提示样例与运行结果。
- `V5` 入口流语义验证：通过条件：`AUTHENTRYFLOW` 的入口变体属于 `username_password`、`email_password`、`phone_code`、`passwordless_code`、`sso_redirect`、`oauth_button`、`magic_link` 之一，且输入区、触发区、提交区和辅助动作区职责清晰。失败原因：入口变体超出合法值域、入口职责混乱或辅助动作不可解释。证据：`AUTHENTRYFLOW` 对应的产品 section、入口结构样例与字段映射。
- `V6` 场景最小必要性验证：通过条件：保留全部 `B*` 时，`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW/FEEDBACKFLOW/GUARDFLOW` 可被完整解释，其中入口内反馈与场景级反馈都能找到稳定承接。失败原因：移除任一 `B*` 后，至少一个场景边界失去承接结构，或入口内反馈、主体状态反馈与检查回路之间出现脱节。证据：`B1~B3` 到 `R1~R5` 的组合映射与校验结论。
