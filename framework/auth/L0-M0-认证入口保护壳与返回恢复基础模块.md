# 认证入口保护壳与返回恢复基础模块:AuthEntryProtectedShellReturnRecoveryBase

@framework

本模块定义 auth 子域里最小的一层：前端怎样把“登录入口、访问前拦截、登录后返回原页”这三件事接起来。认证场景里会用到的输入框、触发器、容器、提示和页面跳转，frontend 已经提供了，所以 auth 不再重复造原子，只在这一层说明四件事：登录入口怎么组织、需要登录的目标怎么先拦住、登录成功后怎么回原页面、失败后前端怎么给出可观察提示。本模块不直接定义具体业务页面清单、认证服务契约或项目实例值，只负责把前端基础原子组织成认证流程的前端编排结构；如果要再接上认证请求、会话结果和错误结果这些服务契约，则由更上层场景模块组合 `L0.M0` 与 `L0.M1`。本模块也不负责“已登录但无权限”的场景，那属于后续更高一层的权限或访问控制标准。


## 1. 能力声明（Capability Statement）

- `C1` 认证入口组织能力：把输入、授权触发和提交组织成一个稳定的登录入口。
- `C2` 保护检查编排能力：把访问前拦截、运行中失效占位和检查通过前包裹组织成一个稳定流程。
- `C3` 返回原页面能力：把“记住用户原本要去哪里、登录成功后尽量回去、回不去时走默认页面”组织成稳定流程。
- `C4` 反馈能力：把登录失败、会话失效、取消登录和重新认证时的反馈组织成稳定流程。
- `C5` 非能力项：不负责具体受保护页面清单、项目文案、接口路径、权限判定与前后端技术选型。

## 2. 边界定义（Boundary / 参数）

<!--   说明：这一段回答“登录入口这层页面结构怎么摆，用户先看到什么、后做什么”。 -->
- `AUTHENTRYSHELL` 认证入口壳边界：登录页里哪些区域先出现、用户从哪里触发登录、提交动作怎么组织，这些顺序要稳定。来源：`C1`。
<!--   说明：这一段回答“哪些东西算受保护目标，进这些地方之前要不要先过登录检查”。这里最好把受保护目标至少分成三类来看：页面或路由进入前的保护、某个动作触发前的保护，以及运行中会话失效后对当前目标的重新认证切回。这样讨论时就不会只盯着‘页面要不要登录’，而漏掉按钮动作或运行中失效。 -->
- `PROTECTEDTARGET` 受保护目标边界：哪些页面、路由、动作或资源进入前必须先检查登录，哪些目标允许匿名访问，以及检查不通过后转去哪里，这些规则要稳定。受保护目标至少应能覆盖 `page_route`、`action_trigger` 和 `runtime_reauth` 三类。来源：`C1 + C2 + C3`。
<!--   说明：这一段回答“系统用什么外层结构把受保护页面先拦住、检查，再决定要不要继续进入”。 -->
- `PROTECTEDSHELL` 保护壳边界：访问前拦截、失效占位和检查通过前包裹目标的顺序要稳定。来源：`C2`。
<!--   说明：这一段回答“登录后回哪里，这个返回动作靠什么外层结构承住”。 -->
- `RETURNSHELL` 返回恢复壳边界：怎么记录原目标、什么时候恢复原目标、什么时候转到默认页面，这套顺序要稳定。来源：`C3`。
<!--   说明：这一段回答“提示信息放在哪里、怎么出现，用户能不能稳定看到”。 -->
- `FEEDBACKSHELL` 反馈壳边界：登录失败、会话失效、取消登录和重新认证时，前端用什么方式把反馈展示出来，要保持稳定。来源：`C4`。

## 3. 最小可行基（Minimum Viable Bases）

<!--   说明：`frontend.L1.M1` 负责承接输入与部分提交字段，`frontend.L1.M2` 负责认证入口所需的容器与区块组织，`frontend.L1.M4` 负责失败反馈、状态标记和可观察反馈；若入口采用跳转或按钮授权，触发动作可由更基础的 frontend 原子承接后再并入本编排。 -->
- `B1` 认证入口基础结构：frontend.L1.M1[R1,R2] + frontend.L1.M2[R2] + frontend.L1.M4[R1,R2]。来源：`C1 + C4 + AUTHENTRYSHELL + FEEDBACKSHELL`。
<!--   说明：`frontend.L3.M0` 负责页面进入、路由跳转和目标恢复，是认证编排里返回原页面与默认回退的前端承接面。这里的默认回退可以直白理解成：原目标无法恢复时，不再停在不确定状态，而是回到一个稳定的默认页，例如工作台首页、个人首页或项目首页。 -->
- `B2` 返回恢复基础结构：frontend.L3.M0[R1,R2]。来源：`C3 + RETURNSHELL`。
<!--   说明：`所有需要登录才能进入的目标，都不能直接暴露出来，而要先包进一个前端保护壳里，由这层壳决定是继续等待检查结果、显示占位，还是切去后续流程；它只定义“先包裹目标、再交给更上层决定是否继续进入”的保护壳结构，本身不承担会话结果判定。借助了frontend.L3.M0页面运行和编排能力 -->
- `B3` 受保护目标保护壳结构：frontend.L3.M0[R1]。来源：`C1 + C2 + C3 + PROTECTEDTARGET + PROTECTEDSHELL`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 所有受保护目标先做认证检查
  - `R1.1` 参与基：`B2 + B3`。
  - `R1.2` 组合方式：将受保护目标保护壳结构与返回恢复结构串联后，页面、路由和动作在进入前必须先经过认证检查，并预留统一返回恢复承接；因此所有受保护目标都不得绕过保护壳直接暴露到主流程。
  - `R1.3` 输出能力：`C2 + C3`。
  - `R1.4` 边界绑定：`PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL`。
- `R2` 认证入口必须接上完整回路
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将认证入口结构与返回恢复结构串联后，认证入口的提交、辅助动作和授权触发都必须接入稳定返回恢复承接；因此认证入口不能只是一个静态表单或孤立按钮，而必须真的接到后续流程。
  - `R2.3` 输出能力：`C1 + C3`。
  - `R2.4` 边界绑定：`AUTHENTRYSHELL/RETURNSHELL/FEEDBACKSHELL`。
- `R3` 失效后要能重新回到入口
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将受保护目标保护壳、认证入口结构和返回恢复结构串联后，当前目标一旦需要重新认证，流程必须沿既有保护壳回路切换到认证入口，并保持原目标恢复承接；因此失效反馈不能脱离保护壳与返回恢复结构单独出现。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`AUTHENTRYSHELL/PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL/FEEDBACKSHELL`。
- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：只有认证入口结构但没有返回恢复结构，登录后不知道该回哪里；只有保护壳结构但没有返回恢复结构，受保护目标失去统一恢复出口；只有返回恢复结构但没有认证入口结构，重新进入流程时没有稳定入口；只有入口与返回恢复结构但没有保护壳结构，受保护目标在进入前无法被统一检查。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`AUTHENTRYSHELL/PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL/FEEDBACKSHELL`。

## 5. 验证（Verification）

- `V1` 认证入口有效性：通过条件：登录入口能稳定承接输入、授权触发、提交和失败反馈。失败原因：缺少输入承接、触发出口、提交出口或失败反馈任一结构。证据：`AUTHENTRYSHELL/FEEDBACKSHELL` 对应的结构映射、生成页面与交互样例。
- `V2` 保护壳有效性：通过条件：所有受保护页面、路由或动作进入前都能先落入稳定保护壳。失败原因：受保护目标未被承接、进入前缺少保护壳或保护壳与返回恢复承接脱节。证据：`PROTECTEDTARGET/PROTECTEDSHELL` 对应的结构映射、页面进入样例与运行结果。
- `V3` 返回恢复有效性：通过条件：登录成功后能恢复到原目标，或转到默认页面。失败原因：原目标未记录、返回恢复链路缺失或默认页面不可解释。证据：`RETURNSHELL` 对应的结构映射、路由恢复样例与运行结果。
- `V4` 编排闭环有效性：通过条件：登录入口、访问前检查和返回恢复结构能形成稳定的前端闭环。失败原因：入口、保护壳和返回恢复三者之间至少有一处无法串联。证据：`AUTHENTRYSHELL/PROTECTEDTARGET/RETURNSHELL/FEEDBACKSHELL` 对应的结构映射与交互样例。
- `V5` 编排必要性：通过条件：保留全部 `B*` 时，`V1~V4` 可同时成立。失败原因：移除任一 `B*` 后，认证入口、保护壳、返回恢复或前端编排闭环至少一项失效。证据：`B1~B3` 到 `R1~R4` 的组合映射与校验结论。
- `V6` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
