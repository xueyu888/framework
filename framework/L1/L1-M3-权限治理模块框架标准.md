# 权限治理模块:AuthorizationGovernanceModule

@framework

## 1. 能力声明（Capability Statement）

- `C1` 主体资源解析能力：提供主体、动作、资源与上下文的统一解析能力。
- `C2` 鉴权判定能力：提供允许、拒绝与原因输出能力。
- `C3` 范围治理能力：提供资源范围、可见范围与动作范围的一致性约束能力。
- `C4` 决策输出能力：提供下游模块可消费的标准化鉴权结果能力。
- `C5` 非能力项：不负责业务文件接收、状态推进与追溯记录写入。

## 2. 边界定义（Boundary / 参数）

- `SUBJECT` 主体边界：调用主体标识与来源必须明确。来源：`C1 + C2`。
- `ACTION` 动作边界：可判定动作集合必须显式定义。来源：`C1 + C2`。
- `RESOURCE` 资源边界：资源标识与资源范围必须可解析。来源：`C1 + C3`。
- `CONTEXT` 上下文边界：判定依赖的请求上下文必须显式输入。来源：`C1 + C2 + C3`。
- `DECISION` 决策边界：鉴权结果必须稳定输出 `allow/deny` 与原因。来源：`C2 + C4`。
- `SCOPE` 范围边界：同一资源在不同范围下的可见与可操作语义必须一致。来源：`C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 鉴权输入解析模块：L0.M0[R1,R3]。来源：`C1 + SUBJECT + ACTION + RESOURCE + CONTEXT`。
- `B2` 鉴权决策生成模块：L0.M0[R3]。来源：`C2 + DECISION + ACTION + RESOURCE + CONTEXT`。
- `B3` 权限范围治理模块：L0.M0[R3,R4]。来源：`C3 + C4 + RESOURCE + SCOPE + DECISION`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 判定输入组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：通过鉴权输入解析模块统一解析主体、动作、资源与上下文，形成稳定鉴权输入。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SUBJECT/ACTION/RESOURCE/CONTEXT`。
- `R2` 决策生成组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：基于统一输入通过鉴权决策生成模块输出允许或拒绝决策，并附带原因说明。
  - `R2.3` 输出能力：`C1 + C2 + C4`。
  - `R2.4` 边界绑定：`ACTION/RESOURCE/CONTEXT/DECISION`。
- `R3` 范围治理组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将资源范围与可见范围约束收敛到统一权限范围治理模块，供功能模块复用。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`RESOURCE/SCOPE/DECISION/CONTEXT`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：缺失主体、动作、资源、上下文或无法输出稳定决策的权限治理方案无效。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`SUBJECT/ACTION/RESOURCE/CONTEXT/DECISION/SCOPE`。

## 5. 验证（Verification）

- `V1` 输入解析完整性：鉴权输入必须满足 `SUBJECT/ACTION/RESOURCE/CONTEXT` 约束。
- `V2` 决策稳定性：同一输入下输出决策必须满足 `DECISION` 一致性约束。
- `V3` 范围一致性：资源范围与动作范围必须满足 `SCOPE` 约束。
- `V4` 输出可消费性：鉴权结果必须可被功能模块直接消费。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
