# 测试契约转化节点基模块:NodeCatalog

@framework

## 0.目的
把输入文件物化为节点目录，目录中每个 value 满足 L0

## 1. 能力声明（Capability Statement）

- `C1` 节点存储能力：定义能存储测试节点定义的能力。
- `C2` 节点查询能力：定义能查询测试节点定义的能力。
- `C3` 契约转换能力：定义能将api契约文件转换为可被测试节点使用的格式的能力。
- `C4` 人工修订能力：人工对测试节点定义进行修正的能力。
- `C5` 来源追踪能力：定义能追踪测试节点定义来源的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责保存运行时状态资源，不负责决定怎么测试怎么进行，不负责运行节点。

## 2. 边界定义（Boundary / 参数）

- `P1` 契约输入边界：静态，从待测应用导出的唯一可信api契约文件。来源：`C1+C3`。
- `P2` 补丁输入边界：静态，导入补丁文件，用于修正或补充契约定义。来源：`C4 + C5`。
- `P3` 节点集输出边界：动态，可读取的节点定义集合。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 节点解析映射基：解析待测程序产生的api契约文件和补丁文件，将其转换为可被测试节点使用的格式，解析规则见文档。来源：`C1 + P1`。
- `B2` 配置块基：auto_test.L0.M1[R2]。来源：`C2 + P2`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 结构装配组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先固定结构接口，再收敛状态通路与装配关系。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`P1 + P2 + P3`。
- `R2` 上层展开组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：在装配结果上叠加扩展约束，形成可继续生长的模块结构。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`P1 + P2 + P3`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：脱离下层结构直接按功能清单拼接，或违反边界约束的组合均无效。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`P1 + P2 + P3`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基/组合方式/输出能力或失效结论/边界绑定。
- `V3` 结构闭环性：`R1~R2` 输出能力并集必须覆盖 `C1~C3`。
- `V4` 边界符合性：所有有效组合必须满足绑定的参数边界。
- `V5` 最小必要性：移除任一 `B*` 后，`V3` 或 `V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 边界绑定` 或 `R* -> N* + 边界绑定` 的通过或失败结论。
