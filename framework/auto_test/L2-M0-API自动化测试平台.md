#  API自动化测试平台:EnglishName

@framework

## 0.目的
一个以契约为中心的 API 自动化及编排测试平台

## 1. 能力声明（Capability Statement）

- `C1` 契约与状态规则归一化能力：从 OpenAPI 契约和补充文件中得到统一的 API 测试模型、检查要求与 stateful 关系定义。
- `C2` 单接口模糊测试能力：基于契约生成边界、等价类、异常输入、模糊输入，并执行单接口探索。
- `C3` 自动场景探索能力：基于 stateful 关系定义自动形成多步 API 调用场景，减少人工编排成本。
- `C4` 显式场景编排能力：对无法由契约自动推导的业务场景，使用 pytest-bdd 进行显式组织与复用。
- `C5` 统一 HTTP 执行能力：通过统一的 httpx 运行层执行各类测试请求，并承接认证、上下文和请求响应采集。
- `C6` 统一结果沉淀与可视化能力：将各组件的执行结果、证据、日志、附件统一沉淀到 Allure 展示和记录。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责项目管理、差异对比。
- `N2` 非职责声明：不负责脱离契约与补充规则的纯业务推理式自动场景生成。

## 2. 边界定义（Boundary / Parameter 参数）

- `P1` 契约与补充规则文件：OpenAPI 契约文件及其补充文件；补充文件中可承载检查要求、stateful 关系定义、前后置条件、数据提取规则。来源：`C1 + C2 + C3`。
- `P2` 运行上下文：base_url、认证信息、headers、cookies、环境配置、共享变量、会话状态。来源：`C5`。
- `P3` 显式场景编排文件：pytest-bdd 对应的 feature / step 文件及补充配置。来源：`C4`。
- `P4` 报告输出配置：Allure 结果目录、附件策略、报告入口。来源：`C6`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 结构装配组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先固定结构接口，再收敛状态通路与装配关系。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`P1 + P2 + P3`。
- `R2` 上层展开组合
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：在装配结果上叠加扩展约束，形成可继续生长的模块结构。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`P1 + P2 + P3`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：脱离下层结构直接按功能清单拼接，或违反边界约束的组合均无效。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`P1 + P2 + P3`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基/组合方式/输出能力或失效结论/边界绑定。
- `V3` 结构闭环性：`R1~R2` 输出能力并集必须覆盖 `C1~C3`。
- `V4` 边界符合性：所有有效组合必须满足绑定的参数边界。
- `V5` 最小必要性：移除任一 `B*` 后，`V3` 或 `V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 边界绑定` 或 `R* -> N* + 边界绑定` 的通过或失败结论。
