# 令牌与放行判定编排模块:TokenAllowanceDecisionOrchestration

@framework

本模块把令牌事实和角色权限关系接成一条放行判断线。它只负责 4 件事：令牌怎么成立、令牌怎么校验、角色权限关系怎么取到、最终怎么输出允许或拒绝；不负责密码校验和失败锁定。

## 1. 能力声明（Capability Statement）

- `C1` 角色权限判定能力：根据角色集合、权限集合和映射关系输出放行依据。
- `C2` 令牌签发校验能力：根据主体和令牌事实完成签发、校验与失效判断。
- `C3` 放行判断能力：根据令牌事实和角色权限关系输出允许或拒绝结论。
- `C4` 非能力项：不负责密码校验、失败锁定和安全记账。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 主体绑定边界：任一令牌事实都必须绑定到一个明确主体。来源：`C2 + C3`。
- `ATOKTTL` 访问令牌时长边界：访问令牌有效时长和签发时点必须显式声明。来源：`C2 + C3`。
- `RTOKTTL` 刷新令牌时长边界：刷新令牌有效时长必须显式声明，且必须长于访问令牌。来源：`C2 + C3`。
- `ROLESET` 角色集合边界：角色集合、权限集合和映射关系只能来自声明过的集合。来源：`C1 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 令牌签发校验器：L0.M0[R1] + L0.M1[R1]：把已定位主体和访问令牌、刷新令牌事实接到一起，能完成访问令牌与刷新令牌的签发、校验和失效判断。来源：`C2 + C3 + USERKEY + ATOKTTL + RTOKTTL`。
- `B2` 角色-权限映射表：L0.M1[R2]：存放角色集合、权限集合和角色到权限映射，能根据主体声明输出当前放行依据。来源：`C1 + C3 + ROLESET`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 角色权限映射成立
  - `R1.1` 参与基：`B2`。
  - `R1.2` 组合方式：先固定角色集合、权限集合和映射关系，形成稳定放行依据。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`ROLESET`。
- `R2` 令牌签发校验成立
  - `R2.1` 参与基：`B1`。
  - `R2.2` 组合方式：先把主体绑定事实和令牌寿命事实接成一条完整令牌校验结果。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL`。
- `R3` 放行判断成立
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：在可校验令牌结果上补齐角色权限映射，形成最终允许或拒绝判断。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL/ROLESET`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2`。
  - `R4.2` 组合方式：令牌未绑定主体、令牌已失效或角色权限关系缺失时仍继续放行的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL/ROLESET`。

## 5. 验证（Verification）

- `V1` 角色权限映射有效性：角色-权限映射表必须满足 `ROLESET` 约束。
- `V2` 令牌签发校验有效性：令牌签发校验器必须满足 `USERKEY/ATOKTTL/RTOKTTL` 约束。
- `V3` 放行判断有效性：同一主体必须在同一角色权限映射下输出稳定允许或拒绝结论。
- `V4` 最小必要性：移除任一 `B*` 后，`V2~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
