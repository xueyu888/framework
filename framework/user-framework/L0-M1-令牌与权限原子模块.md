# 令牌与权限原子模块:TokenPermissionAtoms

@framework

本模块定义用户框架里最底层的令牌与权限部件。它只回答 4 件事：访问令牌怎么放、刷新令牌怎么放、令牌寿命和撤销怎么放、角色权限映射怎么放；不负责用户凭据校验和失败锁定。

## 1. 能力声明（Capability Statement）

- `C1` 令牌主体定义能力：定义访问令牌和刷新令牌怎样稳定绑定到一个主体。
- `C2` 令牌寿命定义能力：定义过期、轮换与撤销事实怎样被稳定表达。
- `C3` 权限映射定义能力：定义角色集合、权限集合与角色到权限关系怎样被稳定表达。
- `C4` 非能力项：不负责密码校验、失败锁定和安全事件记账。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 主体绑定边界：任一访问令牌和刷新令牌都必须绑定到一个明确主体。来源：`C1`。
- `ATOKTTL` 访问令牌时长边界：访问令牌有效时长和签发时点必须显式声明。来源：`C1 + C2`。
- `RTOKTTL` 刷新令牌时长边界：刷新令牌有效时长必须显式声明，且必须长于访问令牌。来源：`C1 + C2`。
- `ROLESET` 角色集合边界：角色集合、权限集合和映射关系只能来自声明过的集合。来源：`C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 访问令牌位基：存放主体绑定、签发时点和过期时点，能表达一个访问令牌当前是否有效。来源：`C1 + ATOKTTL + USERKEY`。
- `B2` 刷新令牌位基：存放家族标识、轮换关系、撤销标记和过期时点，能表达一个刷新令牌当前是否还能续签。来源：`C1 + C2 + RTOKTTL + ATOKTTL`。
- `B3` 角色权限位基：存放角色集合、权限集合和映射关系，能表达角色到权限的放行依据。来源：`C3 + ROLESET`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 令牌事实成立
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先把访问令牌位和刷新令牌位挂到同一主体上，形成完整令牌事实。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL`。
- `R2` 权限映射成立
  - `R2.1` 参与基：`B3`。
  - `R2.2` 组合方式：先固定角色位、权限位和映射位，形成稳定角色权限关系。
  - `R2.3` 输出能力：`C3`。
  - `R2.4` 边界绑定：`ROLESET`。
- `R3` 放行事实成立
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把可用令牌事实和角色权限关系接到同一主体上，形成可用于上层放行判断的完整事实。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL/ROLESET`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：令牌未绑定主体、刷新令牌不长于访问令牌或角色权限映射缺失的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL/ROLESET`。

## 5. 验证（Verification）

- `V1` 令牌事实有效性：访问令牌位和刷新令牌位必须满足 `USERKEY/ATOKTTL/RTOKTTL` 约束。
- `V2` 权限映射有效性：角色位、权限位和映射位必须满足 `ROLESET` 约束。
- `V3` 放行事实有效性：同一主体必须能同时取到可用令牌事实和角色权限关系。
- `V4` 最小必要性：移除任一 `B*` 后，`V2~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
