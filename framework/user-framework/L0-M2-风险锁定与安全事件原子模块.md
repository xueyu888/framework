# 风险锁定与安全事件原子模块:RiskLockingSecurityEventAtoms

@framework

本模块定义用户框架里最底层的失败锁定和安全事件部件。它只回答 4 件事：连续失败次数怎么放、锁定状态怎么放、关键安全事件怎么放、当前安全状态怎么读；不负责密码证明和权限映射。

## 1. 能力声明（Capability Statement）

- `C1` 连续失败次数感知能力：存放一个用户当前连续失败次数和最近一次失败时间。
- `C2` 锁定状态能力：存放一个用户是否锁定、锁定从何时开始、到何时结束以及是否已解锁。
- `C3` 安全事件能力：存放登录失败、锁定、解锁、签发、续签、撤销和角色变更这类安全事件。
- `C4` 当前安全状态获取能力：根据失败次数、锁定状态和最近一次安全事件，读出一个用户当前安全状态和最近一次关键变更。
- `C5` 非能力项：不负责密码证明、令牌签发和角色权限关系定义。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 归属边界：任一失败次数、锁定状态和安全事件都必须归到一个明确用户。来源：`C1 + C2 + C3 + C4`。
- `FAILMAX` 登录失败次数边界：连续失败阈值必须显式声明，达到阈值后必须能推导为锁定。来源：`C1 + C4`。
- `LOCKTIME` 锁定时长边界：锁定开始条件、锁定持续时长和解锁落点必须显式声明。来源：`C2 + C4`。
- `EVENTTYPE` 事件类型边界：登录失败、锁定、解锁、签发、续签、撤销与角色变更只能落到声明过的事件类型中。来源：`C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 连续失败计数槽：存放当前连续失败次数和最近一次失败时间，能明确表示该用户离锁定阈值还差几次。来源：`C1 + FAILMAX`。
- `B2` 锁定状态槽：存放锁定开始时间、锁定截止时间和解锁标记，能明确表示该用户现在是否锁定、锁到什么时候、是否已经解锁。来源：`C2 + LOCKTIME`。
- `B3` 安全事件条目：存放事件类型、发生时间和归属用户，能明确表示一次登录失败、锁定、解锁、签发、续签、撤销或角色变更事实。来源：`C3 + USERKEY + EVENTTYPE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 失败锁定成立
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：把连续失败计数和锁定状态挂到同一用户上，形成当前失败锁定结果。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME`。
- `R2` 安全事件记录成立
  - `R2.1` 参与基：`B3`。
  - `R2.2` 组合方式：把事件类型、发生时间和归属用户固定下来，形成可追认的安全事件条目。
  - `R2.3` 输出能力：`C3`。
  - `R2.4` 边界绑定：`USERKEY/EVENTTYPE`。
- `R3` 当前安全状态成立
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把失败锁定结果和最近一次安全事件挂到同一用户上，形成当前安全状态结果。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/EVENTTYPE`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：失败次数无阈值、锁定状态无截止位或事件记录无归属用户的组合无效。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/EVENTTYPE`。

## 5. 验证（Verification）

- `V1` 连续失败计数有效性：连续失败计数槽必须满足 `FAILMAX` 约束。
- `V2` 锁定状态有效性：锁定状态槽必须满足 `LOCKTIME` 约束。
- `V3` 安全事件条目有效性：安全事件条目必须满足 `USERKEY/EVENTTYPE` 约束。
- `V4` 当前安全状态有效性：同一用户必须能同时读到当前是否锁定和最近一次关键安全事件。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
