# 风险锁定与安全事件原子模块:RiskLockingSecurityEventAtoms

@framework

本模块定义用户框架里最底层的失败锁定和安全事件部件。它只回答 4 件事：失败次数怎么放、锁定状态怎么放、解锁状态怎么放、安全事件怎么放；不负责密码证明和权限映射。

## 1. 能力声明（Capability Statement）

- `C1` 失败锁定定义能力：定义失败次数和锁定状态怎样被稳定表达。
- `C2` 事件记账定义能力：定义安全事件怎样被稳定记载。
- `C3` 状态回看定义能力：定义当前安全状态和最近一次关键变更怎样被稳定回看。
- `C4` 非能力项：不负责密码证明、令牌签发和角色权限关系定义。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 归属边界：任一失败记录、锁定状态和安全事件都必须归到一个明确用户。来源：`C1 + C2 + C3`。
- `FAILMAX` 登录失败次数边界：连续失败阈值必须显式声明。来源：`C1`。
- `LOCKTIME` 锁定时长边界：锁定持续时长和解锁落点必须显式声明。来源：`C1 + C3`。
- `EVENTTYPE` 事件类型边界：登录失败、锁定、解锁、签发、续签、撤销与角色变更只能落到声明过的事件类型中。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 失败计数位基：存放当前连续失败次数和最近失败时间，能表达当前失败计数是否接近或达到锁定阈值。来源：`C1 + FAILMAX`。
- `B2` 锁定状态位基：存放锁定起点、锁定截止时间和解锁标记，能表达当前用户是否处于锁定中。来源：`C1 + C3 + LOCKTIME`。
- `B3` 安全事件位基：存放事件类型、事件时间和归属用户，能表达一次关键安全变化事实。来源：`C2 + C3 + USERKEY + EVENTTYPE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 失败锁定成立
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先把失败计数位和锁定状态位挂到同一用户上，形成失败锁定事实。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME`。
- `R2` 事件记账成立
  - `R2.1` 参与基：`B3`。
  - `R2.2` 组合方式：先固定事件类型位、事件时序位和归属位，形成可追认事件记录。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`USERKEY/EVENTTYPE`。
- `R3` 状态链成立
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把失败锁定事实和事件记录挂到同一用户上，形成可回看当前状态和最近关键变更的安全状态链。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/EVENTTYPE`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：失败次数无阈值、锁定状态无截止位或事件记录无归属用户的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/EVENTTYPE`。

## 5. 验证（Verification）

- `V1` 失败锁定有效性：失败计数位和锁定状态位必须满足 `FAILMAX/LOCKTIME` 约束。
- `V2` 事件记账有效性：事件类型位、事件时序位和归属位必须满足 `USERKEY/EVENTTYPE` 约束。
- `V3` 状态链有效性：同一用户必须能同时回看当前锁定状态和最近关键变更。
- `V4` 最小必要性：移除任一 `B*` 后，`V2~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
