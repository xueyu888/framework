# 用户身份认证与访问控制编排模块:UserAuthenticationAndAccessControlOrchestration

@framework

本模块收敛用户身份认证与访问控制的运行编排结构。它只负责 6 件事：用户怎么被唯一定位、密码复杂度怎么校验、失败与锁定怎么记录、访问令牌与刷新令牌怎么签发和失效、一个令牌当前能不能放行、关键安全变化怎么记录和回放；不负责注册页文案、组织模型、第三方身份供应商名单和 OAuth2/OIDC 接入实现。

## 1. 能力声明（Capability Statement）

- `C1` 用户身份定位能力：把用户主标识、登录标识与用户可用状态收敛为稳定身份入口。
- `C2` 密码强度与登录防护能力：把密码复杂度、摘要校验、失败计数与锁定恢复收敛为稳定登录防护线。
- `C3` 令牌签发与失效控制能力：把访问令牌、刷新令牌、轮换与撤销收敛为稳定令牌线。
- `C4` 放行判断能力：根据用户状态、令牌状态与角色权限映射，稳定输出“允许”或“拒绝”。
- `C5` 安全状态记账能力：把登录失败、锁定、解锁、签发、续签、撤销与角色变更记成稳定安全记录。
- `C6` 非能力项：不负责注册页形态、组织树模型、第三方 IdP 名单和多因子通道实现。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 用户标识边界：同一登录标识在有效用户集合中只能指向一个主用户，禁用用户不得继续进入令牌签发链。来源：`C1 + C3 + C4`。
- `PWDRULE` 密码复杂度边界：最小长度、字符种类下限、禁用模式与历史密码复用窗口必须显式声明。来源：`C2`。
- `FAILMAX` 登录失败次数边界：连续失败阈值必须显式声明，到阈值后必须转入锁定态。来源：`C2 + C5`。
- `LOCKTIME` 锁定时长边界：锁定持续时长、提前解锁条件必须显式声明。来源：`C2 + C5`。
- `ATOKTTL` 访问令牌时长边界：访问令牌有效时长必须显式声明，且每个访问令牌只绑定一个主体和一个签发时刻。来源：`C3 + C4`。
- `RTOKTTL` 刷新令牌时长边界：刷新令牌有效时长必须显式声明，且必须长于访问令牌。来源：`C3 + C5`。
- `ROLESET` 角色集合边界：可分配角色与可判断权限只能来自一组声明过的角色-权限集合。来源：`C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 用户身份存取器：L1.M0[R1]：存放和获取用户主标识、登录标识、状态位与禁用位，能根据登录标识定位唯一用户记录。来源：`C1 + USERKEY`。
- `B2` 密码规则与加密器：L1.M0[R2]：负责密码复杂度规则校验、密码摘要生成与密码比对，能根据用户输入输出当前凭据是否通过。来源：`C1 + C2 + USERKEY + PWDRULE`。
- `B3` 失败计数与锁定记录器：L1.M2[R1]：存放连续失败次数、锁定截止时间与解锁标记，能根据一次登录结果更新当前是否允许继续登录的锁定状态。来源：`C2 + FAILMAX + LOCKTIME`。
- `B4` 令牌签发与校验器：L1.M1[R2]：负责访问令牌与刷新令牌的签发、校验、轮换与撤销判断，能根据已通过身份输出当前可用令牌。来源：`C3 + USERKEY + ATOKTTL + RTOKTTL`。
- `B5` 角色-权限映射表：L1.M1[R1]：存放角色集合、权限集合与角色到权限映射，能根据用户或令牌声明输出放行判断依据。来源：`C4 + ROLESET`。
- `B6` 安全状态监控器：L1.M2[R3]：监测并记录登录失败、锁定、解锁、签发、续签、撤销与角色变更，能根据用户定位当前安全状态和最近一次关键变更。来源：`C5 + USERKEY + FAILMAX + LOCKTIME + RTOKTTL + ROLESET`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 身份登录成立
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：先用用户身份存取器定位唯一用户，再用密码规则与加密器校验当前凭据，最后由失败计数与锁定记录器决定通过、拒绝或锁定。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`USERKEY/PWDRULE/FAILMAX/LOCKTIME`。
- `R2` 令牌成立
  - `R2.1` 参与基：`B1 + B4`。
  - `R2.2` 组合方式：在身份登录通过后，由令牌签发与校验器签发访问令牌和刷新令牌，并同时写下过期、轮换与撤销事实。
  - `R2.3` 输出能力：`C1 + C3`。
  - `R2.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL`。
- `R3` 放行判断成立
  - `R3.1` 参与基：`B1 + B4 + B5`。
  - `R3.2` 组合方式：先确认令牌对应主体仍然有效且未过期、未撤销，再由角色-权限映射表输出允许或拒绝判断。
  - `R3.3` 输出能力：`C4`。
  - `R3.4` 边界绑定：`USERKEY/ATOKTTL/RTOKTTL/ROLESET`。
- `R4` 安全记账成立
  - `R4.1` 参与基：`B3 + B4 + B6`。
  - `R4.2` 组合方式：把失败计数、锁定恢复、签发续签撤销与角色变更持续写入安全状态监控器，形成可检查安全记录。
  - `R4.3` 输出能力：`C2 + C3 + C5`。
  - `R4.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/ATOKTTL/RTOKTTL/ROLESET`。
- `R5` 用户访问闭环成立
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R5.2` 组合方式：把身份定位、凭据校验、锁定防护、令牌签发、放行判断和安全记账接成一条完整用户访问闭环。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`USERKEY/PWDRULE/FAILMAX/LOCKTIME/ATOKTTL/RTOKTTL/ROLESET`。
- `R6` 禁止组合
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R6.2` 组合方式：登录标识不唯一、密码复杂度未声明、过期或撤销令牌仍可放行、锁定状态未拦截、角色变更不更新判断依据或关键安全变化未记账的组合无效。
  - `R6.3` 输出能力：`C6`。
  - `R6.4` 边界绑定：`USERKEY/PWDRULE/FAILMAX/LOCKTIME/ATOKTTL/RTOKTTL/ROLESET`。

## 5. 验证（Verification）

- `V1` 身份唯一性：同一登录标识只能定位到一个有效用户，禁用用户不得通过 `R1`。
- `V2` 密码复杂度有效性：新设或更新密码必须满足 `PWDRULE` 约束，不满足时不得进入签发链。
- `V3` 锁定恢复有效性：连续失败达到 `FAILMAX` 后必须锁定 `LOCKTIME`，到期或显式解锁前不得再次签发令牌。
- `V4` 令牌签发与失效有效性：通过 `R2` 的用户必须得到满足 `ATOKTTL/RTOKTTL` 约束的令牌对，且被撤销令牌不得再次放行。
- `V5` 放行判断有效性：同一有效主体在相同 `ROLESET` 映射下必须输出稳定允许或拒绝结论。
- `V6` 安全记账有效性：登录失败、锁定恢复、签发续签撤销与角色变更必须写入明确安全记录。
- `V7` 最小必要性：移除任一 `B*` 后，`V3~V6` 至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
