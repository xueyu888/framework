# 风险锁定与安全记账编排模块:RiskLockingSecurityLedgerOrchestration

@framework

本模块把登录锁定结果、关键安全事件记录和当前安全状态读取接成一条安全状态线。它只负责 4 件事：当前是否允许继续登录怎么判断、关键安全事件怎么记录、当前安全状态怎么读取、上层怎么拿到稳定安全结果；不负责密码校验和角色权限判定。

## 1. 能力声明（Capability Statement）

- `C1` 登录锁定判断能力：根据失败次数、锁定截止时间和解锁标记输出当前是否允许继续登录。
- `C2` 安全记账能力：把签发、续签、撤销和角色变更记录成可回看的安全事件。
- `C3` 安全状态汇总能力：把失败锁定事实和关键变更记录汇总为当前安全状态。
- `C4` 非能力项：不负责密码校验和角色权限判定。

## 2. 边界定义（Boundary / 参数）

- `USERKEY` 归属边界：登录锁定结果和安全事件都必须归到一个明确用户。来源：`C1 + C2 + C3`。
- `FAILMAX` 登录失败次数边界：连续失败阈值必须显式声明。来源：`C1 + C3`。
- `LOCKTIME` 锁定时长边界：锁定持续时长和解锁落点必须显式声明。来源：`C1 + C3`。
- `RTOKTTL` 刷新令牌时长边界：续签和撤销事件必须能落到一个明确的刷新令牌有效期事实上。来源：`C2 + C3`。
- `ROLESET` 角色集合边界：角色变更事件必须能落到声明过的角色集合上。来源：`C2 + C3`。
- `EVENTTYPE` 事件类型边界：登录失败、锁定、解锁、签发、续签、撤销与角色变更只能落到声明过的事件类型中。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 登录锁定判定器：L0.M2[R1]：存放连续失败次数、锁定截止时间和解锁标记，能直接输出当前是否允许继续登录。来源：`C1 + FAILMAX + LOCKTIME`。
- `B2` 安全事件记录器：L0.M1[R1,R2] + L0.M2[R2]：按用户串起签发、续签、撤销和角色变更这类关键安全事件，能追到事件类型、发生时间和关联事实。来源：`C2 + USERKEY + RTOKTTL + ROLESET + EVENTTYPE`。
- `B3` 安全状态表：L0.M2[R3]：汇总登录锁定结果和最近一次关键安全事件，能直接读出当前安全状态和最近一次关键变更。来源：`C1 + C2 + C3 + USERKEY + FAILMAX + LOCKTIME + EVENTTYPE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 失败锁定成立
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：把连续失败次数、锁定截止时间和解锁标记固定到同一用户上，形成当前登录锁定结果。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`FAILMAX/LOCKTIME`。
- `R2` 安全事件成立
  - `R2.1` 参与基：`B2`。
  - `R2.2` 组合方式：把签发、续签、撤销和角色变更串成带类型、时间和归属的安全事件流水。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`USERKEY/RTOKTTL/ROLESET/EVENTTYPE`。
- `R3` 当前安全状态成立
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把当前登录锁定结果和最近一次关键安全事件汇总到同一用户上，形成可直接读取的当前安全状态。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/RTOKTTL/ROLESET/EVENTTYPE`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：锁定状态未建立、关键变更未记账或安全状态无法回看时仍继续输出安全结果的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`USERKEY/FAILMAX/LOCKTIME/RTOKTTL/ROLESET/EVENTTYPE`。

## 5. 验证（Verification）

- `V1` 登录锁定有效性：登录锁定判定器必须满足 `FAILMAX/LOCKTIME` 约束。
- `V2` 安全事件有效性：安全事件流水必须满足 `USERKEY/RTOKTTL/ROLESET/EVENTTYPE` 约束。
- `V3` 当前安全状态有效性：同一用户必须能直接读到当前是否锁定和最近一次关键安全变更。
- `V4` 最小必要性：移除任一 `B*` 后，`V2~V3` 至少一项失败。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
