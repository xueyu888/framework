# 登录认证账户组件:LoginAuthAccountComponent

@framework

## 1. 能力声明（Capability Statement）

- `C1` 账户鉴权基建建立能力：初始化可用于身份验证的账户实体和凭证绑定。
- `C2` 登录态决策生成能力：闭环处理外部请求，输出是否允许登录的最终决策。
- `C3` 凭证关系维护能力：在生命周期内更新标识与凭证的映射关系。
- `C4` 非能力项：不负责具体 Token（如 JWT）的签发和路由会话拦截，不涉及用户昵称/头像等业务资料。

## 2. 边界定义（Boundary / 参数）

- `P1` 标识槽位空间：定义系统接受的标识类型（如 `phone`, `email`, `username`）。来源：`C1/C2`。
- `P2` 存储媒介约束：定义账户实体与映射关系的持久化要求（结构化落盘）。来源：`C1/C3`。
- `P3` 流程图依赖约束：定义鉴权流水线的合法流转路径（必须先校验格式，再比对凭证）。来源：`C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 账户身份锚点实体（Account Identity Anchor）：承载全局唯一 ID（UID）的基础结构，是所有鉴权行为的最终归属节点。来源：`C1 + P2`。
- `B2` 标识寻址图谱（Identity Addressing Map）：以 `[登录标识 (如手机号)]` 为 Key，`[凭证摘要, 账户UID]` 为 Value 的寻址数据结构。来源：`C1 + C3 + P1`。
- `B3` [引用] `L0.LoginInputFormatValidator`：负责输入数据的格式与长度约束拦截。来源：`C1/C2/C3 + P3`。
- `B4` [引用] `L0.IdentityVerification`：负责提取出的内部凭证与外部输入凭证的比对。来源：`C2/C3 + P3`。
- `B5` 鉴权状态机（Auth State Machine）：描述输入请求从 `[初始] -> [格式合规] -> [凭证匹配] -> [鉴权通过]` 的有向无环图结构，用于编排所有基的执行顺序。来源：`C2 + P3`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 账户与凭证创建组合 (注册流)
  - `R1.1` 参与基：`B1 + B2 + B3 + B5`。
  - `R1.2` 组合方式：输入穿透 `B3`（格式校验图谱）后，状态机 `B5` 推进至实例化 `B1`（生成锚点UID），并在 `B2`（寻址图谱）中插入一条全新的 `[标识 -> 凭证摘要 + UID]` 记录。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`P1/P2/P3`。

- `R2` 身份决策闭环组合 (登录流)
  - `R2.1` 参与基：`B2 + B3 + B4 + B5`。
  - `R2.2` 组合方式：输入穿透 `B3` 后，通过标识在 `B2`（寻址图谱）获取预存凭证；状态机 `B5` 将预存凭证与外部凭证投入 `B4` 计算比对；`B4` 输出 Match 状态后，`B5` 终结于鉴权通过态并输出寻址到的 UID。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`P1/P2/P3`。

- `R3` 凭证变更流水线组合 (改密流)
  - `R3.1` 参与基：`B2 + B3 + B4 + B5`。
  - `R3.2` 组合方式：新旧凭证双双穿透 `B3`，旧凭证经 `B2` 寻址并投入 `B4` 完成原身份验证（Match）后；状态机 `B5` 触发 `B2`（寻址图谱）更新对应标识的 Value 节点（覆写新凭证摘要）。
  - `R3.3` 输出能力：`C3`。
  - `R3.4` 边界绑定：`P1/P2/P3`。

- `R4` 禁止组合 (越权/短路组合)
  - `R4.1` 参与基：`B1 + B2 + B4 + B5` (绕过 `B3`)。
  - `R4.2` 组合方式：未挂载 `B3`（输入格式拦截）直接向 `B2/B4` 投递脏数据，破坏状态机 `B5` 的准入前提。
  - `R4.3` 输出能力：`C4`（非法无效）。
  - `R4.4` 边界绑定：违反 `P3`。
