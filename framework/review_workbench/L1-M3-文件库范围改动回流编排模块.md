# 文件库范围改动回流编排模块:ReviewWorkbenchFileLibraryScopeMutationReturnOrchestration

@framework

本模块定义文件库处理场景前半段里“怎样在当前范围下做范围改动，以及这些改动怎样回到查询和局部反馈上”这条链路。它位于 `review_workbench` 的编排层，关心的是上传、新建目录、删除、移动和重命名怎样建立在文件库当前范围上下文与范围展示承接面上，并在改动完成后把范围节点、当前范围、局部信息和查询重新接稳，而不是关心具体目录字段、接口路径或具体触发入口实现。

本模块建立在 `review_workbench.L0.M1`、`review_workbench.L0.M2`、`review_workbench.L0.M6`、`review_workbench.L0.M7`、`review_workbench.L0.M10` 与外部领域 `review_workbench_backend.L0.M3` 这些领域原子之上；而其中的范围上下文、查询条件这些领域原子继续显式承接 `frontend.L1.M0`、`frontend.L1.M4`、`frontend.L1.M5`、`frontend.L1.M9`、`frontend.L1.M12` 和 `frontend.L1.M15`。其中 `frontend.L1.M5` 对应筛选候选项与当前选中值的临时展开选择面结构，`frontend.L1.M4` 主要对应查询触发、清空重置与改动入口。

这里的范围结构改动不再由单个复合 `L0` 模块承担，而是拆成三个更纯的共享原子：`L0.M7` 负责对象位置与挂接结构，`L0.M6` 负责改动目标与载荷结构，`L0.M10` 负责结果影响与可观察回流结构。也就是说，本模块不直接重写改动入口承载面、范围树承接面或查询摘要基础，而是把下层已经稳定的领域原子编排成“当前范围下改动 -> 树与范围回流 -> 查询重新稳定”的文件库前半段支路。

本模块只定义这条范围结构改动支路本身，不在 `L1` 层内声明自己默认承接哪个同层查询模板；若项目需要把文件库范围查询主链与本支路共同挂接到同一具体场景，应由 `L2-M0` 在更高层显式组合，而不是在本模块正文中隐式写成 `L1 -> L1` 依赖。

对于“范围改动怎样发给后端契约、树、当前范围和查询怎样回流”这件事，它向下复用外部领域 `review_workbench_backend.L0.M3` 的范围结构变更与查询失效契约，以及 `review_workbench.L0.M6`、`L0.M7`、`L0.M10` 的范围改动共享原子结构，不在这里重新定义请求和返回结构。

## 1. 能力声明（Capability Statement）
- `C1` 当前范围下发起范围改动的能力：支持在文件库当前范围树节点和当前范围下发起上传、新建目录、删除、移动与重命名。
- `C2` 范围改动回流能力：支持范围改动后同步更新范围树、当前范围状态和局部信息。
- `C3` 查询重新稳定能力：支持范围改动后让当前筛选、结果命中和查询摘要重新稳定。
- `C4` 后端契约消费能力：支持范围改动稳定消费后端契约中的改动请求、改动返回与查询失效回流。
- `C5` 平台统计回接能力：支持范围改动完成后，将其中包含的统计变化回接到平台统计集合并保持统一平台语义。
- `N1` 非职责声明：不负责结果区业务处理、不负责多场景切换、不负责具体范围树承接面和改动入口实现。

## 2. 边界定义（Boundary / 参数）
- `MUTATION_SCOPE` 改动边界：上传、新建目录、删除、移动与重命名允许在不同对象和节点上发生，但每次改动都必须附着文件库当前范围树节点和当前范围。来源：`C1`。
- `RETURN_SCOPE` 回流边界：改动后当前路径、当前范围状态和焦点回退策略允许变化，但改动完成后必须能解释当前工作范围与当前焦点落点。来源：`C2`。
- `TREE_SCOPE` 树刷新边界：改动后范围树、局部信息和受影响节点允许采用不同刷新策略，但改动完成后必须至少有一处出现可解释更新。来源：`C2`。
- `QUERY_SCOPE` 查询边界：改动后筛选摘要、结果命中和查询上下文允许刷新、失效或重算，但必须能解释当前查询为何变化以及当前查询状态是否仍成立。来源：`C3`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：范围改动请求、改动返回和查询失效信息允许映射为不同后端契约字段，但三者都必须稳定绑定到后端契约请求与返回。来源：`C4`。
- `TRACE_SCOPE` 追踪边界：改动对象、当前范围和改动后落点允许变化，但任一次改动都必须能追溯“改的是谁、发生在哪、改后落到哪”。来源：`C1 + C2`。
- `PLATFORM_STAT_RETURN_SCOPE` 平台统计回接边界：范围改动完成后，若结果影响中包含统计变化，则该变化允许通过不同回接路径进入平台统计集合，但不得停留在范围树局部刷新、查询失效说明或不可观察内部状态中。来源：`C5`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 当前范围对象位置基：`L0.M1[R1,R3] + L0.M7[R1,R2]`。来源：`C1 + MUTATION_SCOPE + TRACE_SCOPE`。
- `B2` 改动目标与载荷基：`L0.M6[R1,R2,R3]`。来源：`C1 + MUTATION_SCOPE + TRACE_SCOPE`。
- `B3` 范围改动影响回流基：`L0.M10[R1,R2,R3] + L0.M1[R4,R5]`。来源：`C2 + RETURN_SCOPE + TREE_SCOPE`。
- `B4` 查询重新稳定基：`L0.M2[R2,R3] + review_workbench_backend.L0.M3[R3]`。来源：`C3 + QUERY_SCOPE`。
- `B5` 改动后端契约消费基：`review_workbench_backend.L0.M3[R1,R2,R3] + L0.M6[R2,R3] + L0.M10[R1,R3]`。来源：`C4 + BACKEND_CONTRACT_SCOPE`。
- `B6` 平台统计回接基：`L0.M10[R1,R3] + L0.M11[R3,R4]`。来源：`C5 + PLATFORM_STAT_RETURN_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 范围改动必须附着当前范围
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：任一上传、新建目录、删除、移动或重命名都必须能说明“改的是当前范围中的哪个对象或容器、它当前挂在哪、准备改到哪里”；不能把范围改动写成脱离当前范围对象位置与改动载荷的孤立动作。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`MUTATION_SCOPE/TRACE_SCOPE`。
- `R2` 范围节点更新必须和当前范围一起回流
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：范围改动后，范围节点更新不能和当前范围状态回退、局部信息变化分离；否则用户无法解释当前上下文落在什么范围节点上、当前范围是否已经完成替换。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/TRACE_SCOPE`。
- `R3` 范围改动之后必须重新稳定查询
  - `R3.1` 参与基：`B3 + B4`。
  - `R3.2` 组合方式：上传、新建目录、删除、移动或重命名之后，当前筛选摘要、结果命中和查询上下文至少要有一处发生可解释更新；不能只改树，不让查询重新稳定。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`TREE_SCOPE/QUERY_SCOPE`。
- `R4` 范围改动编排必须消费后端契约
  - `R4.1` 参与基：`B1 + B2 + B4 + B5`。
  - `R4.2` 组合方式：文件库当前范围上下文与当前范围下的范围改动，不仅界面上要讲得清楚，还要能稳定变成服务请求、节点返回、范围状态回流与查询失效信息；否则这套编排没法复用。
  - `R4.3` 输出能力：`C1 + C3 + C4`。
  - `R4.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R5` 文件库范围改动前半段支路闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：当前范围上下文下的范围改动、范围节点更新、当前范围状态回流、查询重新稳定和后端契约消费要共同组成“进入文件库范围上下文 -> 确认当前范围 -> 发起范围改动 -> 看到范围节点与范围状态更新 -> 查询重新稳定”的完整前半段支路。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R5.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE/TRACE_SCOPE`。
- `R6` 范围改动完成后必须回到当前范围下的查询上下文
  - `R6.1` 参与基：`B3 + B4`。
  - `R6.2` 组合方式：范围改动完成后，当前范围状态、筛选摘要、结果命中或查询上下文必须重新稳定到“可继续当前范围处理”的状态；不能只更新树和范围状态，却让文件库列表上下文断开或变成不可解释的旧查询状态。
  - `R6.3` 输出能力：`C2 + C3`。
  - `R6.4` 边界绑定：`RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE`。
- `R7` 范围改动中的统计变化必须回接平台统计集合
  - `R7.1` 参与基：`B3 + B6`。
  - `R7.2` 组合方式：上传、新建目录、删除、移动或重命名完成后，若结果影响中包含统计变化，则该变化必须回接到平台统计集合当前状态，并保持统一平台语义；不能只更新范围树、当前范围状态或查询上下文，而平台统计集合没有任何可解释更新。
  - `R7.3` 输出能力：`C2 + C5`。
  - `R7.4` 边界绑定：`RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。
- `R8` 无效方案
  - `R8.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R8.2` 组合方式：以下方案无效：范围改动能做，但不属于当前范围对象位置；改动载荷存在，但讲不清目标对象或目标落点；树节点变了但当前范围讲不清；查询结果和筛选摘要保持旧状态；界面上有删除、移动和重命名，但接不上稳定服务请求、范围状态回流与查询失效回流；结果影响已包含统计变化，但平台统计集合没有任何可解释更新；树和范围状态更新了，但无法回到当前范围下继续查询或处理。
  - `R8.3` 失效结论：`N1`。
  - `R8.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE/TRACE_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。

## 5. 验证（Verification）
- `V1` 改动附着有效性：任一范围改动都必须能追溯到文件库当前范围中的对象位置、目标对象与当前范围。
- `V2` 范围回流有效性：范围改动后，当前范围状态、焦点回退和树刷新都必须能被解释。
- `V3` 查询重稳有效性：范围改动后，筛选摘要、结果命中或查询上下文必须至少有一处重新稳定。
- `V4` 后端契约消费有效性：范围改动编排必须稳定绑定到改动请求、改动返回和查询失效回流。
- `V5` 查询回接有效性：范围改动完成后，必须能回到当前范围下继续查询或处理，不能停留在不可解释的断开状态。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
