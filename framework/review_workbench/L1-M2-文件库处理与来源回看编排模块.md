# 文件库处理与来源回看编排模块:ReviewWorkbenchFileLibraryProcessingAndSourceTraceOrchestration

@framework

本模块定义文件库处理与来源回看模板后半段是怎么接起来的，也就是“从对象结果里打开当前文件 -> 进入对象工作资源中的分块预览与处理 -> 做纠错、分类、确认等块级动作 -> 必要时进入来源定位与原位回看 -> 看到反馈回来”。它位于 `review_workbench` 的编排层，关心的是对象工作资源怎样从当前结果对象打开、分块预览与块处理怎样承接、来源怎样原位回看、反馈怎样回到当前工作上下文，而不是具体有哪些列、有哪些状态或用哪个接口。

本模块建立在 `review_workbench.L0.M3~M5` 与外部领域 `review_workbench_backend.L0.M2`、`review_workbench_backend.L0.M4` 这些领域原子之上；而这些领域原子继续显式承接 `frontend.L1.M4`、`frontend.L1.M5`、`frontend.L1.M9`、`frontend.L1.M12` 和 `frontend.L1.M15`。其中 `frontend.L1.M5` 主要用于候选项选择与临时展开选择面，`frontend.L1.M4` 主要用于打开、切换、关闭、确认与其他动作触发。也就是说，本模块不再单列维护 `L0` 层中的对象结果组织、分块预览工作资源或来源回看工作资源原子，而是把下层已经稳定的前端领域原子与外部后端契约编排成“打开对象工作资源 -> 查看来源 / 执行动作 -> 收到反馈”的后半段主链。

本模块只定义文件库处理后半段模板本身，不在 `L1` 层内声明自己默认承接哪个同层查询模板；若项目需要把文件库查询主链、结构改动支路与本模块共同挂接为一个具体文件库场景，应由 `L2-M0` 在更高层显式组合，而不是在本模块正文中隐式写成 `L1 -> L1` 依赖。

对于“对象工作资源怎样打开、来源定位怎样请求、块级动作怎样提交、结果怎样回流”这件事，它向下复用外部领域 `review_workbench_backend.L0.M2` 的工作面打开与查看契约，以及 `review_workbench_backend.L0.M4` 的对象动作与结果回流契约，不在这里重定义请求和返回结构；而“来源查看面如何承接、怎样保持原位回看上下文可解释”则由本模块与共享对象 / 反馈底座共同承担。

这里的来源回看不再依赖单个复合 `L0` 外加隐含底座，而是由 `L0-M4` 的对象来源 / 查看锚点结构、`L0-M5` 的来源反馈结构，以及本模块中的来源定位 / 回看编排共同组成文件库对象工作资源中的来源回看支路。

## 1. 能力声明（Capability Statement）
- `C1` 文件库对象工作资源切换能力：支持文件库固定锚点工作面与多个对象工作资源在同一工作台里切换，并通过已有工作资源返回之前打开过的处理内容。
- `C2` 分块预览与处理承接能力：支持当前对象工作资源稳定承接分块预览、块序列、当前块和块级处理动作。
- `C3` 块级处理动作回流能力：支持纠错、分类、确认、标记、跳转和批量处理，并把结果返回到当前处理上下文。
- `C4` 来源定位与原位回看能力：支持从当前对象或当前块进入来源文件、定位到原位并保持来源上下文可解释。
- `C5` 持续反馈能力：支持处理后同步更新块状态、局部反馈、平台统计集合与全局反馈提示。
- `C6` 后端契约消费能力：支持对象工作资源打开、来源定位、块级动作和结果回流稳定消费后端契约。
- `C7` 平台统计回接能力：支持块级处理动作完成后，将其中包含的统计变化回接到平台统计集合并保持统一平台语义。
- `N1` 非职责声明：不负责定义具体业务动作名，不负责审核提交流程，不负责历史差异回放，也不负责具体预览控件实现。

## 2. 边界定义（Boundary / 参数）
- `WORKSET_SCOPE` 工作资源边界：固定锚点工作面、对象工作资源和返回既有工作资源的方式允许变化，但任一时刻必须存在可解释当前工作资源，且切换与关闭后的回退结果必须可解释。来源：`C1`。
- `DETAIL_SCOPE` 处理内容边界：当前文件、当前块、块处理信息、空态和加载态允许变化，但处理内容必须落在当前对象工作资源与当前对象上下文中，并能识别当前文件与当前块。来源：`C2`。
- `ACTION_SCOPE` 动作边界：块级动作、批量动作和查看动作允许采用不同动作集合，但每个动作都必须附着当前对象工作资源中的明确对象、当前块或对象集合。来源：`C3`。
- `SOURCE_SCOPE` 来源边界：来源回看允许通过当前块、片段或源文件对象进入，但任一次来源回看都必须能追溯到当前对象工作资源中的明确来源。来源：`C4`。
- `FEEDBACK_SCOPE` 反馈边界：动作执行后的状态变化允许落到对象工作资源、固定结果入口、局部反馈区、平台统计集合或全局反馈区，但动作完成后必须至少有一处持续可观察反馈。来源：`C5`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：对象工作资源打开、来源定位、动作提交和汇总回流允许映射为不同后端契约字段，但每类动作都必须稳定绑定到对应请求与返回。来源：`C6`。
- `PLATFORM_STAT_RETURN_SCOPE` 平台统计回接边界：块级处理动作完成后，若结果影响中包含统计变化，则该变化允许通过不同回接路径进入平台统计集合，但不得停留在对象工作资源局部、局部反馈区或不可观察内部状态中。来源：`C7`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 工作资源集合基：`L0.M3[R1,R2,R3,R4]`。来源：`C1 + WORKSET_SCOPE`。
- `B2` 分块预览与处理基：`L0.M3[R1,R2] + L0.M4[R2,R3]`。来源：`C2 + DETAIL_SCOPE`。
- `B3` 块级动作基：`L0.M4[R2,R3] + L0.M5[R2]`。来源：`C3 + ACTION_SCOPE`。
- `B4` 来源定位与原位回看基：`L0.M4[R3,R4] + L0.M5[R1,R3]`。来源：`C4 + SOURCE_SCOPE`。
- `B5` 反馈基：`L0.M5[R2,R3] + L0.M4[R4]`。来源：`C5 + FEEDBACK_SCOPE`。
- `B6` 后端契约消费基：`review_workbench_backend.L0.M2[R1,R2] + review_workbench_backend.L0.M4[R1,R3]`。来源：`C6 + BACKEND_CONTRACT_SCOPE`。
- `B7` 平台统计回接基：`L0.M10[R1,R3] + L0.M11[R3,R4]`。来源：`C7 + PLATFORM_STAT_RETURN_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 文件库对象工作资源必须承接打开来源
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：文件库对象工作资源必须属于当前工作资源集合，并且要能说明自己是从当前结果对象集合中的哪个文件项打开的；已存在于工作资源集合中的对象工作资源可以作为返回之前打开过的处理内容的入口，但该工作资源首次打开时仍必须来自当前结果对象集合中的明确文件项来源；不能把对象工作资源写成无来源的孤立页面。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/SOURCE_SCOPE`。
- `R2` 块级动作必须挂在分块预览中的当前块结构上
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：任何块级处理动作都必须作用在当前对象工作资源中的当前块、某一组块或当前对象之上；不能出现脱离分块预览与处理上下文的黑盒操作。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`DETAIL_SCOPE/ACTION_SCOPE`。
- `R3` 来源回看必须挂在当前块或当前对象上
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：任何来源定位与原位回看都必须从当前对象工作资源中的当前对象或当前块进入，打开后的来源面和原位定位也要能说明当前来源对象、当前块与当前对象工作资源；不能出现脱离当前工作资源内容的孤立来源预览。
  - `R3.3` 输出能力：`C2 + C4`。
  - `R3.4` 边界绑定：`DETAIL_SCOPE/SOURCE_SCOPE`。
- `R4` 动作之后必须有反馈
  - `R4.1` 参与基：`B3 + B5`。
  - `R4.2` 组合方式：动作执行后，当前对象工作资源、固定锚点工作面、局部反馈区、平台统计集合或状态提示里至少要有一处能看出变化，并且能追到当前工作资源、当前对象与当前块。
  - `R4.3` 输出能力：`C3 + C5`。
  - `R4.4` 边界绑定：`ACTION_SCOPE/FEEDBACK_SCOPE`。
- `R5` 文件库对象工作资源、来源回看和动作必须消费后端契约
  - `R5.1` 参与基：`B2 + B3 + B4 + B6`。
  - `R5.2` 组合方式：分块预览与处理不能只停留在界面层，还要对应稳定的对象工作资源打开请求与结果返回；来源定位也要对应稳定的来源请求和来源返回；块级动作也要对应稳定的动作请求和动作结果，否则这套后半段没法复用。
  - `R5.3` 输出能力：`C2 + C3 + C4 + C6`。
  - `R5.4` 边界绑定：`DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R6` 块级处理动作中的统计变化必须回接平台统计集合
  - `R6.1` 参与基：`B3 + B5 + B7`。
  - `R6.2` 组合方式：块级处理动作完成后，若结果影响中包含统计变化，则该变化必须回接到平台统计集合当前状态，并保持统一平台语义；它可以与局部反馈或状态提示并存，但不能只停留在对象工作资源局部、局部反馈区或不可观察内部状态中。
  - `R6.3` 输出能力：`C3 + C5 + C7`。
  - `R6.4` 边界绑定：`ACTION_SCOPE/FEEDBACK_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。
- `R7` 文件库处理场景后半段闭环
  - `R7.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6 + B7`。
  - `R7.2` 组合方式：当前结果对象集合中的打开动作、对象工作资源切换、分块预览与处理、来源定位与原位回看、块级动作、后端契约消费、反馈回流与平台统计回接要共同组成“在当前结果对象集合中打开文件项 -> 进入对象工作资源中的分块预览与处理 -> 查看来源 / 做块级动作 -> 收到结果 -> 看到反馈并完成必要的平台统计回接”的完整后半段。
  - `R7.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7`。
  - `R7.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/FEEDBACK_SCOPE/BACKEND_CONTRACT_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。
- `R8` 无效方案
  - `R8.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6 + B7`。
  - `R8.2` 组合方式：以下方案无效：只有工作资源切换，没有对象工作资源；只有分块预览没有当前块；只有块级动作没有块归属；来源回看可以打开，但无法说明来自哪个当前块或不能定位到源文件原位；动作执行后没有任何反馈；动作结果已包含统计变化但平台统计集合没有任何可解释更新；界面上有预览、来源和动作，但接不上稳定服务返回。
  - `R8.3` 失效结论：`N1`。
  - `R8.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/FEEDBACK_SCOPE/BACKEND_CONTRACT_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。

## 5. 验证（Verification）
- `V1` 工作面承接有效性：当前对象工作资源必须能追溯到当前结果对象集合中的打开来源，并解释当前对象上下文。
- `V2` 块级动作归属有效性：对象工作资源里的任一动作都必须有明确当前块或对象归属。
- `V3` 来源定位有效性：任一来源回看都必须能追溯到当前块或当前对象，并在来源面中定位到原位。
- `V4` 反馈回流有效性：任一块级动作执行后，必须至少有一处持续可观察反馈出现可解释变化。
- `V5` 切换返回有效性：切换或关闭工作资源后，当前工作面必须可解释。
- `V6` 后端契约消费有效性：对象工作资源、来源定位和块级动作回流必须稳定绑定到打开返回、来源返回、动作请求、动作结果和汇总返回。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
