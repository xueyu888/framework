# 平台级统计集合承接原子模块:ReviewWorkbenchPlatformLevelStatisticsSetMountAtoms

@framework

本模块定义 `review_workbench` 作为统一审查平台时最小的“平台级统计集合如何成立”结构原子。它关注的不是某一个具体场景里的局部摘要、范围统计或一次动作后的临时反馈，而是跨场景持续成立、在平台壳长期承接面中稳定存在、并能随着文件操作、审核动作或状态变化而更新的一组统计集合结构。
本模块位于 `review_workbench` 的原子层。它只负责把平台级统计项身份、统计集合、平台统计集合承接面与统计更新附着关系表达清晰，供 `L1-M0` 等上层编排复用；它不在这里定义具体统计项名称、统计文案、聚合公式、后端实现或具体刷新策略，也不负责成功失败等反馈提示。

## 1. 能力声明（Capability Statement）

- `C1` 平台级统计项身份表达能力：将跨场景持续成立的统计项组织成稳定身份结构。
- `C2` 平台级统计集合表达能力：将多个平台级统计项组织成稳定集合，并附着统一平台成立。
- `C3` 跨场景统计持续能力：将当前场景切换前后仍然成立的统计承接关系组织成稳定结构。
- `C4` 平台统计更新附着能力：将文件操作、审核动作、状态变化等引发的统计更新附着到平台当前状态中。
- `N1` 非职责声明：不负责具体统计项名称、不负责聚合公式、不负责局部范围统计、不负责后端统计实现。

## 2. 边界定义（Boundary / 参数）

- `PLATFORM_STAT_ITEM_SCOPE` 平台统计项边界：统计项种类、命名和排序允许变化，但每个统计项都必须具备稳定身份，不能退化成仅靠展示文案识别的临时数字。
- `PLATFORM_STAT_SET_SCOPE` 平台统计集合边界：统计项数量、组合方式与展示组织允许变化，但它们必须共同附着统一平台成立，而不是分散附着到单一场景内部。
- `CROSS_SCENE_STAT_SCOPE` 跨场景统计边界：当前场景允许切换，但平台级统计集合不得因场景切换失效，且切换前后都必须能解释当前统计仍属于同一统一平台。
- `PLATFORM_STAT_UPDATE_SCOPE` 平台统计更新边界：统计值允许因平台中的动作与状态变化而更新，但更新后必须仍能解释为平台当前统计状态的一部分，而不是不可追溯的孤立快照。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 平台统计项身份基：frontend.L1.M15[R1]。由稳定标记、身份区分与状态附着约束成平台级统计项身份结构。来源：`C1 + PLATFORM_STAT_ITEM_SCOPE`。
- `B2` 平台统计集合基：frontend.L1.M12[R1,R2] + frontend.L1.M9[R1]。由集合结构、集合内组织关系与展示承接面约束成平台级统计集合结构。来源：`C2 + PLATFORM_STAT_SET_SCOPE`。
- `B3` 跨场景统计持续基：frontend.L1.M12[R2] + frontend.L1.M15[R2]。由当前集合状态、切换后持续可识别关系和状态反馈约束成跨场景统计持续结构。来源：`C3 + CROSS_SCENE_STAT_SCOPE`。
- `B4` 平台统计更新附着基：frontend.L1.M4[R1,R2] + frontend.L1.M15[R1,R2]。由状态变化触发、更新后的状态标记与持续可观察反馈约束成平台统计更新附着结构。来源：`C4 + PLATFORM_STAT_UPDATE_SCOPE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 平台级统计必须先有稳定统计项身份
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：平台级统计集合必须由具备稳定身份的统计项组成；不能只有一组数字或文案，而无法解释每一个统计项是谁、如何进入同一集合。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`PLATFORM_STAT_ITEM_SCOPE/PLATFORM_STAT_SET_SCOPE`。

- `R2` 平台级统计集合必须附着统一平台而不是单一场景
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：平台级统计集合在文件库、审核、历史等场景切换前后都必须继续成立，并能解释自己仍属于同一统一平台；不能把平台统计误写成某个单一场景内部的局部摘要。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`PLATFORM_STAT_SET_SCOPE/CROSS_SCENE_STAT_SCOPE`。

- `R3` 平台中的动作与状态变化必须能回到平台统计更新
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：文件操作、审核动作、状态变化等引发的统计变化，必须回到平台级统计集合承接面中形成当前统计状态；不能让统计更新停留在不可观察的内部变化，或退化为某个场景局部的临时数值。
  - `R3.3` 输出能力：`C2 + C4`。
  - `R3.4` 边界绑定：`PLATFORM_STAT_SET_SCOPE/PLATFORM_STAT_UPDATE_SCOPE`。

- `R4` 平台级统计集合承接闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：平台级统计项身份、统计集合、跨场景持续关系与统计更新附着关系必须共同形成“平台长期统计集合承接面”；既能在场景切换时保持成立，也能在平台状态变化后完成统计更新。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`PLATFORM_STAT_ITEM_SCOPE/PLATFORM_STAT_SET_SCOPE/CROSS_SCENE_STAT_SCOPE/PLATFORM_STAT_UPDATE_SCOPE`。

- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：只有一组数字但没有稳定统计项身份；统计集合只能在单一场景里成立；切换场景后统计承接面消失；动作已经引发状态变化但平台统计没有任何可观察更新；把具体统计项名称、聚合公式或后端实现直接写回本模块。
  - `R5.3` 失效结论：`N1`。
  - `R5.4` 边界绑定：`PLATFORM_STAT_ITEM_SCOPE/PLATFORM_STAT_SET_SCOPE/CROSS_SCENE_STAT_SCOPE/PLATFORM_STAT_UPDATE_SCOPE`。

## 5. 验证（Verification）

- `V1` 统计项身份有效性：任一平台级统计项都必须能识别稳定身份，而不是只剩展示文案。
- `V2` 统计集合平台归属有效性：任一平台级统计集合都必须能追溯到统一平台，而不是某个单一场景内部。
- `V3` 跨场景持续有效性：场景切换前后，平台级统计集合必须仍然成立且可解释。
- `V4` 统计更新有效性：平台中的动作或状态变化后，平台统计必须至少出现一处可追溯更新。
- `V5` 前端承接有效性：平台级统计项、统计集合、持续关系与更新关系必须能追溯到 frontend 的触发、集合浏览、展示承接与状态反馈结构。
- `V6` 反馈边界纯度：本模块中不得混入成功失败提示、临时反馈消息或状态提示承接语义。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
