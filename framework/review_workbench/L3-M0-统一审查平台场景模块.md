# 统一审查平台场景模块:ReviewWorkbenchUnifiedPlatformScenarios

@framework

本模块定义 `review_workbench` 在统一平台层如何把已经成立的平台前置编排与具体场景实例组织成统一审查平台。它关注的不是某个具体场景内部的范围树、审核结论、文件处理链路或历史差异结构，而是统一平台如何以唯一入口承接多个合法场景实例，并保持唯一当前场景、统一主工作区归属与平台级上下文成立。

本模块位于 `review_workbench` 的统一平台层。向下它依赖 `L1-M0` 已经建立的平台导航到场景挂接前置编排，并承接 `L2-M0`、`L2-M1`、`L2-M2` 这类具体场景模块作为统一平台中的合法场景实例。向上它为项目配置、代码实现与验证视图提供“统一平台壳 + 合法场景实例集合 + 唯一当前场景归属”这一平台级入口。

本模块不重新定义平台导航、顶部工具位、平台反馈或具体场景内部主线的编排细节；这些分别由 `L1-M0` 与各 `L2/L1` 场景模块承担。本模块只回答：什么样的平台承接关系才算统一审查平台成立。

关于 `review_workbench` 的模块树分层、默认扩展顺序以及“新增场景优先落 `L2`、仅在出现新的稳定模板时才上提 `L1`、仅在出现新的共享结构原子时才上提 `L0`”这类跨模块规则，统一说明见同目录下的 `README.md`。

## 1. 能力声明（Capability Statement）
- `C1` 统一平台承接能力：支持统一审查平台以唯一入口承接多个合法具体场景实例。
- `C2` 唯一当前场景归属能力：支持平台在任一时刻维持唯一当前场景，并把该场景稳定挂接到统一主工作区中。
- `C3` 平台级上下文持续成立能力：支持平台在场景切换、场景挂接或挂接失败后，仍保持平台级工具位、平台级反馈与当前工作落点的可解释性。
- `C4` 多场景平台闭环能力：支持把平台入口、合法场景实例集合、唯一当前场景归属与平台级上下文接成统一且可扩展的平台主线。
- `N1` 非职责声明：不直接替代 `L1-M0` 的平台前置编排，不直接定义任何具体场景内部的对象定位、文件处理、审核结论或历史追溯主线，也不直接定义具体界面实现。

## 2. 边界定义（Boundary / 参数）
- `ENTRY_SCOPE` 平台入口边界：统一平台允许从不同入口形态进入，但进入后必须能识别合法场景实例集合与唯一当前场景；不允许进入后处于无场景归属的悬空状态。来源：`C1 + C2`。
- `SCENE_INSTANCE_SCOPE` 场景实例边界：统一平台允许承接不同具体场景，但被承接对象必须是 `review_workbench` 中已声明、满足平台内具体场景层约束的 `L2` 场景模块；不允许把未声明页面流、临时流程或场景内部子步骤伪装成平台场景实例。来源：`C1 + C4`。
- `PLATFORM_CONTEXT_SCOPE` 平台上下文边界：平台级工具位、平台反馈、主工作区和当前工作落点的具体表达方式允许变化，但只要当前仍属于统一平台，就必须保持平台级语义与持续可解释性；不允许进入具体场景后退化为仅对该场景局部成立的临时上下文。来源：`C2 + C3`。
- `SCENE_ISOLATION_SCOPE` 场景隔离边界：统一平台允许声明“当前是谁、挂接到哪里、如何切换”，但不得重写具体场景内部的对象定位、审核、处理或追溯主线；这些必须继续由对应 `L2/L1` 模块承担。来源：`C1 + C4 + N1`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 平台前置编排基：`L1.M0[R5]`。来源：`C2 + C3 + ENTRY_SCOPE + PLATFORM_CONTEXT_SCOPE`。
- `B2` 合法场景实例集合基：`L2.M0[B1] + L2.M1[B1] + L2.M2[B1]`。来源：`C1 + C4 + SCENE_INSTANCE_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 统一平台只能承接合法场景实例集合中的成员
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：统一平台中的可切换场景与当前被挂接场景都必须取自合法场景实例集合，且任一当前场景都必须能定位到集合中的唯一成员；不允许在平台层临时拼出未声明的并行场景实例。
  - `R1.3` 输出能力：`C1 + C2 + C4`。
  - `R1.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE`。
- `R2` 统一平台必须通过平台前置编排承接当前场景
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：统一平台必须先以 `L1-M0` 已成立的平台前置编排作为统一承接面，再在该承接面中挂接当前场景实例，并保持平台级工具位、平台反馈与当前工作落点持续可解释；不允许绕过平台前置编排，直接把某个具体场景写成没有平台上下文的独立页面。
  - `R2.3` 输出能力：`C2 + C3 + C4`。
  - `R2.4` 边界绑定：`ENTRY_SCOPE/PLATFORM_CONTEXT_SCOPE/SCENE_INSTANCE_SCOPE`。
- `R3` 统一平台不得吞并具体场景内部主线
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：统一平台只负责承接合法场景实例、维持唯一当前场景归属并保持平台级上下文成立；一旦需要说明文件处理、审核决策、历史追溯或其他场景内部主线，必须下沉到对应 `L2/L1` 模块，而不能在本模块中混成平台唯一主线。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`SCENE_ISOLATION_SCOPE/SCENE_INSTANCE_SCOPE`。
- `R4` 统一平台主线闭环
  - `R4.1` 参与基：`B1 + B2`。
  - `R4.2` 组合方式：统一平台入口、合法场景实例集合、唯一当前场景归属与平台级上下文要共同组成“进入平台 -> 识别合法场景集合 -> 确定唯一当前场景 -> 在统一承接面中挂接当前场景 -> 持续保持平台级上下文成立”的完整平台主线；不能把其中任一段拆成脱离前后文的孤立状态。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE/PLATFORM_CONTEXT_SCOPE/SCENE_ISOLATION_SCOPE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2`。
  - `R5.2` 组合方式：以下方案无效：只有统一平台壳没有合法场景实例集合；只有具体场景没有统一平台入口；进入具体场景后平台级工具位与平台反馈失去平台语义；切换场景后当前落点不可解释；把未声明页面流伪装成平台场景实例；把不同具体场景的内部主线直接混成本模块的唯一主线。
  - `R5.3` 失效结论：`N1`。
  - `R5.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE/PLATFORM_CONTEXT_SCOPE/SCENE_ISOLATION_SCOPE`。

## 5. 验证（Verification）
- `V1` 平台入口有效性：任一平台进入后，都必须能识别合法场景实例集合与唯一当前场景。
- `V2` 场景归属有效性：任一当前场景都必须能挂接到统一平台承接面中，并解释当前场景归属。
- `V3` 平台上下文有效性：任一场景切换、场景挂接或挂接失败后，平台级工具位、平台反馈与当前工作落点都必须仍保持平台级语义与可解释性。
- `V4` 场景隔离有效性：本模块不得直接输出具体场景内部主线结论；凡涉及文件处理、审核、追溯等内部主线，必须能追溯到对应 `L2/L1` 模块。
- `V5` 平台承接有效性：统一平台中的当前挂接场景必须能定位到合法场景实例集合中的唯一成员，且平台承接主链必须能追溯到 `L1-M0`，而不是在本模块中重新拼出并行真相源。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
