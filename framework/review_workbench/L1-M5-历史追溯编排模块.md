# 历史追溯编排模块:ReviewWorkbenchHistoryTraceOrchestration

@framework

本模块定义统一审查平台中历史追溯主线是怎么接起来的，也就是“定位对象或范围 -> 查看历史序列 -> 打开某次变更 -> 理解差异与元信息”。它位于 `review_workbench` 的编排层，关心的是历史序列如何进入工作资源、差异如何打开、元信息如何稳定承接，而不是具体恢复策略或底层对比算法实现。

本模块建立在 `review_workbench.L0.M4`、`review_workbench.L0.M5`、`review_workbench.L0.M9` 与外部领域 `review_workbench_backend.L0.M1`、`review_workbench_backend.L0.M2`、`review_workbench_backend.L0.M5` 这些领域原子之上。其中 `L0.M4` 提供对象列表与查看入口底座，`L0.M5` 提供反馈与摘要底座，`L0.M9` 提供历史序列、差异回看与前端元信息承接底座，`review_workbench_backend.L0.M1` 提供历史列表查询契约，`review_workbench_backend.L0.M2` 提供节点打开与查看契约，`review_workbench_backend.L0.M5` 提供历史请求与差异返回契约。

本模块只定义历史追溯模板本身，不在 `L1` 层内声明自己默认承接哪个同层平台前置模板；若项目需要把平台前置编排与历史追溯主线共同挂接为一个具体追溯场景，应由 `L2-M2` 与 `L3-M0` 在更高层显式组合，而不是在本模块正文中隐式写成 `L1 -> L1` 依赖。

这里的历史追溯不再依赖单个复合 `L0` 外加隐含底座，而是由 `L0-M4` 的对象与查看入口结构、`L0-M5` 的反馈与摘要结构，以及 `L0-M9` 的历史序列 / 差异 / 元信息结构共同组成历史追溯输入。

## 1. 能力声明（Capability Statement）
- `C1` 历史序列查看能力：支持在历史追溯主线中稳定查看对象历史序列。
- `C2` 历史节点打开能力：支持从历史序列进入某个历史节点的差异查看面。
- `C3` 差异与元信息理解能力：支持在当前历史节点下查看差异和变更元信息。
- `C4` 追溯后端契约消费能力：支持历史查询、节点打开和差异回看稳定消费后端契约。
- `N1` 非职责声明：不负责文件块纠错动作、不负责审核决策规则、不负责具体恢复实现。

## 2. 边界定义（Boundary / 参数）
- `HISTORY_SCOPE` 历史序列边界：对象历史列表、版本节点和当前节点允许变化，但任一次历史查看都必须附着当前对象或当前范围，并能识别当前历史节点。来源：`C1`。
- `DIFF_SCOPE` 差异查看边界：差异片段、前后版本和查看承载面允许变化，但任一次差异查看都必须追溯到当前历史节点及明确前后版本关系。来源：`C2 + C3`。
- `META_SCOPE` 元信息边界：操作者、时间、说明和关联上下文字段允许变化，但任一当前变更都必须至少保留一组可解释元信息。来源：`C3`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：历史查询、节点打开和差异返回允许映射为不同后端契约字段，但三者都必须稳定绑定到对应请求与返回。来源：`C4`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 历史序列与节点基：`L0.M9[R1,R2] + L0.M4[R1,R2]`。来源：`C1 + C2 + C3 + HISTORY_SCOPE + META_SCOPE`。
- `B2` 差异回看与元信息基：`L0.M9[R1,R2,R3] + L0.M4[R3,R4] + L0.M5[R1,R3]`。来源：`C2 + C3 + DIFF_SCOPE`。
- `B3` 后端契约消费基：`review_workbench_backend.L0.M1[R1,R2] + review_workbench_backend.L0.M2[R1,R2] + review_workbench_backend.L0.M5[R1,R2] + L0.M5[R3]`。来源：`C4 + BACKEND_CONTRACT_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 差异查看必须由历史节点打开
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：任何差异查看都必须能说明来自哪个对象、哪个历史节点和哪一组前后版本；不能直接构造无历史来源的差异页面。
  - `R1.3` 输出能力：`C1 + C2 + C3`。
  - `R1.4` 边界绑定：`HISTORY_SCOPE/DIFF_SCOPE/META_SCOPE`。
- `R2` 历史追溯主线必须消费后端契约
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：历史序列查询、历史节点打开和差异回看都必须稳定接到服务契约中；否则这套历史追溯主线没法复用。
  - `R2.3` 输出能力：`C1 + C2 + C4`。
  - `R2.4` 边界绑定：`HISTORY_SCOPE/DIFF_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R3` 历史追溯主线闭环
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：对象历史序列、历史节点打开、差异回看和元信息理解要共同组成“定位对象 -> 查看历史序列 -> 打开某次变更 -> 理解差异与元信息”的完整历史追溯主线。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`HISTORY_SCOPE/DIFF_SCOPE/META_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：只有历史列表没有当前节点语义；只有差异内容没有历史来源；历史节点可以点开但没有元信息；界面上有历史和差异，但接不上稳定服务。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`HISTORY_SCOPE/DIFF_SCOPE/META_SCOPE/BACKEND_CONTRACT_SCOPE`。

## 5. 验证（Verification）
- `V1` 历史来源有效性：任一差异查看都必须能追溯到明确历史节点与当前对象来源。
- `V2` 差异承接有效性：任一差异片段都必须能说明明确前后版本关系。
- `V3` 元信息可解释性：任一历史节点都必须至少带有一组可解释元信息。
- `V4` 后端契约消费有效性：历史查询、节点打开和差异返回必须稳定绑定到对应请求与返回。
- `V5` 最小必要性：移除任一 `B*` 后，`V1~V4` 至少一项失效。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
