# 全局反馈与状态提示原子模块:ReviewWorkbenchGlobalFeedbackAndStatusPromptAtoms

@framework

本模块定义 `review_workbench` 领域最小的全局反馈与状态提示结构。推导链为：`framework/frontend/L0-M0-承载与绑定原子模块.md` + `framework/frontend/L1-M9-承载-展示与容器-原子模块.md` + `framework/frontend/L1-M15-反馈-标记与反馈-原子模块.md` -> 本模块。也就是说，工作台中的成功提示、失败提示、处理中提示、空态提示与状态条提示，不单独重写一套全局提示基础，而是把前端已有的承载壳、展示容器和附着反馈约束成“工作台级反馈提示”这组领域原子。
本模块只回答两个最小问题：一是“工作台中的反馈如何进入稳定可观察位置”，二是“状态变化如何被持续提示给用户”。它不负责平台级统计集合、不负责总量汇总、不负责跨场景计数面板；这些长期统计结构统一由 `L0-M11` 单独承担。

## 1. 能力声明（Capability Statement）

- `C1` 全局反馈内容表达能力：将成功、失败、进行中、空态与提醒等反馈组织成稳定反馈内容结构。
- `C2` 状态提示持续承接能力：将状态变化挂接在稳定位置并持续提示当前状态。
- `C3` 反馈变化可观察能力：将动作后的反馈变化组织成至少一处持续可观察提示。
- `N1` 非职责声明：不负责平台级统计项集合、不负责计数汇总面板、不负责局部范围统计。

## 2. 边界定义（Boundary / 参数）

- `GLOBAL_FEEDBACK_SCOPE` 全局反馈边界：成功、失败、空态、处理中和提醒提示的组合允许变化，但动作或状态变化发生后必须至少导出一处可观察反馈。
- `STATUS_PROMPT_SCOPE` 状态提示边界：状态提示的位置、样式和持续时长允许变化，但必须存在稳定承接位置，且不能随单次动作丢失。
- `FEEDBACK_UPDATE_SCOPE` 反馈更新边界：反馈内容允许因动作结果与当前状态变化而更新，但更新后必须仍能解释为工作台当前反馈而不是孤立消息。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 全局反馈内容基：frontend.L1.M15[R1,R2]。由成功、失败、空态与处理中等反馈标记约束成工作台级反馈内容结构；它回答“反馈内容是什么”，不回答“挂在哪里”或“如何随动作更新”。来源：`C1 + GLOBAL_FEEDBACK_SCOPE`。
- `B2` 状态提示承接基：frontend.L0.M0[R1,R2] + frontend.L1.M9[R2]。由稳定承载面和展示容器约束成工作台级状态提示位置与持续承接关系；它回答“反馈和状态提示挂在哪里并如何持续成立”，不负责定义反馈内容本身。来源：`C2 + STATUS_PROMPT_SCOPE`。
- `B3` 反馈变化附着基：frontend.L1.M4[R1,R2] + frontend.L1.M15[R1,R2]。由状态变化触发、反馈更新和持续可观察约束成反馈变化附着结构；它回答“动作或状态变化后，反馈如何进入新的可观察状态”，不替代稳定承接位置本身。来源：`C3 + FEEDBACK_UPDATE_SCOPE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 反馈提示必须进入稳定承接位置
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：在 frontend 的承载壳和展示容器结构上，工作台级反馈提示必须挂接到稳定的状态提示区或全局反馈区，不能只以瞬时消息存在。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`GLOBAL_FEEDBACK_SCOPE/STATUS_PROMPT_SCOPE`。

- `R2` 动作后的反馈变化必须可观察
  - `R2.1` 参与基：`B1 + B3`。
  - `R2.2` 组合方式：处理成功、失败、空态或进行中等反馈，在动作完成或状态变化后必须至少进入一处可持续观察的位置，不能停留在不可追溯内部变化。
  - `R2.3` 输出能力：`C1 + C3`。
  - `R2.4` 边界绑定：`GLOBAL_FEEDBACK_SCOPE/FEEDBACK_UPDATE_SCOPE`。

- `R3` 全局反馈提示原子闭环
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：反馈内容、状态提示承接和反馈变化附着必须共同形成“有状态变化 -> 有工作台反馈 -> 有稳定提示位置”的闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`GLOBAL_FEEDBACK_SCOPE/STATUS_PROMPT_SCOPE/FEEDBACK_UPDATE_SCOPE`。

- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：跳过 frontend 承载壳与反馈结构直接声明工作台状态；只有临时消息没有稳定承接位置；把平台级统计集合或计数面板混入反馈原子；反馈出现后无法持续提示当前状态。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`GLOBAL_FEEDBACK_SCOPE/STATUS_PROMPT_SCOPE/FEEDBACK_UPDATE_SCOPE`。

## 5. 验证（Verification）

- `V1` 反馈可识别性：工作台中必须至少存在一处可识别的全局反馈提示。
- `V2` 状态提示可观察性：成功、失败、空态或处理中状态发生后，必须至少有一处可观察提示。
- `V3` 稳定承接性：反馈提示与状态提示都必须挂接在稳定工作台级位置，且不能随单次动作丢失。
- `V4` 前端承接有效性：反馈内容、状态提示承接和反馈变化附着必须能追溯到 frontend 的承载绑定、展示容器和附着反馈结构。
- `V5` 统计边界纯度：本模块中不得出现平台级统计项集合、计数面板或跨场景统计集合语义。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
