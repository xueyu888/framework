# 审核提交场景模块:ReviewWorkbenchReviewSubmissionScenarios

@framework

本模块定义 `review_workbench` 中审核提交场景作为一个具体场景实例，如何在统一平台中承接平台前置编排、平台级统计持续关系与审核型场景模板主线。它关注的不是审核型链路内部每一段如何独立成立，而是“审核提交”这个具体场景如何以统一身份进入平台、持续承接平台级统计、审核工作栏、待审对象查看、审核对象打开、审核结论提交与回流反馈。

本模块位于 `review_workbench` 的具体场景层。向下它依赖 `L1-M0` 和 `L1-M4` 这两个编排模块：其中 `L1-M0` 负责平台前置编排与平台级统计持续承接，`L1-M4` 负责审核型场景主线。

本模块不重新定义审核工作栏、审核对象打开、审核结论提交或审核回流各自如何编排；这些属于 `L1-M4`。本模块只回答：当“审核提交”被用作统一平台中的一个具体场景实例时，什么样的场景承接关系才算成立。

## 1. 能力声明（Capability Statement）
- `C1` 审核提交场景实例承接能力：支持“审核提交”作为统一平台中的一个具体场景实例被识别和承接。
- `C2` 平台统计持续承接能力：支持审核提交场景在进入、切换和审核过程中持续承接统一平台中的平台级统计。
- `C3` 审核工作栏承接能力：支持审核提交场景在同一场景上下文中承接当前审核焦点、说明输入区和待审 / 已审列表。
- `C4` 审核对象与结论承接能力：支持审核提交场景承接待审核对象查看、审核对象打开与审核结论提交。
- `C5` 审核回流承接能力：支持审核提交场景在审核完成后承接对象状态、统计和提示变化，并保持可解释回流。
- `C6` 审核提交场景闭环能力：支持把场景进入、平台级统计、审核工作栏、待审核对象、审核结论与回流反馈接成统一且可解释的审核提交场景实例主线。
- `N1` 非职责声明：不替代文件库处理场景，不替代历史追溯场景，不直接定义审核型模板内部的工作栏、对象打开、结论提交或回流编排细节，也不直接定义原子字段或具体界面实现。

## 2. 边界定义（Boundary / 参数）
- `SCENE_INSTANCE_SCOPE` 场景实例边界：审核提交场景允许采用不同承载方式、入口配置和工作面组织，但它只能作为统一平台中的一个具体场景实例被承接；至于它是否属于统一平台中的合法场景实例集合，由 `L3-M0` 单独约束。来源：`C1 + C6`。
- `PLATFORM_STAT_CONTINUITY_SCOPE` 平台统计持续边界：审核提交场景允许有不同审核步骤和工作面组织方式，但进入、切换或审核过程中都不得让平台级统计脱离统一平台承接面或失去当前统计语义。来源：`C2 + C5 + C6`。
- `SIDEBAR_SCOPE` 审核工作栏边界：当前审核焦点摘要、说明输入区和待审 / 已审列表允许采用不同布局，但必须共同落在审核提交场景中的同一审核工作栏承接面。来源：`C3`。
- `REVIEW_SCOPE` 审核上下文边界：待审核列表、对象详情和审核结论允许随实例变化，但任一次审核都必须能解释当前审核对象、列表来源与当前审核场景上下文。来源：`C4`。
- `FLOW_SCOPE` 审核回流边界：审核后的状态变化允许通过审核工作栏、列表区、审核工作面、平台统计区或全局反馈区表达，但审核完成后必须至少有一处持续可观察反馈，且审核工作栏中必须能看到局部变化。来源：`C5 + C6`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 平台前置场景承接基：`L1.M0[R3,R6]`。来源：`C1 + C2 + SCENE_INSTANCE_SCOPE + PLATFORM_STAT_CONTINUITY_SCOPE`。
- `B2` 审核提交承接基：`L1.M4[R5]`。来源：`C3 + C4 + C5 + C6 + SIDEBAR_SCOPE + REVIEW_SCOPE + FLOW_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 审核提交场景必须在统一平台统计持续成立的前提下进入
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：审核提交场景进入后，必须继续承接统一平台中的平台级统计集合，并保持该统计集合的统一平台归属语义；不能把审核提交场景写成进入后就切断平台统计的孤立场景。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`SCENE_INSTANCE_SCOPE/PLATFORM_STAT_CONTINUITY_SCOPE`。
- `R2` 审核提交场景必须承接完整审核闭环
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：审核提交场景进入后，必须以统一场景身份承接审核工作栏、待审核对象查看、审核对象打开、审核结论提交与回流反馈；不能只保留其中某一段孤立存在，也不能把本模块扩写成审核型模板的完整替代定义。
  - `R2.3` 输出能力：`C1 + C3 + C4 + C5 + C6`。
  - `R2.4` 边界绑定：`SCENE_INSTANCE_SCOPE/SIDEBAR_SCOPE/REVIEW_SCOPE/FLOW_SCOPE`。
- `R3` 审核完成后的回流必须与平台统计持续承接兼容
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：审核完成后的对象状态变化、队列变化、提示变化和统计变化，既要在审核工作栏或其他可观察区域中可解释，也不得破坏统一平台中的平台级统计持续承接关系。
  - `R3.3` 输出能力：`C2 + C5 + C6`。
  - `R3.4` 边界绑定：`PLATFORM_STAT_CONTINUITY_SCOPE/FLOW_SCOPE`。
- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2`。
  - `R4.2` 组合方式：以下方案无效：审核提交场景进入后平台级统计失去统一平台语义；审核提交场景没有统一场景身份；只有审核工作栏没有待审核列表来源；只有待审核列表没有审核结论；只有审核触发入口没有当前审核对象；审核完成后没有任何回流反馈或审核工作栏局部变化；把审核提交场景扩写成审核型模板的完整替代定义。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`SCENE_INSTANCE_SCOPE/PLATFORM_STAT_CONTINUITY_SCOPE/SIDEBAR_SCOPE/REVIEW_SCOPE/FLOW_SCOPE`。

## 5. 验证（Verification）
- `V1` 场景实例承接有效性：任一审核提交任务都必须能说明自己是统一平台中的当前审核提交场景实例。
- `V2` 平台统计持续有效性：审核提交场景进入、切换或审核过程中，平台级统计集合必须持续保持统一平台归属语义。
- `V3` 审核工作栏承接有效性：任一审核提交任务都必须能说明当前审核工作栏中的审核焦点、未选择态或待审 / 已审列表归属。
- `V4` 审核对象有效性：任一审核结论都必须能追溯到待审核列表来源，并解释当前审核对象。
- `V5` 回流可见性有效性：任一审核完成后必须至少有一处持续可观察反馈出现可解释变化，且审核工作栏中必须能看到局部变化。
- `V6` 场景承接有效性：审核提交场景必须能追溯到 `L1-M0` 和 `L1-M4`；本模块验证的是审核提交场景实例的承接来源链是否成立，至于当前场景是否属于统一平台中的合法场景实例集合，由 `L3-M0` 单独验证。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
