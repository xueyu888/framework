# review_workbench 领域总览

本文件是 `framework/review_workbench` 的领域级总览入口。它不替代各个 `L0/L1/L2/L3` 模块正文，而是回答三个跨模块问题：

1. 这个领域当前的模块树如何分层
2. 后续新增场景时应优先改哪一层
3. 什么情况才值得继续新增 `L1` 或 `L0`

`review_workbench` 当前不再被理解成“单一审查处理工作台”，而是被理解成：

`统一平台 + 平台内具体场景 + 场景模板 + 共享原子`

## 1. 当前分层

### `L0` 共享原子层

`L0` 只负责定义跨多个场景稳定复用的共享结构原子。它回答的是：

- 平台壳层与场景承接如何成立
- 平台级统计如何跨场景持续成立
- 范围上下文、查询上下文与工作面承接如何成立
- 对象列表、查看入口与全局反馈如何成立
- 文件与目录改动载荷、位置附着如何成立
- 审核对象附着与结论载荷如何成立
- 历史序列、差异承接与统一结果回流如何成立

这里的 `L0` 默认为前端邻域真相源：
- 它负责前端承接、前端表达与前端可观察反馈。
- 它不直接负责请求契约字段、回流契约字段、动作编码或历史差异编码。
- 这些服务侧语义统一外移到 `review_workbench_backend`。

当前主要模块包括：

- `L0-M0` 统一审查平台壳层与场景承接原子模块
- `L0-M1` 范围上下文与局部摘要原子模块
- `L0-M2` 查询条件与筛选上下文原子模块
- `L0-M3` 工作集与工作面承接原子模块
- `L0-M4` 对象列表与查看入口原子模块
- `L0-M5` 全局反馈与状态提示原子模块
- `L0-M6` 文件与目录改动目标与载荷原子模块
- `L0-M7` 文件与目录位置附着原子模块
- `L0-M8` 审核对象附着与结论载荷原子模块
- `L0-M9` 历史序列与差异承接原子模块
- `L0-M10` 结果影响与可观察回流原子模块
- `L0-M11` 平台级统计集合承接原子模块
- `L0-M12` 当前焦点统计承接原子模块

其中有几类共享原子需要特别注意边界：
- `L0-M6` 负责文件与目录结构改动的目标与最小载荷，不负责对象位置本体，也不负责结果回流。
- `L0-M7` 负责文件与目录对象位置、父子挂接与位置连续性，不负责改动载荷与结果回流。
- `L0-M8` 负责审核对象附着与结论载荷，不负责后续结果回流。
- `L0-M9` 负责历史序列、差异承接与元信息在前端如何承接，不负责历史请求 / 返回契约。
- `L0-M10` 负责动作完成后的结果影响、焦点结果与可观察回流，不负责动作目标本体或动作载荷本体。
- `L0-M11` 负责平台级统计项身份、统计集合、跨场景持续关系与统计更新附着，不负责具体统计项名称、聚合公式与后端统计实现。
- `L0-M12` 负责当前范围、当前对象、当前文件、当前工作页签或当前块这类唯一当前焦点上的局部统计承接；它不负责跨场景平台级统计，也不负责具体统计项名称、聚合公式与后端统计实现。

### `L1` 场景模板层

`L1` 负责定义一类场景如何由 `L0` 原子组合成稳定主线。它回答的是：

- 平台前置编排如何把当前场景稳定接到统一承接面中
- 某一类场景的前半段和后半段如何闭环
- 某类支撑支路如何回接主线

当前主要模块包括：

- `L1-M0` 平台导航到场景挂接编排模块
- `L1-M1` 文件库查询与清单编排模块
- `L1-M2` 文件库处理与来源回看编排模块
- `L1-M3` 文件库范围改动回流编排模块
- `L1-M4` 审核工作栏与结论回流编排模块
- `L1-M5` 历史追溯编排模块

其中 `L1-M0` 负责平台前置编排：也就是把平台壳、场景导航、顶部全局工具位、平台级统计集合承接面、主工作区承接面与平台反馈接成稳定前半段，但它不负责判断“哪些 `L2` 才属于统一平台中的合法场景实例集合”；这项职责由 `L3-M0` 单独承担。

当前 `L1` 层默认按“前端结构 + 外部后端契约消费”共同编排：
- 文件库查询与清单主链主要组合 `L0-M1`、`L0-M2`、`L0-M4`、`L0-M5`，并复用 `review_workbench_backend.L0-M1`；它主要回答“当前范围 -> 查询 -> 对象列表结果 -> 文件清单结果组织 -> 结果反馈”这条前半段模板主线。
- 文件库处理与来源回看主链主要组合 `L0-M3`、`L0-M4`、`L0-M5`，并复用 `review_workbench_backend.L0-M2` 与 `review_workbench_backend.L0-M4`；它主要回答“从当前结果对象打开对象工作面 -> 分块预览与处理 -> 来源定位 / 原位回看 -> 动作反馈”这条后半段模板主线。
- 文件库范围改动回流支路主要组合 `L0-M1`、`L0-M2`、`L0-M6`、`L0-M7`、`L0-M10`，并复用 `review_workbench_backend.L0-M3`；它主要回答“当前范围下做结构改动 -> 范围节点与范围状态回流 -> 查询重新稳定”这条支撑支路，而不是替代文件库查询主链本身。
- 审核工作栏与结论回流主链主要组合 `L0-M3`、`L0-M4`、`L0-M5`、`L0-M8`、`L0-M10`，并复用 `review_workbench_backend.L0-M1`、`L0-M2`、`L0-M4`。
- 历史追溯主链主要组合 `L0-M4`、`L0-M5`、`L0-M9`，并复用 `review_workbench_backend.L0-M1`、`L0-M2`、`L0-M5`。

这里要特别固定一条写法约束：`L1` 只定义“模板自身如何由 `L0` 原子与外部领域契约组成”，不在正文里声明“默认承接另一个 `L1` 的结果”。若需要把多条 `L1` 模板主线或支路共同挂接到一个具体场景，统一由 `L2` 显式组合；若需要把平台前置编排与多个具体场景实例共同挂接到统一平台，统一由 `L3` 显式组合。不要在 `L1` 正文里形成仅靠叙述存在的同层依赖图。

### `L2` 平台内具体场景层

`L2` 负责定义统一平台中的具体场景实例如何承接对应模板主线。它回答的是：

- 当前平台里具体有哪些场景实例
- 每个具体场景实例如何承接统一平台与对应模板
- 每个具体场景实例如何保持自己的场景上下文，而不替代 `L1` 模板层或 `L3` 平台层职责

当前主要模块包括：

- `L2-M0` 文件库处理场景模块
- `L2-M1` 审核提交场景模块
- `L2-M2` 历史追溯场景模块

这里的 `L2` 默认不负责重新定义模板主线内部的编排细节，也不负责判断“哪些场景属于统一平台中的合法场景实例集合”。前者继续由对应 `L1` 模板模块承担，后者继续由 `L3-M0` 单独承担。`L2` 主要回答的是：某个具体场景实例如何以统一身份进入平台，并承接已经成立的模板主线。

### `L3` 统一平台层

`L3` 负责定义统一平台本身如何成立，并如何挂接 `L2` 具体场景。它回答的是：

- 统一平台如何承接合法场景实例集合
- 当前场景如何以唯一归属被挂接到统一承接面
- 平台如何保持平台级上下文成立而不吞并具体场景内部主线

当前主要模块包括：

- `L3-M0` 统一审查平台场景模块

## 2. 默认扩展规则

后续新增场景时，默认按下面顺序判断：

### 规则 A：优先新增 `L2`

如果新需求只是平台里多了一个具体场景，但它本质上仍属于现有模板之一，那么默认新增 `L2` 模块，不先改 `L1` 或 `L0`。

典型例子：

- 新增一个“抽检审核场景”，但它仍然属于审核型主线
- 新增一个“版本回放场景”，但它仍然属于追溯型主线

这类变化优先落在 `L2`。

### 规则 B：只有出现新的稳定场景类型时，才新增 `L1`

如果新场景无法自然归入文件库处理型、审核型、追溯型等已有模板，而且这种差异是跨实例稳定复用的，那么才新增新的 `L1` 模板模块。

判断标准不是“当前项目觉得不一样”，而是：

- 是否形成新的稳定主线
- 是否会被多个未来场景复用
- 是否不是现有 `L1` 通过组合就能覆盖

### 规则 C：只有出现新的共享结构原子时，才新增 `L0`

如果多个 `L1` 模板都开始重复依赖某个新的稳定结构，而这个结构又不能被现有 `L0` 原子表达，那么才允许新增 `L0`。

判断标准是：

- 是否跨多个场景模板复用
- 是否是结构原子，而不是功能清单
- 是否不是某个 `L1/L2` 模块内部的实例化差异

## 3. 修改判定规则

为了避免未来继续把平台结构改乱，后续修改时优先按下面方式判断：

### 情况 1：只是当前场景字段变了

例如新增字段、筛选项、列表列、统计项、审核意见字段、历史展示字段。

处理方式：

- 优先改 `project.toml` 或后续实例化配置
- 不优先改 framework

### 情况 2：只是多了一个同类型场景

例如新增一个新的审核场景、历史场景、对象处理场景。

处理方式：

- 优先新增 `L2`

### 情况 3：出现了新的场景主线

例如新场景既不是文件库处理型，也不是审核型，也不是追溯型，而且会稳定复用。

处理方式：

- 新增 `L1`

### 情况 4：多个模板都在重复借同一种新结构

例如多个 `L1` 都开始重复定义新的共享对象关系、共享承接面、共享契约结构。

处理方式：

- 新增或重构 `L0`

## 4. 当前作者入口建议

如果后续继续改 `review_workbench`，默认阅读顺序建议是：

1. 先读本文件，判断当前需求应落在哪一层
2. 再读 `L3-M0`，确认平台层承接与治理语义
3. 再读 `L1-M0`，确认平台前置编排如何承接当前场景
4. 如果是具体场景，先找对应 `L2`
5. 如果 `L2` 不够，再看是否应落到某个现有 `L1`
6. 只有在多个 `L1` 都出现共享缺口时，才回到 `L0`

## 5. 当前总原则

`review_workbench` 后续扩展时，默认新增具体场景，不默认新增底层结构。

换句话说：

- 优先扩 `L2`
- 谨慎扩 `L1`
- 更谨慎扩 `L0`

只有这样，统一平台壳、具体场景、场景模板和共享原子之间的单向收敛关系才会稳定。

## 6. 边界与验证写法约定

为了让 `review_workbench` 的 framework 文档更适合作为后续 Config / Code / Evidence 的稳定上游，后续新增或修改模块时，默认采用下面这组写法约定：

- 边界项优先写“允许变化的维度 + 必须成立的最小结构条件”，不要只写“需要稳定”。
- 边界项优先回答“什么可以变、变到哪里仍然合法、最少要保留什么结构”，不要退回对象名词清单。
- 涉及当前焦点的边界，优先写成“唯一当前场景 / 唯一当前范围 / 唯一当前工作面”这类可判定条件。
- 涉及来源、归属、定位的边界，优先写成“必须说明来源 / 归属 / 锚点 / 落点”，而不是泛写“可追溯”。
- 涉及反馈的边界，优先写成“状态变化后至少有一处持续可观察反馈”，不要只写“有反馈”。
- 验证项应直接复用边界语义，优先写成“必须能识别什么、必须说明什么、至少出现什么变化”，避免和边界口径脱节。
- 验证项默认要能区分“结构没成立”与“实例值不同”这两类问题，因此尽量避免把具体现实例值写进 `V*`。
- 若某个模块新增了新的边界写法或验证写法，优先先看这组约定是否也要同步更新到 `README`。

## 7. 设计约束小结

为了避免 `review_workbench` 后续继续长回“同层依赖、复合原子、实现口吻混入 framework”这几类问题，新增或重写模块时默认再检查下面这组约束：

- `L0` 只写自足共享原子；若一个模块必须依赖多个 `L0` 才成立，它更可能应被拆回更纯原子，或上提给 `L1` 编排。
- 平台级长期统计属于独立共享结构；若它在多个场景切换前后都成立，应优先沉成独立 `L0` 原子，而不是混入单一场景摘要或平台反馈实例值。
- `L1` 只写“模板如何由 `L0` 与外部领域契约组成”，不要在正文里声明“默认承接另一个 `L1` 的结果”。
- `L2` 只负责具体场景实例承接模板；不要在 `L2` 里重新发明模板主线，也不要吞并 `L3` 的平台治理职责。
- `L3` 只负责统一平台如何承接合法场景实例集合；不要把文件处理、审核、追溯等内部主线改写成平台唯一主线。
- `B*` 优先写成稳定结构单元，而不是动作阶段名或流程阶段名；动作链条优先交给 `R*` 表达。
- 工作页签、对象页签、辅助回看页签等长期工作承接关系应优先沉成工作资源结构，而不是只以 UI 页签名词描述。
- 结果区应优先表达“对象结果承接 + 结果角色结构”，不要把主标识列、状态列、聚合列和操作列直接退化成具体字段清单。
- framework 正文优先使用“承接面、触发入口、附着关系、回流、契约字段”这类结构口吻，不要退回“按钮、控件、接口返回字段”这类实现口吻。
- 若某个新问题只在单一场景实例内成立，优先落到 `L2` 或项目配置，不要为了当前实例差异扩写新的 `L0`。
- 若某个新结构会被多条 `L1` 模板稳定复用，再考虑把它沉成新的 `L0`；否则宁可保持在上层显式组合。
- 若文档里需要写“复用某模块”，优先把复用关系写进结构化 `B* / R* / V*` 或上层组合关系，不要只留在说明性正文里。
- 每次较大调整后，都应至少复查一次：`L0` 是否仍自足、`L1` 是否仍无同层隐式依赖、README 是否仍和当前模块树一致。
### 外部后端契约依赖
`review_workbench` 当前把后端契约真相外移到独立领域 `review_workbench_backend`。也就是说：
- `review_workbench` 继续负责前端邻域中的原子、编排与场景。
- `review_workbench_backend` 负责统一审查平台服务侧的请求 / 返回 / 回流后端契约。
- `review_workbench` 中若提到“消费后端契约”，默认是复用 `review_workbench_backend.L0.M0~M5`，而不是在本目录重复维护一份后端契约真相。
- 对于审核与追溯这类场景，`review_workbench` 里的 `L0-M8`、`L0-M10`、`L0-M9` 只负责审核对象附着 / 结论载荷、结果影响与回流、历史承接与前端可观察结构；对应的动作请求、历史请求、差异返回等契约语义，由 `review_workbench_backend` 中对应模块承担。
- 对于文件库这类场景，`L0-M6`、`L0-M7` 与 `L0-M10` 分别负责改动载荷、对象位置与结果回流这些前端结构承接；对应的结构改动请求 / 失效回流、查看请求 / 返回契约分别由 `review_workbench_backend.L0-M3`、`L0-M2` 承担。
