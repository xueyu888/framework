# 对象列表与查看入口原子模块:ReviewWorkbenchObjectListActionAndInspectionEntryAtoms

@framework

本模块定义 `review_workbench` 领域最小的结果对象承接、结果角色结构与查看入口结构。推导链为：`framework/frontend/L1-M4-动作-触发-原子模块.md` + `framework/frontend/L1-M9-承载-展示与容器-原子模块.md` + `framework/frontend/L1-M12-浏览-集合浏览-原子模块.md` + `framework/frontend/L1-M15-反馈-标记与反馈-原子模块.md` -> 本模块。也就是说，平台中的对象结果集合、主标识列、属性/状态/聚合/操作角色、行级动作、批量动作、查看入口和侧边承接面，不单独重写表格或抽屉基础，而是把前端已有的集合浏览、展示容器、触发入口和标记反馈约束成“结果承接与查看入口”这组共享领域原子。

本模块定义统一审查平台中的结果对象承接、结果角色与查看入口共享结构。它回答“当前结果对象集合如何被承接”“结果中的主标识、属性、状态、聚合和操作角色如何成立”以及“如何从结果对象进入来源查看、审核查看或历史查看承接面”这三个最小问题。它不负责范围树与查询条件，也不直接代表文件库、审核或追溯任一具体场景；它的主要作用是作为 `L0-M8`、`L0-M9`、`L0-M10` 以及对应 `L1` 场景模板的共享结构。

## 1. 能力声明（Capability Statement）
- `C1` 对象集合表达能力：将列表、表格、卡片中的对象集合与空态组织成稳定结构。
- `C2` 结果主标识承接能力：将对象结果中的主标识、主要展示字段与行结构组织成稳定承接。
- `C3` 结果角色表达能力：将属性角色、状态角色、聚合角色和操作角色组织成稳定结果结构。
- `C4` 动作入口表达能力：将行级动作、批量动作与上下文查看动作组织成稳定动作结构。
- `C5` 查看入口表达能力：将来源查看、审核查看、历史查看等进入承接面组织成稳定查看结构。
- `N1` 非职责声明：不负责工作集切换，不负责平台统计集合或全局反馈提示，不负责后端事务实现，也不直接定义文件库、审核型或追溯型场景实例。

## 2. 边界定义（Boundary / 参数）
- `OBJECTSET` 对象集合边界：对象数量、排序和空态形式允许变化，但结果成立时必须存在可识别对象集合，空态时必须能解释当前结果状态。
- `OBJECTROW` 对象行边界：主要字段、行结构和展示顺序允许变化，但每个对象都必须具备稳定对象标识，并能解释行与对象的承接关系。
- `RESULT_ROLE_SCOPE` 结果角色边界：属性角色、状态角色、聚合角色和操作角色允许变化，但每种角色都必须附着对象结果成立，不能退化成无法解释归属的孤立列片段。
- `OBJECTACTION` 对象动作边界：行级动作、批量动作和查看动作入口允许变化，但每个动作都必须附着对象或对象集合，不能脱离对象归属独立存在。
- `INSPECTION_ENTRY` 查看入口边界：来源、审核、历史等查看入口种类允许变化，但每个入口都必须说明由哪个对象进入哪个承接面。
- `INSPECTION_ANCHOR` 查看锚点边界：定位方式、片段边界和定位恢复策略允许变化，但进入查看承接面后必须能解释当前定位锚点及恢复关系。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 对象集合基：frontend.L1.M12[R1,R2] + frontend.L1.M15[R2]。由集合结构、当前项、空态与状态反馈约束成对象集合和当前结果状态结构。来源：`C1 + OBJECTSET`。
- `B2` 主标识承接基：frontend.L1.M9[R1,R2] + frontend.L1.M12[R2]。由展示容器、元信息和当前项导航约束成对象标识、主要展示字段与主标识结构。来源：`C2 + OBJECTROW`。
- `B3` 结果角色基：frontend.L1.M9[R2] + frontend.L1.M15[R1]。由展示容器、状态标记与角色附着关系约束成属性角色、状态角色、聚合角色和操作角色结构。来源：`C3 + RESULT_ROLE_SCOPE`。
- `B4` 动作入口基：frontend.L1.M4[R1,R2] + frontend.L1.M15[R1]。由行级动作、批量动作、查看入口和状态标记约束成动作结构。来源：`C4 + OBJECTACTION`。
- `B5` 查看入口基：frontend.L1.M9[R2] + frontend.L1.M12[R2] + frontend.L1.M4[R1]。由侧边承接面、对象导航和查看触发入口约束成来源查看、审核查看、历史查看等进入结构。来源：`C5 + INSPECTION_ENTRY + INSPECTION_ANCHOR`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 对象集合必须导出对象结构
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：在 frontend 的集合浏览与展示容器结构上，对象集合必须能说明有哪些对象被展示、对象如何被承接以及每个结果对象的主标识是什么；不能只有一个“表格区域”而无对象行结构。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`OBJECTSET/OBJECTROW`。
- `R2` 结果角色必须附着对象结果
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：属性角色、状态角色、聚合角色和操作角色都必须附着对象结果成立；不能只有若干列片段，而无法解释它们附着于哪个对象结果。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`OBJECTROW/RESULT_ROLE_SCOPE`。
- `R3` 动作必须附着对象
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：在 frontend 的触发结构上，动作入口必须附着于对象结果行或对象结果集合，不能定义脱离结果承接的孤立动作。
  - `R3.3` 输出能力：`C2 + C4`。
  - `R3.4` 边界绑定：`OBJECTROW/OBJECTACTION/RESULT_ROLE_SCOPE`。
- `R4` 查看入口必须附着对象
  - `R4.1` 参与基：`B2 + B5`。
  - `R4.2` 组合方式：在 frontend 的展示容器和导航结构上，任何查看入口都必须能说明“当前是由哪个对象进入哪个查看承接面”；不能只有承接面内容而没有对象来源与定位锚点。
  - `R4.3` 输出能力：`C2 + C5`。
  - `R4.4` 边界绑定：`OBJECTROW/INSPECTION_ENTRY/INSPECTION_ANCHOR`。
- `R5` 结果承接与查看入口共享闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：对象结果集合、主标识结构、结果角色、动作入口与查看入口必须共同形成“展示结果对象 -> 识别主标识 -> 通过不同结果角色理解对象 -> 对对象执行动作 / 进入查看承接面”的稳定共享结构；该结构可被文件库、审核和追溯等专用原子继续复用。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`OBJECTSET/OBJECTROW/RESULT_ROLE_SCOPE/OBJECTACTION/INSPECTION_ENTRY/INSPECTION_ANCHOR`。
- `R6` 无效方案
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：以下方案无效：跳过 frontend 集合、展示和触发结构直接声明对象工作面；只有空表格没有对象结构；只有若干列片段但没有主标识；结果角色无法说明附着于哪个对象；只有操作入口没有对象归属；批量动作无法说明作用于哪一组对象；查看承接面可以点开但无法说明对应哪个对象或当前查看位置。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`OBJECTSET/OBJECTROW/RESULT_ROLE_SCOPE/OBJECTACTION/INSPECTION_ENTRY/INSPECTION_ANCHOR`。

## 5. 验证（Verification）
- `V1` 对象可识别性：结果成立时必须存在可识别对象集合，并且每个对象结果都必须具备稳定对象标识。
- `V2` 主标识有效性：任一结果对象都必须能识别主标识结构，而不是只剩若干属性片段。
- `V3` 结果角色有效性：任一属性角色、状态角色、聚合角色或操作角色都必须能说明附着的对象结果归属。
- `V4` 查看入口有效性：任一查看入口都必须同时说明对象来源、承接面归属与当前定位锚点。
- `V5` 共享底座有效性：本模块必须能被文件库、审核型或追溯型专用原子复用，而不是被误写成某个具体场景的唯一真相源。
- `V6` 前端承接有效性：对象结果、结果角色、对象动作和查看入口必须能追溯到 frontend 的触发、展示容器、集合浏览和标记反馈结构。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
