# 审核工作栏与结论回流编排模块:ReviewWorkbenchReviewSidebarAndDecisionReturnOrchestration

@framework

本模块定义统一审查平台中审核工作栏与结论回流主线是怎么接起来的，也就是“在审核工作栏中识别当前审核焦点并查看待审 / 已审对象结果队列 -> 打开当前审核对象的只读审核工作资源 -> 给出审核结论 -> 看到结果回流”。它位于 `review_workbench` 的编排层，关心的是审核工作栏怎样承接当前审核焦点与队列、审核对象如何进入只读审核工作资源、审核结论如何表达、结论如何回流，而不是具体审核规则、权限流转或表单控件实现。

本模块建立在 `review_workbench.L0.M3`、`review_workbench.L0.M4`、`review_workbench.L0.M5`、`review_workbench.L0.M8`、`review_workbench.L0.M10` 与外部领域 `review_workbench_backend.L0.M1`、`review_workbench_backend.L0.M2`、`review_workbench_backend.L0.M4` 这些领域原子之上。其中 `L0.M3` 提供工作资源切换底座，`L0.M4` 提供对象结果与查看入口底座，`L0.M5` 提供全局反馈与状态提示底座，`L0.M8` 提供审核对象附着与结论载荷底座，`L0.M10` 提供结果影响与可观察回流底座，`review_workbench_backend.L0.M1` 提供审核列表查询契约，`review_workbench_backend.L0.M2` 提供审核对象打开与查看契约，`review_workbench_backend.L0.M4` 提供审核动作请求与结果回流契约。

本模块只定义审核型场景模板本身，不在 `L1` 层内声明自己默认承接哪个同层平台前置模板；若项目需要把平台前置编排与审核型场景主线共同挂接为一个具体审核场景，应由 `L2-M1` 与 `L3-M0` 在更高层显式组合，而不是在本模块正文中隐式写成 `L1 -> L1` 依赖。

## 1. 能力声明（Capability Statement）
- `C1` 审核工作栏承接能力：支持在审核型场景中稳定承接当前审核焦点、待审 / 已审列表和审核输入区。
- `C2` 审核队列查看能力：支持在审核型场景中稳定查看待审核对象列表、已审核对象列表或其他审核队列分区。
- `C3` 审核对象打开能力：支持从审核工作栏中的队列项进入当前对象的只读审核工作资源并保留对象上下文。
- `C4` 审核结论提交能力：支持对当前审核对象给出通过、驳回、退回或补充意见等结论。
- `C5` 审核回流能力：支持审核完成后把对象状态、统计和反馈回流到可观察区域。
- `C6` 审核后端契约消费能力：支持审核列表、审核打开、审核提交和结果回流稳定消费后端契约。
- `C7` 平台统计回接能力：支持审核结论完成后，将其中包含的统计变化回接到平台统计集合并保持统一平台语义。
- `N1` 非职责声明：不负责文件块纠错动作、不负责历史差异回看、不负责具体审核业务规则。

## 2. 边界定义（Boundary / 参数）
- `SIDEBAR_SCOPE` 审核工作栏边界：审核工作栏、当前焦点摘要、输入区和局部队列概览允许采用不同布局，但必须共同落在同一审核工作栏承接面中。来源：`C1`。
- `REVIEW_LIST_SCOPE` 审核列表边界：待审核对象列表、已审核对象列表、筛选和摘要允许变化，但这些审核队列必须落在当前审核场景上下文中，并能识别当前审核对象来源。来源：`C2`。
- `REVIEW_DETAIL_SCOPE` 审核详情边界：审核对象、审核上下文和审核工作资源允许变化，但任一次审核打开都必须稳定挂接到当前对象上，且审核工作资源只负责查看当前对象的块预览、来源或审核信息，不承担可编辑纠错动作。来源：`C3`。
- `DECISION_SCOPE` 审核结论边界：结论类型、结论载荷和附加意见允许变化，但每次结论提交都必须附着当前审核对象。来源：`C4`。
- `FLOW_SCOPE` 回流边界：审核后的对象状态、统计与反馈允许通过不同区域表达，但审核完成后必须至少有一处持续可观察回流，且审核工作栏中必须能观察到局部变化。来源：`C5`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：审核列表、审核对象打开、审核提交和回流允许映射为不同后端契约字段，但每类审核动作都必须稳定绑定到对应请求与返回。来源：`C6`。
- `PLATFORM_STAT_RETURN_SCOPE` 平台统计回接边界：审核结论完成后，若结果影响中包含统计变化，则该变化允许通过不同回接路径进入平台统计集合，但不得停留在审核工作栏局部、对象局部或不可观察内部状态中。来源：`C7`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 审核工作栏基：`L0.M3[R1,R2] + L0.M4[R1,R2]`。来源：`C1 + C2 + SIDEBAR_SCOPE + REVIEW_LIST_SCOPE`。
- `B2` 审核工作资源基：`L0.M3[R1] + L0.M4[R1,R3]`。来源：`C3 + REVIEW_DETAIL_SCOPE`。
- `B3` 审核结论与回流基：`L0.M8[R1,R2,R3] + L0.M5[R1,R2,R3] + L0.M10[R1,R2,R3]`。来源：`C4 + C5 + DECISION_SCOPE + FLOW_SCOPE`。
- `B4` 后端契约消费基：`review_workbench_backend.L0.M1[R1,R2] + review_workbench_backend.L0.M2[R1,R2] + review_workbench_backend.L0.M4[R1,R3]`。来源：`C6 + BACKEND_CONTRACT_SCOPE`。
- `B5` 平台统计回接基：`L0.M10[R1,R3] + L0.M11[R3,R4]`。来源：`C7 + PLATFORM_STAT_RETURN_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 审核工作栏必须先承接当前审核焦点
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：审核型场景中的审核工作栏必须先能说明当前审核焦点是谁，或明确处于未选择态，并且工作栏中要能稳定承接待审核 / 已审核等审核队列，再由这些队列项打开当前审核对象；不能只有输入区和提交入口，而没有当前审核焦点与审核队列语义。
  - `R1.3` 输出能力：`C1 + C2 + C3`。
  - `R1.4` 边界绑定：`SIDEBAR_SCOPE/REVIEW_LIST_SCOPE/REVIEW_DETAIL_SCOPE`。
- `R2` 审核工作资源必须由待审核列表打开并回接工作栏
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：审核工作资源必须能说明来自哪个待审核或已审核对象结果项以及当前审核对象是谁，同时审核工作栏中的当前对象摘要、输入区和局部队列概览也要同步回接到该当前审核对象；审核工作资源本身只负责查看当前对象的块预览、来源和审核信息，不承担可编辑纠错动作；不能直接构造无列表来源的审核页面。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`SIDEBAR_SCOPE/REVIEW_LIST_SCOPE/REVIEW_DETAIL_SCOPE`。
- `R3` 审核结论必须附着当前审核对象并带出工作栏回流
  - `R3.1` 参与基：`B1 + B3`。
  - `R3.2` 组合方式：任何审核结论都必须作用于当前审核对象，并在结论提交后带出对象状态、局部队列概览或提示变化；不能出现脱离审核对象的黑盒结论，也不能只在中部列表变化而审核工作栏无感。
  - `R3.3` 输出能力：`C1 + C3 + C4 + C5`。
  - `R3.4` 边界绑定：`SIDEBAR_SCOPE/REVIEW_DETAIL_SCOPE/DECISION_SCOPE/FLOW_SCOPE`。
- `R4` 审核型场景必须消费后端契约
  - `R4.1` 参与基：`B2 + B3 + B4`。
  - `R4.2` 组合方式：审核列表、审核对象打开、审核结论提交和结果回流都必须稳定接到服务契约中；否则这套审核型场景没法复用。
  - `R4.3` 输出能力：`C4 + C5 + C6`。
  - `R4.4` 边界绑定：`REVIEW_LIST_SCOPE/DECISION_SCOPE/FLOW_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R5` 审核结论中的统计变化必须回接平台统计集合
  - `R5.1` 参与基：`B3 + B5`。
  - `R5.2` 组合方式：审核结论完成后，若结果影响中包含统计变化，则该变化必须回接到平台统计集合当前状态，并保持统一平台语义；它可以与审核工作栏局部回流或全局反馈并存，但不能只停留在审核工作栏局部、对象局部或不可观察内部状态中。
  - `R5.3` 输出能力：`C5 + C7`。
  - `R5.4` 边界绑定：`FLOW_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。
- `R6` 审核型场景闭环
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：审核工作栏、待审核 / 已审核队列、只读审核工作资源、审核结论、结果回流和平台统计回接要共同组成“在审核工作栏中识别当前审核焦点 -> 查看待审 / 已审队列 -> 打开当前对象的只读审核工作资源 -> 给出审核结论 -> 在工作栏与其他可观察区域看到回流反馈并完成必要的平台统计回接”的完整审核型主线。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7`。
  - `R6.4` 边界绑定：`SIDEBAR_SCOPE/REVIEW_LIST_SCOPE/REVIEW_DETAIL_SCOPE/DECISION_SCOPE/FLOW_SCOPE/BACKEND_CONTRACT_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。
- `R7` 无效方案
  - `R7.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R7.2` 组合方式：以下方案无效：只有审核触发入口没有审核对象；只有审核工作栏没有待审 / 已审队列来源；只有审核对象没有可提交结论；审核工作资源里混入可编辑纠错动作而不是只读审核查看；审核完成后没有任何状态、统计或工作栏局部回流；结果影响已包含统计变化但平台统计集合没有任何可解释更新；界面上有审核动作但接不上稳定服务。
  - `R7.3` 失效结论：`N1`。
  - `R7.4` 边界绑定：`SIDEBAR_SCOPE/REVIEW_LIST_SCOPE/REVIEW_DETAIL_SCOPE/DECISION_SCOPE/FLOW_SCOPE/BACKEND_CONTRACT_SCOPE/PLATFORM_STAT_RETURN_SCOPE`。

## 5. 验证（Verification）
- `V1` 审核工作栏承接有效性：任一审核型场景都必须存在可识别审核工作栏，并能说明当前审核焦点或未选择态。
- `V2` 列表来源有效性：任一审核工作面都必须能追溯到待审核列表来源，并识别当前审核对象。
- `V3` 结论归属有效性：任一审核结论都必须有明确审核对象归属。
- `V4` 回流可见性有效性：任一审核完成后，必须至少有一处持续可观察反馈出现可解释变化，且审核工作栏中必须能看到局部变化。
- `V5` 后端契约消费有效性：审核列表、审核打开、审核提交和回流必须稳定绑定到对应请求与返回。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
