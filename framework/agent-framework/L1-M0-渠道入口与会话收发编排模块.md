# 渠道入口与会话收发编排模块:ChannelIngressSessionDispatchOrchestration

@framework

本模块把渠道入口、会话定位、消息收发和出站路由接成一条会话收发线。它只负责 4 件事：平台消息怎么转成入站事件、会话键怎么稳定定位、会话记录怎么续写、最终结果怎么回到目标渠道；不负责提示上下文装配、模型调用和后台任务触发。

## 1. 能力声明（Capability Statement）

- `C1` 渠道入站接收能力：根据已启用渠道和平台消息输出稳定入站事件。
- `C2` 会话键定位能力：根据渠道标识、聊天标识和显式 override 输出稳定会话键。
- `C3` 会话记录续写能力：根据会话键持续加载和追加 JSONL 会话记录。
- `C4` 出站发送路由能力：根据出站结果和目标渠道输出唯一发送路径。
- `C5` 非能力项：不负责提示上下文装配、模型调用、记忆整理和后台任务触发。

## 2. 边界定义（Boundary / 参数）

- `CHANNELSET` 渠道集合边界：只有已启用并完成初始化的渠道才能接收入站事件和发送出站消息。来源：`C1 + C4`。
- `SESSIONKEY` 会话键边界：同一聊天默认只能收敛为一个 `channel:chat_id` 会话键。来源：`C2 + C3`。
- `HISTORYFMT` 会话记录格式边界：会话记录必须能稳定写成 JSONL metadata line 和 message line。来源：`C3`。
- `SENDTARGET` 出站目标边界：任一出站结果都必须绑定一个明确渠道和一个明确聊天目标。来源：`C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 入出站消息总线：L0.M0[R1]：接收入站事件、排队出站事件，能发布和消费待处理消息事件。来源：`C1 + C4 + CHANNELSET + SENDTARGET`。
- `B2` 会话键定位器：L0.M0[R2]：固定 `channel:chat_id` 组合键和显式 override，能为同一聊天输出稳定会话键。来源：`C2 + SESSIONKEY`。
- `B3` 会话记录存取器：L0.M0[R3]：把消息记录和整理游标持续写入 JSONL，能按会话键加载和续写会话记录。来源：`C2 + C3 + SESSIONKEY + HISTORYFMT`。
- `B4` 渠道适配管理器：L0.M0[R4]：初始化已启用渠道并维护发送路由，能把出站事件投递到唯一目标渠道。来源：`C1 + C4 + CHANNELSET + SENDTARGET`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 渠道入口成立
  - `R1.1` 参与基：`B1 + B2 + B4`。
  - `R1.2` 组合方式：先由渠道适配管理器接住平台消息，再由入出站消息总线发布入站事件，最后由会话键定位器为该事件固定唯一会话键。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`CHANNELSET/SESSIONKEY`。
- `R2` 会话续写成立
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：先固定会话键，再由会话记录存取器按该键加载并续写 JSONL 会话记录。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`SESSIONKEY/HISTORYFMT`。
- `R3` 出站路由成立
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：先由入出站消息总线接住出站结果，再由渠道适配管理器按目标渠道和聊天目标执行唯一发送路由。
  - `R3.3` 输出能力：`C4`。
  - `R3.4` 边界绑定：`CHANNELSET/SENDTARGET`。
- `R4` 会话收发闭环成立
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：把渠道入口、会话定位、会话续写和出站路由接成一条完整会话收发闭环。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`CHANNELSET/SESSIONKEY/HISTORYFMT/SENDTARGET`。
- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：未启用渠道仍接收消息、同一聊天映到多个会话键、会话记录无法稳定续写或出站结果无明确目标的组合无效。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 边界绑定：`CHANNELSET/SESSIONKEY/HISTORYFMT/SENDTARGET`。

## 5. 验证（Verification）

- `V1` 渠道入口有效性：只有 `CHANNELSET` 内渠道才能通过 `R1` 进入入站链。
- `V2` 会话键有效性：同一聊天在相同 override 条件下必须输出稳定 `SESSIONKEY`。
- `V3` 会话续写有效性：同一会话键必须能稳定加载和续写满足 `HISTORYFMT` 的 JSONL 记录。
- `V4` 出站路由有效性：同一出站结果必须只能落到一个满足 `SENDTARGET` 约束的目标渠道。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
