# 测试列管理模块:TestColumnsManagement

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取所有列列表能力：测试系统一次性拉取并返回当前所有已配置的列定义数据集合的能力。
- `C2` 测试查询列能力：测试根据特定参数进行模糊匹配或精确过滤，从而查询目标列的能力。
- `C3` 测试新增列能力：测试向系统中添加新的列属性配置或字段元数据的能力。
- `C4` 测试更新列配置能力：测试对系统中已存在的列属性进行信息修改和保存的能力。
- `C5` 测试删除单列记录能力：测试从系统中精确移除单个列定义数据的能力。
- `C6` 测试批量删除列记录能力：测试一次性从系统中清除多条列定义数据的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `GETCOLUMNSALL` 边界：is_banned (属性: 可选，类型: boolean | null) 来源：`C1`。
- `GETCOLUMNSSEARCH` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围: >= 1)
                          limit (属性: 可选，类型: integer，默认值: 20，有效范围: >= 1)
                          column_name (属性: 可选，类型: string | null)
                          column_alias (属性: 可选，类型: string | null)
                          is_banned (属性: 可选，类型: boolean | null) 来源：`C2`。
- `POSTCOLUMNADD` 边界：column_name (属性: 必选，类型: string)
                       column_alias (属性: 必选，类型: string)
                       is_banned (属性: 必选，类型: boolean)
                       column_type (属性: 必选，类型: string)
                       metadata_json (属性: 必选，类型: object) 来源：`C3`。
- `POSTCOLUMNUPDATE` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id)
                          column_name (属性: 可选，类型: string | null，有效范围: 长度 >= 1)
                          column_alias (属性: 可选，类型: string | null，有效范围: 长度 >= 1)
                          is_banned (属性: 可选，类型: boolean | null)
                          column_type (属性: 可选，类型: string | null，有效范围: 长度 >= 1)
                          metadata_json (属性: 可选，类型: object) 来源：`C4`。
- `DELETECOLUMNID` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id) 来源：`C5`。
- `DELETECOLUMNS` 边界：无，来源：`C6`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 单基调用有效原则
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`。
  - `R1.2` 组合方式：每个基独立调用，每个基每个参数均满足第 2 节边界参数有效条件；对依赖 id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：在边界有效条件下分别稳定导出 `C1~C6` 的正向能力。
  - `R1.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R2` 单基调用无效原则
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`。
  - `R2.2` 组合方式：每个基独立调用；对每个基的每个参数，针对每一类适用的无效输入（字段缺失、类型不符、越界值、无效标识等）至少执行 1 次，保证覆盖率。
  - `R2.3` 失效结论：`N1`（边界无效条件下应进入失败态 `!=2000`）。
  - `R2.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R3` 随机组合有效原则
  - `R3.1` 参与基：`{B1, B2, B3, B4, B5, B6}` 的随机子集。
  - `R3.2` 组合方式：在各基边界参数有效前提下，合法随机组合需要满足：
  - `R3.2.a` 含 `B4/B5` 时，`id` 必须来自本轮 `B1` 列表、本轮 `B2` 查询结果，或本轮 `B3` 新增返回值。
  - `R3.2.b` 含 `B6` 时，`B6` 必须作为该轮清理链路末步。
  - `R3.2.c` 同轮对同一 `id` 不允许“先删后改”或“删除后继续删除/更新”。
  - `R3.3` 输出能力：在满足 `R3.2` 约束下，导出 `C1~C6` 的高覆盖正向能力。
  - `R3.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R4` 特殊组合非有效原则
  - `R4.1` 参与基：`B4`、`B5`、`B6` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若出现“无来源直调依赖基、`B6` 非末步、删除后继续操作同一目标、跨轮串用失效标识”等情况，就判定为非有效组合处理。
  - `R4.3` 失效结论：`N1`（触发 `R4.2` 的特殊组合应进入失败态 `!=2000`）。
  - `R4.4` 边界绑定：`POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS + GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD`。

## 5. 验证（Verification）

- `V1` 单基调用有效验证（对应 `R1`）：通过 = `B1~B6` 有效样例均 `2000`；失败 = 任一样例 `!=2000`；证据 = 单 API 有效请求/响应表。

- `V2` 单基调用无效验证（对应 `R2`）：通过 = `B1~B6` 无效样例均 `!=2000`；失败 = 任一样例 `2000`；证据 = 单 API 无效请求/响应表。

- `V3` 随机有效组合验证（对应 `R3`）：通过 = 随机有效组合满足 `R3.2` 且全 `2000`；失败 = 约束不满足或任一 `!=2000`；证据 = 组合命中 + 标识溯源 + 响应日志。

- `V4` 特殊无效组合验证（对应 `R4`）：通过 = 触发 `R4.2` 的组合均 `!=2000`；失败 = 任一 `2000`；证据 = 无效样例集 + 规则命中记录。

- `V5` 分层与优先级验证：通过 = 顺序“先 `R1/R2` 后 `R3/R4`”，且 `R1/R2` 全通过后才进组合测试，冲突按 `R4>R3`；失败 = 顺序/门槛/判定错误；证据 = 流水线阶段日志。

- `V6` 能力导出与问题分层校验：通过 = `C1~C6` 均有合法 `R->C` 导出链，失败样例可分“结构问题/实例问题”；失败 = 能力未覆盖或不可分层；证据 = 覆盖报告 + 失败分类报告。

- `V7` 覆盖率与稳定性验证：通过 = 固定 seed 多轮下覆盖达标、每个 B* 命中 >= 1、关键链路“新增→查询→更新→删除”稳定且成功率达阈值；失败 = 覆盖缺失/命中为0/顺序漂移/偶发失败；证据 = 覆盖矩阵 + 轮次统计 + 归因报告。

- `V8` 结论判定：通过 = `V1~V7` 全通过；失败 = 任一失败；证据 = 汇总判定单。
