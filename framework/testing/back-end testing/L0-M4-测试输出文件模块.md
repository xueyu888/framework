# 测试输出文件模块:TestOutputFile

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取输出文件能力：测试系统根据传入的参数，成功提取并返回最终处理生成的结果文件或输出数据的能力。
- `C2` 测试获取异常列表能力：测试系统获取任务运行过程中产生的异常记录明细，或整体执行结果统计汇总数据的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `POSTGETOUTPUTFILE` 边界：execute_id (属性: 必选，类型: string)
                           node_id (属性: 必选，类型: string)
                           range_start (属性: 必选，类型: integer)
                           range_end (属性: 必选，类型: integer) 来源：`C1`。
- `POSTGETERRORLIST` 边界：execute_id (属性: 必选，类型: string)
                          node_id (属性: 必选，类型: string)
                          range_start (属性: 必选，类型: integer)
                          range_end (属性: 必选，类型: integer) 来源：`C2`。

## 3. 最小可行基（Minimum Viable Bases）
    
- `B1` 测试获取输出文件基：
       由
         调用接口: POST /api/get_output_file
         参数：无
         请求体：execute_id
                node_id
                range_start
                range_end
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取输出文件结构。来源：`C1 + POSTGETOUTPUTFILE`。

- `B2` 测试获取异常列表基：
       由
         调用接口: POST /api/get_error_list
         参数：无
         请求体：execute_id
                node_id
                range_start
                range_end
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取异常列表结构。来源：`C2 + POSTGETERRORLIST`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 单基调用有效原则
  - `R1.1` 参与基：`B1`、`B2`。
  - `R1.2` 组合方式：每个基独立调用，每个基每个参数均满足第 2 节边界参数有效条件（`range_start <= range_end`）；对依赖 execute_id/node_id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：在边界有效条件下分别稳定导出 `C1~C2` 的正向能力。
  - `R1.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R2` 单基调用无效原则
  - `R2.1` 参与基：`B1`、`B2`。
  - `R2.2` 组合方式：每个基独立调用；对每个基的每个参数，针对每一类适用的无效输入（字段缺失、类型不符、`range_start > range_end` 等）至少执行 1 次，保证覆盖率。
  - `R2.3` 失效结论：`N1`（边界无效条件下应进入失败态 `!=2000`）。
  - `R2.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R3` 随机组合有效原则
  - `R3.1` 参与基：`{B1, B2}` 的随机子集。
  - `R3.2` 组合方式：在各基边界参数有效前提下，合法随机组合需要满足：
  - `R3.2.a` 组合形态仅允许 `{B1}`、`{B2}`、`{B1+B2}`。
  - `R3.2.b` 命中 `{B1+B2}` 时，`execute_id/node_id/range_start/range_end` 必须同轮一致。
  - `R3.2.c` 全部参与基参数均满足边界有效条件。
  - `R3.3` 输出能力：在满足 `R3.2` 约束下，导出 `C1~C2` 的高覆盖正向能力。
  - `R3.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R4` 特殊组合非有效原则
  - `R4.1` 参与基：`B1 + B2` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若组合违反 `R3.2` 约束（同轮上下文不一致、跨轮串用标识、窗口不一致），就判定为非有效组合处理。
  - `R4.3` 失效结论：`N1`（触发 `R4.2` 的特殊组合应进入失败态 `!=2000`）。
  - `R4.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

## 5. 验证（Verification）

- `V1` 单基调用有效验证（对应 `R1`）：通过 = `B1`、`B2` 有效样例均 `2000`；失败 = 任一样例 `!=2000`；证据 = 单 API 有效请求/响应表。

- `V2` 单基调用无效验证（对应 `R2`）：通过 = `B1`、`B2` 无效样例均 `!=2000`；失败 = 任一样例 `2000`；证据 = 单 API 无效请求/响应表。

- `V3` 随机有效组合验证（对应 `R3`）：通过 = 随机有效组合满足 `R3.2` 且全 `2000`；失败 = 约束不满足或任一 `!=2000`；证据 = 组合命中 + 参数一致性报告 + 响应日志。

- `V4` 特殊无效组合验证（对应 `R4`）：通过 = 触发 `R4.2` 的组合均 `!=2000`；失败 = 任一 `2000`；证据 = 无效样例集 + 规则命中记录。

- `V5` 分层与优先级验证：通过 = 顺序“先 `R1/R2` 后 `R3/R4`”，且 `R1/R2` 全通过后才进组合测试，冲突按 `R4>R3`；失败 = 顺序/门槛/判定错误；证据 = 流水线阶段日志。

- `V6` 能力导出与问题分层校验：通过 = `C1`、`C2` 均有合法 `R->C` 导出链，失败样例可分“结构问题/实例问题”；失败 = 能力未覆盖或不可分层；证据 = 覆盖报告 + 失败分类报告。

- `V7` 覆盖率与稳定性验证：通过 = 固定 seed 多轮下覆盖达标、每个 B* 命中 >= 1、关键链路“输出文件查询→异常列表查询”稳定且成功率达阈值；失败 = 覆盖缺失/命中为0/顺序漂移/偶发失败；证据 = 覆盖矩阵 + 轮次统计 + 归因报告。

- `V8` 结论判定：通过 = `V1~V7` 全通过；失败 = 任一失败；证据 = 汇总判定单。
