# 测试执行图模块:TestExecutionDiagram

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试创建/启动执行任务能力：测试系统在后台生成并启动一个新的执行任务的能力。
- `C2` 测试查询执行记录列表能力：测试获取历史或正在运行的执行任务列表的能力，支持分页及按状态、时间等条件筛选。
- `C3` 测试删除执行记录能力：测试清理或取消执行任务的能力。
- `C4` 测试获取指定任务完成状态/结果能力：测试查询指定任务是否完成，或拉取该任务执行完毕后的最终结果数据的能力。
- `C5` 测试查询单次执行详情或事件日志能力：测试获取任务详细信息与运行过程日志的能力，用于展示执行图进度或排查错误。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不校验无效参数/无效组合场景下具体错误码语义的一致性，仅校验该场景是否进入失败态（返回码非 `2000`）。

## 2. 边界定义（Boundary / 参数）

- `POSTEXECUTE` 边界：graph_id (属性: 必选，类型: string)
                     data_src (属性: 必选，类型: list[ExecuteInputParamsDTO]，有效范围: 至少 1 项)
                     data_src[].node_id (属性: 必选，类型: string)
                     data_src[].file_id (属性: 必选，类型: string)
                     data_src[].file_name (属性: 必选，类型: string) 来源：`C1`。
- `GETEXECUTE` 边界：limit (属性: 可选，类型: integer，默认值: 10，有效范围: [1, 100])
                    page (属性: 可选，类型: integer，默认值: 1，有效范围: >= 1)
                    file_name (属性: 可选，类型: string | null，有效范围: 默认所有文件)
                    graph_name (属性: 可选，类型: string | null，有效范围: 默认所有图)
                    state (属性: 可选，类型: string | null，默认值: all)
                    graph_id (属性: 可选，类型: string | null，有效范围: 默认所有图)
                    device_name (属性: 可选，类型: string | null，有效范围: 默认所有设备) 来源：`C2`。
- `DELETEEXECUTE` 边界：execute_id_list (属性: 必选，类型: list[string]，有效范围: 已存在的 execute_id 集合且非空) 来源：`C3`。
- `GETEXECUTECOMPLETEDID` 边界：execute_id (属性: 必选，类型: string，有效范围: 已存在的 execute_id) 来源：`C4`。
- `GETEXECUTETID` 边界：execute_id (属性: 必选，类型: string，有效范围: 已存在的 execute_id) 来源：`C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 测试创建/启动执行任务基：
       由
         调用接口: POST /api/execute
         参数：无
         请求体：graph_id
                data_src
                data_src[].node_id
                data_src[].file_id
                data_src[].file_name
         调用结果：调用成功状态码返回2000，且响应中包含 execute_id；调用失败状态码返回!2000
       构成的测试创建/启动执行任务结构。来源：`C1 + POSTEXECUTE`。

- `B2` 测试查询执行记录列表基：
       由
         调用接口: GET /api/execute
         参数：limit
              page
              file_name
              graph_name
              state
              graph_id
              device_name
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试查询执行记录列表结构。来源：`C2 + GETEXECUTE`。

- `B3` 测试删除执行记录基：
       由
         调用接口: DELETE /api/execute
         参数：无
         请求体：execute_id_list
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试删除执行记录结构。来源：`C3 + DELETEEXECUTE`。

- `B4` 测试获取指定任务完成状态/结果基：
       由
         调用接口: GET /api/execute/completed/{execute_id}
         参数：execute_id
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取指定任务完成状态/结果结构。来源：`C4 + GETEXECUTECOMPLETEDID`。

- `B5` 测试查询单次执行详情或事件日志基：
       由
         调用接口: GET /api/execute/{execute_id}
         参数：execute_id
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试查询单次执行详情或事件日志结构。来源：`C5 + GETEXECUTETID`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 单基调用有效原则
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`。
  - `R1.2` 组合方式：每个基独立调用，每个基每个参数均满足第 2 节边界参数有效条件；对依赖 execute_id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：在边界有效条件下分别稳定导出 `C1~C5` 的正向能力。
  - `R1.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R2` 单基调用无效原则
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`。
  - `R2.2` 组合方式：每个基独立调用；对每个基的每个参数，针对每一类适用的无效输入（字段缺失、类型不符、越界值、无效标识等）至少执行 1 次，保证覆盖率。
  - `R2.3` 失效结论：`N1`（边界无效条件下应进入失败态 `!=2000`）。
  - `R2.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R3` 随机组合有效原则
  - `R3.1` 参与基：`{B1, B2, B3, B4, B5}` 的随机子集。
  - `R3.2` 组合方式：在各基边界参数有效前提下，合法随机组合需要满足：
  - `R3.2.a` 含 `B3/B4/B5` 时，`execute_id` 必须来自本轮 `B1` 返回值，或本轮 `B2` 结果里选中的记录。
  - `R3.2.b` 含 `B3` 时，`execute_id_list` 必须非空且列表元素都可追溯到本轮有效来源。
  - `R3.2.c` 同一轮里如果同时有 `B3 + B4/B5`，顺序必须是先查后删，且删除必须放在目标链路最后一步。
  - `R3.2.d` 不允许跨轮继续使用已经删除或失效的 execute_id，也不允许中途替换目标记录。
  - `R3.3` 输出能力：在满足 `R3.2` 约束下，导出 `C1~C5` 的高覆盖正向能力。
  - `R3.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R4` 特殊组合非有效原则
  - `R4.1` 参与基：`B3`、`B4`、`B5` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若链路里出现“无来源直调依赖基、先删后查、删完还查被删目标、跨轮继续用失效标识”等情况，就判定为非有效组合处理。
  - `R4.3` 失效结论：`N1`（触发 `R4.2` 的特殊组合应进入失败态 `!=2000`）。
  - `R4.4` 边界绑定：`DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID + POSTEXECUTE + GETEXECUTE`。

## 5. 验证（Verification）

- `V1` 单基调用有效验证（对应 `R1`）：通过 = `B1~B5` 有效样例均 `2000`；失败 = 任一样例 `!=2000`；证据 = 单 API 有效请求/响应表。

- `V2` 单基调用无效验证（对应 `R2`）：通过 = `B1~B5` 无效样例均 `!=2000`；失败 = 任一样例 `2000`；证据 = 单 API 无效请求/响应表。

- `V3` 随机有效组合验证（对应 `R3`）：通过 = 随机有效组合满足 `R3.2` 且全 `2000`；失败 = 约束不满足或任一 `!=2000`；证据 = 组合命中 + 标识溯源 + 响应日志。

- `V4` 特殊无效组合验证（对应 `R4`）：通过 = 触发 `R4.2` 的组合均 `!=2000`；失败 = 任一 `2000`；证据 = 无效样例集 + 规则命中记录。

- `V5` 分层与优先级验证：通过 = 顺序“先 `R1/R2` 后 `R3/R4`”，且 `R1/R2` 全通过后才进组合测试，冲突按 `R4>R3`；失败 = 顺序/门槛/判定错误；证据 = 流水线阶段日志。

- `V6` 能力导出与问题分层校验：通过 = `C1~C5` 均有合法 `R->C` 导出链，失败样例可分“结构问题/实例问题”；失败 = 能力未覆盖或不可分层；证据 = 覆盖报告 + 失败分类报告。

- `V7` 覆盖率与稳定性验证：通过 = 固定 seed 多轮下覆盖达标、每个 B* 命中 >= 1、关键链路稳定且成功率达阈值；失败 = 覆盖缺失/命中为0/顺序漂移/偶发失败；证据 = 覆盖矩阵 + 轮次统计 + 归因报告。

- `V8` 结论判定：通过 = `V1~V7` 全通过；失败 = 任一失败；证据 = 汇总判定单。
