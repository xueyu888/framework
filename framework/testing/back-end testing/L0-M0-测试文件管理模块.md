# 测试文件管理模块:TestFileManagement

@framework

## 0. 目标 (Goal)

- 本模块描述测试文件的基本测试结构（增、删、查）。
- 本模块涉及接口无论业务成功还是失败，HTTP 状态码均返回 `200`；测试判定只看响应体业务码，成功为 `2000`，失败为 `!=2000`。

## 1. 最小结构基（Minimal Structural Bases）

- `B1` 测试文件上传正例基：满足文件上传接口`POSTFILE`参数边界有效且device_name依赖已有资源。
- `B2` 测试文件上传反例基：满足文件上传接口`POSTFILE`参数边界至少一个无效（如字段缺失、类型不符、越界值、无效标识等），或device_name不依赖已有资源。
- `B3` 测试查询文件列表正例基：满足测试查询所有文件列表接口`GETFILE`参数边界有效且device_name、table_name、user均依赖已有资源。
- `B4` 测试查询文件列表反例基：满足测试查询所有文件列表接口`GETFILE`参数边界至少一个无效（如字段缺失、类型不符、越界值、无效标识等），或device_name、table_name、user至少存在一个不依赖已有资源。
- `B5` 测试查询单个文件正例基：满足测试查询单个特定文件接口`GETSINGLEFIL`参数边界有效且hash_table_name依赖已有资源。
- `B6` 测试查询单个文件反例基：满足测试查询单个特定文件接口`GETSINGLEFIL`参数边界至少一个无效（如字段缺失、类型不符、越界值、无效标识等），或hash_table_name不依赖已有资源。
- `B7` 测试删除文件正例基：满足测试删除文件接口`DELETEFILE`参数边界有效且id依赖已有资源。
- `B8` 测试删除文件反例基：满足测试删除文件接口`DELETEFILE`参数边界至少一个无效（如字段缺失、类型不符、越界值、无效标识等），或id依赖已有资源。

## 2. 基排列组合（Base Arrangement / Combination）

- `R1` `文件管理全量场景矩阵`：由 `<{B1, B2}, {B3, B4}, {B5, B6}, {B7, B8}>` 形成 `2^4 = 16` 个文件管理测试场景组合，覆盖上传、列表查询、单文件查询、删除四个测试位点的全量正反例排列，用于联合导出 `C1`、`C2`、`C3`、`C4`。
- `R2` `全正常链路组合`：由 `<B1, B3, B5, B7>` 形成“上传成功 -> 列表查询成功 -> 单文件查询成功 -> 删除成功”的完整正常测试链路，导出全正常场景下的 `C1`、`C2`、`C3`、`C4`。
- `R3` `单异常扰动组合`：由 `R1` 中“恰有一个测试位点取反例基，其余测试位点取对应正例基”的组合形成 `4` 个单异常场景，用于验证单点异常输入或单点依赖缺失时，对应接口稳定失败、其余接口仍按各自边界独立判定，联合导出单异常场景下的 `C1`、`C2`、`C3`、`C4`。
- `R4` `多异常联动组合`：由 `R1` 中“恰有两个或三个测试位点取反例基，其余测试位点取对应正例基”的组合形成 `10` 个多异常场景，用于验证多处异常同时存在时，文件管理各接口仍按参数边界与依赖约束稳定返回业务失败，联合导出多异常场景下的 `C1`、`C2`、`C3`、`C4`。
- `R5` `全异常链路组合`：由 `<B2, B4, B6, B8>` 形成上传、列表查询、单文件查询、删除全部落在反例基上的完整异常测试链路，用于验证文件管理模块在全异常输入下的失败判定一致性，导出全异常场景下的 `C1`、`C2`、`C3`、`C4`。

## 3. 边界定义（Boundary）

### 3.1 接口定义（IO / Ports）

- `POSTFILE`：OST /api/file （测试文件上传接口）
- `GETFILE`：GET /api/file（测试查询所有文件列表接口）
- `GETSINGLEFIL`：GET /api/single/file/{hash_table_name}（测试查询单个特定文件接口）
- `DELETEFILE`：DELETE /api/file/{id}（测试删除文件接口）

### 3.2 参数边界（Parameter Constraints）

- `POSTFILE` 边界：
                  参数：无
                  请求体：file (属性: 必选，类型: string($binary), 有效范围: 格式txt|csv 编码utf-8) 
                         table_name (属性: 可选，类型: string | null)
                         device_name (属性: 必选，类型: string)
                         skip（属性: 可选，类型: integer, 有效范围: [`0`, `上传文件总行数 - 1`]）来源：`C1`。
- `GETFILE` 边界：
                  参数：page (属性: 可选，类型: integer，默认值: 1，有效范围: >= 1)
                       limit (属性: 可选，类型: integer, 默认值: 20，有效范围: >= 1)
                       table_name (属性: 可选，类型: string | null)
                       user（属性: 可选，类型: string | null）
                       device_name（属性: 可选，类型: string | null）
                  请求体：无，来源：`C2`。
- `GETSINGLEFILE` 边界：
                  参数：hash_table_name (属性: 必选，类型: string) 
                  请求体：无，来源：`C3`。
- `DELETEFILE` 边界：
                  参数：id (属性: 必选，类型: string) 
                  请求体：无，来源：`C4`。

## 4. 能力声明（Capability Statement）

- `C1` 测试文件上传能力：测试将用户本地的文件数据发送并保存到服务器上的能力。
- `C2` 测试查询所有文件列表能力：测试获取服务器上已经上传的文件列表的能力。
- `C3` 测试查询单个特定文件能力：测试系统会根据特定文件的哈希值能返回单个特定文件的详细属性或内容的能力。
- `C4` 测试删除文件能力：测试系统会根据传入指定的 ID 把对应的文件从服务器中彻底删除的能力。
