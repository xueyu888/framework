# 测试文件上传模块:TestFileUpload

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试文件上传能力：测试将用户本地的文件数据发送并保存到服务器上的能力。
- `C2` 测试查询所有文件列表能力：测试获取服务器上已经上传的文件列表的能力。
- `C3` 测试查询单个特定文件能力：测试系统会根据特定文件的哈希值能返回单个特定文件的详细属性或内容的能力。
- `C4` 测试删除文件能力：测试系统会根据传入指定的 ID 把对应的文件从服务器中彻底删除的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不校验无效参数/无效组合场景下具体错误码语义的一致性，仅校验该场景是否进入失败态（返回码非 2000）。

## 2. 边界定义（Boundary / 参数）

- `POSTFILE` 边界：file (属性: 必选，类型: string($binary), 有效范围: 格式txt|csv 编码utf-8) 
                  table_name (属性: 可选，类型: string | null)
                  device_name (属性: 必选，类型: string，有效范围: 已存在的device_name)
                  skip（属性: 可选，类型: integer, 有效范围: [`0`, `上传文件总行数 - 1`]）来源：`C1`。
- `GETFILE` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围: >= 1)
                  limit (属性: 可选，类型: integer, 默认值: 20，有效范围: >= 1)
                  table_name (属性: 可选，类型: string | null)
                  user（属性: 可选，类型: string | null）
                  device_name（属性: 可选，类型: string | null）来源：`C2`。
- `GETSINGLEFILE` 边界：hash_table_name (属性: 必选，类型: string, 有效范围: 已存在的hash_table_name) 来源：`C3`。
- `DELETEFILE` 边界：id (属性: 必选，类型: string, 有效范围: 已存在的文件 id) 来源：`C4`。

       构成的测试删除文件结构。来源：`C4 + DELETEFILE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 单基调用有效原则
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`。
  - `R1.2` 组合方式：每个基独立调用，每个基每个参数均满足第 2 节边界参数有效条件；对依赖 hash_table_name/id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：在边界有效条件下分别稳定导出 `C1~C4` 的正向能力。
  - `R1.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R2` 单基调用无效原则
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`。
  - `R2.2` 组合方式：每个基独立调用；对每个基的每个参数，针对每一类适用的无效输入（如字段缺失、类型不符、越界值、无效标识等）至少执行 1 次，保证覆盖率。
  - `R2.3` 失效结论：`N1`（边界无效条件下应进入失败态 `!=2000`）。
  - `R2.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R3` 随机组合有效原则
  - `R3.1` 参与基：`{B1, B2, B3, B4}` 的随机子集。
  - `R3.2` 组合方式：在各基边界参数有效前提下，合法随机组合需要满足：
  - `R3.2.a` 含 `B3/B4` 时，用的标识必须是“本轮新鲜拿到的”，要么来自本轮 B1 返回值，要么来自本轮 B2 结果里选中的记录。
  - `R3.2.b` 同一轮里如果同时有 `B3 + B4` ，顺序必须是先查后删，而且“删除”必须放在这条目标链路的最后一步。
  - `R3.2.c` 不允许跨轮继续用已经删掉或失效的标识，也不允许中途把标识对应目标换成别的记录。
  - `R3.3` 输出能力：在满足 `R3.2` 约束下，导出 C1~C4 的高覆盖正向能力。
  - `R3.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R4` 特殊组合非有效原则
  - `R4.1` 参与基：`B3`、`B4` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若链路里出现“没来源就直接调、先删再查、删完还查被删目标、跨轮继续用失效标识”这类情况，就判定为非有效组合处理。
  - `R4.3` 失效结论：`N1`（触发 `R4.2` 的特殊组合应进入失败态 `!=2000`）。
  - `R4.4` 边界绑定：`GETSINGLEFILE + DELETEFILE + POSTFILE + GETFILE`。

## 5. 验证（Verification）

- `V1` 单基调用有效验证（对应 `R1`）：通过 = `B1~B4` 有效样例均 `2000`；失败 = 任一样例 `!=2000`；证据 = 单 API 有效请求/响应表。

- `V2` 单基调用无效验证（对应 `R2`）：通过 = `B1~B4` 无效样例均 `!=2000`；失败 = 任一样例 `2000`；证据 = 单 API 无效请求/响应表。

- `V3` 随机有效组合验证（对应 `R3`）：通过 = 随机有效组合满足 `R3.2` 且全 `2000`；失败 = 约束不满足或任一 `!=2000`；证据 = 组合命中+标识溯源+响应日志。

- `V4` 特殊无效组合验证（对应 `R4`）：通过 = 触发 `R4.2` 的组合均 `!=2000`；失败=任一 `2000`；证据 = 无效样例集+规则命中记录。

- `V5` 分层与优先级验证：通过 = 顺序“先 `R1/R2` 后 `R3/R4`”，且 `R1/R2` 全通过后才进组合测试，冲突按 `R4>R3`；失败 = 顺序/门槛/判定错误；证据 = 流水线阶段日志。

- `V6` 能力导出与问题分层校验：通过 = `C1~C4` 均有合法 `R->C` 导出链，失败样例可分“结构问题/实例问题”；失败 = 能力未覆盖或不可分层；证据 = 覆盖报告+失败分类报告。

- `V7` 覆盖率与稳定性验证：通过 = 固定 seed 多轮下覆盖达标、每个 B* 命中 >= 1、关键链路稳定且成功率达阈值；失败 = 覆盖缺失/命中为0/顺序漂移/偶发失败；证据 = 覆盖矩阵+轮次统计+归因报告。

- `V8` 结论判定：通过 = `V1~V7` 全通过；失败 = 任一失败；证据 = 汇总判定单。
