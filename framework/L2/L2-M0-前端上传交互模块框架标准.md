# 前端上传交互模块:FrontendUploadInteractionModule

@framework

## 1. 能力声明（Capability Statement）

- `C1` 文件选择能力：提供本地文件选择与文件信息展示能力。
- `C2` 前端预校验能力：提供文件大小、文件类型与空文件等本地预校验能力。
- `C3` 上传触发能力：提供上传请求发起与请求状态切换能力。
- `C4` 结果反馈能力：提供成功、失败与拒绝结果的页面反馈能力。
- `C5` 非能力项：不负责服务端文件保存、共享治理与后续业务处理。

## 2. 边界定义（Boundary / 参数）

- `SELECT` 选择边界：文件选择入口、单文件选择语义与文件信息展示范围必须明确。来源：`C1 + C4`。
- `LOCALCHECK` 本地校验边界：浏览器侧大小、类型、空文件检查规则必须明确。来源：`C1 + C2`。
- `TRIGGER` 触发边界：上传按钮、请求发起时机与重复点击策略必须明确。来源：`C2 + C3`。
- `FEEDBACK` 反馈边界：等待、成功、失败与拒绝结果的展示语义必须明确。来源：`C3 + C4`。
- `FILE` 文件边界：文件对象及其基础属性必须可被前端读取与使用。来源：`C1 + C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文件选择交互基：L1.M0[R1]。来源：`C1 + SELECT + FILE + FEEDBACK`。
- `B2` 本地校验交互基：L1.M0[R1,R3]。来源：`C1 + C2 + FILE + LOCALCHECK + FEEDBACK`。
- `B3` 上传触发反馈基：L1.M0[R1,R3]。来源：`C3 + C4 + TRIGGER + FEEDBACK + FILE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 文件选择组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先选择文件，再展示文件基础信息，形成可上传的前端起点。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SELECT/FILE/FEEDBACK`。
- `R2` 本地校验组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在触发上传前完成本地大小、类型与空文件检查，失败时直接给出反馈。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`FILE/LOCALCHECK/FEEDBACK`。
- `R3` 触发反馈组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：在选择与本地校验通过后发起上传请求，并对等待、成功、失败结果进行反馈。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`TRIGGER/FEEDBACK/FILE/LOCALCHECK`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：未选择文件即触发上传、跳过本地校验或反馈状态不完整的交互方案无效。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`SELECT/LOCALCHECK/TRIGGER/FEEDBACK`。

## 5. 验证（Verification）

- `V1` 选择有效性：文件选择流程必须满足 `SELECT` 约束。
- `V2` 校验有效性：本地校验必须满足 `LOCALCHECK` 约束。
- `V3` 触发有效性：上传触发必须满足 `TRIGGER` 约束。
- `V4` 反馈一致性：页面反馈必须满足 `FEEDBACK` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
