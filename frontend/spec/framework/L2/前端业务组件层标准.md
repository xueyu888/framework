# 前端业务组件层标准（L2）

## 0. 规范关系
本文件是前端业务组件层（业务语义层）标准，遵循：
- `standards/L0/规范总纲与树形结构.md`
- `standards/L1/框架设计核心标准.md`
- `standards/L1/可追溯性标准.md`
- `standards/L1/可删减性标准.md`
- `framework/L0/前端框架标准.md`

## 1. 目标（Goal）
业务组件层负责封装业务语义与业务交互规则，输出可复用、可测试、可组合的业务能力单元。

## 2. 边界定义（Boundary）
- 业务组件可以组合基础组件，但不得直接绑定具体页面容器
- 业务组件不得直接读取路由对象或页面实例
- 业务组件输入输出契约必须显式定义并稳定演进

## 3. 模块（最小可行基，Module）
- `M1 语义渲染模块`：业务语义 UI 表达
- `M2 基础组件编排模块（最小可行基）`：以基础组件作为业务组件的最小装配单元
- `M3 契约模块`：`Props/Events` 输入输出边界与状态约束
- `M4 质量模块`：业务组件测试与回归验证

说明：
- 业务组件层最小可行基定义到基础组件层面，业务组件应通过组合基础组件构建业务语义。

## 4. 组合原则（Combination Principles）
先定义模块如何组合，再定义组合后的判定约束。

模块组合关系：
- `C0` 整体拓扑（总关系）
  - 主链路（业务能力构建）：`M1 -> M2 -> M3`
  - 质量回路（验证闭环）：`M3 -> M4 -> M3`
  - 结构说明：`M1` 负责语义表达，`M2` 负责基础组件装配，`M3` 负责输入输出契约，`M4` 负责质量回归
- `C1` `M1 语义渲染模块 -> M2 基础组件编排模块`
  - 在 `C0` 中对应主链路第一跳
  - 通俗理解：业务语义必须落在基础组件编排上，不直接依赖页面容器
- `C2` `M2 基础组件编排模块 -> M3 契约模块`
  - 在 `C0` 中对应主链路第二跳
  - 通俗理解：编排后的业务组件必须对外提供清晰稳定的 `Props/Events` 契约
- `C3` `M3 契约模块 -> M4 质量模块`
  - 在 `C0` 中对应质量回路第一跳
  - 通俗理解：契约与交互规则必须进入测试验证，不可只停留在文档声明
- `C4` `M4 质量模块 -> M3 契约模块`
  - 在 `C0` 中对应质量回路第二跳
  - 通俗理解：测试反馈应回流修正契约定义，形成可演进闭环

组合约束（由组合关系导出）：
- `R1`：业务组件必须定义明确契约（props/events，对应 `C2`）
- `R2`：业务组件依赖通过页面层注入，不直接依赖路由容器（对应 `C1`）
- `R3`：业务组件不得直接耦合业务 API SDK（对应 `C1/C2`）
- `R4`：业务组件必须有最小测试覆盖（渲染/交互/异常，对应 `C3/C4`）
- `R5`：业务组件主结构应由基础组件构成，不得在业务组件中重复发明基础交互原语（对应 `C1`）

## 5. 验证（Verification）
通过条件：  
- 模块组合关系 `C1-C4` 均可在组件实现中追溯
- 契约变更可追溯并有影响说明
- 组件可脱离单页面上下文复用
- 关键交互规则通过回归测试
- 业务组件层规则违反项为 0
- 业务组件可追溯到基础组件清单（而非页面容器耦合）
