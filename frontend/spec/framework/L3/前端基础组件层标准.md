# 前端基础组件层标准（L2）

## 0. 规范关系
本文件是前端基础组件层（基础能力层）标准，遵循：
- `standards/L0/规范总纲与树形结构.md`
- `standards/L1/框架设计核心标准.md`
- `standards/L1/可追溯性标准.md`
- `standards/L1/可删减性标准.md`
- `framework/L0/前端框架标准.md`

## 1. 目标（Goal）
基础组件层提供无业务语义的通用交互与视觉能力，保障一致性、可访问性和复用效率。

## 2. 边界定义（Boundary）
- 基础组件不得包含业务字段命名或业务流程分支
- 基础组件可来源于开源 UI（如 shadcn/ui），但需二次封装并纳入统一规范
- 基础组件必须支持主题 token 与样式扩展机制

## 3. 模块（最小可行基，Module）
- `M1 开源 UI 基座模块（最小可行基）`
  - 以三方 UI 框架作为基础能力来源（默认：shadcn/ui）
  - 图标组件库默认：lucide-react
- `M2 基础元素组合模块（最小可行基）`
  - 以原子元素和开源组件组合形成通用组件（Button/Input/Dialog/Table 等）
  - 组合过程必须保持“无业务语义”
- `M3 规范化封装模块`
  - 对三方组件进行统一命名、属性约束、样式约束、文档约束
- `M4 样式与可访问性模块`
  - Tailwind token/主题变量
  - 语义标签、键盘可达、焦点管理

说明：
- 基础组件层最小可行基定义为“开源三方 UI 框架 + 基础元素组合”。
- 允许替换三方框架，但必须在项目实例标准中声明替代方案与迁移策略。

## 4. 组合原则（Combination Principles）
先定义模块如何组合，再定义组合后的判定约束。

模块组合关系：
- `C0` 整体拓扑（总关系）
  - 主链路（能力构建）：`M1 -> M2 -> M3 -> M4`
  - 结构说明：`M1` 提供能力来源，`M2` 负责元素组合，`M3` 负责规范化封装，`M4` 负责样式与可访问性闭环
- `C1` `M1 开源 UI 基座模块 -> M2 基础元素组合模块`
  - 在 `C0` 中对应第一跳
  - 通俗理解：基础组件能力来源必须明确，并在白名单范围内组合
- `C2` `M2 基础元素组合模块 -> M3 规范化封装模块`
  - 在 `C0` 中对应第二跳
  - 通俗理解：组合后的组件必须统一命名、属性与来源登记，不能直接裸用
- `C3` `M3 规范化封装模块 -> M4 样式与可访问性模块`
  - 在 `C0` 中对应第三跳
  - 通俗理解：封装完成后必须满足 token 化样式与 a11y 基线
- `C4` `M4 样式与可访问性模块 -> M3 规范化封装模块`
  - 在 `C0` 中对应质量回路
  - 通俗理解：样式/a11y 检查结果必须回流修正组件封装

组合约束（由组合关系导出）：
- `R1`：基础组件不得访问业务接口或耦合业务字段（对应 `C1/C2`）
- `R2`：基础组件必须支持 token 化样式扩展（对应 `C3`）
- `R3`：基础组件必须满足 a11y 基线（语义、焦点、键盘，对应 `C3/C4`）
- `R4`：基础组件 API 破坏性变更必须提供迁移说明（对应 `C2/C4`）
- `R5`：每个基础组件必须标注来源：三方组件、元素组合、或二次封装（对应 `C2`）
- `R6`：元素组合必须基于可维护白名单（开源组件 + 原子元素），禁止隐式复制业务组件逻辑（对应 `C1`）
- `R7`：更换三方 UI 基座时，必须提供等价能力对照与兼容验证结果（对应 `C1/C3`）

## 5. 验证（Verification）
通过条件：
- 模块组合关系 `C1-C4` 均可在组件实现中追溯
- 基础组件在多个业务场景可复用
- 样式 token 合规率达标
- 可访问性检查通过
- 基础组件层规则违反项为 0
- 三方来源与封装来源可追溯（来源登记完整）
- 元素组合组件可通过一致性检查（命名、属性、样式、a11y）
