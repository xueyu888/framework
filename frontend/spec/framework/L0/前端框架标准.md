# 前端框架标准（L2）
框架标准“定骨架”
## 0. 规范关系
本文件为前端框架领域总标准，遵循：
- `standards/L0/规范总纲与树形结构.md`
- `standards/L1/框架设计核心标准.md`
- `standards/L1/可追溯性标准.md`
- `standards/L1/可删减性标准.md`

本文件定位：定义“前端系统是否成立”的框架级标准，不承载业务组件与基础组件实现细节。

## 1. 目标（Goal）
建立可复用、可追溯、可验证的前端框架基线，使项目在统一框架约束下稳定交付页面能力。

可度量目标：
- 一致性：路由与页面入口规范违规项可统计并收敛
- 可用性：核心页面路由可达率可验证
- 稳定性：页面级异常具备兜底与恢复能力
- 交付效率：新增页面在既定框架下可快速接入

## 2. 边界定义（Boundary）
### 2.1 技术边界（默认）
- `Framework`：React
- `CSS`：Tailwind CSS
- `UI`：shadcn/ui
- `Icon`：lucide-react
- `Build Tool`：Vite
- `Runtime`：Browser SPA（如需 SSR，须在项目实例标准中声明）

### 2.2 范围边界
- 本标准仅约束框架级能力：路由、页面入口、状态闭环、异常兜底、可追溯性
- 业务语义组件与基础组件细则由对应层标准定义，不在本文件展开实现细节

### 2.3 质量边界
- 页面入口必须可访问
- 页面状态必须可判定
- 关键失败路径必须可恢复

## 3. 模块（最小可行基，Module）
最小可行基分为“必选模块”和“可选增强模块”。

必选模块（框架成立条件，最小成立条件）：
- `M1 路由注册模块`
  - 定义路由路径、页面入口、访问策略
- `M2 页面入口模块`
  - 页面挂载点与页面级初始化逻辑
- `M3 页面状态闭环模块`
  - 页面最小状态集合：`loading / success / error / empty`
- `M4 路由兜底模块`
  - 未命中路由（404）、异常回退、权限失败处理

可选增强模块（按项目复杂度启用）：
- `M5 消息中心模块（非必要）`
  - 用于跨页面或跨模块发布/订阅通知事件（如全局 toast、系统公告、会话失效广播）
  - 不作为主链路强制依赖
- `M6 数据中心模块（非必要）`
  - 用于跨页面共享查询结果与会话级数据（如用户信息、配置缓存、字典数据）
  - 不作为主链路强制依赖
- `M7 服务层模块（非必要）`
  - 用于统一封装 API 调用、超时/重试、错误映射、DTO 转换
  - 不作为主链路强制依赖，但启用后页面与业务组件应通过服务契约访问后端能力

说明：
- 若项目需要业务组件层/基础组件层规则，应在后续层级标准中扩展，不回填本标准主模块定义。
- `M5` 启用时必须在项目实例标准中声明事件边界、事件命名规范、失败降级策略。
- `M6` 启用时必须在项目实例标准中声明数据域边界、缓存策略、一致性策略、失效策略。

## 4. 组合原则（Combination Principles）
先定义模块如何组合，再定义组合后的判定约束。

模块组合关系：
- `C0` 整体拓扑（总关系）
  - 必选主链路（正常访问）：`M1 -> M2 -> M3`
  - 必选回流链路（异常恢复）：`M1 -> M4 -> M2 -> M3`
  - 可选事件链路：`M2/M3 -> M5`
  - 可选数据链路：`M2/M3 <-> M6`
  - 可选服务链路：`M2/M3 <-> M7`
  - 可选服务数据链路：`M7 <-> M6`
  - 结构说明：`M1` 负责分发，`M2` 负责汇合，`M3` 负责状态终态承载，`M4` 负责异常回流，`M5/M6/M7` 负责跨页面协同增强
- `C1` `M1 路由注册模块 -> M2 页面入口模块`
  - 在 `C0` 中对应主链路第一跳
  - 通俗理解：先有路由，才有页面入口；每条路由都要“指向一个页面”
- `C2` `M2 页面入口模块 -> M3 页面状态闭环模块`
  - 在 `C0` 中对应主链路第二跳与回流链路终点
  - 通俗理解：进入页面后，页面必须能在 `loading/success/error/empty` 之间切换
- `C3` `M1 路由注册模块 -> M4 路由兜底模块`
  - 在 `C0` 中对应回流链路第一跳
  - 通俗理解：路由失败时不能崩掉，必须有 404/权限失败/异常回退入口
- `C4` `M4 路由兜底模块 -> M2 页面入口模块`
  - 在 `C0` 中对应回流链路第二跳
  - 通俗理解：兜底页不是终点，用户要能回到正常页面继续操作（回退/重试/重定向）
- `C5` `M2/M3 -> M5 消息中心模块（可选）`
  - 在 `C0` 中对应可选事件链路
  - 通俗理解：页面入口与状态变化可发布标准化事件，但消息中心不能反向接管页面主流程
- `C6` `M2/M3 <-> M6 数据中心模块（可选）`
  - 在 `C0` 中对应可选数据链路
  - 通俗理解：页面可从数据中心读取共享数据并回写，但必须保持状态闭环可判定
- `C7` `M6 数据中心模块 -> M3 页面状态闭环模块（可选回流）`
  - 通俗理解：数据命中/失效/失败必须映射为页面状态变化，而非静默失败
- `C8` `M5 消息中心模块 -> M4 路由兜底模块（可选协同）`
  - 通俗理解：系统级故障事件可触发统一兜底动作，但兜底策略仍由路由层定义
- `C9` `M2/M3 <-> M7 服务层模块（可选）`
  - 通俗理解：页面入口与状态编排通过服务层读取/提交数据，避免直接耦合 API SDK
- `C10` `M7 服务层模块 <-> M6 数据中心模块（可选）`
  - 通俗理解：服务层可读写缓存与共享数据，但缓存一致性与失效策略仍由数据中心声明

组合约束（由组合关系导出）：
- `R1`：每个页面必须通过路由可达，禁止“不可达页面入口”
- `R2`：每条路由必须绑定唯一页面入口（对应 `C1`）
- `R3`：页面入口必须具备最小状态闭环（`loading/success/error/empty` 至少可触发其一，对应 `C2`）
- `R4`：必须存在路由兜底策略（404/权限失败/异常回退，对应 `C3`）
- `R5`：框架变更必须可追溯到实现文件与符号（映射登记，对应 `C1/C3`）
- `R6`：规则结果必须可判定为“有效/无效”，并输出失败原因
- `R7`：技术栈若偏离默认基线，必须在项目实例标准中声明替代方案与理由
- `R8`：启用 `M5` 时，事件必须具备稳定命名、发布来源、消费边界与幂等策略（对应 `C5/C8`）
- `R9`：启用 `M6` 时，数据必须具备域归属、TTL/失效策略、一致性级别声明（对应 `C6/C7`）
- `R10`：`M5/M6` 仅可增强主链路，禁止替代 `M1-M4` 的必选职责（对应 `C0/C5/C6`）
- `R11`：启用 `M7` 时，服务契约必须显式定义（input/output/error），且可追溯到实现（对应 `C9`）
- `R12`：启用 `M7` 时，页面层/业务组件层禁止直接依赖裸 `fetch/axios` 与业务 API SDK（对应 `C9/C10`）

## 5. 验证（Verification）
通过条件：
- `framework_instance_valid == true`（满足必选约束 `R1-R7`）
- 模块组合关系 `C1-C4` 均可在工程文件中追溯
- `route_reachability_target > route_reachability_baseline`
- `page_error_recovery_target > page_error_recovery_baseline`
- 验证输出必须包含失败原因列表（failure reasons）

可选模块启用条件（按需）：
- 启用 `M5` 时：`R8`、`R10` 必须通过，且事件清单与消费边界可追溯
- 启用 `M6` 时：`R9`、`R10` 必须通过，且数据域清单与一致性策略可追溯
- 启用 `M7` 时：`R11`、`R12` 必须通过，且服务契约与错误映射策略可追溯

验证口径（框架级）：
- 路由覆盖率（核心页面）
- 页面入口可达率
- 异常兜底触发成功率
- 规则违规项数量
- 可选事件链路有效率（启用 `M5` 时）
- 可选数据链路一致性达标率（启用 `M6` 时）
- 可选服务链路成功率（启用 `M7` 时）
