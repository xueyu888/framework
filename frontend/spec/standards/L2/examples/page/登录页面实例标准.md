# 登录页面实例标准（L2 Example）

## 0. 实例元数据
- `instance_id`：PAGE-LOGIN-001
- `category`：page
- `source_standard`：`standards/L2/前端页面标准.md`
- `version`：0.1.0
- `status`：draft
- `owner`：frontend

## 1. 目标（Goal）
在登录页面中落实页面层规范，确保用户可完成登录、失败可恢复、行为可观测。

目标指标：
- 登录流程完成率提升（相对基线）
- 登录失败后恢复成功率可统计
- 页面关键状态与关键事件可观测

## 2. 边界定义（Boundary）
- 路由：`/login`
- 页面输入：账号、密码（可扩展验证码）
- 页面输出：登录成功跳转、失败提示、恢复入口
- 页面状态：`loading/success/error/empty`
- 异常场景：网络失败、凭证错误、服务端异常、权限限制

## 3. 模块（最小可行基，Module）
- `M1 页面入口模块`
  - 登录页路由挂载
  - 未登录访问时可达
- `M2 业务组件编排模块（最小可行基）`
  - 业务组件：`LoginForm`
  - 业务组件：`LoginErrorNotice`
  - 页面主结构由业务组件编排
- `M3 页面状态模块`
  - 提交中：`loading`
  - 登录成功：`success`
  - 登录失败：`error`
  - 初始空态：`empty`
- `M4 页面可观测模块`
  - 埋点：页面曝光、提交点击、登录成功、登录失败
  - 恢复动作：重试登录、返回上一页、联系支持（可选）

## 4. 组合原则（Combination Principles）
- `P-R1` 页面主结构必须通过业务组件组织：
  - 判定：登录页主区至少由 `LoginForm` 作为业务组件承载
- `P-R2` 页面必须定义四态切换条件：
  - 判定：`loading/success/error/empty` 触发条件明确且可测
- `P-R3` 页面必须提供错误恢复路径：
  - 判定：登录失败后至少存在“重试”路径
- `P-R4` 页面必须声明关键行为埋点：
  - 判定：曝光、提交、成功、失败四类事件均已登记
- `P-R5` 页面直接使用基础组件仅限壳层与容器：
  - 判定：业务语义逻辑不直接写在基础组件拼装层

## 5. 验证（Verification）
通过条件：
- 登录成功路径自动化用例通过
- 凭证错误与网络异常路径自动化用例通过
- `loading/success/error/empty` 状态切换可观测
- 恢复路径（重试）可触发且可观测
- 页面主结构可追溯到业务组件清单

失败原因（示例）：
- 未定义 `error -> retry` 恢复链路
- 页面主结构直接由基础组件拼装并承载业务语义
- 关键埋点缺失（如登录失败事件未登记）
