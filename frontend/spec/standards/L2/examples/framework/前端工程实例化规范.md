# 前端工程实例化规范（L2 Example）

## 0. 实例元数据
- `instance_id`：FE-PROJECT-001
- `category`：framework
- `source_standard`：`standards/L2/前端框架标准.md`
- `version`：0.1.0
- `status`：draft
- `owner`：frontend

## 1. 目标（Goal）
将前端框架标准实例化为可执行的工程基线，确保页面路由可组织、可访问、可验证，并支持持续扩展。

目标指标：
- 核心页面路由可达率：>= 99%
- 页面异常兜底触发成功率：>= 99%
- 新页面接入（路由 + 入口 + 状态闭环）平均工时持续下降

## 2. 边界定义（Boundary）
技术边界（默认）：
- `Framework`：React
- `CSS`：Tailwind CSS
- `UI`：shadcn/ui
- `Icon`：lucide-react
- `Build Tool`：Vite
- `Runtime`：Browser SPA

工程范围边界：
- 本实例优先约束框架级能力：路由、页面入口、状态闭环、异常兜底、可追溯性
- 业务组件和基础组件细则由对应子标准约束，不在本实例重复定义

## 3. 模块（最小可行基，Module）
- `M1 路由注册模块`
  - 输出工程路由表（path、name、entry、auth）
- `M2 页面入口模块`
  - 每个路由绑定唯一页面入口组件
- `M3 页面状态闭环模块`
  - 页面入口必须具备最小状态集合：`loading/success/error/empty`
- `M4 路由兜底模块`
  - 404、权限失败、异常回退页

## 4. 组合原则（Combination Principles）
- `R1` 必须声明默认技术栈，若替换需写明替代方案与原因
- `R2` 所有页面必须通过路由可达，不允许不可达入口
- `R3` 每条路由必须绑定唯一页面入口组件
- `R4` 页面入口必须具备状态闭环并可触发验证
- `R5` 必须配置 404 / 权限失败 / 异常回退三类兜底
- `R6` 路由与页面入口映射必须可追溯到实现文件与符号
- `R7` 本实例规则必须可判定有效/无效，并输出失败原因

## 5. 验证（Verification）
通过条件：
- `framework_instance_valid == true`（满足 R1-R7）
- 路由覆盖检查通过（核心页面）
- 页面入口可达检查通过
- 异常兜底检查通过
- 验证结果输出 failure reasons（若失败）

建议校验清单：
- 路由清单文件存在且可解析
- 路由 path 唯一
- 页面入口组件存在
- 404/403/error 路由存在

## 6. 工程落地（Instantiation）
目录约定（示例）：
```text
src/
├── app/
│   ├── router/
│   │   ├── routes.ts
│   │   └── guards.ts
│   └── entry/
├── pages/
│   ├── login/
│   ├── dashboard/
│   └── not-found/
├── business/
└── components/
    └── base/
```

必备产物：
- 路由表：`src/app/router/routes.ts`
- 页面入口：`src/pages/**/index.tsx`
- 兜底页面：`src/pages/not-found/index.tsx`
- 校验脚本：`scripts/check_routes.py`（可选，建议）
