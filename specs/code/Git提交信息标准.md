# Git提交信息标准

## 1. 目标

本标准定义仓库内 Git 提交信息的最小要求，确保历史记录可读、可追溯、可用于发布与审计，而不是留下无法解释的临时说明。

## 2. 适用范围

- 本仓库内所有 Git 提交
- 本仓库内面向公开发布、版本整理与规范修订的提交

## 3. 强制要求

- 提交标题必须使用中文自然语言书写，直接说明本次提交实际完成了什么。
- 可以保留必要的英文术语、路径、命令、版本号、标识符或框架名，但标题主体不得退化成纯英文模板。
- 禁止只写 `fix`、`feat`、`update`、`wip`、`tmp`、`test`、`1` 等无法独立解释变更结果的标题。
- 若一次工作同时包含多个可独立审查的目标，应拆成多次提交，避免把无关修改混进同一条提交信息。
- 面向公开发布的提交信息必须能单独支撑回溯，读者仅看提交标题也应能判断本次改动的大致范围与意图。

## 4. 推荐结构

- 首行直接写中文结果句，例如“修复框架树画布拖拽视口模型”。
- 若需要补充上下文、验证方式或兼容性说明，可在正文继续使用中文分点说明。
- 提交正文中引用文件路径、命令、版本号或代码符号时，可保留原样，不必强行翻译。

## 5. 示例

- 合格：`修复框架树画布拖拽视口模型`
- 合格：`分离框架树与证据树默认入口`
- 合格：`发布 Shelf AI 0.0.44 并补齐双语版本说明`
- 不合格：`fix framework tree`
- 不合格：`update`
- 不合格：`wip`
