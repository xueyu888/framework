# 发布与版本说明标准

## 1. 目标

本标准定义仓库内公开发布物的版本说明要求，确保发布记录可读、可安装、可追溯，而不是只保留自动生成的 compare 链接。

## 2. 适用范围

- GitHub Release
- 可安装产物发布，例如 VSCode `.vsix`
- 其它面向外部使用者的版本化交付物

## 3. 强制要求

- 每次公开发布必须附带人工编写的版本说明，不得只依赖自动生成说明。
- 版本说明必须同时包含中文和英文两部分。
- 版本说明必须与发布版本一一对应，按版本落库保存。
- 版本说明必须基于“上一发布版本 tag 到当前发布版本 tag”的完整提交区间汇总（例如 `shelf-ai-v0.1.20..shelf-ai-v0.1.21`），不得只按单次会话、单个 PR 或单条提交写说明。
- 发布页必须附带可直接安装或直接使用的正式产物；只有源码压缩包不算完成发布。
- 若本次发布主要是流程修复、包装修复或文档修复，也必须明确写清楚，不得伪装成功能发布。

## 4. 版本说明最低结构

每份版本说明至少应包含以下内容：

- `## 中文说明`
- `## English Notes`
- 明确写出覆盖的提交区间（上一 tag 到当前 tag）
- 本次变更摘要
- 对使用者可见的影响
- 安装或获取方式（若发布产物需要手动安装）

若存在迁移、兼容性变化、已知限制或回滚提示，应额外写明。

## 5. Shelf AI 约束

对于 `tools/vscode/shelf-ai`：

- 每个公开版本必须提供 `tools/vscode/shelf-ai/release-notes/<version>.md`
- GitHub Release 正文必须直接使用该文件内容
- 发布资产中必须包含对应版本的 `.vsix`
- release notes 必须显式包含完整提交区间字符串：`shelf-ai-v<prev>..shelf-ai-v<current>`
- 若缺少 release notes 文件、缺少双语段落、缺少区间字符串或缺少 `.vsix`，则发布流程必须失败

`knowledge_base_basic` 约束：

- `projects/knowledge_base_basic` 仅用于本地验证、演示和回归测试
- 不得为 `knowledge_base_basic` 创建新的 GitHub Release、版本说明或安装包发布
- 若仓库内存在历史 `knowledge_base_basic` 发布物，应优先删除并停止继续维护

## 6. 执行原则

- 发布说明属于交付物本身的一部分，不是可选附属品。
- 能被用户下载到的版本，必须能从发布页直接看懂“这版改了什么、怎么安装、是否有注意事项”。
