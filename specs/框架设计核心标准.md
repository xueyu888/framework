# 框架设计标准（核心）

## 0. 目标（可选）

框架文档可以在开头记录设计目标、问题入口、关注对象或期望导出的结果类型，也可以省略不写。

目标的作用是提供探索起点，而不是充当结构正确性的第一性依据。若后续结构推导揭示出更稳定、更一般或更具生成性的组织方式，应允许修正目标表述，而不是反向让结构服从初始目标。

## 0.1 定位
本文件定义框架设计核心方法，不包含工程执行细节。

核心范围分为两部分：
1. 结构定义
2. 五项设计方法：能力声明、边界定义、最小结构基、基组合原则、验证

本文件采用以下基本立场：
- 结构是框架的第一性对象。
- 框架不是某个项目的模板，而是 AI 编程时代的人和 AI 之间的共同结构语言。
- 模块是若干结构基按明确关系拼装后的结果，不是额外命名的壳。
- 通用框架既可以是基础通用框架，也可以是领域通用框架；判断标准是“是否跨实例稳定复用”，而不是“是否完全无领域语义”。
- 设计意图或初始目标可以记录，但不属于框架核心必选项；若结构推导揭示出更稳定、更一般或更具生成性的组织方式，应允许修正目标表述，而不是反向让结构服从初始目标。
- 层级模型不是固定模板；是否分层、分多少层、每层语义为何，均由具体框架自行声明。
- 本文件只定义框架设计的语义核心。

## 0.2 结构定义（Structure）
人们谈论一个事物时，往往最先看到的是它的功能：置物架能放东西，桥能承重，语言能表达思想，程序能处理输入，组织能完成任务，生物能生长和适应。

但功能不是凭空出现的。一个事物之所以能够稳定地、重复地生成某种结果，并不是因为“功能”这个词本身具有力量，而是因为其内部已经存在某种更基础的组织方式，使这种结果得以成立。这个更基础的东西，就是结构。

结构不是结果，不是表现，不是实现，也不是实例；它不是“发生了什么”，而是“为什么能够这样发生”，是事物内部各要素之间得以稳定成立、相互配合并持续生成结果的组织关系。

功能是结构的表现，变化是结构的变形，演化是结构的重组。

页面如此，程序如此，组织如此，语言如此，机械如此，生命亦如此。凡能生成结果者，背后必有结构。

因此，在本框架中，结构是最基础、最原始、也是最核心的概念。

如需进一步展开理解“什么是结构”以及“最小结构基如何参与生成更高层结构”，可选阅读 [结构详解：从最小结构基到万物生成](../docs/extended-reading/结构详解-从最小可行基到万物生成.md)。该文档属于扩展阅读，不作为规范、代码实现或验证判定的直接依据。

## 1. 能力声明（Capability Statement）
定义框架负责导出的正向能力范围，以及明确不负责的非职责范围。

要求：
- 能力声明必须回答“该框架能稳定导出什么能力”。
- 非职责声明必须回答“该框架不负责什么”。
- 能力不是结构，能力是基在参数条件下经由组合规则稳定导出的结果。
- 同一组基与规则在不同参数下可以导出不同能力集合；能力应以结构导出关系表达。
- `C*` 只保留正向能力，不得再混入“非能力项 / 不负责什么 / 排除范围”这类负向语义。
- “不负责什么 / 排除范围 / 非职责”必须使用独立的 `N*` 声明；`N*` 不属于正向能力集合，也不进入 `B -> R -> C+` 导出链。
- 在 Markdown 表达上，`N*` 可位于 `## 1. 能力声明` 之内，但语义上必须视为独立的非职责声明集合，而不是正向能力列表的延伸项。

## 2. 边界定义（Boundary）
定义框架适用范围与限制条件。

参数描述的不是“变化本身”，而是“结构成立时允许变化的维度、范围与成立条件”。
在本标准中，参数是“结构内变量”：变量属于既定结构，用于表达结构内部可调维度，不负责生成新的结构单元或关系。

也就是说，参数回答的是：
- 什么可以变
- 变到哪里仍然算该结构的合法展开
- 超出哪里就不再属于当前结构问题域

因此，参数不是对 `B*` 的配套参数清单，也不是某个实例的具体取值；参数首先约束的是结构展开空间，而不是反过来从现有基定义里抽取说明项。
参数在框架层定义“变量及其合法域”，在实例层（如 `project.toml`）被赋值后才形成可量化的具体结果。

说明：
- 作者源术语现在统一使用 `Parameter / 参数`。
- 当前实现与 canonical 为兼容既有链路，仍保留 `boundary_*`、`FrameworkBoundary*` 等历史字段和类名；这些兼容命名不改变本节的作者侧术语。

要求：
- 每个参数项都可测量
- 每个参数项都可校验
- 参数项需直接服务于能力成立条件、适用域或验证条件
- 参数用于约束基的展开空间，不用于定义具体实例值
- 参数项应写成“变量维度 + 合法域 + 判定条件”的可执行表达，不得只写抽象口号
- 参数应优先表达“允许变化的维度 + 合法范围/作用域/窗口/条件”，而不是对象名词清单或对某个 `B*` 的附属注释
- 参数不是 `B*` 本体，不得把结构单元、动作步骤或状态机节点直接当作参数命名
- 参数不是实例值；应表达为可测的合法区间、作用域、窗口、阈值或判定条件，而不是直接写死某个具体值

## 3. 最小结构基（Minimal Structural Bases）
定义最小结构基（Minimal Structural Base）。

要求：
- 基必须能被稳定定位到某个模块内结构单元。
- 框架设计必须先从更基础、更通用的结构开始；若某类框架天然存在更细的通用原子，应优先把这些原子落在最低可复用结构层，而不是先合并成过粗的结构族
- 基是当前模块内最小、可解耦、可复用、可组合、跨实例稳定的结构单元
- 基的“最小”是相对于当前模块的最小可行表达边界，而不是要求所有 `B*` 在整个框架引用图里都是全局叶子
- 一切最小结构基都必须是结构，不能把动作、状态、目标或功能清单直接当作基
- `B*` 必须是抽象、可复用、跨实例稳定的结构单元
- 每个 `B*` 都必须是边界清晰、语义自足的结构件；不得把依赖脑补才能成立的残片当作基
- 框架文档必须把模块本身写清楚：由哪些结构基组成，每个结构基的边界是什么，基与基如何连接，组合原则如何判定成立。
- 命名和定义必须让不同读者指向同一个对象，而不是各自脑补；例如“向量”可以直接指向较明确的对象，“表示”则更容易让不同人理解成不同东西。凡是这类会让人各自理解的词，都不得直接作为结构基命名或定义。
- 最小结构基应尽量收敛自由度，在满足当前层生成要求的前提下，选择自由度最少且最合适的结构
- 基的命名与定义必须优先描述组织形式，不得用动作词、状态词或目标词替代结构定义
- 功能必须由结构导出，不得把功能清单直接当作基定义
- 若多个模块只是重复同一拆分轴而不提供新的独立结构选择，则应主动合并模块、压缩层级与收敛 `B*` 数量，直到每个保留下来的模块和基都代表真实独立的结构差异
- 若框架声明层级，则更高层 `B*` 可以由更低层模块组合而成；未继续向下展开的最低被引用层基必须保持自足，且根层 `L0` 的 `B*` 在本框架内部必须自足，不得再引用本框架其它模块。
- 若框架显式依赖外部模块，则必须显式声明该依赖；外部依赖是否合法优先由引用图的单向性、存在性与无环性判断，被引模块的 `Lx` 标签仅作为总体规划参考，不得代替真实依赖审查。
- 基不直接承接“能力归属”语义；能力由规则在组合层导出，不再要求在基层建立 `B -> C+` 的直接映射函数。
- 每个基必须至少绑定一个参数边界；记参数边界集合为 `P`，则基到参数边界的关系应满足 `H: B -> 𝒫(P) \\ {∅}`。
- 每个基必须进入至少一条合法组合规则；可由多条规则复用，并在不同参数边界下参与不同导出路径。
- 若两个基在同一模块内的边界集合、规则参与集合与下游影响集合完全一致，则视为伪拆分，必须合并或重构。

### 3.1 最小结构基拆分原则

> 拆分应持续进行，直到结构达到正交解耦且继续拆分不再新增结构收益；若已到达当前框架采用的最底层公理、原始关系或基元层，则在当前框架语义下不可再分。

最小结构基的拆分，以**正交解耦**与**结构收益**为判断标准。
拆分应持续进行，直到结构达到正交解耦，且继续拆分不再新增结构收益；或已到达当前框架采用的最底层公理、原始关系或基元层为止。

#### 3.1.1 必须再分

若候选结构内部仍包含可**独立变化**、可**独立受约束**、可**独立参与组合**或可**独立影响能力成立**的结构因素，则说明该结构尚未解耦，必须继续拆分。

关键词：未解耦、独立变化、独立组合

#### 3.1.2 不必再分

若候选结构已经达到正交解耦，且继续拆分不再新增**独立能力**、**独立约束**、**独立组合方式**或**独立验证点**；并且拆分后的部分除恢复原结构外，不再形成新的有意义组合，则该结构不必继续拆分。

关键词：已解耦、无新增收益、组合无意义

#### 3.1.3 不可再分

若候选结构已经拆至当前框架采用的最底层**公理**、**原始关系**或**基元层**，则在当前框架语义下不可再分。
此处“不可再分”是指：继续拆分将转入更低层的新问题域，而非绝对意义上的不能继续分析。

关键词：已触底、公理层、换层

#### 3.1.4 拆分停止条件

一个结构可以停止拆分，当且仅当满足下列条件之一：

1. 已达到正交解耦，且继续拆分不再新增结构收益；
2. 已到达当前框架采用的最底层公理、原始关系或基元层。

不得以“名字熟悉”“已经能用”“像一个现成组件/对象”“表现形态完整”作为停止拆分的理由。

#### 3.1.5 术语说明

“**组合无意义**”是指：拆分所得部分除恢复原结构外，不能形成具有独立语义的合法组合。
“**结构收益**”是指：继续拆分后新增的独立能力、独立约束、独立组合方式或独立验证点。

## 4. 组合原则（Combination Principles）
定义基如何形成有效系统。

要求：
- 显式模块引用关系是框架层次判断的第一性事实；`L0/L1/L2/...` 只是当前框架内部的局部投影、阅读分组和规划标注，不能反向覆盖真实依赖方向
- 组合原则只针对已定义的最小结构基进行选择、排序、约束与闭合，不应在组合阶段产生新的基；若需要新的独立结构单元，必须回到“最小结构基”层显式定义
- 组合原则是“结构生成约束”：它负责判定哪些基组合合法、失效或不适用，不负责参数变量本身的定义与实例赋值
- 基与基之间的拼装关系必须显式写成规则，不能依赖脑补
- 组合规则必须可表达为显式条件
- 规则间不可互相冲突
- 组合结果必须可判定“有效/无效”
- 每条规则至少声明一种结果：正向导出能力或失效结论；两者不可同时为空
- 合法组合规则必须说明参与基、组合方式、导出能力与参数绑定；禁止/无效/不适用规则必须说明参与基、组合方式、失效结论与参数绑定
- 规则应能区分合法组合、非法组合与不适用情形
- 组合原则的作用，是把最小结构基潜在的无序排列组合空间，从无限、发散、不可判定的状态，收敛为有限、可解释、可验证的合法组合集合
- 组合必须沿“更基础结构 -> 更高抽象结构”的单向依赖展开，不得反向污染
- 若框架声明层级，则本框架内部的显式引用必须只指向更低的局部层；局部层差是否为 `1` 属于规划选择，不再单独作为硬约束
- 若框架声明层级，则禁止同层引用、反向引用与任何循环依赖
- 允许同一下层节点被多个更高节点复用，但每个组合节点必须至少由一个更低层节点组成
- 从树结构看，未继续向下展开的基础结构落在叶子节点，其它高层结构可由下层节点逐层组成
- 若更高层领域框架建立在更基础的通用框架之上，应显式声明该依赖方向；更基础的通用结构不应反向依赖更高层领域结构
- 若框架显式依赖外部更基础通用框架，该依赖必须只承担承接面，不得替代当前框架内部的结构定义责任
- 记组合原则集合为 `R`。组合原则对基的参与关系记为 `I: R -> 𝒫(B) \\ {∅}`；`I(r)` 表示规则 `r` 使用的基集合，因此一个组合原则可以使用一个或多个基。
- 组合原则对能力的导出关系记为 `O: R -> 𝒫(C+) \\ {∅}`；`O(r)` 表示规则 `r` 导出的能力集合，因此一个组合原则可以导出一个或多个能力。
- 非法/不适用规则对非职责声明的归因关系可记为 `F: R -> 𝒫(N) \\ {∅}`；`F(r)` 表示规则 `r` 的失效结论集合。`F(r)` 不属于正向导出结果，不得伪装成 `O(r)`。
- 对每个正向能力 `c ∈ C+`，至少必须存在一条合法导出链 `g(c) -> r -> c`，即至少存在一个组合原则 `r`，使得 `g(c) ∈ I(r)` 且 `c ∈ O(r)`。

## 5. 验证（Verification）
定义如何判断框架结论是否成立。

要求：
- 必须有通过条件（pass criteria）
- 必须有失败原因（failure reasons）
- 必须有证据（evidence）
- 必须可自动验证层级与组合合法性
- 必须可区分“结构定义问题”与“实例配置问题”
- 必须验证“声明的能力是否真的由这些基在参数条件下经由规则导出”
- 当参数被实例化为配置 section 时，验证必须覆盖“参数 -> 主 section”的稳定映射，并能定位到对应配置落点
- 当框架被用作生成式实现输入时，验证应覆盖“框架 -> 实现”一致性，并将生成代码、测试结果与运行样例作为证据层产物
- 当框架被用作项目生成源时，验证还必须覆盖“实例目录内无手写项目代码”和“当前 generated 产物可由框架与实例配置重新生成且完全一致”
- 只有当框架本身是比较式或优化式框架时，才强制要求 `baseline/target`

## 6. AI 编程分层职责（强制）

### 6.1 Framework 的职责边界

- `Framework` 负责定义共同结构、公理、能力范围、参数、组合规则、验证方式与合法展开空间。
- `Framework` 首先定义基、参数与规则，不首先定义功能清单、实例值或实现细节。
- `Framework` 只回答“什么结构成立、什么组合合法、哪些参数允许变化”，不直接回答某个具体产品最终长什么样。
- `Framework` 不得固化项目专属品牌、实例文案、主题值、知识内容、技术栈、库名、文件结构、类名、函数名或运行时实现技巧。
- `framework/*.md` 的模块正文必须对下游文件与 section 落点无感；不得把 `project.toml`、具体文件名或 `[exact.frontend.surface]` 这类项目 section token 写进模块说明、规则与验证条目。层间职责由顶层规范与治理规则承接，不应混入具体框架模块定义。
- `Framework` 可以定义默认规则、派生约束与合法值域，但这些规则必须保持跨实例稳定复用。
- 当框架被用作 AI 编程输入语言时，框架文档仍然只定义结构标准；生成代码、测试与运行产物属于实现与证据层，不反向定义框架本体。

### 6.2 Unified Project Config 的职责边界

- 统一项目配置写在 `projects/<project_id>/project.toml` 中。
- `framework` 负责声明当前项目启用哪些 framework 根模块。
- `communication` 负责定义人与 AI 协作时使用的结构化沟通要求。
- `exact` 负责定义 Code 层精确消费的字段、路径、参数、策略与槽位。
- `communication` 不得替代结构字段，也不得直接被 Code 层消费。
- `exact` 不得改写 framework 已限定的结构参数。
- `project.toml` 应默认使用中文注释，并在文件头与主要 section 前说明职责边界。

### 6.3 层间映射与泄漏约束

- 允许的映射：
  - `Framework -> Config`：由 `FrameworkModule` 约束 `ConfigModule` 的合法展开空间。
  - `Config -> Code`：Code 只消费 `ConfigModule.exact_export` 并导出具体运行时代码。
  - `Code -> Evidence`：生成测试结果、运行样例、构建产物与验证证据。
- 禁止的泄漏：
  - `project.toml` 反向修改 `Framework` 的 `B*`、`R*`、边界定义或合法值域。
  - `Code` 直接读取 framework markdown 或 `communication`。
  - `Code` 成为产品真相、框架定义或实现配置的第一性事实来源。
  - `Evidence` 被当作人工编辑源反向定义 `Framework` 或 `project.toml`。
- 当框架被用作项目代码生成源时，项目行为只能由 `framework/*.md` 与 `projects/<project_id>/project.toml` 驱动；`projects/<project_id>/generated/*` 属于生成产物，不得作为人工编辑源。

## 7. 外部关联
本文件的外部关联由顶层规范树定义：
- `specs/规范总纲与树形结构.md`
- `specs/框架文档Lint标准.md`
- `specs/code/Python实现质量标准.md`
- 层级传导方向：`L0 -> L1 -> L2`
- framework 与 Config / Code / Evidence 的正式绑定由 `src/project_runtime/` 的四层编译器给出。
- 当框架参数被开放到项目层时，其关联落点是 `projects/<project_id>/project.toml` 的显式 section；参数不应被拆散到不可追溯的杂项层。
- 框架文档表达、作者入口与 lint 约束由 `specs/框架文档Lint标准.md` 承接；本文件不重复定义文档协议。
- 追溯、物化、变更传导和 lint 由顶层规范与校验脚本处理，不属于本文件的最小定义。
