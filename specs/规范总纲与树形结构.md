# 规范总纲与树形结构（L0）

## 1. 目的

本文件定义仓库唯一主链、目录边界、项目配置入口和物化规则。

## 2. 目录与分层

- `specs/`
  放 L0/L1 规范文档。
- `framework/<domain>/Lx-Mn-*.md`
  放正式 framework 模块文档；Markdown 是作者源。
- `framework_drafts/<domain>/Lx-Mn-*.md`
  放未发布草稿；草稿不进入正式项目物化链。
- `src/project_runtime/`
  放 Framework / Config / Code / Evidence 四层编译器与运行时装配逻辑。
- `projects/<project_id>/project.toml`
  放统一 Config 层配置，固定分为 `project / framework / communication / exact`。
- `projects/<project_id>/generated/`
  放 canonical JSON 及其派生视图；不是人工编辑源。

```text
specs/
├── 规范总纲与树形结构.md
├── 框架设计核心标准.md
├── 框架文档Lint标准.md
├── 可追溯性标准.md
└── code/

framework/
├── frontend/
├── knowledge_base/
├── backend/
└── ...

src/
└── project_runtime/

projects/
└── <project_id>/
    ├── project.toml
    └── generated/
```

## 3. 主链语义

仓库唯一主链为：

`Framework -> Config -> Code -> Evidence`

其中：

- `Framework`
  定义结构、公理、边界、基、规则与验证语义。
- `Config`
  只把 framework 允许的空间具体化为 `communication` 与 `exact`。
- `Code`
  只消费 `exact`，并在此层第一次落地到实现、运行时装配与代码绑定。
- `Evidence`
  只消费 Code export，负责验证结果、快照、追溯材料和 GUI 视图输入。

## 4. 强制约束

- Framework Markdown 继续是作者源，不得被 schema 或配置反向生成替代。
- 每个 framework 文件必须对应一个独立 `FrameworkModule` class。
- 每个 `B*` 必须对应一个独立 `Base` class。
- 每个 `R*` 必须对应一个独立 `Rule` class。
- `project.toml` 是唯一项目配置入口。
- Config 只能表达 `communication` 与 `exact` 两类配置。
- Code 不得直接读取 framework markdown 或 `communication`。
- `projects/*/generated/canonical.json` 是唯一机器真相源。
- 其它视图只能是 canonical 的派生物。
- 禁止直接编辑 `projects/*/generated/*`。
- 代码实现默认遵循“简洁等效优先”：写任何代码前，必须先评估是否存在更简洁的等效方案；若存在，应优先采用更简洁方案。
- 若必须采用更复杂实现，必须能说明必要性（如正确性、性能、兼容性或可测试性），且不得为“未来可能用到”提前引入复杂度。

## 5. 物化规则

- 修改 `framework/*.md` 后，必须重新物化相关项目并更新 canonical 与派生视图。
- 修改 `projects/<project_id>/project.toml` 后，必须重新物化项目。
- 修改 `src/project_runtime/` 后，必须验证四层编译结果与 Evidence 输出仍然成立。
- 修改派生视图生成脚本后，必须确保其输入仍然只来自 canonical。

## 6. 验证命令

```bash
uv run mypy
uv run python scripts/materialize_project.py
uv run python scripts/validate_canonical.py
uv run python scripts/validate_canonical.py --check-changes
```

## 7. 合并门槛

- 仓库只保留 `Framework -> Config -> Code -> Evidence` 这条主链
- `project.toml` 切片与四层编译结果一致
- canonical JSON 可重新物化
- 派生视图明确来自 canonical
