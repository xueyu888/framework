# 规范总纲与树形结构（L0）

## 1. 目的

本文件定义仓库的唯一主链、标准目录边界、项目配置入口和变更传导规则。

## 2. 目录与分层

- `specs/`
  承载 L0/L1 规范文档。
- `framework/<domain>/Lx-Mn-*.md`
  承载正式 framework 模块文档；Markdown 是作者源。
- `framework_drafts/<domain>/Lx-Mn-*.md`
  承载未发布草稿；草稿不进入正式项目物化链。
- `src/framework_packages/`
  承载 framework package contract、registry 和每个 framework 文件对应的 package 入口类。
- `projects/<project_id>/project.toml`
  承载统一项目配置；逻辑上必须区分 `selection / truth / refinement / narrative`。
- `projects/<project_id>/generated/`
  承载 canonical JSON 及其派生视图；不是人工编辑源。

```text
specs/
├── 规范总纲与树形结构.md
├── 框架设计核心标准.md
├── 框架文档Lint标准.md
├── 可追溯性标准.md
└── code/

└── <project_id>/
    ├── project.toml
    └── generated/
```

## 3. 主链语义

仓库唯一主链为：

`Framework Markdown -> Package Registry -> Project Config -> Code -> Evidence`

其中：

- `Framework Markdown`
  定义结构、公理、边界、组合规则与验证语义。
- `Package Registry`
  建立 `framework file <-> package entry class <-> module id` 的正式绑定。
- `Project Config`
  决定本项目选择哪些 framework roots、哪些产品真相、哪些实现细化。
- `Code`
  由 package compile 和 runtime assembly 收敛出的运行时面。
- `Evidence`
  编译证据、运行证据、治理视图和校验报告。

## 4. 强制约束

- Framework Markdown 继续是作者源，不得被 schema 反向生成替代。
- 每个 framework 文件必须对应一个 package。
- 每个 package 必须有唯一入口 class，并实现统一 contract。
- registry 是 framework 与 package 的唯一绑定真相。
- `project.toml` 是唯一项目配置入口。
- 配置必须按模块树逐层切片分发。
- `projects/*/generated/canonical_graph.json` 是唯一机器真相源。
- `generation_manifest.json`、`derived_governance_manifest.json`、`derived_governance_tree.json`、严格报告和覆盖报告只能是 canonical 的派生视图。
- 禁止直接编辑 `projects/*/generated/*`。

## 5. 变更传导

- 修改 `framework/*.md` 后，必须评估对应 package、project config 和 generated evidence。
- 修改 `src/framework_packages/` 中的 contract、registry 或 package compile 逻辑后，必须重新物化项目并验证 canonical 与派生视图。
- 修改 `projects/<project_id>/project.toml` 后，必须重新物化项目。
- 修改派生视图生成脚本后，必须确保其输入仍然只来自 canonical。

## 6. 验证命令

```bash
uv run mypy
uv run python scripts/materialize_project.py
uv run python scripts/validate_strict_mapping.py
uv run python scripts/validate_strict_mapping.py --check-changes
```

## 7. 合并门槛

- 主链仍然是 `Framework Markdown -> Package Registry -> Project Config -> Code -> Evidence`
- registry 绑定无冲突、无未实现、无悬空 package
- `project.toml` 切片与 package contract 一致
- canonical JSON 可重新物化
- 派生视图明确来自 canonical
