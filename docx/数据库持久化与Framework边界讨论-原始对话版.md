## 第 1 轮

### 用户

对于关系型数据库的表格来说，是不是谁建表谁维护比较好？比如说，A程序动态建立了一个表，表格有哪些列，建立哪些索引，表格的约束，事务这些内容就统一由A程序进行负责，其他程序只管数据的读写。

### 助手

通常是的，但要稍微改成一句更准确的话:

更好的原则不是“谁建表谁维护”，而是“谁拥有这张表背后的业务语义，谁维护这张表的结构与演进”。

也就是说，如果 A 程序创建这张表，而且这张表本质上是 A 的领域模型一部分，那么通常就应该由 A 负责：

- 表结构：列、类型、默认值
- 约束：主键、唯一约束、外键、检查约束
- 索引：建什么索引、为什么建
- 数据演进：迁移、回填、废弃字段
- 写入语义：事务边界、一致性规则

其他程序最好只做两类事：

- 按约定读写数据
- 不擅自修改 schema 和核心约束

这样做的好处很明显：

- 避免多个程序同时改 schema，责任边界清楚
- 业务规则集中，不容易出现“表能写进去，但语义已经错了”
- 迁移和回滚更可控
- 索引和约束不会被不同程序各自优化到互相打架

但有几个前提要注意：

1. 其他程序不能“随便写”
如果别的程序直接写表，却不了解 A 的业务约束，就容易破坏数据一致性。很多时候“其他程序只管读写”这句话太宽了，最好改成“其他程序只能按 A 定义的接口/契约读写”。

2. 事务不一定只由 A 独占
如果一次业务操作跨多个系统或多个表，事务边界可能不是单个程序能完全控制。此时要么通过服务接口收口，要么明确最终一致性方案，而不是默认大家都直接碰表。

3. “共享表”通常是高风险设计
如果一张表被多个程序共同拥有、共同改结构，后期几乎一定会出现演进困难。能避免就避免。

4. 建表者不一定等于所有者
有时 A 只是初始化工具、脚本或平台组件，真正理解业务的是 B 团队。那维护权应该归业务所有者，不该机械地归“最早建表的人”。

更实用的落地建议是：

- 一张表只设一个 schema owner
- 由 owner 负责 migration
- 其他程序尽量通过 API、存储过程、消息或受控读写层访问
- 至少要有明确的数据契约文档，写清楚哪些字段谁能写、哪些只能读

如果你愿意，我可以继续帮你把这个问题展开成几种架构场景对比，比如：
1. 多程序直连同库同表
2. 一个 owner，其它程序只读
3. 一个 owner，其它程序经 API 写
4. 每个程序各自持有自己的表，靠消息同步

我可以顺手告诉你哪种最稳。

## 第 2 轮

### 用户

那我有个问题，按照shelf-ai的约束和当前目录中的specs中的标准来看，如果某个模块需要维护数据库，那从框架的角度来看应该怎么描述呢？这个模块中是不是要搭建一个数据库的框架？

### 用户

我不是在针对Chunk_Tagging_V2，我只是在单纯提问

### 助手

按照 shelf 这套规范，答案通常是：

不是先问“要不要搭一个数据库框架”，而是先问“数据库在这个模块里是不是一条需要被 framework 显式声明的业务边界”。

如果只是实现细节，比如模块最终用 MySQL/Postgres/SQLite 落盘，但对上层来说它只是“保存一下结果”，那不需要专门建数据库 framework。因为 framework 只该定义稳定的结构、公理、边界和合法组合，不该先落到技术栈、库名、ORM、表名这类实现细节，这一点在 [specs/框架设计核心标准.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/框架设计核心标准.md) 和 [specs/规范总纲与树形结构.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/规范总纲与树形结构.md) 里是明确的。

更合适的做法是分两种情况：

1. 如果“存取数据库”本身是模块对外能力的一部分，就建“持久化接口/记录仓储接口”这种 framework 模块。
这个模块描述的是业务语义，不是数据库产品说明。它应当写：
- 能力：查询什么、写回什么、按什么作用域读写
- 边界：最少字段、唯一定位条件、状态合法值、单次作用范围、一致性要求、幂等要求
- Base：查询接口、写入接口、状态回写接口
- Rule：这些接口如何组合成完整业务能力

仓库里现成的例子就是 [framework/Chunk_Tagging/L1-M1-数据库接口.md](/home/wuyz/Ability_Enhance/frame/shelf/framework/Chunk_Tagging/L1-M1-数据库接口.md)，它写的是“查询/写回的结构语义”，不是“怎么建表”。

2. 如果数据库只是模块内部实现细节，就不要在 framework 里单独起数据库模块。
这时 framework 只写业务对象和业务流转，最多在 `N*` 里明确“本模块不负责底层数据库实现细节”。仓库里这种写法可以参考 [framework/Chunk_Tagging/L3-M0-标注审核模块.md](/home/wuyz/Ability_Enhance/frame/shelf/framework/Chunk_Tagging/L3-M0-标注审核模块.md#L12)。

所以，“数据库相关内容”在 shelf 里最好这样分层：

- `framework`
  写业务上必须稳定的存储语义，比如“这类记录必须能被唯一定位”“状态只能这样流转”“写回至少包含哪些字段”
- `exact`
  写代码要精确消费的实现参数，比如连接方式、库类型、表名、字段映射、事务策略、批量大小
- `code`
  真正实现 SQL、ORM、migration、索引、事务

也就是说，表结构里的内容不要一股脑都塞进 framework。可以这样判断：

- “这个约束换了，业务语义就变了” -> 进 framework
- “这个约束主要是为了实现/性能/部署” -> 进 exact 或 code

举几个典型归类：

- 进 `framework`
  - 记录必须由哪些业务字段唯一定位
  - 写回时哪些字段是必填
  - 状态字段允许哪些值、如何流转
  - 单次操作作用于一条还是一批、一个对象还是一个文档域
- 进 `exact`
  - 用 postgres 还是 mysql
  - 具体表名、列名映射
  - 是否启用某类索引策略
  - 事务隔离级别、重试次数、连接池参数
- 留在 `code`
  - 建表 SQL
  - migration 脚本
  - ORM model
  - 索引实现细节

所以结论很直接：

不是“模块里要维护数据库，就一定要搭一个数据库框架”，而是“当数据库读写契约本身已经构成稳定的业务边界时，才把它抽成 framework 模块”；而且这个模块应该是“领域持久化接口框架”，不是“通用数据库框架”。

如果你愿意，我下一步可以直接给你一个通用模板，写成“某模块需要持久化时，framework 该怎么写 C/P/B/R/V”。

## 第 3 轮

### 用户

这里的主要问题是，我现在需要将数据存入到数据库中，但是目前数据库中并没有对应的表格，也就是说我需要先新建一个表格，并且后续可能会维护、调整这个表格的约束、事务之类的东西。这样的话应该怎么做呢？感觉这一套是很大的东西。而且不太好通过framework来搭建。几乎是必然涉及到数据库的一些底层内容。

### 助手

是的，这件事本来就很大，而且你感受到的困难点是对的:

“不存在表，需要建表，后续还要维护约束、索引、事务”这整套内容，确实不应该全塞进 framework。

按 shelf 这套分层，更合理的理解是：

`framework` 只定义“这张表承载的业务语义和稳定约束”  
`exact` 只定义“代码实现要精确消费的数据库参数与映射”  
`code` 才真正负责“建表、迁移、索引、事务、ORM/SQL 实现”

这和 [specs/框架设计核心标准.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/框架设计核心标准.md) 以及 [specs/规范总纲与树形结构.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/规范总纲与树形结构.md) 是一致的。

你现在这种场景，最稳的做法不是去建一个“数据库底层框架”，而是建一个“该模块拥有的数据对象 + 持久化接口”框架。也就是把数据库当成“该模块的持久化边界”来描述，而不是把数据库产品本身当成 framework 对象。

可以这样拆：

1. 在 framework 里定义“记录对象”
比如 `评分记录`、`任务记录`、`状态记录`。
这里写的是业务语义字段：
- 哪些字段是这条记录的身份
- 哪些字段是必填
- 哪些状态值合法
- 哪些关系必须成立
- 哪些更新必须原子发生

2. 在 framework 里再定义“记录仓储接口”或“持久化接口”
这里写的是能力和边界：
- 能写入什么
- 能按什么条件查询
- 能更新什么状态
- 单次操作作用于一条还是一批
- 哪些写入必须幂等
- 哪些修改必须在一个事务语义里完成

3. 明确 `N*`
把不该放进 framework 的东西明确排除掉：
- 不负责具体表名
- 不负责 SQL 方言
- 不负责 ORM 模型写法
- 不负责索引实现细节
- 不负责事务隔离级别的具体枚举值
- 不负责 migration 文件组织方式

然后真正“很大”的那部分，落到下面两层：

`exact` 适合放：
- 数据库类型
- 表名
- 字段映射
- 主键/唯一键映射策略
- 是否启用某些索引策略
- 事务模式开关
- migration 入口、schema version

`code` 适合放：
- 建表 SQL / Alembic / ORM migration
- 索引创建
- 外键实现
- 事务包装
- 回填脚本
- schema upgrade / downgrade

最关键的一条判断线是：

如果某个东西变了，会改变“业务语义是否成立”，它应进 framework。  
如果某个东西变了，主要影响“怎么实现、怎么优化、怎么部署”，它应进 exact 或 code。

例如：

进 framework：
- 一条记录必须由哪些业务字段唯一定位
- 写回必须同时更新哪些状态字段
- 某个状态流转是否合法
- 某次提交是否要求原子语义

不进 framework：
- 表叫 `tag_score_records` 还是 `chunk_tagging_scores`
- 用 PostgreSQL 还是 MySQL
- 建 BTree 还是 Hash 索引
- 隔离级别是 `READ COMMITTED` 还是 `REPEATABLE READ`
- ORM 用 SQLAlchemy 还是手写 SQL

所以你现在该怎么做，实践上通常是这条线：

1. 先补 framework 中的“业务记录”与“持久化接口”语义
2. 再在 `project.toml` 的 `exact` 里补数据库精确参数
3. 再在 code 层写 migration / DDL / transaction
4. 后续如果只是索引优化、事务实现调整，通常只改 exact/code
5. 后续如果连“记录语义、唯一性、状态合法值”都变了，才需要先改 framework

仓库里现成的参考方向其实已经有了：[framework/Chunk_Tagging/L1-M1-数据库接口.md](/home/wuyz/Ability_Enhance/frame/shelf/framework/Chunk_Tagging/L1-M1-数据库接口.md)。它写的是“数据库接口的业务边界”，而不是“数据库底层实现”。这种方向是对的。

如果你愿意，我可以下一步直接给你一个通用骨架，专门回答这件事：
“当一个模块需要自建并维护数据库表时，framework 文件应该怎么写成 `C/P/B/R/V`。”

## 第 4 轮

### 用户

我感觉，这一套框架，是不是静态的框架？它不包含我刚才说的那种，运行过程中发生变化的应对措施？

### 助手

可以这么理解：

对，`framework` 在 shelf 里本质上是“偏静态”的。  
但这里的“静态”不是指它只能描述一张不变的图纸，而是指它主要负责定义：

- 稳定的结构
- 长期成立的边界
- 合法的变化空间
- 动态行为必须遵守的规则

它不直接承载“运行过程中具体怎么应对变化”的实现细节。

所以你刚才那个直觉是对的：  
像“运行中发现表不存在怎么办”“怎么自动建表”“如何迁移旧表”“事务失败怎么重试”“线上怎么兼容新旧 schema”这些，核心落点通常不在 `framework`，而在 `exact` 和 `code`。

更准确地说，shelf 里是这样分的：

`framework` 回答：
- 哪类数据对象存在
- 它们的业务语义是什么
- 哪些字段/状态/关系是稳定约束
- 哪些变化是合法的
- 哪些操作必须原子、幂等、可追溯

`exact` 回答：
- 当前项目决定采用什么数据库策略
- 表名、字段映射、版本策略、迁移模式、事务策略这些精确参数是什么

`code` 回答：
- 真正如何建表、改表、迁移、回滚、重试、补偿、加索引、包事务

所以如果你问：

“framework 包不包含运行过程中发生变化的应对措施？”

答案是：

- 包含“应对措施的规则边界”
- 不包含“应对措施的具体执行机制”

比如数据库场景里，`framework` 可以描述：

- 记录有哪几种状态
- 状态允许怎样流转
- 写入和状态更新必须在同一事务语义内完成
- schema 演进必须保持哪些兼容边界
- 哪些字段一旦落库就不可回写
- 哪些失败允许重试，哪些失败必须人工介入

但它通常不描述：

- 用哪种 migration 工具
- 启动时自动建表还是单独发布 migration
- DDL 怎么写
- 事务隔离级别具体取什么
- 重试次数、退避策略、锁实现方式

所以你可以把 shelf 的 `framework` 理解成：

“动态系统的静态宪法”  
而不是  
“运行时调度器本身”。

你的问题也暴露出一个很关键的点：  
如果一个需求的大头是“运行时演化和存储实现”，那它天然就不会主要靠 framework 搭起来。framework 只负责把这件事的业务边界钉住，避免实现层失控。

如果你愿意，我可以下一步直接帮你把这句话落成一个非常实用的模板：

“当某模块需要在运行中建表、迁移、维护事务时，framework 里该写什么，不该写什么。”

## 第 5 轮

### 用户

先讲一下，什么叫如果一个需求的大头是“运行时演化和存储实现”，那它天然就不会主要靠framework搭起来。framework 只负责把这件事的业务边界钉住，避免实现层失控。
然后落实成一个非常实用的模板：“当某模块需要在运行中建表、迁移、维护事务时，framework 里该写什么，不该写什么。”

### 助手

**先讲概念**

“如果一个需求的大头是运行时演化和存储实现，那它天然就不会主要靠 framework 搭起来”的意思是：

这个需求的难点，不在“业务结构是什么”，而在“系统运行起来以后，怎么把这些结构安全地落到真实数据库里，并持续演进”。

比如下面这些工作，都是典型的“运行时演化和存储实现”：

- 表不存在时怎么创建
- 线上已有旧表时怎么迁移到新结构
- 新旧 schema 怎么兼容一段时间
- 哪些写入要放进同一事务
- 失败后怎么重试、回滚、补偿
- 索引什么时候加、怎么避免锁表
- 并发写入时怎么保证唯一性和一致性

这些问题的共同点是：它们高度依赖具体数据库、具体部署环境、具体实现策略。  
而 shelf 里的 `framework` 按规范主要负责“稳定结构、边界、合法组合空间”，不是直接承载这些技术实现细节。这个边界在 [specs/框架设计核心标准.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/框架设计核心标准.md) 和 [specs/规范总纲与树形结构.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/规范总纲与树形结构.md) 里是一致的。

所以更准确地说：

- `framework` 负责定义：这类数据在业务上是什么、必须满足什么、允许怎样变化
- `exact` 负责定义：当前项目决定怎样把这些边界精确落到数据库方案上
- `code` 负责实现：建表、迁移、事务、索引、重试、兼容、回填

也就是说，`framework` 不是把整套数据库工程做出来，而是给数据库工程划红线。  
它像“宪法”，不是“施工队”。

**一个很实用的判断法**

如果某条内容回答的是下面这些问题，它更像 `framework`：

- 这张表在业务上存的到底是什么对象
- 一条记录怎样才算合法
- 哪些字段构成业务身份
- 哪些状态允许流转，哪些不允许
- 哪些更新必须被视为同一次业务提交
- 哪些失败可以重试，哪些失败必须人工介入

如果某条内容回答的是下面这些问题，它更像 `exact` 或 `code`：

- 表名到底叫什么
- 用 PostgreSQL/MySQL/SQLite 哪个
- 用 ORM 还是 SQL
- 索引怎么建
- migration 文件怎么组织
- 事务隔离级别怎么配
- 启动时自动建表还是发布前跑 migration

**落成模板**

下面这个模板比较适合“某模块需要在运行中建表、迁移、维护事务”时的 framework 写法。

```md
# 记录持久化接口:RecordPersistenceInterface

@framework

0. 目标（Target）
声明一个持久化接口模块，用于将某类业务记录写入持久化存储，并支持后续按业务身份查询、状态更新与版本演进约束。

## 1. 能力声明（Capability Statement）

- `C1` 记录写入能力：支持写入一条或一批合法业务记录。
- `C2` 记录查询能力：支持按业务身份或业务作用域查询已有记录。
- `C3` 状态更新能力：支持对已有记录执行合法状态流转。
- `C4` 原子提交能力：支持将一次业务提交中要求同时成立的字段更新作为同一原子操作处理。
- `C5` 演进兼容能力：支持在结构演进过程中保持既有业务身份与核心状态语义不失真。

- `N1` 不提供具体建表实现能力：不负责 DDL、ORM model 或 migration 脚本实现。
- `N2` 不提供具体数据库选型能力：不负责声明 PostgreSQL/MySQL/SQLite 等具体产品。
- `N3` 不提供性能优化实现能力：不负责索引类型、分区策略、连接池和锁实现细节。

## 2. 边界定义（Boundary / Parameter 参数）

- `P1` 记录身份边界：每条记录必须能被一组稳定业务字段唯一定位。
- `P2` 写入完整性边界：新写入记录必须至少包含最小必填字段集合。
- `P3` 状态合法值边界：状态字段只能落入已声明的合法值域。
- `P4` 状态流转边界：状态只能按声明的流转路径变化，不允许越级或逆向非法变更。
- `P5` 原子更新边界：同一次业务提交要求同时成立的字段，必须作为同一原子语义处理。
- `P6` 幂等边界：重复提交同一业务请求时，不得生成语义重复且彼此冲突的记录。
- `P7` 演进兼容边界：结构调整后，不得破坏既有业务身份字段、核心状态字段及其解释语义。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 记录身份：唯一定位单条业务记录的身份结构。
- `B2` 最小记录载荷：创建记录所需的最小字段集合。
- `B3` 查询接口：按业务身份或作用域读取记录的接口。
- `B4` 写入接口：创建记录的接口。
- `B5` 状态更新接口：执行合法状态流转的接口。
- `B6` 原子提交单元：要求在一次提交中共同成立的字段更新集合。

## 4. 基组合原则（Base Combination Principles）

- `R1` 创建记录组合
  - `R1.1` 参与基：`B1 + B2 + B4`
  - `R1.2` 组合方式：接收最小合法记录载荷后，以稳定业务身份写入一条新记录。
  - `R1.3` 输出能力：`C1`
  - `R1.4` 参数绑定：`P1 + P2 + P6`

- `R2` 查询记录组合
  - `R2.1` 参与基：`B1 + B3`
  - `R2.2` 组合方式：按记录身份或业务作用域查询已有记录。
  - `R2.3` 输出能力：`C2`
  - `R2.4` 参数绑定：`P1`

- `R3` 状态流转组合
  - `R3.1` 参与基：`B1 + B5`
  - `R3.2` 组合方式：对已有记录执行一次合法状态更新，并保持状态解释一致。
  - `R3.3` 输出能力：`C3`
  - `R3.4` 参数绑定：`P3 + P4`

- `R4` 原子提交组合
  - `R4.1` 参与基：`B1 + B5 + B6`
  - `R4.2` 组合方式：对同一次业务提交要求共同成立的字段更新，以单次原子语义完成。
  - `R4.3` 输出能力：`C4`
  - `R4.4` 参数绑定：`P5`

- `R5` 演进兼容组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`
  - `R5.2` 组合方式：当记录结构演进时，仍保持业务身份、核心状态和值域解释稳定。
  - `R5.3` 输出能力：`C5`
  - `R5.4` 参数绑定：`P1 + P3 + P7`

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须可由对应 `C* + P*` 推导得到。
- `V2` 规则完整性：每个 `R*` 必须明确参与基、组合方式、输出能力与参数绑定。
- `V3` 状态合法性：任意有效状态更新都必须满足 `P3 + P4`。
- `V4` 原子语义符合性：任意声明为同次提交的字段更新都必须满足 `P5`。
- `V5` 幂等符合性：任意重复提交都不得破坏 `P6`。
- `V6` 演进兼容性：结构演进后仍必须满足 `P1 + P3 + P7`。
```

**framework 里该写什么**

重点写“业务上不可随便变”的东西：

- 记录是什么业务对象
- 哪些字段构成业务身份
- 哪些字段是最小必填
- 状态有哪些合法值
- 状态怎样流转
- 哪些更新必须原子完成
- 什么叫幂等
- 演进时什么语义绝不能破坏

一句话概括：  
写“业务约束”，不写“数据库技巧”。

**framework 里不该写什么**

尽量不要写这些：

- 具体表名、列名
- 数据库产品名
- ORM 类名、SQL 文件名
- 索引类型、分区方式
- migration 工具名
- 事务隔离级别具体枚举
- 锁实现、连接池、重试参数
- 启动时自动建表还是人工发版建表

这些更适合放到 `exact` 或 `code`。

**一个简单落点表**

- `framework`
  - `record_identity`
  - `required_fields`
  - `state_machine`
  - `atomicity_rule`
  - `idempotency_rule`
  - `compatibility_rule`

- `exact`
  - `db.kind`
  - `table_name`
  - `column_mapping`
  - `unique_key_mapping`
  - `migration_strategy`
  - `transaction_strategy`

- `code`
  - migration
  - DDL
  - repository
  - transaction wrapper
  - retry / rollback / backfill

**最重要的实操建议**

如果你发现自己在 framework 里已经开始写：

- 表结构变更步骤
- 索引优化方案
- DDL 执行顺序
- 事务隔离级别配置
- 线上迁移流程

那通常说明你已经越过 framework 边界了，应该往 `exact` / `code` 收。

如果你愿意，我下一步可以继续帮你把这个模板再压缩成一个“最小可用版”，专门适合你现在这种“先建表、后续持续演进”的场景。

### 用户

我感觉你还是没有回答清楚我的问题，如果我的需求大头是“运行时演化和存储实现”，是不是就不适合用framework来描述了？

### 助手

是，但要分成“完全不适合”和“不适合作为主描述方式”两层来看。

更准确的回答是：

如果你的需求大头确实是“运行时演化和存储实现”，那么它通常不适合主要用 `framework` 来描述；  
但也不是说完全不能用 `framework`，而是 `framework` 只适合描述其中那一小部分稳定的业务边界。

也就是说，这种需求往往是：

- `framework` 只占 20%
- `exact + code` 占 80%

你真正困难的部分，比如：

- 运行中自动建表
- schema 迁移
- 旧数据兼容
- 事务与并发
- 重试、补偿、回滚
- 索引和性能策略

这些都不是 framework 的强项。因为这些东西的核心问题不是“结构是否成立”，而是“系统在真实运行环境里怎么安全工作”。

所以结论可以直接说成：

1. 如果你要表达的是“业务上到底存什么、哪些字段必须有、哪些状态合法、哪些更新必须原子”，适合用 `framework`。
2. 如果你要表达的是“表怎么建、怎么迁移、怎么兼容、事务怎么做、失败怎么处理”，不适合主要用 `framework`，应放到 `exact` 和 `code`。
3. 如果一个需求 80% 都是第 2 类内容，那它就不是一个“主要靠 framework 搭起来”的需求。

最实用的判断句是：

如果不写数据库实现，需求的主体就几乎没剩下什么，那它就不适合作为一个以 framework 为主的模块。  
如果去掉数据库实现后，仍然有清晰的业务对象、状态机、约束、输入输出边界，那它就适合先写 framework，再落实现。

所以对你这个问题，我会给一个明确答案：

是的，若需求的大头是运行时演化和存储实现，它就不适合主要用 framework 描述。

你也可以把它理解成：

- `framework` 适合描述“这个系统在业务上必须成为什么样”
- `runtime/storage engineering` 适合描述“这个系统在工程上必须怎么活下来”

## 第 7 轮

### 用户

为什么exact + code 占 80%？这两个部分的内容不是通过framework一步步生成的吗？如果framework中没有相关的说明，我如何保证exact + code 能实现我想要的内容呢？

### 助手

你这个追问很关键，核心误区就在这里：

`exact + code` 不是“从 framework 自动、完整、唯一地一步步生成出来”的。  
更准确地说，shelf 里的关系是：

- `framework` 定义合法结构和不可越界的业务边界
- `exact` 在当前项目里把其中需要落地的部分实例化、精确化
- `code` 根据 `exact` 去实现
- `evidence` 验证实现有没有真的满足前面三层

所以 `framework` 更像“约束空间”，不是“完整施工蓝图”。

**为什么我说 `exact + code` 可能占 80%**

这个 `80%` 不是数学比例，是在说“工程工作量和实现复杂度”大头常常在那里。因为很多问题并不是“定义不定义得出来”，而是“即使定义了，也还有大量实现选择”。比如：

- `framework` 可以规定“某类记录必须唯一”
- 但它不能唯一决定你用唯一索引、悲观锁、乐观锁、幂等键还是 upsert

再比如：

- `framework` 可以规定“某些字段更新必须原子”
- 但它不能唯一决定你用单库事务、分阶段提交、补偿事务还是消息 outbox

也就是说，`framework` 能定义“必须满足什么”，但往往不能唯一推出“应该怎么实现”。

**那如果 framework 没写，怎么保证 exact + code 实现我想要的内容？**

答案是：靠分层保证，不是只靠 framework 一层保证。

可以把“保证”拆成三种：

1. `framework` 保证业务边界不漂移  
比如：
- 哪些字段构成业务身份
- 哪些状态合法
- 哪些流转允许
- 哪些更新必须被视为同一次业务提交

2. `exact` 保证当前项目选择被明确说清楚  
比如：
- 用什么数据库
- 表怎么映射
- 唯一键怎么落
- migration 策略是什么
- 事务策略是什么

3. `code + evidence` 保证系统真的按这些要求运行  
比如：
- migration 真能跑
- 事务真能兜住
- 并发下不出错
- 回归测试通过
- 验证脚本和运行证据成立

所以不是“framework 不写就没法保证”，而是：

- 业务语义上的保证主要来自 `framework`
- 项目方案上的保证主要来自 `exact`
- 可执行正确性的保证主要来自 `code + evidence`

**更关键的一点**

如果某件事“你非常想保证”，先问自己它属于哪一类：

- 如果它是稳定业务规则，就应该进 `framework`
- 如果它是当前项目的实现选择，就应该进 `exact`
- 如果它是具体工程做法，就应该进 `code`
- 如果它是“是否真的生效”，就应该靠 `evidence` 验证

**举个最直接的例子**

你想保证“评分记录不会重复”。

这句话可能有三种落点：

- `framework`
  - “一条评分记录必须由 `doc_id + chunk_id + tag_id` 唯一定位”
- `exact`
  - “当前项目把这组业务身份映射为某张表的唯一键”
- `code`
  - “通过唯一索引 + upsert 实现”
- `evidence`
  - “并发写入测试证明没有重复记录”

这四层合起来，才是完整保证链。

**所以你真正该怎么理解**

不是“framework 没写实现细节，我就没法保证系统行为”，  
而是“framework 只负责保证那些应该稳定的业务语义；其余保证要靠 exact、code 和 evidence 接力完成”。

反过来说，如果你希望 framework 一层直接推出完整数据库工程，那它就会被迫塞进大量：

- 表名
- 索引
- migration
- 事务策略
- 兼容方案

这会让 framework 失去“稳定结构语言”的角色，变成半实现文档，最后反而更难维护。

**一句话结论**

`framework` 不是用来穷尽实现的，它是用来防止实现偏离业务语义的。  
你想要的“完整保证”不是只靠 framework，而是靠 `framework -> exact -> code -> evidence` 这一整条链。

如果你愿意，我可以下一步继续把这个问题讲透成一个判断表：

- 什么内容必须写进 `framework`
- 什么内容应该只写进 `exact`
- 什么内容只能在 `code/evidence` 里保证

## 第 9 轮

### 用户

那我再换个角度，如果我已经完成了framework的搭建， 并且已经转换为了代码运行起来了。这个时候我想修改其中一个，我自己定义的存入数据库的数据记录的字段。对于数据库来说就是要在一个表中增加一列。这种情况下，数据库是需要停机维护吗？还是支持我动态维护呢？以及我需要怎么修改framework？以及通过framework生成的程序，是否会与之前产生较大差异？这样的差异会不会导致程序运行的稳定性或者性能变差？

### 助手

是分情况的，但先给你一个直接结论：

不是一改字段就一定要停机。  
如果只是“给表增加一列”，而且这列是兼容式新增，通常可以做成不停机或近似不停机的动态演进。  
真正危险的不是“加一列”本身，而是你是否同时引入了：

- 非兼容读写
- 大量历史数据回填
- 新约束
- 新索引
- 更重的事务语义

这和 shelf 的分层是对得上的：`framework` 只定义边界，[specs/框架设计核心标准.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/框架设计核心标准.md) 与 [specs/规范总纲与树形结构.md](/home/wuyz/Ability_Enhance/frame/shelf/specs/规范总纲与树形结构.md) 都是这个意思；真正“怎么平滑改数据库”主要是 `exact + code` 的事情。

**先回答停机还是动态维护**

一般有两类：

- 适合动态维护
  - 新增的是可空列，或有安全默认值
  - 老代码不依赖这列
  - 新代码能容忍这列暂时为空
  - 不马上加重型唯一约束/大索引
  - 不要求一次性把全量旧数据立刻改完

- 可能需要维护窗口，甚至短暂停机
  - 新列一上来就必须 `NOT NULL`
  - 要立刻回填海量历史数据
  - 要加唯一约束、外键、重索引
  - 这列进入高频查询条件或排序
  - 它改变了主事务路径或记录身份定义

所以更稳的方式不是“直接改完”，而是走兼容式演进，也就是常见的 `expand -> migrate -> contract`：

1. 先扩表：加新列，但先保持兼容，通常先允许为空。
2. 再发代码：新代码开始识别/写入这个字段，但仍兼容老数据。
3. 再回填：把旧数据慢慢补齐。
4. 再收紧：确认稳定后，再加 `NOT NULL`、唯一约束、索引，或切换读路径。
5. 最后清理：删掉旧逻辑或旧字段。

**在 shelf 里你该怎么改 framework**

关键先判断：这个新字段是不是“业务语义字段”。

如果是业务语义字段，比如：
- 记录身份的一部分
- 状态机的一部分
- 下游判断必需字段
- 审核/追踪/归因的稳定字段

那应该先改 `framework`。通常要改两类地方：

- 记录对象模块
  - 把这个字段加进对应 `B*` / `P*`
- 持久化接口模块
  - 写清楚这个字段是“写入必填”还是“可选补充”
  - 是否参与查询、更新、唯一定位、事务原子语义

如果这个字段只是实现型字段，比如：
- `created_at`
- `updated_at`
- 内部重试次数
- 分片键
- 纯技术性追踪字段

那往往不必先改 `framework`，更多是改 `exact + code`。

所以不是“加一列就一定改 framework”，而是：

- 改了业务语义，改 `framework`
- 只改实现落法，不一定改 `framework`

**程序会不会和之前差很多**

可能会，但不一定是坏事。

如果这个字段只是“兼容式新增、可选、非热点”，生成程序的差异通常比较小，主要会体现在：
- 新字段映射
- 序列化/反序列化
- 写入逻辑
- 少量校验

如果这个字段牵涉：
- 身份唯一性
- 状态流转
- 查询条件
- 事务边界
- 批处理或索引路径

那差异就会比较大，而且这时风险主要来自数据库演进策略，不是来自 `framework` 本身。

**稳定性和性能会不会变差**

会不会变差，主要看你怎么演进，不是看“有没有 framework”。

风险大的点通常是：

- 强行一次性加必填字段
- 直接全量回填大表
- 新列立刻进高频查询
- 新索引/唯一约束直接压到线上
- 事务里多做了很多事情
- 新代码不兼容旧数据

相反，如果你走兼容式步骤，风险通常可控，性能影响也能拆开观察。

**最实用的一句话**

如果你要改的是“记录的业务语义”，就先改 `framework`，再让 `exact + code` 做兼容式数据库演进。  
如果你要改的只是“数据库实现细节”，就不要硬往 `framework` 里塞，直接在 `exact + code` 里做迁移方案。

如果你愿意，我下一步可以直接把这个问题继续收敛成一个非常具体的操作模板：

“在 shelf 中，已有系统要给数据库记录新增一个字段时，framework、exact、code 分别怎么改。”

## 第 10 轮

### 用户

不太懂，换个问题。如果我的framework已经写好了，我约束好了数据库中的表格的结构，或者说一行记录有哪些字段。但是如何这个时候数据库中没有改表格，需要新建表格。我是不是要针对这个表格存在性进行一个检测？并且针对这个检测+建表的模块编写对应的framework？

### 助手

通常**不用**。

更准确地说：

如果你的 `framework` 已经约束了“这类记录是什么、有哪些字段、哪些字段合法”，而数据库里还没有对应的表，那么：

- “要不要先检测表是否存在”
- “不存在时要不要自动建表”
- “什么时候建表，启动时建还是发布时建”

这些通常是 **`exact + code` 的实现策略问题**，**不必专门再为‘检测表存在性 + 建表’单独写一个 framework 模块**。

核心原因是：

“表存在性检测”本身一般不是业务语义，更多是基础设施就绪策略。  
它回答的是“系统怎么落地运行”，不是“业务结构是什么”。

你可以这样理解：

**你已经有的 framework 负责：**
- 这张表对应什么业务记录
- 一条记录有哪些业务字段
- 哪些字段必填
- 哪些状态合法
- 哪些约束属于业务边界

**你现在缺的，不是新的业务 framework，而是实现层的 schema readiness 机制：**
- 检测表是否存在
- 不存在时执行 migration / create table
- 已存在但版本旧时执行 upgrade
- 保证这套过程幂等、安全、可重复执行

这更像下面三种方案之一：

1. **发布前 migration**
最常见。程序启动前先跑 migration，程序本身不负责建表。  
这时通常不需要额外 framework。

2. **启动时 bootstrap**
程序启动时检测表，不存在就创建。  
这时也通常主要改 `exact + code`，不需要单独 framework。

3. **运行时懒初始化**
第一次真正写入时才检测并建表。  
也还是实现策略，通常不建议先抽成 framework，除非这是你的稳定产品能力。

## 什么时候才值得写成 framework

只有当“存储结构就绪”本身已经变成了**稳定、可复用、需要被上层显式依赖的能力**时，才值得抽成 framework。

比如你明确要做的是：

- 模块具备“自举存储”能力
- 模块启动后可自动检测并初始化自己的持久化结构
- 模块支持结构版本升级和兼容校验
- 上层模块会显式依赖这个“存储就绪能力”

这时才可以考虑写一个类似：

- `存储结构就绪模块`
- `持久化结构初始化模块`
- `Schema 演进编排模块`

但即便如此，framework 里写的也应该是：
- 能力：检测、初始化、升级、幂等
- 边界：只作用于本模块拥有的表；重复执行不破坏已有数据；升级必须保持兼容

而不是：
- 具体 SQL
- 具体 `CREATE TABLE`
- 具体 migration 文件

## 对你这个问题的直接回答

你的场景下，大概率应该这样做：

- **不用**为了“表存在性检测”单独写一个 framework
- 在已有 framework 里继续保持“记录结构/业务字段”的定义
- 在 `exact` 里声明 schema 管理策略
  - 例如：`predeploy_migration` / `bootstrap_on_start`
- 在 `code` 里实现
  - 检测表是否存在
  - 不存在时建表或执行 migration
  - 已存在时检查版本并升级

## 最实用的判断句

如果你描述的是：

- “这条记录该长什么样”

用 `framework`。

如果你描述的是：

- “数据库里还没有这张表时，系统该怎么把它弄出来”

通常用 `exact + code`，不是单独再补一个 framework。

“已有记录 framework 不变时，`exact` 和 `code` 应该怎么设计表存在性检测 + 自动建表机制。”

## 第 11 轮

### 用户

实现层的 schema readiness 机制是怎么来的呢？以及你说的exact是怎么来的呢？

### 助手

这两个东西都**不是凭空来的**，也**不是 framework 自动长出来的**。更准确地说：

**`schema readiness` 机制来自“项目实现决策”**  
**`exact` 来自“把 framework 允许的空间，实例化成当前项目的精确配置”**

可以拆开看。

**1. `schema readiness` 是怎么来的**

它本质上是在回答一个工程问题：

“程序启动或运行时，怎么确认数据库 schema 已经处于可用状态？”

这通常由项目在 `exact + code` 层共同决定，而不是由 framework 单独推出。

它常见有几种来源：

- 发布策略决定：发布前先跑 migration
- 启动策略决定：程序启动时 `ensure_schema()`
- 运维策略决定：单独有 `init-db` / `upgrade-db` 命令
- 兼容策略决定：只检查版本，不自动改表

也就是说，`schema readiness` 机制通常来自你对这个项目的实现选择，比如：

- 我们要不要自动建表
- 要不要自动迁移
- 不兼容时是拒绝启动还是尝试升级
- 是启动前处理，还是运行时懒处理

这些都是**项目级工程策略**，不是 framework 本体。

**2. `exact` 是怎么来的**

`shelf` 里，`exact` 不是“自动推导出来的唯一答案”，而是：

**在 framework 已经定义好边界后，当前项目对这些边界的具体落点。**

也就是：

- `framework` 说：必须有这样一种记录，必须满足这些字段和约束
- `exact` 说：这次项目里我们具体用哪个库、哪张表、哪些列映射、哪种 schema 管理策略
- `code` 去实现

所以 `exact` 的来源通常是：

- 人先定义 `framework`
- 人或 AI 在 `projects/<project_id>/project.toml` 里补 `exact`
- 编译/物化流程把 `exact` 导出给 `CodeModule`
- 代码只消费 `exact_export`

换句话说，`exact` 是**项目实例化配置**，不是 framework 的全文翻译。

**3. 三层之间到底是什么关系**

可以用一句最直接的话理解：

- `framework` 定义“必须满足什么”
- `exact` 定义“这次项目具体怎么落”
- `code` 定义“实际怎么做出来”

比如你有一个“评分记录”：

`framework` 会写：
- 记录必须包含 `doc_id / chunk_id / tag_id / score`
- 一条记录由哪些业务字段唯一定位
- 哪些状态流转合法

`exact` 会写：
- 用 `postgres`
- 表名叫 `chunk_tag_scores`
- 业务身份映射到唯一键 `(doc_id, chunk_id, tag_id)`
- schema 策略是 `bootstrap_on_start` 或 `predeploy_migration`

`code` 会写：
- `CREATE TABLE`
- migration
- `ensure_schema()`
- repository / transaction

**4. 所以你担心的点，正确答案是**

如果 framework 已经写好了“记录结构”，但数据库还没有这张表，那么：

- framework 负责说明“应该存在什么结构”
- exact 负责说明“本项目怎么把这结构落到数据库”
- code 负责“检测是否存在，不存在时怎么创建或升级”

所以 `schema readiness` 机制不是从 framework 自动生成的，而是**由项目在 exact 中声明、由 code 中实现**。

**5. 一个很短的例子**

你可以把 `exact` 想成这种东西：

```toml
[exact.chunk_tagging.persistence]
db_kind = "postgres"
table_name = "chunk_tag_scores"
schema_strategy = "bootstrap_on_start"
schema_version = 3
auto_create_table = true
auto_migrate_compatible = false
```

然后 `code` 根据这些值做：

- 启动时检查表是否存在
- 不存在则建表
- 已存在则检查版本
- 版本不兼容则拒绝启动或提示 migration

而 framework 不需要写：

- `CREATE TABLE ...`
- `ALTER TABLE ...`
- 启动时 SQL 检查顺序

**6. 最后一句收拢**

所以：

- `schema readiness` 来自项目的实现与部署策略
- `exact` 来自 framework 约束下的项目实例化选择
- 它们都不是 framework 自动完整推出的，而是 framework 约束下的人机共同落地结果

如果你愿意，我可以下一步直接给你画一个非常具体的链路图：

`framework 记录定义 -> project.toml exact -> code ensure_schema/migration -> 数据库真实表`

答案是：

**不能保证。**

而且按照当前标准，严一点说：

- 如果 framework 没有给出相关边界或允许的 projection
- 那 `exact` 不能凭空发明一个“官方受支持”的建表策略字段
- AI 也不应该假装这个需求已经被 framework 覆盖了

这不是 AI 的小失误，而是**当前框架表达能力的边界问题**。

也就是说，你这个担心是成立的：

如果 framework 只描述了“记录有哪些字段”，  
但没有描述“持久化结构是否需要自举/检测/迁移”这种稳定要求，  
那 AI 生成的 `project.toml` 很可能不会自动把这件事补全。

**所以更准确的说法应该改成**

不是：

“framework 写完了，AI 再补 exact，就自然能把建表问题处理掉。”

而是：

“只有当 framework 已经给了足够的合法展开空间，AI 才能在 exact 里可靠地补项目级实现策略。”

**这会导出一个更严格的结论**

如果你希望“表不存在时检测并建表”成为一个**可被稳定推导、可被 AI 一致处理、可被校验**的能力，那么它不能只停留在我口头说的“工程实现建议”，而应该满足下面两者之一：

1. 上游 framework 里已经有对应的稳定能力/边界  
比如存在“持久化结构就绪”“schema bootstrap”“存储初始化策略”这类模块或参数空间。

2. 仓库级标准明确规定某类 exact 字段是所有持久化模块的公共保底槽位  
比如统一规定所有持久化模块都允许声明 `schema_strategy`、`bootstrap_mode`、`migration_policy`。

如果这两者都没有，那这件事就只是“实现层临时约定”，不是 framework 驱动的正式能力。

**所以你现在问到的，其实是一个架构缺口**

你问得非常准：

这意味着：

- 它不是完全不能做
- 但它现在还不是一个被框架标准严密承接的能力
- 如果希望 AI 稳定处理，就需要先把这部分能力上提到 framework 或仓库级规范
