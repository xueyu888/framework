# Chunk Tagging 整体思路

首先明确当前最小可用版本要落的是什么东西：

1. `documents` 表：存放所有文档，包含 `doc_id`、文档名、文档类型、文档领域等文档级信息。
2. `chunks` 表：存放所有文档分块结果，包含 `doc_id`、`chunk_id`、文本内容等字段；其中 `chunk_id` 只在当前文档内部唯一，因此使用 `(doc_id, chunk_id)` 作为联合主键。
3. `tags` 表：存放所有标签定义，包含 `tag_id`、标签名、适用文档类型、适用文档领域，以及该标签对应的规则配置。
4. `chunk_tags` 表：存放文本块与标签之间的评分映射关系，包含 `doc_id`、`chunk_id`、`tag_id`、匹配评分、审核状态等字段。

## 一、四张核心表的职责

### 1. `documents`

这张表负责回答：

- 当前系统里有哪些文档
- 每个文档的名字是什么
- 每个文档属于什么类型
- 每个文档属于什么领域

### 2. `chunks`

这张表负责回答：

- 某个文档被拆成了哪些文本块
- 每个文本块在该文档中的编号是什么
- 每个文本块的内容是什么

这里要注意：

- `chunk_id` 不是全局唯一
- `chunk_id` 只表示“该文本块在当前文档中的编号”
- 因此 `chunks` 表应使用 `(doc_id, chunk_id)` 作为联合主键

### 3. `tags`

这张表负责回答：

- 系统里有哪些标签
- 每个标签适用于哪些文档类型
- 每个标签适用于哪些文档领域
- 每个标签有哪些匹配规则配置

规则配置在当前版本中至少应支持：

- `match_type`：精确匹配或正则匹配
- `pattern`：关键词或正则表达式
- `scope`：标题或正文
- `weight`：规则权重

### 4. `chunk_tags`

这张表负责回答：

- 哪个文本块命中了哪个标签
- 该命中的匹配评分是多少
- 当前是否需要人工审核
- 当前审核结果是什么

因为 `chunks` 使用 `(doc_id, chunk_id)` 作为联合主键，所以 `chunk_tags` 里也必须带上：

- `doc_id`
- `chunk_id`

这样才能稳定指向具体文本块。

## 二、标注流程

在对文本块进行标注时，整体流程如下：

1. 先确定本次要标注的是哪一个文档。
2. 从 `documents` 表中获取该文档对应的文档类型与文档领域。
3. 从 `chunks` 表中筛选出该文档对应的全部文本块。
4. 从 `tags` 表中筛选出适用于该文档类型与文档领域的标签定义。
5. 将每个 `chunk` 针对每个候选 `tag` 依次执行匹配评分。
6. 将评分结果写入 `chunk_tags` 表。

## 三、评分结果的落表策略

当前版本建议将所有评分结果都保留在 `chunk_tags` 表中，而不是直接删除低分结果。

推荐按以下方式处理：

- 评分低于 `0.5`：标记为低置信度，不进入直接通过结果
- 评分处于 `0.5~0.8`：标记为待人工审核
- 评分高于 `0.8`：标记为高置信度，可直接通过

这样做的好处是：

- 便于审核回看
- 便于后续调整阈值
- 便于回溯算法效果
- 不会因为“直接删除低分记录”而丢失证据

## 四、当前版本的核心结论

当前版本的整体结构应统一为：

- `documents`：文档主表
- `chunks`：文本块主表，联合主键为 `(doc_id, chunk_id)`
- `tags`：标签定义表，内含规则配置
- `chunk_tags`：文本块与标签的评分映射关系表

在这个结构下：

- 文档级信息放在 `documents`
- 分块结果放在 `chunks`
- 标签和规则配置放在 `tags`
- 实际评分结果与审核状态放在 `chunk_tags`

这套结构既能支撑当前最小版本，也给后面扩展文档类型、领域、规则治理和审核流程留出了空间。
