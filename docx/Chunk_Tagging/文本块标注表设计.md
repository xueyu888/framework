# 文本块标注表设计

这份文档整理的是当前这一版最小可用设计思路。

当前版本的目标不是只做“文本块和标签的简单映射”，而是先把以下四类核心对象的框架搭起来：

- 文档
- 文本块
- 标签
- 文本块与标签的评分映射关系

同时保留以下两个维度：

- 文档类型
- 文档领域

---

## 一、整体思路

当前版本统一采用 4 张核心表：

- `documents`：文档表
- `text_chunks`：文本块表
- `tags`：标签定义表
- `chunk_tags`：文本块与标签的评分映射表

这样设计的原因是：

1. 文档级信息和文本块级信息需要分开管理
2. 文本块编号只在单文档内唯一，因此需要显式保留 `doc_id`
3. 标签不仅有名字，还要带适用范围和规则配置
4. 文本块与标签之间是带评分的多对多关系

---

## 二、表结构

## 1. `documents`

作用：存储所有文档的文档级信息。

| 字段名 | 含义 | 说明 |
|---|---|---|
| `doc_id` | 文档主键 | 唯一标识一个文档 |
| `document_name` | 文档名 | 文档的人类可读名称 |
| `document_type` | 文档类型 | 用于筛选适用标签 |
| `domain` | 文档领域 | 用于筛选适用标签 |
| `created_at` | 创建时间 | 可选 |

主键：

- `PRIMARY KEY (doc_id)`

说明：

- `documents` 承接文档级身份信息
- `document_type` 和 `domain` 先保留在当前版本中，作为后续标签筛选的输入

---

## 2. `text_chunks`

作用：存储文档拆分后的文本块。

这里的 `chunk_id` 只在当前文档内部唯一，因此采用联合主键。

| 字段名 | 含义 | 说明 |
|---|---|---|
| `doc_id` | 文档 ID | 外键，指向 `documents.doc_id` |
| `chunk_id` | 文本块编号 | 只在当前文档内部唯一 |
| `content` | 文本块内容 | 需要标注的具体文本 |
| `chunk_type` | 文本块类型 | 可选，例如标题块、正文块 |
| `created_at` | 创建时间 | 可选 |

主键：

- `PRIMARY KEY (doc_id, chunk_id)`

外键：

- `FOREIGN KEY (doc_id) REFERENCES documents(doc_id)`

说明：

- 这里不使用全局单列 `id`
- `chunk_id` 只表示“当前文本块在当前文档中的编号”
- `doc_id + chunk_id` 组合后才能稳定定位一个文本块

---

## 3. `tags`

作用：存储标签定义及其规则配置。

| 字段名 | 含义 | 说明 |
|---|---|---|
| `tag_id` | 标签主键 | 唯一标识一个标签 |
| `tag_name` | 标签名称 | 例如 `人物`、`时间`、`风险点` |
| `document_type` | 适用文档类型 | 该标签适用于哪些文档类型 |
| `domain` | 适用领域 | 该标签适用于哪些文档领域 |
| `rule_config` | 规则配置 | 存储该标签的一组匹配规则 |
| `description` | 标签说明 | 可选，用于解释标签含义 |

主键：

- `PRIMARY KEY (tag_id)`

说明：

- 标签本身是独立对象
- 当前版本中，规则配置先直接内嵌在标签定义里
- `rule_config` 至少应支持一组规则项，每条规则项至少包含：
  - `match_type`
  - `pattern`
  - `scope`
  - `weight`

---

## 4. `chunk_tags`

作用：存储文本块与标签之间的实际评分结果与审核状态。

因为 `text_chunks` 使用 `(doc_id, chunk_id)` 作为联合主键，所以映射表必须同时带上 `doc_id` 和 `chunk_id`。

| 字段名 | 含义 | 说明 |
|---|---|---|
| `doc_id` | 文档 ID | 与 `chunk_id` 一起指向 `text_chunks` |
| `chunk_id` | 文本块编号 | 与 `doc_id` 一起定位文本块 |
| `tag_id` | 标签 ID | 外键，指向 `tags.tag_id` |
| `confidence` | 匹配评分 | `0~1` 范围内的数值 |
| `review_status` | 审核状态 | 例如待审核、已通过、已驳回 |
| `review_note` | 审核备注 | 可选 |
| `created_at` | 创建时间 | 可选 |

主键：

- `PRIMARY KEY (doc_id, chunk_id, tag_id)`

外键：

- `FOREIGN KEY (doc_id, chunk_id) REFERENCES text_chunks(doc_id, chunk_id)`
- `FOREIGN KEY (tag_id) REFERENCES tags(tag_id)`

说明：

- 一个文本块可以对应多个标签
- 一个标签也可以对应多个文本块
- 这是一个标准的多对多关系表
- 与普通多对多表不同的是，这里还需要承载评分与审核状态

---

## 三、表之间的关系

当前版本的关系可以先理解成：

1. `documents` 和 `text_chunks`
   一对多
   一个文档可以拆出多个文本块

2. `documents` 和 `tags`
   不是直接外键关系
   而是通过 `document_type` 与 `domain` 做适配筛选

3. `text_chunks` 和 `tags`
   多对多
   通过 `chunk_tags` 建立关系

---

## 四、推荐的最小流程

当前版本可以先按下面的流程使用：

1. 将文档写入 `documents`
2. 对文档执行分块，并写入 `text_chunks`
3. 维护一份标签集合及规则配置，写入 `tags`
4. 根据文档类型和领域筛选候选标签
5. 对该文档的每个文本块针对每个候选标签执行评分
6. 将评分结果写入 `chunk_tags`
7. 按评分阈值决定自动通过或人工审核

---

## 五、一个简化理解

可以把这 4 张表这样记：

- `documents`：文档主表
- `text_chunks`：文档分块结果表
- `tags`：标签定义与规则配置表
- `chunk_tags`：实际评分与审核结果表

---

## 六、当前设计的核心结论

按当前已经确定的方向，这套设计应统一为：

1. 保留文档表 `documents`
2. 文本块表 `text_chunks` 使用 `(doc_id, chunk_id)` 联合主键
3. 标签表 `tags` 保留文档类型、领域和规则配置
4. 文本块与标签通过 `chunk_tags` 做多对多评分映射
5. `chunk_tags` 需要显式带上 `doc_id`
6. 标注结果不直接删除低分记录，而是保留评分并进入不同审核流转

这套设计适合作为当前版本的统一落点，后面还可以继续扩展：

- 更细的规则治理
- 标签版本
- 审核人
- 审核时间
- 标注版本
- 人工修订记录

---

## 七、工程上的常见做法

在真实项目里，这种设计非常常见：

- 文档表承接文档级信息
- 分块表承接文档内容拆分结果
- 标签表承接标签与规则配置
- 中间表承接实际评分与审核状态

只要当前核心问题是：

- 文档怎么存
- 文本块怎么存
- 标签和规则怎么存
- 文本块与标签的评分关系怎么存

那这 4 张表就已经足够支撑第一版。
