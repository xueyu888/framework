# 触发过程结构:TriggerProcess

@framework

触发过程结构用于让一个动作能够被明确发起、在过程中维持，并在完成、取消或中断条件下结束，同时对重复触发形成限制。

## 1. 能力声明（Capability Statement）

- `C1` 动作发起能力：让对象能够在明确条件下发起一个动作。
- `C2` 过程维持与结束能力：让已发起动作能够进入过程阶段，并在满足条件时结束。
- `C3` 重复触发限制能力：让对象对重入触发、并发触发或连续重复触发形成明确限制。

### 非职责声明（Non-Responsibility Statement）

- `N1` 不负责结果反馈：不负责成功失败反馈的内容、位置与持续时机。
- `N2` 不负责校验判定：不负责校验规则定义和校验执行结果。
- `N3` 不负责值与字段：不负责值绑定、聚合与结构化组织。

## 2. 边界定义（Boundary / Parameter 参数）

- `P1` 发起条件边界：触发过程结构必须允许定义明确发起条件、触发方式与作用对象。来源：`C1`。
- `P2` 过程边界：触发过程结构必须允许定义维持、完成、取消或中断处理方式。来源：`C2`。
- `P3` 重复限制边界：触发过程结构必须允许定义重复触发、并发触发或重入触发的限制方式。来源：`C3`。

## 3. 最小结构基（Minimal Structural Bases）

- `B1` 发起结构：用于确定动作如何被发起。来源：`C1 + P1`。
- `B2` 过程结构：用于确定动作在过程中如何维持与结束。来源：`C2 + P2`。
- `B3` 重复限制结构：用于确定重复触发与并发触发的限制方式。来源：`C3 + P3`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基本触发过程组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先为动作定义明确发起条件与触发方式，使动作能够被稳定启动。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 参数绑定：`P1`。

- `R2` 受控触发过程组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在动作能够被发起的前提下，引入过程结构，使动作具备明确维持与结束机制。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 参数绑定：`P1 + P2`。

- `R3` 完整触发过程组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：在发起能力与过程机制成立的前提下，引入重复限制结构，使动作对重入与并发形成明确门禁。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 参数绑定：`P1 + P2 + P3`。

- `R4` 无效悬空组合
  - `R4.1` 参与基：`B2` 或 `B3`。
  - `R4.2` 组合方式：只有过程机制或重复限制，而没有动作发起结构。
  - `R4.3` 失效结论：如果没有动作发起入口，则过程机制与重复限制都没有约束对象，不能构成合法触发过程结构。
  - `R4.4` 参数绑定：缺失 `P1`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力或失效结论、参数绑定。
- `V3` 结构闭环性：`R1 ~ R3` 输出能力并集必须覆盖 `C1 ~ C3`。
- `V4` 参数符合性：所有有效组合必须满足绑定的参数约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V3` 或 `V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* + 参数绑定` 或 `R* -> N* + 参数绑定` 的通过或失败结论。
