# 事务子系统:TransactionSubsystem

@framework

## 0. 目标 (Goal)

- 事务子系统用于描述数据库内部承载事务机制的核心结构集合。
- 该子系统由事务入口、事务状态控制、并发协调、日志恢复、存储执行与后台维护等结构共同组成。
- 该文档重点描述事务相关内部结构的分层与组合关系，不直接展开单个事务实例的运行时状态流转细节。

## 1. 最小结构基（Minimal Structural Bases）

- `B1` 事务接口层：数据库事务系统对外暴露事务控制入口的接口层结构。
- `B2` 事务管理层：数据库事务系统内部用于承载事务状态与生命周期管理的控制层结构。
- `B3` 并发控制层：数据库事务系统内部用于承载并发协调与隔离规则的控制层结构。
- `B4` 日志恢复层：数据库事务系统内部用于承载事务日志记录与恢复支撑的持久化层结构。
- `B5` 存储执行层：数据库事务系统内部用于承载事务读写落地执行的存储层结构。
- `B6` 后台维护层：数据库事务系统内部用于承载后台刷写、清理与维护任务的支撑层结构。

## 2. 基排列组合（Base Arrangement / Combination）

- `R1` `事务执行链`：由 `{B1, B2, B3, B5}` 共同消费事务控制输入与 SQL 操作输入，形成一次事务内的执行过程。
- `R2` `事务提交链`：由 `{B1, B2, B4, B5}` 在提交时共同确认事务结果可持久化，并导出 `C1`。
- `R3` `事务回滚链`：由 `{B1, B2, B3, B5}` 在失败或显式撤销时共同终止本次事务影响，并导出 `C2`。
- `R4` `事务恢复链`：由 `{B4, B5, B6}` 在异常中断后协同恢复已提交事务状态，并导出 `C3`。

## 3. 边界定义（Boundary）

### 3.1 接口定义（IO / Ports）

- `TX_BEGIN_IN`：运行时输入接口。载荷类型为事务开始指令；基数为 `0..1`；语义为显式开启一个事务上下文。
- `SQL_OP_IN`：运行时输入接口。载荷类型为一条或多条 SQL 操作请求；基数为 `0..n`；语义为属于当前事务的数据库读写或结构操作。
- `TX_COMMIT_IN`：运行时输入接口。载荷类型为事务提交指令；基数为 `0..1`；语义为请求数据库确认当前事务全部生效。
- `TX_ROLLBACK_IN`：运行时输入接口。载荷类型为事务回滚指令；基数为 `0..1`；语义为请求数据库撤销当前事务未完成或不应保留的修改。
- `TX_RESULT_OUT`：运行时输出接口。载荷类型为事务执行结果；基数为 `1`；语义为事务成功、失败或回滚后的可观测结果。

### 3.2 参数边界（Parameter Constraints）

- `P1` 生命周期约束：一条 SQL 操作若被视为事务内操作，则必须能够归属到某个明确事务上下文。
- `P2` 状态约束：任一事务在任一时刻必须处于可判定状态，例如活动中、已提交或已回滚。
- `P3` 提交约束：事务对外可见前，必须满足日志已按实现要求记录完成。
- `P4` 隔离约束：并发事务之间必须遵循某种明确定义的隔离规则，不能任意暴露中间状态。
- `P5` 回滚约束：未提交事务产生的效果必须能够被撤销、屏蔽或判定为对外不可见。
- `P6` 恢复约束：数据库异常退出后，系统必须能够区分“已提交结果”和“未提交结果”。

## 4. 能力声明（Capability Statement）

- `C1` 原子提交能力：能够把一组操作作为一个整体提交，并在提交成功后把该结果视为正式生效。
- `C2` 回滚撤销能力：能够在失败、异常或主动取消时阻止事务的中间结果污染最终数据状态。
- `C3` 崩溃恢复能力：能够在系统中断后恢复已提交事务，并避免未提交事务伪装成已完成结果。
- `C4` 并发隔离能力：能够在多个事务并行执行时，通过锁、版本或等价机制控制可见性与冲突。
- `N1` 非职责声明：不直接负责业务规则本身是否正确，业务一致性仍需应用层和数据库约束共同保证。
- `N2` 非职责声明：不默认等价于分布式事务；跨服务、跨数据库的一致性问题不属于该模块的最小职责边界。
