# 事务接口层:TransactionInterfaceLayer

@framework

## 0. 目标 (Goal)

- 事务接口层负责接收客户端发起的事务控制请求，并把这些请求转化为数据库内部可处理的事务动作。
- 该层的核心职责是识别事务边界、绑定当前连接上下文、校验当前事务状态，并把合法请求分发给后续事务模块。
- 该层不直接实现锁、MVCC、WAL、崩溃恢复等事务底层机制。

## 1. 最小结构基（Minimal Structural Bases）

- `B1` 输入接收器：事务接口层内部用于承载事务控制输入接入的入口结构件。
- `B2` 边界识别器：事务接口层内部用于承载事务边界判定的识别结构件。
- `B3` 会话绑定器：事务接口层内部用于承载事务请求与连接上下文绑定关系的关联结构件。
- `B4` 状态检查器：事务接口层内部用于承载事务状态合法性判定的校验结构件。
- `B5` 指令分发器：事务接口层内部用于承载事务动作内部路由关系的分发结构件。
- `B6` 结果返回器：事务接口层内部用于承载事务控制结果对外反馈的返回结构件。

## 2. 基排列组合（Base Arrangement / Combination）

- `R1` `事务控制请求接入链`：由 `{B1, B2, B3}` 共同把外部事务控制请求转化为带上下文的内部事务动作，导出 `C1`。
- `R2` `事务合法性校验链`：由 `{B2, B3, B4}` 共同判断当前事务边界动作是否可执行，导出 `C2`。
- `R3` `事务动作分发链`：由 `{B4, B5, B6}` 共同完成事务动作的内部路由、执行反馈与结果返回，导出 `C3`。

## 3. 边界定义（Boundary）

### 3.1 接口定义（IO / Ports）

- `tx_control_in`：运行时输入接口。载荷类型为 SQL 事务控制语句或驱动事务控制调用；基数为 `0..n`；语义为客户端发起的 `BEGIN`、`COMMIT`、`ROLLBACK`、自动提交切换等事务控制请求。
- `sql_op_in`：运行时输入接口。载荷类型为单条 SQL 操作请求；基数为 `0..n`；语义为需要被判定是否归属当前事务上下文的普通 SQL 语句。
- `session_ctx_in`：运行时输入接口。载荷类型为结构化对象；基数为 `1`；语义为当前连接、会话或后端执行上下文信息。
- `tx_action_out`：运行时输出接口。载荷类型为结构化对象；基数为 `1`；语义为已完成边界识别和状态校验的内部事务动作。
- `tx_feedback_out`：运行时输出接口。载荷类型为结构化对象或错误文本；基数为 `1`；语义为返回给客户端的事务控制结果、状态变化或错误信息。

### 3.2 参数边界（Parameter Constraints）

- `P1` 控制指令可识别性：约束 `tx_control_in` 必须能够被明确判定为事务开始、提交、回滚、自动提交切换或无效请求之一。
- `P2` 会话唯一绑定性：约束一次事务控制动作只能归属到一个明确的 `session_ctx_in`，不得跨连接漂移。
- `P3` 边界一致性：约束 `B2` 识别出的事务边界必须与当前会话状态一致，例如未开启事务时不得提交已不存在的事务。
- `P4` 状态可判定性：约束 `B4` 对每个事务请求都必须能给出明确状态判断，不能让未定义状态流入下游。
- `P5` 分发完备性：约束所有合法事务动作都必须被 `B5` 分发到明确下游，所有非法动作都必须被 `B6` 明确返回错误或拒绝结果。

## 4. 能力声明（Capability Statement）

- `C1` 事务入口接收能力：能够稳定接收来自 SQL 文本与驱动 API 的事务控制请求，并把它们纳入统一入口。
- `C2` 事务边界判定能力：能够识别显式事务边界与隐式事务切换，并把普通 SQL 归属到正确事务上下文。
- `C3` 事务动作路由能力：能够在会话绑定和状态检查完成后，把事务控制动作正确分发到后续事务模块并返回执行反馈。
- `N1` 非职责声明：不负责生成事务 ID，也不负责维护完整事务生命周期状态机。
- `N2` 非职责声明：不负责锁冲突处理、MVCC 可见性判定、WAL 写入或崩溃恢复。
- `N3` 非职责声明：不负责业务 SQL 本身的语义优化、执行计划选择或数据页读写实现。
