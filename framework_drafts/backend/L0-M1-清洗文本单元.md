# 清洗文本单元:CleanedKnowledgeTextUnit

@framework

## 0. 目标 (Goal)
- 清洗文本单元定义“已完成预处理并可稳定读取”的文本对象。
- 该单元聚焦文本内容、顺序位置与文档归属三个核心结构，不承担检索召回、重排打分与回答生成职责。

## 1. 最小结构基（Minimal Structural Bases）

- `B1` 文本块：清洗后的文本内容。
- `B2` 序号（`seq`）：文本块在所属文档中的顺序位置。
- `B3` 文档归属索引（`doc_id`）：文本块所属文档的唯一归属标识。

## 2. 基排列组合（Base Arrangement / Combination）

- `R1` `唯一定位组合`：由 `{B2, B3}` 形成唯一定位键，导出 `C1`。
- `R2` `可读文本组合`：由 `{B1, B2, B3}` 形成可读取文本单元，导出 `C2`。
- `R3` `归属追溯组合`：由 `{B3}` 形成文档归属追溯能力，导出 `C3`。
- `R4` `排列等价规则`：`<B1, B2, B3>` 在不改变字段语义时的排列顺序不影响 `C1 ~ C3` 的可导出性。

## 3. 边界定义（Boundary）

### 3.1 接口定义（IO / Ports）

- `cleaned_text_in`：运行时输入接口，接收单个清洗后文本块。
- `seq_in`：运行时输入接口，接收单个文本序号。
- `doc_id_in`：运行时输入接口，接收单个文档归属索引。
- `cleaned_text_unit_out`：运行时输出接口，导出 `{doc_id, seq, cleaned_text}` 结构化文本单元。
- `doc_trace_out`：运行时输出接口，导出用于归属追溯的 `doc_id`。

### 3.2 参数边界（Parameter Constraints）

- `P1` 序号值域：约束 `seq_in` 与 `B2` 的值域为无符号 64 位整型。
- `P2` 文档归属索引值域：约束 `doc_id_in` 与 `B3` 的值域为无符号 64 位整型。
- `P3` 文本长度：约束 `cleaned_text_in` 与 `B1` 的合法长度范围。
- `P4` 文本语言：约束 `cleaned_text_in` 与 `B1` 的允许语言集合。
- `P5` 联合唯一性：约束 `(doc_id, seq)` 在同一数据域内必须唯一。

## 4. 能力声明（Capability Statement）

- `C1` 单元定位能力：支持通过 `(doc_id, seq)` 唯一定位文本单元。
- `C2` 原文读取能力：支持读取文本单元的文本块内容。
- `C3` 归属追溯能力：支持将文本单元追溯到其所属文档索引。
- `N1` 非职责声明：不负责原始文档清洗策略、切块算法、纠错与去噪实现。
- `N2` 非职责声明：不负责自然语言查询理解、语义检索、向量召回、排序与重排。
- `N3` 非职责声明：不负责版本链、修改历史、来源血缘与页码偏移等复杂追溯。
- `N4` 非职责声明：不负责恢复清洗前的 PDF、HTML、表格、图片或版式结构。
