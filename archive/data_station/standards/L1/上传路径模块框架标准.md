# 上传模块框架标准（L1）

## 0. 规范关系
本文件为上传模块标准，参考：
- `archive/data_station/standards/L0/数据站框架标准.md`

本文件位于 `L1`，承接 `L0` 对 `M1` 的定义，对上传模块进行框架化约束，并向实现层传导。

## 1. 目标（Goal）
- 构建一个可稳定接收器件手册上传请求的模块，形成从 UI/API 发起上传到服务器入库、状态可见、可追溯的闭环。
- 满足客户公司内部全员可见、工程师可上传的业务要求。

## 2. 边界定义（Boundary）
说明：本节将上传模块视为一个整体黑盒，不拆分内部实现细节。
- `UploadBoundary.event_scope`：事件边界。仅覆盖“用户发起上传”到“系统返回上传结果”的单次上传事件，不包含后续处理、审核、检索流程。
- `UploadBoundary.domain_scope`：业务对象边界。纳入文件流、上传元信息、上传任务记录；排除处理/审核结论等非上传事件对象。
- `UploadBoundary.lifecycle_scope`：生命周期边界。仅覆盖“发起上传 -> 接收传输 -> 校验 -> 接收成功/拒绝”阶段。
- `UploadBoundary.path_scope`：路径边界。仅允许定义内的上传 UI 路径与 API 路由触发上传事件；越界路径不进入上传处理链路。
- `UploadBoundary.input_contract`：输入边界。输入限定为文件流与必填元信息（如文档类型、来源、上传人）；仅触发状态变化或副作用的输入计为有效输入。
- `UploadBoundary.output_contract`：输出边界。输出限定为上传结果响应与可观察变化（界面反馈、状态变化、副作用写入）。
- `UploadBoundary.state_contract`：状态边界。上传事件状态仅允许 `upload_pending -> receiving -> validating -> accepted/rejected` 合法流转。
- `UploadBoundary.authority_contract`：权限边界。上传结果在客户公司内部可见；上传操作仅对工程师角色开放。
- `UploadBoundary.quality_contract`：质量边界。定义文件格式、大小上限、必填元信息与完整性校验规则；不达标直接拒绝并给出原因。
- `UploadBoundary.traceability_contract`：追溯边界。上传关键操作必须记录操作者、时间、来源、对象标识、判定结果与原因码。

去重约束：
- 事件表示本次上传是否闭环完成，生命周期表示阶段进度，状态表示当前节点状态，质量表示输入与结果是否达标。

权限约束：
- 鉴权调用统一使用 `M5` 提供的接口：`authorize(subject, action, resource, context)`。
- 上传模块映射示例：`action in {upload.create, upload.retry, upload.cancel, upload.view}`，`resource in {upload_task, manual_document}`。
- 上传权限判定由统一接口返回，模块内不维护多套权限规则。

## 3. 模块（最小可行基，Module）
- `M1_1` 路径门控子模块：校验上传 UI/API 路径是否命中 `path_scope`，越界路径直接拒绝。
- `M1_2` 上传入口子模块：接收 UI 上传请求，完成参数解析、统一鉴权接口调用、上传任务初始化。
- `M1_3` 文件接收子模块：执行文件传输与服务器落盘，支持上传中断失败反馈。
- `M1_4` 元信息校验子模块：校验必填元信息、文件格式与大小约束，执行标准化归一。
- `M1_5` 状态提交子模块：维护上传任务状态机，完成幂等控制与最终状态提交。
- `M1_6` 结果出站子模块：对外返回上传结果，触发 `M3` 提供的统一追溯记录接口，并向 `M3/M4` 输出可消费事件。

说明：
- `M1_1-M1_6` 共同构成 `M1` 上传模块的最小可行基，不再按功能继续拆分。
- `authority/quality/traceability` 为横切约束，作用于 `M1_1-M1_6` 全链路。
- 追溯能力由 `M3` 同模块托管，`M1_6` 仅负责触发追溯记录，不承担追溯规则与存储实现。

## 4. 组合原则（Combination Principles）
- 子模块组合按 `R1 -> R2 -> R3` 编排。
- `R1` 路径与建单：`M1_1` 先完成路径门控，`M1_2` 再完成请求校验与建单，状态进入 `upload_pending`。
- `R2` 接收校验：`M1_3` 完成文件接收，`M1_4` 完成元信息与质量校验；任一失败即写入失败状态并返回可观察错误。
- `R3` 提交出站：`M1_5` 提交最终状态并保证幂等，`M1_6` 输出结果并触发统一追溯记录；成功后文档状态为 `已接入/待处理`，在 `M4` 可见可达。

## 5. 验证（Verification）
判定公式：
- `PASS = BoundaryAssertions(all) AND CombinedOutcomeChecks(all) AND GoalChecks(all)`

组合验证说明：
- 组合过程遵循第 4 节组合原则，验证阶段仅对组合形成的结果进行判定。

通过条件：
- `V-B0` 路径边界通过：仅定义内 UI 路径/API 路由可触发上传处理，越界路径被拒绝或路由不到上传模块。
- `V-B1` 输入边界通过：仅将触发状态变化或副作用的输入计入功能输入，且输入符合“文件流 + 必填元信息”约束。
- `V-B2` 输出边界通过：每个关键操作都形成上传结果响应与可观察变化（界面/状态/副作用）。
- `V-B3` 状态边界通过：仅发生上传状态机定义内流转，非法流转全部被拦截。
- `V-B4` 权限边界通过：上传模块所有鉴权决策均经 `M5` 提供的统一 `authorize(subject, action, resource, context)` 接口执行；`action/resource` 映射一致且可审计。
- `V-B5` 质量边界通过：上传质量规则（格式/大小/必填/完整性）全部达标。
- `V-B6` 追溯边界通过：上传事件最小追溯字段经 `M3` 统一追溯接口记录且完整可查。
- `V-C1` 组合结果完整：上传请求在通过路径门控后可创建唯一任务并进入 `upload_pending`，并完成接收、校验、提交出站。
- `V-C2` 组合结果一致：成功路径状态为 `已接入/待处理` 且在 `M4` 可见；失败路径返回明确可观察反馈。
- `V-C3` 组合结果稳态：重复提交同一请求时，状态、副作用与追溯记录保持一致。
- `V-G1` 目标闭环通过：样本文档可走通“UI上传 -> 接收 -> 校验 -> 入库 -> 可见”。
- `V-G2` 目标稳态通过：重复提交同一上传请求时，状态与副作用保持幂等一致。

失败原因至少包含：
- 任一边界断言失败（`V-B0` 至 `V-B6`）。
- 任一组合结果约束失败（`V-C1` 至 `V-C3`）。
- 目标闭环或幂等一致性失败（`V-G1` 或 `V-G2`）。

证据记录要求：
- 采用验收矩阵记录 `case_id`、输入、期望界面变化、期望状态变化、期望副作用、实际结果、证据位置。
