# 数据站框架标准（L0）

## 0. 规范关系
本文件为数据站平台标准，必须遵循：
- `standards/L0/规范总纲与树形结构.md`
- `standards/L1/框架设计核心标准.md`
- `standards/L1/可追溯性标准.md`
- `standards/L1/可删减性标准.md`
本文件位于 `L0`，定义数据站平台的目标、边界、模块与组合原则，并向实现层传导。


## 1. 目标（Goal）
- 构建一个面向器件手册的数据站平台，形成从上传、结构化处理、人工审核到可检索复用的闭环，确保过程可追溯、结果可审计、数据可持续迭代。

## 2. 边界定义（Boundary）
说明：本节边界定义将数据站平台视为一个整体黑盒，不拆分内部模块边界。

- `GoalBoundary.domain_scope`：业务对象边界。规定纳入与排除的数据对象范围。
- `GoalBoundary.lifecycle_scope`：生命周期边界。规定平台覆盖的阶段与顺序（接入、处理、审核、展示、检索），不表示对象状态或质量结论。
- `GoalBoundary.path_scope`：路径边界。规定功能对外暴露的 UI 路径与 API 路由范围；仅命中有效路径且满足输入契约时才触发功能处理。
- `GoalBoundary.input_contract`：输入边界。仅将会触发状态变化或副作用的输入认定为功能输入，并规定其类型、来源与最小必需元信息。
- `GoalBoundary.output_contract`：输出边界。输出以可观察变化定义，包括界面变化、状态变化与副作用变化（记录写入、通知触发、导出产物）。
- `GoalBoundary.state_contract`：状态边界。规定对象状态集合、流转条件与合法路径，不复用生命周期阶段名。
- `GoalBoundary.authority_contract`：权限边界。规定角色职责、操作权限与可见范围；系统交付后客户公司内部全员可见，上传/处理/审核能力向内部工程师开放；权限规则由统一权限能力层集中管理。
- `GoalBoundary.quality_contract`：质量边界。规定质量维度、判定规则与阈值；判定结果可作为状态流转条件。
- `GoalBoundary.traceability_contract`：追溯边界。规定全流程留痕要求（操作记录、版本关系、责任归属），并由 `M3` 统一托管追溯记录接口。

去重约束：
- 生命周期表示阶段进度，状态表示对象当前业务状态，质量表示结果达标情况。

权限接口唯一规范（L0 Only）：
- `IAuthorizer.authorize(subject, action, resource, context) -> AuthDecision` 为全系统唯一鉴权接口契约。
- `subject/action/resource/context` 字段语义、`allow/deny` 判定语义、错误码与审计字段在 `L0` 定义。
- `IAuthorizer` 由 `M5` 对外提供，`M1/M2/M3/M4` 仅调用接口，不单独建设权限服务。
- `L1` 仅允许引用该契约并给出模块内 `action/resource` 映射，不得重定义角色规则与判定语义。

追溯接口唯一规范（L0 Only）：
- `ITracer.record(event) -> trace_id` 为全系统唯一追溯记录接口契约。
- `event` 字段语义（`subject/action/resource/result/reason/version/request_id/timestamp`）在 `L0` 定义。
- `ITracer` 由 `M3` 对外提供，`M1/M2/M4` 仅调用接口，不单独建设追溯服务。
- `L1` 仅允许引用该契约并给出模块内触发点与字段映射，不得重定义追溯字段语义与判定规则。



## 3. 模块（最小可行基，Module）
- `M1` 上传模块：用于提供器件手册上传能力（上传入口、文件接收、入库建单）
- `M2` 数据处理模块：支持授权用户对器件手册中的内容进行结构化处理（清洗：提取、精炼，审核）
- `M3` 状态管理模块：用于对上传的器件手册进行状态管理（是否清洗，是否审核，是否确认，版本管理等功能），并同模块托管统一追溯记录接口
- `M4` 结果展示与检索模块：用于在客户公司内部以网页形式提供数据可见可达能力（上传后即见、列表检索、详情展示、版本信息、审核结论、复用/导出入口）
- `M5` 权限管理模块：用于统一管理角色与权限策略，并对 `M1-M4` 提供统一鉴权接口（逻辑独立，当前与主服务同体部署）


说明：
- `M4` 对应 `GoalBoundary.output_contract`，并受 `GoalBoundary.authority_contract` 与 `GoalBoundary.traceability_contract` 约束。
- `M5` 对应 `GoalBoundary.authority_contract`，以统一接口方式作用于 `M1-M4`。
- `GoalBoundary.quality_contract`、`GoalBoundary.traceability_contract` 为横切约束，作用于 `M1-M5` 全链路。
- 权限管理逻辑由 `M5` 独立封装并统一提供 `IAuthorizer` 接口，物理部署当前与主服务同体。
- 追溯能力由 `M3` 同模块托管并对外提供统一追溯接口，各模块在关键节点调用该接口，不在模块内重复实现追溯存储与检索规则。

## 4. 组合原则（Combination Principles）
- 模块组合按 `R1 -> R2 -> R3` 编排。
- `R1` 上传可见：`M1` 完成上传后即生成文档记录，并在 `M4` 中可见可达（状态为已接入/待处理）。
- `R2` 处理回写：`M2` 仅处理已由 `M1` 接入的文档；处理与审核结果必须回写，并在 `M4` 同步展示。
- `R3` 状态与权限约束：`M3` 对 `M1/M2/M4` 的操作执行状态校验、版本留痕与责任归属；系统交付后客户公司内部全员可见，内部工程师可执行上传、处理、审核与检索共享操作；`M1/M2/M3/M4` 权限判定均通过 `M5` 提供的统一鉴权接口执行。
- 关键步骤追溯：`R1-R3` 每一步在成功与失败路径都必须触发统一追溯记录接口。


## 5. 验证（Verification）
判定公式：
- `PASS = BoundaryAssertions(all) AND CombinedOutcomeChecks(all) AND GoalChecks(all)`

组合验证说明：
- 组合过程遵循第 4 节组合原则，验证阶段仅对组合形成的结果进行判定。

通过条件：
- `V-B0` 路径边界通过：仅定义内的 UI 路径/API 路由可触发对应功能，越界路径请求被拒绝或路由不到功能处理。
- `V-B1` 输入边界通过：仅将触发状态变化或副作用的输入计入功能输入。
- `V-B2` 输出边界通过：每个关键操作至少产生一种可观察变化（界面/状态/副作用）。
- `V-B3` 状态边界通过：仅存在合法状态流转，非法流转全部被拦截。
- `V-B4` 权限边界通过：客户公司内部全员可见；上传/处理/审核仅工程师角色可执行；`M1-M4` 鉴权决策均由 `M5` 提供且来源一致可审计。
- `V-B5` 质量边界通过：质量规则与阈值达标后才允许进入下一状态。
- `V-B6` 追溯边界通过：关键操作均通过 `M3` 提供的统一追溯接口记录，且具备 `subject/action/resource/result/reason/version/request_id/timestamp` 字段。
- `V-C1` 组合结果完整：组合后形成“上传 -> 处理 -> 审核 -> 展示/检索”可执行链路。
- `V-C2` 组合结果一致：接入后文档可见可达，处理/审核结果与展示结果保持一致。
- `V-C3` 组合结果受控：状态校验、权限校验与追溯留痕在全链路生效。
- `V-G1` 目标闭环通过：样本文档可走通“上传 -> 处理 -> 审核 -> 展示 -> 检索”。
- `V-G2` 目标复用通过：检索结果可被复用，且记录可复核。

失败原因至少包含：
- 任一边界断言失败（`V-B0` 至 `V-B6`）。
- 任一组合结果约束失败（`V-C1` 至 `V-C3`）。
- 目标闭环或复用能力失败（`V-G1` 或 `V-G2`）。

证据记录要求：
- 采用验收矩阵记录 `case_id`、输入、期望界面变化、期望状态变化、期望副作用、实际结果、证据位置。
